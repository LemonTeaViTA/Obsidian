# 检索召回与优化

## 〇、查询理解与意图识别

### 为什么需要查询理解

传统 RAG 系统的典型问题：所有用户查询无差别地走向量检索流程（query → 向量化 → 检索 → 拼 Prompt → 生成）。这导致：
- 计算型查询（"保额 50 万、免赔额 1 万、花费 8 万能赔多少？"）被送去搜索知识库，LLM 试图从文档中"推理"出数值结果，而非直接调用计算模块
- 数据查询型（"我的理赔申请进度"）去搜通用知识，返回"一般需要 5-15 个工作日"而非用户的具体记录
- 闲聊型查询触发不必要的检索，浪费资源

**核心问题：该精确的变成了模糊的，该确定的变成了概率的。**

正确做法：在检索前加入 **Query 理解层**，先识别用户意图，再路由到对应的处理链路。

### 意图分类体系

最基本的意图分类至少包含四种类型：

| 意图类型 | 典型场景 | 处理链路 | 示例 |
|---------|---------|---------|------|
| **知识问答类** | 查询概念、规则、流程 | 向量检索 + LLM 生成 | "重疾险的等待期是多少天？" |
| **计算求解类** | 包含明确数值参数的计算 | 计算模块（公式/代码） | "保额 50 万、免赔 1 万、花费 8 万能赔多少？" |
| **数据查询类** | 查询个人业务数据 | NL2SQL / 结构化查询 | "我上个月的理赔申请进度？" |
| **闲聊类** | 寒暄、感谢、无关话题 | 直接 LLM 回复或引导 | "你好""谢谢""今天天气不错" |

### 意图识别实现方案

#### 方案一：规则匹配（Rule-based）

**实现：** 维护关键词映射表，通过正则表达式或关键词匹配判断意图。

```python
def classify_intent_rule(query: str) -> str:
    # 计算类关键词
    if any(kw in query for kw in ["多少钱", "怎么算", "赔付金额", "计算"]):
        return "calculation"
    # 数据查询类关键词
    if any(kw in query for kw in ["进度", "状态", "我的申请", "我的订单"]):
        return "data_query"
    # 闲聊类关键词
    if any(kw in query for kw in ["你好", "谢谢", "再见", "天气"]):
        return "chitchat"
    # 默认为知识问答
    return "knowledge_qa"
```

**优点：** 零延迟，可控性强，无需训练数据
**缺点：** 覆盖面有限，用户换个说法就匹配不上

#### 方案二：机器学习分类模型（ML-based）

**实现：** 使用 BERT 或轻量级文本分类模型训练四分类器。

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

class IntentClassifier:
    def __init__(self, model_path):
        self.tokenizer = AutoTokenizer.from_pretrained(model_path)
        self.model = AutoModelForSequenceClassification.from_pretrained(model_path)
        self.labels = ["knowledge_qa", "calculation", "data_query", "chitchat"]
    
    def classify(self, query: str) -> tuple[str, float]:
        inputs = self.tokenizer(query, return_tensors="pt", truncation=True, max_length=128)
        with torch.no_grad():
            outputs = self.model(**inputs)
            probs = torch.softmax(outputs.logits, dim=-1)
            confidence, pred_idx = torch.max(probs, dim=-1)
        return self.labels[pred_idx.item()], confidence.item()
```

**优点：** 识别同义表达，鲁棒性强
**缺点：** 需要几百条标注数据训练，有推理延迟（10-30ms）

**训练数据示例：**
```
"重疾险等待期多久？" → knowledge_qa
"帮我看看能拿到多少补偿" → calculation
"我的理赔单到哪一步了" → data_query
"今天天气真好" → chitchat
```

#### 方案三：LLM 零样本分类（LLM-based）

**实现：** 用 LLM 做 Few-shot 分类，无需训练。

```python
def classify_intent_llm(query: str, llm_client) -> str:
    prompt = f"""你是一个意图分类助手。请判断用户问题属于以下哪一类：
1. knowledge_qa - 知识问答（查询概念、规则、流程）
2. calculation - 计算求解（包含数值参数需要计算）
3. data_query - 数据查询（查询个人业务数据）
4. chitchat - 闲聊（寒暄、感谢等）

用户问题：{query}

只返回类别名称，不要解释。"""
    
    response = llm_client.generate(prompt, max_tokens=20)
    return response.strip()
```

**优点：** 无需训练，泛化能力强，能处理边界 case
**缺点：** 延迟较高（100-500ms），成本较高

**实践建议：** 三种方案组合使用
1. 先用规则匹配处理高置信度的明显 case（如包含"多少钱"直接判定为计算类）
2. 规则无法覆盖时，调用 ML 分类器
3. ML 分类器置信度 < 0.6 时，回退到 LLM 分类或默认走知识问答链路

### 实体提取

意图识别后，需要提取查询中的关键约束条件，辅助后续处理。

**提取目标：**
- **时间实体：** "上个月""2023 年 Q2""最近一周"
- **数值实体：** "保额 50 万""免赔额 1 万""花费 8 万"
- **业务实体：** "重疾险""理赔申请""订单号 12345"

**实现方式：**

1. **正则表达式：** 提取结构化数值和时间
```python
import re
from datetime import datetime, timedelta

def extract_entities(query: str) -> dict:
    entities = {}
    
    # 提取金额（支持"万"单位）
    amount_pattern = r'(\d+(?:\.\d+)?)\s*万'
    amounts = re.findall(amount_pattern, query)
    if amounts:
        entities['amounts'] = [float(a) * 10000 for a in amounts]
    
    # 提取时间
    if "上个月" in query:
        last_month = datetime.now() - timedelta(days=30)
        entities['time_range'] = ('start', last_month.strftime('%Y-%m-%d'))
    
    return entities
```

2. **NER 模型：** 使用命名实体识别模型提取业务实体
```python
from transformers import pipeline

ner = pipeline("ner", model="hfl/chinese-roberta-wwm-ext-large")
entities = ner("我的重疾险理赔申请进度")
# 输出：[{'entity': 'B-PRODUCT', 'word': '重疾险'}, ...]
```

### 查询路由（Query Routing）

根据意图分类结果，将查询路由到不同的处理链路：

```python
def route_query(query: str, intent: str, confidence: float, entities: dict):
    # 置信度不足时回退到默认检索
    if confidence < 0.6:
        return retrieval_pipeline(query)
    
    if intent == "knowledge_qa":
        # 走向量检索链路
        return retrieval_pipeline(query, filters=entities)
    
    elif intent == "calculation":
        # 走计算模块
        params = entities.get('amounts', [])
        return calculation_module(params)
    
    elif intent == "data_query":
        # 走 NL2SQL
        sql = nl2sql_generator(query, entities)
        return database_query(sql)
    
    elif intent == "chitchat":
        # 直接 LLM 回复
        return llm_chat(query)
```

**多索引路由：** 如果知识库按主题分成多个索引（如"理赔制度""产品信息""投保指南"），可根据查询主题选择最相关的索引检索，提高精度并减少计算量。

### 容错与回退机制

意图分类器不是 100% 准确，需要设计容错策略：

**1. 置信度阈值：** ML 分类器预测概率 < 0.6 时，回退到默认检索链路
```python
intent, confidence = classifier.classify(query)
if confidence < 0.6:
    intent = "knowledge_qa"  # 保守策略
```

**2. 多路径并行：** 对于模糊 case，同时走两条路径（如既做检索又做计算），取结果更好的一个
```python
if 0.5 < confidence < 0.7:
    result_retrieval = retrieval_pipeline(query)
    result_calculation = calculation_module(query)
    return select_best_result(result_retrieval, result_calculation)
```

**3. 异常回退：** 计算模块返回异常结果（如负数赔付金额）或 NL2SQL 执行报错时，自动回退到检索链路
```python
try:
    result = calculation_module(params)
    if result < 0:  # 异常结果
        raise ValueError("Invalid calculation result")
    return result
except Exception:
    return retrieval_pipeline(query)  # 回退
```

## 一、数据预处理流程

数据预处理是 RAG 系统效果的基础，直接决定了检索召回率的上限。流程分四步：

**1. 文档解析**
针对不同格式采用适配的解析工具提取文本与元数据。PDF 采用布局感知解析，区分正文、标题、表格、页眉页脚，避免非核心内容干扰；Word 提取正文与层级标题；HTML 去除导航栏、广告等冗余内容，提取核心正文。

**2. 文本清洗**
去除多余空格、换行符、特殊符号，统一中英文标点格式，过滤长度过短的无效文本片段（如 < 50 字符），确保文本连贯性与规范性。

**3. 元数据提取**
为每段文本提取对应元数据：文档唯一 ID、文档类型、所属章节标题、更新时间、访问权限等级。这些元数据用于后续检索过滤（按权限、按时间范围缩小候选集）和来源追溯。

**4. 文本分块**
采用递归字符分块策略，按段落 → 句子 → 词语的优先级进行分割。基础分块大小 512 token，块间重叠 50 token，保证每个分块的语义完整性，避免语义单元被截断导致检索偏差。

> 数据来源：[动态知识库的RAG系统混合检索与性能优化研究：融合BM25与稠密向量及RRF重排序实证分析](https://mp.weixin.qq.com/s?__biz=MzU4NTA1MDk4MA==&mid=2247554659&idx=1&sn=707b40f835af03e25f0a40726996dcfc&chksm=fc3809685bf64eb6087cc78afca98bd42d5ff2f9b3caae6c4c1e5d13efe9ac9743383aa2488d&mpshare=1&scene=24&srcid=0405JGVoNVxFK2HT0sRclYvb&sharer_shareinfo=43114a4bb85b93457016cd36be027e4f&sharer_shareinfo_first=43114a4bb85b93457016cd36be027e4f#rd)，数据集：MS MARCO + DuReader。

### 判别式语义分块：超长文档分块新方案

> 内容来源：[RAG语义分块新范式！金山办公提出超长文档通用分块框架](https://mp.weixin.qq.com/s?__biz=MzYzMjE5MDkwOA==&mid=2247486910)
> 论文：https://arxiv.org/pdf/2602.23370

#### 现有分块方案的短板

| 方案 | 代表 | 短板 |
|------|------|------|
| 无监督规则方法 | TextTiling | 靠词汇相似度/LDA，严重依赖人工调参，对复杂文体鲁棒性差 |
| 传统监督式判别模型 | BERT 序列标注 | 受限于 Transformer 二次复杂度，只能处理短文本，超长文档强制截断丢失信息 |
| 生成式大模型分块 | Jina Qwen2 系列 | 逐 token 生成推理慢、成本高；超过数千 token 性能下降；对 prompt 敏感 |

#### 判别式框架：编码-聚合-融合-判别

核心思路：跳出"生成式分块"，将主题分割转化为"相邻句子间是否存在主题边界"的二分类问题。

**5 个核心阶段**：
1. **句子级细粒度建模**：将文档拆分为句子级基础文本块，句子是最小的语义完整单元，提供更细的候选边界粒度
2. **token 级语义编码**：用 Qwen3-0.6B 作为骨干网络，对全文本做 token 级编码
3. **块级语义聚合**：通过注意力池化聚合得到每个句子的块级语义表示
4. **跨块上下文建模**：新增轻量 Transformer 编码器，建模跨句子的上下文依赖关系，既捕捉局部语义变化，也参考全文篇章结构
5. **边界判别预测**：MLP 分类头直接输出相邻句子间的主题边界概率，无需复杂解码

#### 超长文档处理：重叠滑动窗口

- 超长文档按固定长度拆分窗口，相邻窗口保留 10% 重叠区域
- 重叠区域的边界概��取多个窗口预测结果的平均值
- 无需额外参数，支持任意长度文档

#### 启发式后处理：三阶段无参数策略

1. 按预设阈值做初始分块，生成高置信度的粗粒度语义块
2. 超长块递归选取块内边界概率最高的位置二次拆分，直到满足长度约束
3. 超短块对比左右边界语义连续性，自动合并到语义更连贯的一侧

#### 超长块检索适配：向量融合与标量校正（VFSC）

**解决问题**：分块后的超长块超出嵌入模型上下文窗口，截断会导致语义丢失

**方案**：用一个融合向量 + 一个标量因子，严格还原超长块所有子段的平均余弦相似度
- 存储复杂度：O(N) → O(1)
- 检索计算量：O(N) → O(1)
- 零语义损失，与现有 RAG 检索系统无缝衔接

#### 技术特点总结

- 基于 Qwen3-0.6B 轻量化模型，原生支持 13k+ token
- 判别式而非生成式，推理速度快两个数量级
- 适合企业级海量文档处理、超长文档、对推理速度和成本敏感的场景

### 分块策略实战：从固定 512 到语义感知，召回率 67% → 91%

> 内容来源：[阿里面试官怒了：RAG 你 chunk 还在用固定 512](https://mp.weixin.qq.com/s?__biz=MzkzMDIwMzg1Mw==&mid=2247490039)

#### 三代方案演进

| 版本 | 策略 | Recall@5 | 核心问题 |
|------|------|----------|---------|
| V1 | 固定长度 512 token，overlap 50 | 0.67 | 句子中间截断，章节标题丢失，语义残缺 |
| V2 | 句子级切分（按句号/问号等自然边界） | 0.74 | 不理解文档层级结构，表格和列表处理差 |
| V3 | 语义感知切分（层级识别 + 递归细切 + 特殊元素处理） | 0.91 | — |

关键结论：没换 Embedding 模型，没调检索策略，仅改切分方式就提升了 24 个百分点。

#### V3 核心设计

**1. 文档结构识别（多策略融合）**
- 用正则匹配识别多种编号体系（`第X章/条`、`X.X`、`（X）`、`（1）/a)` 等）
- 切分原则：同一章节内容尽量放在同一个 chunk，跨章节绝不合并

**2. 超长章节递归细切**
- 普通章节 max_size = 1024 token
- 关键条款（责任/免责/费率）max_size = 1536 token（这类内容任何截断都会导致语义丢失）
- 先按子小节边界切，没有子小节则按句子边界切

**3. 表格专项处理**
- 小表格（≤300 token）：整体作为一个 chunk
- 大表格（跨页/超 300 token）：按行切分，每个 chunk 复制表头
- 不复制表头的后果：LLM 收到没有表头的数据行，不知道字段含义，只能胡编（表格类问题正确率从 43% → 78%）

**4. 列表项前导句合并**
- 问题：`（1）核辐射` 单独成 chunk 后，向量里没有"不在承保范围"这个关键语义，LLM 会误判为承保
- 方案：识别列表结构，把前导句和所有列表项合并；超长则至少保留前导句到每个 chunk
- 效果：否定性查询召回率从 0.58 → 0.83

#### Overlap 量化实验

| Overlap 大小 | Recall@5 | 存储增加 |
|-------------|----------|---------|
| 0 token | 0.81 | 基准 |
| 50 token | 0.86 | +5% |
| 100 token | 0.89 | +10% |
| 200 token | 0.90 | +20% |
| 300 token | 0.90 | +30% |

100 token 是性价比最优点。在此基础上加"基于句子边界的智能 overlap"（向后扫描到最近句子结尾才截断），避免了 87% 的句子在重叠区域被截断问题，召回率从 0.89 → 0.91。

#### Chunk 元数据设计

```python
@dataclass
class ChunkMetadata:
    doc_id: str          # 文档ID
    chunk_id: str        # chunk唯一ID
    section_path: str    # 层级路径："第3条 保险责任 > 3.2 责任免除"
    chunk_type: str      # "text" | "table" | "list"
    is_key_clause: bool  # 是否关键条款（责任/免责/费率）

```

用途：section_path 用于答案溯源；is_key_clause 用于检索加权（1.5 倍）；prev/next_chunk_id 用于检索到半截信息时自动扩展上下文。

#### 前沿方向

- **语义切分（Semantic Chunking）**：用 Embedding 模型计算相邻句子向量相似度，相似度突变处即为切分边界。理论更精准，但计算成本高（一份文档 30-40 秒 vs 规则方案 2-3 秒），大规模场景暂不实用
- **Late Chunking**：先用长文本 Embedding 对整个文档编码，再在向量层面切分，每个 token 向量都带全文上下文。目前内存占用是传统方案 8-10 倍，处于研究阶段
- **动态 Chunk Size**：根据章节语义密度（用 LLM 打分 0-10）动态调整 size，对多跳推理类问题有 3-5 个百分点提升

## 一、RAG 检索优化的四层框架

### 为什么检索是 RAG 的命脉

LLM 只能根据送进去的 context 来回答，检索召回的内容就是整个系统的天花板。生成层做得再好，如果检索没把相关内容找回来，LLM 也是巧妇难为无米之炊。

**投入产出比：** 生成层的优化（调 prompt、换模型）是锦上添花，而检索层的优化是从根本上提升系统的能力上限。在 RAG 系统里，检索优化是投入产出比最高的环节。

### 四层优化全貌

RAG 检索优化可以从四个层次来理解，每一层解决的问题不同，优化手段也不同：

| 层次 | 核心问题 | 关键技术 | 优先级 |
|------|---------|---------|--------|
| **索引层** | 知识怎么「存」 | Small-to-Big、Parent-Child、摘要索引 | 推荐 |
| **查询层** | 问题怎么「转」 | Query 改写、Multi-Query、HyDE、Step-back | 视场景 |
| **召回层** | 从哪里「找」 | 多路召回（向量 + BM25）、元数据过滤 | 推荐 |
| **重排序层** | 谁「最相关」 | Cross-Encoder Rerank | 强烈推荐 |

**层次关系：** 索引层决定「仓库里放了什么」，查询层决定「用什么钥匙开门」，召回层决定「从哪几扇门进去找」，重排序层决定「把找到的东西里最好的几件带出来」。

### 第一层：索引优化（Small-to-Big）

#### 核心矛盾：检索粒度 vs 上下文完整性

一个 chunk 需要同时完成两个任务：
1. **检索时被找到：** 要求向量语义聚焦，需要小粒度 chunk（如 150 token）
2. **被 LLM 读懂：** 要求上下文完整，需要大粒度 chunk（如 500 token）

**问题：** 小 chunk 检索准但内容太碎，大 chunk 内容完整但检索时语义稀释。

#### 解决方案：Small-to-Big 策略

**核心思路：小块检索、大块使用。**

**方案一：Parent-Child Chunking（父子分块）**

把文档切成两个版本：
- **子 chunk（Child）：** 细粒度（150 token），用于建立向量索引
- **父 chunk（Parent）：** 粗粒度（500 token），用于提供给 LLM

```python
# 数据结构示例
child_chunk = {
    "id": "child_001",
    "parent_id": "parent_001",  # 关联父 chunk
    "content": "重疾险的等待期是指...",  # 150 token
    "embedding": [0.12, 0.34, ...]  # 向量索引
}

parent_chunk = {
    "id": "parent_001",
    "content": "保险产品介绍：重疾险是一种...[完整上下文 500 token]",
    "children": ["child_001", "child_002", "child_003"]
}

# 检索流程
def retrieve_with_parent_child(query):
    # 1. 用子 chunk 的向量检索
    child_results = vector_search(query, index="child_chunks", top_k=10)
    
    # 2. 根据 parent_id 取出对应的父 chunk
    parent_ids = [child["parent_id"] for child in child_results]
    parent_chunks = fetch_parents(parent_ids)
    
    # 3. 父 chunk 去重后返回给 LLM
    return deduplicate(parent_chunks)
```

**优点：** 检索精度高（小块语义聚焦），LLM 理解好（大块上下文完整）
**适用场景：** 通用场景，效果稳定

**方案二：摘要索引（Summary Index）**

为每段内容生成摘要，用摘要建向量索引，检索时用摘要匹配，命中后返回原文。

```python
# 数据结构
document_chunk = {
    "id": "doc_001",
    "summary": "本段介绍重疾险的等待期规则和计算方法",  # 用于检索
    "content": "重疾险等待期详细说明...[原文 500 token]",  # 用于 LLM
    "summary_embedding": [0.23, 0.45, ...]
}

# 检索流程
def retrieve_with_summary(query):
    # 1. 用摘要的向量检索
    summary_results = vector_search(query, index="summary_embeddings", top_k=5)
    
    # 2. 返回原文内容给 LLM
    return [doc["content"] for doc in summary_results]
```

**优点：** 摘要语义更聚焦，命中率更高
**缺点：** 需要 LLM 生成摘要，离线处理成本高
**适用场景：** 文档表述散乱、需要提炼核心意思的场景

**方案三：多粒度分层索引**

同时建立章节级、段落级、句子级三层索引，根据问题类型选择合适粒度。

```python
# 三层索引结构
indexes = {
    "chapter": [...],   # 章节级（1000+ token）
    "paragraph": [...], # 段落级（500 token）
    "sentence": [...]   # 句子级（50-100 token）
}

# 根据问题类型路由
def route_by_query_type(query):
    if is_conceptual_question(query):  # "什么是 RAG"
        return search(query, index="chapter")
    elif is_detail_question(query):    # "退款需要几个工作日"
        return search(query, index="sentence")
    else:
        return search(query, index="paragraph")
```

**优点：** 覆盖不同类型问题
**缺点：** 索引存储成本高（3 倍），路由逻辑复杂
**适用场景：** 问题类型多样、对精度要求极高的场景

### 第二层：查询优化

#### 核心问题：用户提问与知识库表述的鸿沟

即使索引建得再好，用户的提问方式和知识库里的表述方式之间还是会存在鸿沟。

**示例：** 用户问"苹果手机咋截图"，知识库写的是"iPhone 截图操作方法"。两句话意思一样，但向量相似度可能不高（口语 vs 书面语）。

#### 优化方法

**方法一：Query 改写（Query Rewriting）**

用 LLM 把口语化、有歧义的 query 转化成更正式、更精准的书面表达。

```python
def rewrite_query(query, llm_client, conversation_history=None):
    prompt = f"""将用户的口语化问题改写成正式、精准的书面表达。
如果问题中有指代不明的词（如"它""这个"），结合对话历史补全。

对话历史：{conversation_history}
用户问题：{query}

改写后的问题："""
    
    rewritten = llm_client.generate(prompt, max_tokens=100)
    return rewritten.strip()

# 示例
# 输入："它为什么这么贵"
# 输出："iPhone 15 Pro Max 定价偏高的原因是什么"
```

**适用场景：** 用户提问质量差、指代不清、口语化严重

**方法二：Multi-Query 扩展**

把一个问题扩展成 3-5 个不同角度的问法，每种问法单独检索，最后合并去重。

```python
def multi_query_expansion(query, llm_client):
    prompt = f"""将下面的问题改写成 3 个不同角度的问法，每个问法一行。

原问题：{query}

改写后的问题："""
    
    response = llm_client.generate(prompt, max_tokens=200)
    queries = [query] + response.strip().split('\n')  # 保留原问题
    
    # 每个问法单独检索
    all_results = []
    for q in queries:
        results = vector_search(q, top_k=10)
        all_results.extend(results)
    
    # 合并去重
    return deduplicate_by_id(all_results)

# 示例
# 原问题："怎么退货"
# 扩展为：
# 1. "退货流程是什么"
# 2. "如何申请售后退款"
# 3. "商品退换货政策"
```

**关键点：** 原始问题必须保留，因为改写可能丢失细节，原始问题反而最精准。

**适用场景：** 用户提问角度和文档描述角度不对齐

**方法三：HyDE（假设文档嵌入）**

先让 LLM 根据问题生成一段"假设的答案"，然后用假设答案的向量去检索，而不是用原始问题的向量。

```python
def hyde_retrieval(query, llm_client):
    # 1. 生成假设答案
    prompt = f"""请根据问题生成一段假设的答案（100-200字）。

问题：{query}

假设答案："""
    
    hypothetical_answer = llm_client.generate(prompt, max_tokens=200)
    
    # 2. 用假设答案的向量检索
    results = vector_search(hypothetical_answer, top_k=5)
    return results

# 示例
# 问题："重疾险的等待期是多少天"
# 假设答案："重疾险的等待期通常为 90 天或 180 天，具体取决于保险公司和产品类型..."
# 用假设答案的向量检索，和文档风格更接近
```

**原理：** 假设答案和文档都是陈述性文字，风格更接近，向量距离更近。

**注意：** 如果 LLM 生成的假设答案方向错了，反而会把检索带偏。适合知识库领域明确的场景。

**方法四：Step-back Prompting（后退提问）**

把具体问题往上抽象一层，生成一个更通用的背景问题去检索，把背景知识检索回来，再结合背景知识回答具体问题。

```python
def step_back_retrieval(query, llm_client):
    # 1. 生成背景问题
    prompt = f"""将具体问题抽象成一个更通用的背景问题。

具体问题：{query}

背景问题："""
    
    background_query = llm_client.generate(prompt, max_tokens=100)
    
    # 2. 检索背景知识
    background_docs = vector_search(background_query, top_k=3)
    
    # 3. 检索具体问题
    specific_docs = vector_search(query, top_k=3)
    
    # 4. 合并返回
    return background_docs + specific_docs

# 示例
# 具体问题："为什么 transformer attention 要除以 sqrt(d_k)"
# 背景问题："transformer attention 机制的数学原理"
# 先检索背景知识，再回答具体问题
```

**适用场景：** 问题太具体，知识库只有通用背景的情况

### 第三层：召回优化（多路召回）

#### 核心问题：单一检索路径的盲区

单一的向量检索有根本局限：
- **向量检索：** 擅长语义相似，但对精确词语匹配效果差（如"M4 Pro 芯片"）
- **BM25 关键词检索：** 擅长精确匹配，但不理解语义（"怎么退货" vs "售后申请流程"）

#### 解决方案：多路召回 + 融合

**实现：向量检索 + BM25 并行召回，用 RRF 融合**

```python
def multi_path_retrieval(query, top_k=5):
    # 路径 1：向量检索
    vector_results = vector_search(query, top_k=20)
    
    # 路径 2：BM25 关键词检索
    bm25_results = bm25_search(query, top_k=20)
    
    # 路径 3（可选）：元数据过滤
    # metadata_results = search_with_filters(query, filters={"type": "policy"})
    
    # RRF 融合
    fused_results = rrf_fusion([vector_results, bm25_results], smooth_k=60)
    
    return fused_results[:top_k]
```

**元数据过滤：** 如果用户问题包含时间、类型等约束，可以先用元数据过滤缩小候选集。

```python
# 示例：用户问"2023 年的理赔政策"
filters = {
    "year": 2023,
    "doc_type": "policy"
}
results = vector_search(query, filters=filters, top_k=10)
```

**效果：** 多路召回互补盲区，召回率显著提升（详见下文混合检索章节的实验数据）。

### 第四层：重排序优化（Rerank）

#### 核心问题：粗召精度不足

向量相似度 ≠ 语义相关性。向量检索找到的是"在嵌入空间中接近"的文档，但可能实际上不回答问题。

#### 解决方案：Cross-Encoder 精排

**两阶段架构：**
1. **粗召（Recall）：** 向量检索 + BM25 快速召回 20-30 个候选（毫秒级）
2. **精排（Rerank）：** Cross-Encoder 对每个候选打精确相关性分数，取 top-3 到 top-5

**Bi-encoder vs Cross-encoder：**

| 模型类型 | 编码方式 | 精度 | 速度 | 适用阶段 |
|---------|---------|------|------|---------|
| **Bi-encoder** | query 和 chunk 分别编码 | 中等 | 极快（毫秒级） | 粗召 |
| **Cross-encoder** | query + chunk 拼接编码 | 高 | 慢（每对需单独推理） | 精排 |

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch

class Reranker:
    def __init__(self, model_name="BAAI/bge-reranker-v2-m3"):
        self.tokenizer = AutoTokenizer.from_pretrained(model_name)
        self.model = AutoModelForSequenceClassification.from_pretrained(model_name)
    
    def rerank(self, query: str, candidates: list, top_k=5):
        scores = []
        for chunk in candidates:
            # 拼接 query 和 chunk
            inputs = self.tokenizer(
                query, chunk["content"],
                return_tensors="pt",
                truncation=True,
                max_length=512
            )
            with torch.no_grad():
                score = self.model(**inputs).logits.item()
            scores.append((chunk, score))
        
        # 按分数降序排列
        ranked = sorted(scores, key=lambda x: x[1], reverse=True)
        return [chunk for chunk, score in ranked[:top_k]]

# 使用示例
reranker = Reranker()
recall_results = multi_path_retrieval(query, top_k=20)  # 粗召 20 个
final_results = reranker.rerank(query, recall_results, top_k=5)  # 精排取 5 个
```

**常用 Rerank 模型：**
- `BAAI/bge-reranker-v2-m3`：中英双语，效果优异
- `BAAI/bge-reranker-large`：精度更高，速度较慢
- `maidalun1020/bce-reranker-base_v1`：中文场景优化
- Cohere Rerank API / Jina Reranker API：云端服务，无需部署

### 四层优化组合策略

实际落地时不需要全部都上，按业务场景和问题症状来选：

| 层次 | 解决的核心问题 | 推荐程度 | 成本 |
|------|---------------|---------|------|
| 索引优化（Parent-Child） | 检索粒度 vs 上下文完整性 | ⭐⭐⭐⭐⭐ | 低 |
| 查询优化（Multi-Query / HyDE） | 用户提问和知识库表达不对齐 | ⭐⭐⭐ | 中（LLM 调用） |
| 多路召回（向量 + BM25） | 单路检索漏召 | ⭐⭐⭐⭐⭐ | 低 |
| Rerank 精排 | 粗召精度不足 | ⭐⭐⭐⭐⭐ | 中（推理成本） |

**典型生产级搭配：**
```
Parent-Child 索引 + 向量 BM25 多路召回 + Rerank 精排
```

这三层组合基本能覆盖大多数场景的检索质量问题。如果用户提问质量比较差（口语化、指代不清），再额外加上 Query 改写。

**记忆口诀：**
- 索引层保证「存进去的知识可以被找到」
- 查询层保证「搜索的姿势是对的」
- 召回层保证「不漏掉该找到的内容」
- Rerank 层保证「送进 LLM 的是真正有用的内容」

## 二、为什么需要混合检索

单一检索模式各有致命短板：

| 检索方式 | 优势 | 短板 |
|----------|------|------|
| 纯向量检索 | 理解语义、同义词、上下文 | 专有名词、精确编号、关键词匹配弱 |
| 纯 BM25 | 精确关键词匹配，无需训练 | 无法捕捉语义相似性，同义改写全失效 |

以下实验数据来自一篇 RAG 混合检索实证研究（数据集：MS MARCO + DuReader，环境：RTX 3090，每组实验重复 5 次取均值）：

| 检索策略 | Recall@5 | MRR | 端到端准确率 | 响应时间 |
|----------|----------|-----|-------------|----------|
| 纯向量检索 | 0.682 | 0.621 | 0.695 | 862ms |
| 纯 BM25 | 0.615 | 0.587 | 0.652 | 725ms |
| 混合检索（RRF） | 0.827 | 0.783 | 0.827 | 947ms |
| 混合检索 + 重排 | **0.896** | **0.875** | **0.903** | 1086ms |

混合检索 Recall@5 较纯向量提升 **21.3%**，较纯 BM25 提升 **34.5%**。加重排后端到端准确率较基准提升 **29.9%**，响应时间增幅仅 14.7%。

## 二、BM25 原理

### 题目：BM25 和向量检索的区别是什么？

#### 核心解析

两者在完全不同的空间上做匹配：

**BM25（稀疏检索）** 在词频维度上匹配——统计查询词在文档中出现的频率和分布。维度是离散的词汇空间，每个词是一个独立维度。

BM25 核心公式：

$$
\text{score}(q, d) = \sum_{i} \text{IDF}(q_i) \times \frac{tf(q_i, d) \times (k_1 + 1)}{tf(q_i, d) + k_1 \times \left(1 - b + b \times \frac{|d|}{\text{avgdl}}\right)}
$$
- `k1`：词频饱和调节参数，经典取值 1.2（控制词频对分数的影响上限）
- `b`：文档长度归一化参数，经典取值 0.75（防止长文档因词频绝对值高而占优）
- `IDF`：逆文档频率，越稀有的词权重越高

**向量检索（稠密检索）** 在语义嵌入维度上匹配——文本编码成高维稠密向量（768/1024/2048 维），计算余弦相似度。语义相近的词在空间中距离近。

## 三、RRF 融合算法

### 题目：混合检索结果怎么融合？RRF 是什么？

#### 核心解析

两路检索结果的分数量纲不同（向量是余弦相似度 0-1，BM25 是 TF-IDF 加权分），不能直接相加，需要归一化或用排名融合。

**RRF（Reciprocal Rank Fusion，倒数排名融合）** 是最稳健的方案：

```python
def rrf_fusion(result_lists, smooth_k=60):
    doc_score_map = {}
    for single_result in result_lists:
        for rank, (doc_id, _) in enumerate(single_result):
            if doc_id not in doc_score_map:
                doc_score_map[doc_id] = 0
            doc_score_map[doc_id] += 1 / (rank + smooth_k)
    return sorted(doc_score_map.items(), key=lambda x: x[1], reverse=True)
```

`smooth_k=60` 是经典取值，防止排名第 1 的文档分数过高压制其他结果。

**为什么 RRF 比加权求和更稳健：** 加权求和需要对两路分数做 min-max 归一化，归一化对异常值敏感；RRF 只看排名不看分数，天然抗噪。

**另一种方案：加权求和融合**
```
final_score = dense_score × 0.6 + sparse_score × 0.4
```
参数敏感性实验：密集权重 0.6、稀疏权重 0.4 是通用场景最优组合；关键词查询占比高的场景可提升稀疏权重至 0.5-0.7。

**Baize 项目实现：** [[RAG管道设计#混合检索：KNN 向量 + BM25 rescore]] — KNN 宽召回（topK×30）+ BM25 rescore，score 公式 `KNN×0.2 + BM25×1.0`，BM25 主导。

## 四、重排（Rerank）

### 题目：为什么要加重排？重排模型怎么选？

#### 核心解析

向量相似度 ≠ 语义相关性。向量检索找到的是"在嵌入空间中接近"的文档，但文档可能在空间中"接近"却实际上不回答问题。

**两阶段检索架构：**
1. 第一阶段：向量检索快速召回 20-50 个候选（毫秒级）
2. 第二阶段：Cross-Encoder 重排，对每个候选同时读取 query 和文档，输出精确相关性分数

Cross-Encoder 比 Bi-Encoder（向量检索用的）精度高的原因：Bi-Encoder 把 query 和文档分别编码，Cross-Encoder 把两者拼在一起编码，能捕捉 query 和文档之间的细粒度交互。

**重排模型选型：**
- `bge-reranker-v2-m3`：中文场景表现优异，轻量版本 CPU 快速推理，P95 延迟 50ms 以内
- `cross-encoder/ms-marco-MiniLM-L-6-v2`：英文场景经典选择

**实测效果：**
- 加重排后 Recall@5 从 0.827 提升到 0.896
- 端到端准确率从 0.827 提升到 0.903
- 响应时间从 947ms 增加到 1086ms（增幅 14.7%，完全值得）

**面试追问：重排增加了延迟，值得吗？**

在企业级知识库场景，幻觉带来的业务风险远大于 100ms 的延迟开销。工程上可以做路由：简单 FAQ 查询跳过重排保证速度，复杂专业查询启用重排保证质量。

## 五、向量数据库与检索算法

### 为什么 LLM 需要向量数据库

传统关系型数据库（MySQL）通过精确匹配检索结构化数据。但 AI 时代面临的是非结构化数据（图片、视频、文本），这些数据难以用键值对描述。向量数据库通过计算高维空间中的距离来表征相似度，让机器能理解数据的"上下文"和"语义"。

在 RAG 系统中，两者通常并存：
- **向量数据库：** 负责大规模"初筛"，从百万级文档中找出最相关的段落
- **传统数据库：** 负责"元数据"管理，记录文档的作者、发布时间、权限等确切信息

### 向量数据库 vs 传统数据库

| 特性 | 关系型数据库 (SQL) | 向量数据库 (Vector DB) |
|------|-------------------|----------------------|
| **存储数据类型** | 结构化数据（数字、字符串、日期） | 非结构化数据的向量表示 (Embedding) |
| **检索方式** | 精确匹配：基于条件（如 `WHERE ID=1`） | 近似匹配：基于距离（如余弦相似度） |
| **查询结果** | 是或否（匹配或不匹配） | 相似度分值（从最相似到最不相似） |
| **扩展性** | 擅长处理数百万级的行记录 | 专为处理数十亿级的高维向量设计 |
| **典型代表** | MySQL, PostgreSQL | Milvus, Zilliz, Pinecone |

### KNN vs ANN

| 维度 | KNN (精确搜索) | ANN (近似搜索) |
|------|---------------|---------------|
| **计算方式** | 暴力计算：遍历全库，计算所有距离 | 索引检索：只在候选子集中计算距离 |
| **准确率** | 100%（理论最优解） | < 100%（存在微小召回损失） |
| **查询速度** | 随数据量线性增长 O(N)，大数据下极慢 | 极快 O(log N) 或 O(1)，支持亿级数据 |
| **内存消耗** | 低（只需存储原始数据） | 高（需要额外存储索引结构） |
| **适用场景** | 小数据集、对精度要求极高的科研 | 生产环境、RAG、推荐系统、海量检索 |

### 向量检索算法总览

| 算法类型 | 代表算法 | 核心原理 | 优点 | 缺点 |
|---------|---------|---------|------|------|
| **基于图 (Graph)** | HNSW, NSG | 构建分层小世界图，通过节点跳转快速逼近目标 | 性能最强，召回率极高，查询延迟极低 | 索引构建慢，内存占用大 |
| **基于聚类 (Quantization)** | IVF, IVFPQ | 将空间划分为簇，查询时仅搜索相邻簇 | 内存占用低，数据压缩率高 | 召回率受聚类质��影响，存在硬边界误差 |
| **基于哈希 (Hash)** | LSH | 利用局部敏感哈希函数将相似点映射到同个桶 | 计算简单，空间复杂度低 | 高维数据下性能下降明显 |
| **基于树 (Tree)** | Annoy, KD-Tree | 通过超平面不断切分空间，构建二叉树结构 | 适合中低维度，支持静态文件映射 | 高维情况下出现"维度灾难" |

### 向量检索算法详解

#### FLAT（暴力搜索）

直接将每个查询向量与数据集中的每个向量进行比较，100% 召回率。优点是简单可靠、不需要训练或参数配置；缺点是最慢的索引选项，不适合海量数据集。

#### IVF_FLAT（反转文件 + 扁平存储）

双层方法：先用 K-means 聚类将向量空间划分为可管理的区域（IVF），每个聚类内向量以原始形式存储（FLAT）。

**工作原理：**
1. **聚类：** 用 k-means 将向量数据集划分为指定数量的簇，每个聚类有一个中心点
2. **分配：** 每个向量被分配到最近的聚类中心
3. **反向索引：** 建立聚类中心点到向量列表的映射
4. **搜索：** 查询时先比较聚类中心点，选择最有希望的簇，再在簇内搜索

适合需要快速查询响应和高精确度的大规模数据集。

#### IVF_SQ8（反转文件 + 标量量化）

在 IVF 基础上加入标量量化（SQ8）：用 8 位整数替代 32 位浮点数存储每个维度值，大幅减少内存使用量。

**两步加速：**
1. IVF 缩小搜索范围（只搜最相关的簇）
2. SQ8 加快距离计算速度（压缩后的向量计算更快）

适合内存受限但需要较高精度的场景。

#### IVF_PQ（反转文件 + 乘积量化）

**乘积量化（PQ）** 是一种针对高维向量的压缩方法：

1. **维度分解：** 将每个高维向量分解为 `m` 个大小相等的子向量
2. **子空间编码本生成：** 在每个子空间内用 k-means 学习一组代表性中心点
3. **向量量化：** 每个子向量用最近的中心点索引替代
4. **压缩表示：** 存储需求从 D × 32 位减少到 m × nbits 位

**检索时的距离计算：**
- 将查询向量分解为 m 个子向量
- 预计算每个子向量与对应编码本所有中心点的距离，生成 m 个查找表
- 对数据库中的每个 PQ 编码，从查找表中查找并累加距离

IVF_PQ 内存占用比 HNSW 少得多，是大型数据集的实用选择。

#### IVF_RABITQ（反转文件 + 二进制量化）

将 FP32 向量量化为二进制表示，压缩比 1:32，存储效率极高。

**RaBitQ 量化步骤：**
1. **归一化：** 将所有数据向量映射到单位超球面上
2. **随机旋转：** 引入随机正交矩阵旋转原始空���（类似 Johnson-Lindenstrauss 变换）
3. **二值化：** 只保留符号位，正号记为 1，负号记为 0，每个向量变成一个 D-bit 的二进制串

在内存受限的情况下，可替代 IVF_SQ8 和 IVF_FLAT。

#### HNSW（分层导航小世界图）

构建多层图结构，类似不同缩放级别的地图。底层包含所有数据点，上层由从底层采样的数据点子集组成。

**搜索流程：**
1. **入口点：** 从顶层的固定入口点开始
2. **贪婪搜索：** 在当前层贪婪地移动到近邻，直到无法再接近查询向量（达到局部最小值）
3. **层层下降：** 跳转到下层，重复贪婪搜索
4. **最终细化：** 在最底层进行最后的细化步骤，找出最近的邻居

**关键参数：**
- `m`：每层的邻居数，越大索引越准但内存占用越多
- `ef_search`：检索时的候选数，越大越准但越慢
- 实测：`m=16, ef_search=100` 在准确率和延迟之间是较好的平衡

HNSW 是目前性能最强的向量检索算法，召回率极高、查询延迟极低，但索引构建慢、内存占用大。

#### DISKANN（基于磁盘的向量检索）

当数据集大小超过可用 RAM 时（数十亿甚至数万亿向量），标准内存索引方法（HNSW、IVF_FLAT）跟不上。DISKANN 结合两项技术：
- **Vamana 图形：** 基于磁盘的图形索引，高效导航
- **乘积量化 (PQ)：** 内存中的压缩方法，快速计算近似距离

适合超大规模数据集、内存受限的生产环境。

#### ScaNN（Google 的可扩展近邻搜索）

将向量搜索分解为三个阶段：
1. **分区：** 将数据集划分为簇，只关注相关子集
2. **量化：** 采用各向异性向量量化压缩
3. **重新排序：** 对排名靠前的候选向量进行精确内积计算，确保最终结果高度准确

### Milvus 架构特点

Milvus 通过 Growing Segment（实时数据，快速索引）和 Sealed Segment（历史数据，高性能 Immutable 索引）的划分，配合异步 Compaction 机制，解决了"大数据量下索引构建慢"与"实时搜索需求"之间的矛盾。

**混合检索能力：**
- Sparse Vector（稀疏向量）：结合 BM25 算法与 BERT 语义能力
- 多列检索：支持 Dense + Dense（如人脸+步态）或 Scalar + Dense（如证件号+人脸）

**丰富的查询语义：**
- Range Search：寻找"足够相近"的结果，避免无关干扰（不只是 Top K）
- Groupby：解决结果多样性问题（如搜索"狗"时确保包含不同品种）

> 数据来源：[Agent Harnesses 套件：向量数据库与向量检索算法](https://mp.weixin.qq.com/s?__biz=MzIwNDA5NDYzNA==&mid=2247511705&idx=1&sn=e66c77417f6dc77db20d862b9872870e)

### 题目：RAG 为什么不能用 MySQL 存向量？

#### 核心解析

MySQL 没有向量索引，只能暴力遍历。100 万条向量暴力搜索需要 10 秒，HNSW 索引只需 50ms。

**百万级向量的存储规模估算：**
- 100 万文档 × 平均 50 个 chunk = 5000 万条 chunk
- 每条 chunk 存 2048 维向量：5000 万 × 2048 × 4 字节 ≈ **400GB**

**主流向量存储方案对比：**

| 方案 | 适用场景 | 优势 | 劣势 |
|------|----------|------|------|
| ElasticSearch 8.x | 混合检索（向量+关键词） | 天然支持 BM25+向量双路，运维成熟 | 内存占用大 |
| Milvus | 纯向量，超大规模 | 专为向量设计，性能极致 | 运维复杂 |
| Pinecone | 云托管，快速上线 | 零运维 | 成本高，数据出境 |
| Weaviate | 混合检索 | 内置 BM25+向量 | 社区相对小 |
| pgvector | 小规模，已有 PG | 复用现有数据库 | 大规模性能差 |

**Baize 项目选型：** [[RAG管道设计#四、检索召回]] — MySQL 存文档元数据（文件名、路径、用户、分类），ElasticSearch 存向量和关键词，两者配合。MySQL 这边没有大字段，百万级文档也就几 GB，单表扛住。

### 题目：ES 向量检索怎么优化？HNSW 参数怎么调？

#### 核心解析

ES 8.x 支持 HNSW 索引，向量检索复杂度 O(log N)，和维度关系不大。

**HNSW 两个关键参数：**
- `m`：每层的邻居数，越大索引越准但内存占用越多
- `ef_search`：检索时的候选数，越大越准但越慢

实测：`m=16, ef_search=100` 在准确率和延迟之间是较好的平衡。5000 万条 2048 维向量，P99 延迟 100ms 以内。

**三个检索优化思路：**
1. **先过滤再向量搜索**：用 term 查询把候选集缩小（按文档类型、用户权限、时间范围），再做向量相似度计算
2. **冷热分层**：热数据放 SSD，冷数据放 HDD，兼顾性能和成本
3. **语义缓存**：80% 的查询集中在 20% 的热门问题，把检索结果缓存在 Redis，命中率能到 40% 以上，直接省掉向量计算

## 五 bis、Embedding 模型选型

> 内容来源：[2026 Embedding 模型选型指南](https://mp.weixin.qq.com/s?__biz=Mzg4MDAzNjMyMg==&mid=2247487812)

### 题目：Embedding 模型怎么选？看哪些维度？

#### 核心解析

Embedding 模型是 RAG 的地基。地基不稳，上面的分块策略、检索算法、Rerank 都发挥不了作用。选型主要看 5 个维度：

| 维度 | 说明 | 关键考量 |
|------|------|---------|
| **领域匹配度** | 通用模型在垂直领域可能表现差 | 医疗、法律、金融等需要在自己数据上评估，不要迷信 MTEB 分数 |
| **上下文长度** | 单次能处理的最大 token 数 | 长文档选 8192+ tokens（Qwen3 支持 32K），短文本 2048 够用 |
| **向量维度** | 维度越高精度越高，但存储和检索成本也越高 | 高精度选 3072/4096，大规模选 384/512；新模型支持 Matryoshka 训练可灵活截断 |
| **语言支持** | 中文场景需要特别关注 | BGE、Qwen 系列对中文深度优化，OpenAI 对中文"可用但建议实测" |
| **延迟与成本** | API 按量付费 vs 本地部署 | 轻量模型（DistilBERT、MiniLM）延迟更低 |

### 题目：2026 年主流 Embedding 模型有哪些？

#### 核心解析

**中文场景**：
- **BAAI/bge-large-zh**：中文老牌选手，稳定可靠

- **Gemini Embedding 2**：文本/图像/视频/音频/PDF，全能选手
- **Qwen3-VL-Embedding-2B**：开源多模态

### 题目：怎么评估 Embedding 模型的好坏？

#### 核心解析

**核心原则：不要迷信排行榜，在自己数据上测最靠谱。**

**简单评估方法**：构造 query-positive-negative 三元组，计算区分度

```python
# 构造测试用例
test_cases = [
    {"query": "怎么退款", "positive": "退货流程说明", "negative": "公司年假政策"},
    {"query": "如何请假", "positive": "带薪假期申请指南", "negative": "工资发放时间"},
]

# 核心指标：区分度 = 正例平均相似度 - 负例平均相似度
# 区分度越高，模型越能准确区分相关和不相关内容
```

**进阶评估**：使用 RAGAS 框架，评估上下文精度、召回率、忠实度等多维度指标

**评估工具**：RAGAS、DeepEval、Phoenix、TruLens-Eval

### 题目：什么时候需要微调 Embedding 模型？怎么做？

#### 核心解析

**适用场景**：通用模型在垂直领域表现不佳，且有一定规模文档库（几百份以上）

**新思路：合成数据微调**（CustomIR 框架）：
1. 用 LLM 从文档库自动生成多样化查询
2. 生成正例和经过 LLM 验证的难负例
3. 对比学习微调预训练 Embedding 模型

**优势**：无需人工标注，单张 GPU 几小时完成，迭代快

**不需要微调的情况**：数据量很少或通用模型已经够用，直接使用现成模型更划算

### 题目：Embedding 领域有哪些前沿趋势？

#### 核心解析

1. **超大参数模型崛起**：KaLM-Embedding 达到 120 亿参数，Embedding 也在走向"大模型化"
2. **多模态统一嵌入**：Amazon Nova 用单一模型支持文本、图像、视频、音频，简化架构
3. **知识蒸馏让小模型也能打**：LEAF 等框架让小型模型达到接近大型模型的效果
4. **混合检索成为标配**：BGE-M3 同时生成稠密向量和稀疏向量，兼顾关键词精确性和语义理解

## 六、召回率评估体系

### 题目：你说召回率提升了 15%，这个数字怎么来的？

#### 核心解析

这是面试最容易被追死的问题。"感觉更好"不是工程答案，必须有测试集、有指标、有工具。

**测试集构建六步流程：**

1. **确定评估范围**：覆盖哪些领域和文档类型，每个场景至少 50 条 query
2. **设计评估问题**：LLM 生成 500 条初始问题 → 人工筛选保留 200 条（标准：能在知识库找到明确答案、表述自然、不重复）
3. **标注参考答案和支持证据**：每个问题标注正确答案 + 答案来自哪个文档的哪个 chunk（文档 ID、Chunk ID、页码）
4. **覆盖多样性和难度**：
   - 简单事实题：答案在单个 chunk 里
   - 多跳推理题：需要综合多个 chunk
   - 否定性查询："核辐射能赔吗"（答案在免责条款，容易召回错误的承保条款）
   - 口语化查询："孩子在学校摔了能报不"（表述和知识库差异大）
5. **质量审查**：抽样 20% 做交叉验证，初次标注错误率约 8%，交叉验证后降到 2% 以下
6. **持续迭代**：线上 badcase 定期加入测试集，从 200 条增长到 350 条

**检索阶段核心指标：**
- **Recall@K**：前 K 个结果中包含真实相关文档的比例，是检索系统的核心指标
- **Precision@K**：前 K 个结果中有多少是相关的
- **MRR（平均倒数排名）**：第一个正确结果排名的倒数均值，衡量高相关内容是否排在前列
- **NDCG@K**：综合考虑相关性等级和排名位置

**生成阶段核心指标：**
- **忠实度（Faithfulness）**：回答中每句话是否有检索证据支撑，有没有编造不存在的内容
- **答案相关性**：回答是否真正解决了用户的问题
- **端到端准确率**：生成答案的事实正确率

**评估工具选型：**

| 工具 | 适用场景 | 特点 |
|------|----------|------|
| RAGAS | 快速评估，无标注数据 | 不需要 ground truth，LLM 自动判断 |
| LangSmith | 已用 LangChain，有参考答案 | LLM-as-a-Judge，批量对比 |
| TruLens | 系统已上线，需持续监控 | 支持线上实时监控，有 Dashboard |
| DeepEval | 团队有单测习惯 | 类 Pytest，可集成 CI/CD |

**推荐组合：** 离线评估用 RAGAS + 人工抽检（50 条校准自动指标，偏差控制在 3% 以内），线上监控用 TruLens Dashboard 看每日质量指标。

**分维度拆解的价值：** V2 方案在表格类问题召回率只有 0.51，文本类是 0.82——这直接指导了 V3 的优化方向：先解决表格问题。不分维度就看不到这个差距。

## 七、查询优化策略

### 题目：用户 query 很短很模糊，检索效果差怎么办？

#### 核心解析

朴素 RAG 的一个核心问题：用户查询通常很模糊，"什么是 RAG"无法捕捉他们想要的是架构细节、使用案例还是实施指南。

**四种查询优化策略：**

**1. 查询扩展（Query Expansion）**
用 LLM 把简短查询扩展为更详细的版本：
```
输入："什么是 RAG？"
输出："什么是检索增强生成（RAG），它如何将信息检索与语言生成相结合，
      其关键组件和架构是什么，以及它为问答系统提供了哪些优势？"
```
优点：单次 LLM 调用，快；缺点：增加延迟，可能过度指定简单查询。

**2. 多查询 RAG（Multi-Query）**
生成相同问题的 3-4 种不同表述，并行搜索，去重合并结果。
```
原始："如何部署 ML 模型？"
变体1："将机器学习模型部署到生产环境的步骤是什么？"
变体2："ML 模型部署基础设施的最佳实践"
变体3："训练模型的生产部署选项"
```
优点：对模糊查询召回率更好，并行执行保持快速；缺点：4 倍数据库查询，成本更高。

**3. 自反思 RAG（Self-Reflection）**
检索后评估结果相关性（1-5 分），分数低则优化 query 重新搜索，最多迭代 2-3 次。
优点：自校正，能从差的初始结果中恢复；缺点：最高延迟，最贵。

**4. 上下文检索（Contextual Retrieval）**
在 Embedding 之前，用 LLM 为每个 chunk 生成 1-2 句上下文前缀，解释该 chunk 与整个文档的关系：
```
之前："收入增长 40% 至 3.14 亿美元，利润率提高。"
之后："此分块来自 ACME 公司 2024 年 Q2 SEC 文件，讨论季度财务表现。
      收入增长 40% 至 3.14 亿美元，利润率提高。"
```
根据 Anthropic 研究，减少 35-49% 的检索失败。缺点：摄取时每个 chunk 需要一次 LLM 调用，成本高。

> 论文来源：[Anthropic Contextual Retrieval](https://www.anthropic.com/news/contextual-retrieval)

## 七 bis、智能体与自反思 RAG

### 题目：智能体 RAG（Agentic RAG）是什么？什么时候用？

#### 核心解析

**作用：** 给 LLM Agent 提供多个检索工具（语义搜索、完整文档检索、SQL 查询），让它根据查询自主选择用哪个。

**解决的问题：** 不同问题需要不同检索策略——有些问题适合语义搜索，有些需要完整文档，有些需要查结构化数据库。一个固定的检索策略无法覆盖所有场景。

**示例流程：**
```
用户："完整的退款政策是什么？"
Agent 推理：
1. 调用 search_knowledge_base("退款政策") → 找到提及 refund_policy.pdf 的分块
2. 意识到分块不够完整 → 调用 retrieve_full_document("退款政策")
3. 拿到完整文档 → 生成全面答案
```

**适用场景：** 异构数据源（文档 + 数据库 + API）且查询复杂度差异大时。

**缺点：** 实现复杂、行为不太可预测、多步推理延迟高。

**前沿方向：结构感知的动态检索（BookRAG）**

传统 Agentic RAG 的 Agent 在"无结构的向量库"里做决策，BookRAG（港中文）提出了一种更精准的方式：先把文档还原成层级树 + 知识图谱，然后 Agent 根据查询类型动态选择检索策略：

- **单跳查询**：通过实体/关键词直接定位到对应章节子树，只搜这一小块（相当于直接翻到书的对应小节）
- **多跳查询**：先拆子问题，分别定位到不同章节子树检索，再利用章节层级关系合并跨块证据
- **全局聚合查询**：按页面范围、章节层级、模态类型（表格/图片）过滤整棵结构树，快速统计

核心优势是"导航式精搜"替代"盲搜全文"——检索范围从全文几万块缩到某一小节几十块。实验数据：M3DocVQA 的 EM 达 61%（次优基线 43%），Token 消耗降低一个数量级。

局限：子章节内超长内容仍需分块，top-k 固定，未能根据查询意图动态调整检索粒度。

> 论文：https://arxiv.org/pdf/2512.03413

### 题目：多文档场景下单 Agent 不够用怎么办？（SPD-RAG）

#### 核心解析

单智能体 Agentic RAG 面对需要综合十几份文档的问题时会崩：全局检索的跨文档噪声不断带偏检索方向，top-K 必然漏掉关键证据。

**SPD-RAG（Sub-Agent Per Document RAG）** 的核心洞察：多文档问答的瓶颈不是 top-K 多大或上下文多长，而是每份文档是否被充分、独立、深度挖掘。做法是在原有多轮迭代检索的基础上，给每份文档配一个专属子 Agent：

**三层架构：**
1. **协调层**：把用户复杂问题拆解成原子化的信息提取任务清单，广播给所有子 Agent。协调器不需要知道文档内容，只负责拆解。
2. **并行检索层**：每份文档一个子 Agent，检索范围严格限制在自己的文档内（避免跨文档噪声），最多 5 轮迭代检索-推理，输出文档专属的发现报告 + 置信度分数。所有子 Agent 完全并行执行。
3. **合成层**：对所有报告做向量嵌入 → 余弦相似度聚类 → 递归 map-reduce 合并，最终生成答案。

**实验数据（Loong 基准，EMNLP 2024，102 个测试用例）：**

| 方案 | 平均分 | 完美回答率 | 平均成本 |
|------|--------|-----------|---------|
| 普通 RAG | 33.0 | 13.7% | $0.080 |
| Agentic RAG（单 Agent） | 32.8 | 8.8% | $0.098 |
| SPD-RAG | **58.1** | **18.6%** | $0.103 |
| 全上下文基线 | 68.0 | 31.4% | $0.273 |

SPD-RAG 用全上下文基线 37.9% 的成本达到 85.4% 的性能，成本效率（得分/每美元）是全上下文基线的 2.25 倍。学术论文场景最明显：传统 RAG 完美率 0%，SPD-RAG 平均分 60。

> 论文：https://arxiv.org/pdf/2603.08329

### 题目：自反思 RAG（Self-Reflection RAG）是什么？

#### 核心解析

**作用：** 检索后用 LLM 评估结果相关性（1-5 分），分数低则优化 query 重新搜索，最多迭代 2-3 次。

**示例：**
```
迭代 1：query="部署" → 评分 2/5（太模糊）
迭代 2：优化为 "机器学习模型部署到生产环境" → 评分 4/5（结果好，返回）
```

**适用场景：** 答案准确性极重要、延迟可接受的场景（研究应用、复杂查询）。

**缺点：** 最高延迟（2-3 次 LLM 调用）、最贵的策略，对真正困难的查询可能仍然失败。

#### Self-RAG 的 Reflection Tokens 机制

Self-RAG（华盛顿大学 + IBM）是自反思 RAG 的一个重要变体，通过训练 LLM 生成特殊的 Reflection Tokens 实现自主检索决策和质量评估：

| Token | 含义 | 可能值 |
|-------|------|--------|
| `[Retrieve]` | 是否需要检索 | Yes / No |
| `[IsRel]` | 检索内容是否相关 | Relevant / Irrelevant |
| `[IsSup]` | 生成内容是否有文档支撑 | Fully Supported / Partially / No Support |
| `[IsUse]` | 答案整体是否有用 | 1-5 分 |

推理时 LLM 在生成过程中预测这些 token，形成"检索 → 生成 → 评估 → 优化"的闭环。如果 `[IsRel]` 判定为 Irrelevant，自动重新检索；如果 `[IsSup]` 判定为 No Support，重新生成或添加低置信度警告。

**性能数据（Self-RAG 论文）：**
- 事实核查准确率：71%（传统 RAG）→ 81%（Self-RAG）
- 长文本引用准确率：Self-RAG 80%，ChatGPT 71%
- 7B 参数的 Self-RAG 在 TriviaQA 上超越 13B Llama2-chat + RAG

#### 真实案例：金融科技公司三阶段演进

| 阶段 | 方案 | 效果 |
|------|------|------|
| 第一阶段（1 个月） | Router Agent：根据查询类型路由到不同知识源 | 召回率 60% → 78%，P99 延迟 3.5s → 2.1s |
| 第二阶段（2 个月） | Multi-Agent：每个主题域专属 Agent + Meta-Agent 综合 | 多方案对比满意度 65% → 89%，追问率下降 40% |
| 第三阶段（3 个月） | Self-RAG：自动评估相关性和支撑性，低置信度重新检索 | 事实准确率 82% → 93% |

关键收获：分阶段演进（不要一步到位）、Haiku 做路由 + Sonnet 做生成（节省 60% 成本）、基于真实用户查询日志优化。

## 七 ter、知识图谱增强检索

### 题目：什么时候需要在 RAG 中加入知识图谱？

#### 核心解析

**作用：** 把向量搜索和图数据库结合，捕捉实体间的明确关系。

**解决的问题：** 向量搜索找到"语义相似"的文本，但错过实体间的明确关系——如 "CEO of"、"located in"、"reported revenue"。

**对比示例：**

```
查询："谁运营 ACME 公司，第二季度发生了什么变化？"

纯向量搜索结果：
- "ACME 公司 CEO 信息..."
- "第二季度变化包括..."
（两段相似文本，但无法直接回答）

知识图谱结果：
ACME 公司
  ├─ HAS_CEO → Jane Smith
  ├─ REPORTED_REVENUE → $314M
  │   └─ PERIOD → Q2 2024
  └─ LOCATED_IN → California
（直接答："Jane Smith 运营 ACME 公司，Q2 收入增至 3.14 亿美元"）
```

**主流方案：**
- **Graphiti**（开源）：自动从文本提取实体和关系，混合"语义相似 + BM25 + 图遍历 + 时间上下文"。
- **Neo4j + 自建提取流程**：手动定义实体类型，用 LLM 或 NER 模型抽取。

**适用场景：** 实体间关系至关重要的领域（医疗网络、财务系统、研究数据库）。

**缺点：** 需要 Neo4j 等图数据库基础设施，设置和维护复杂，比纯向量慢且贵，需要实体提取流程。

> 项目地址：https://github.com/graphiti-ai/graphiti

**拓展：知识图谱怎么自动构建？** Graphusion（WWW 2025）提出三步走方案：种子实体生成（主题建模确定骨架）→ 候选三元组抽取（检索 + LLM）→ 知识图谱融合（实体合并、冲突消解、跨子图补边）。核心发现：去掉融合步骤后关系质量明显下降，"融合"才是从碎片化子图到全局可推理图谱的关键。另外 RAG 不一定总带来提升——长而噪的文档可能拖累模型，高质量短摘要更友好。

### 题目：三代 Embedding 算法是怎么演进的？

#### 核心解析

**第一代：静态词向量（Word2Vec / GloVe / FastText）**

核心思路：用一个词周围的词来预测这个词。训练后每个词对应一个固定向量。

- Word2Vec：CBOW 和 Skip-gram 两种训练方式，「国王 - 男人 + 女人 ≈ 女王」
- GloVe：对整个语料库的词共现矩阵做分解，对全局统计信息利用更充分
- FastText：把词拆成字符级 n-gram 子词，解决未登录词问题

致命缺陷：静态。「我吃了苹果」和「苹果手机发布了」里的「苹果」向量完全相同，处理不了多义词。

**第二代：上下文相关向量（ELMo / BERT）**

同一个词在不同语境下有不同向量表示。

- ELMo：双向 LSTM，从左往右和从右往左扫描句子
- BERT：Transformer 替代 LSTM，Masked Language Model 预训练，用 [CLS] token 表示整个句子

致命缺陷：检索时必须把两个句子拼在一起跑 BERT，百万级知识库要跑百万次，完全不可接受。具体算：BERT 一次前向传播几十毫秒，百万次就是好几个小时，用户根本等不起。

**第三代：句子级对比学习 Embedding（SBERT / SimCSE / BGE）——RAG 标配**

每个句子独立生成一个向量，直接用余弦相似度比较。知识库所有文档向量提前算好存起来，检索时只算一次查询向量，毫秒级返回。

- SBERT（2019）：bi-encoder 结构，两个句子分别独立过 BERT，各自得到句子级向量。精度比 cross-encoder 低（两个句子不再交互，看不到细微关联），但速度提升几个数量级——在 RAG 这种"速度优先"的场景里，这个取舍完全值得
- SimCSE：对比学习，同一句话做两次 dropout 得到正样本对，解决 BERT 向量「各向异性」问题
- BGE（北京智源）：中文 RAG 场景最常用的开源模型，同时支持 bi-encoder 和 reranker

#### 三代对比总结

| 世代 | 代表模型 | 核心特点 | 主要局限 | RAG 适用性 |
|------|----------|----------|----------|-----------|
| 第一代 | Word2Vec、GloVe、FastText | 词级静态向量 | 无法处理多义词 | 不适用 |
| 第二代 | ELMo、BERT | 上下文动态向量 | 检索时需两两拼接，速度极慢 | 不适用于实时检索 |
| 第三代 | SBERT、SimCSE、BGE、E5 | 句子级 bi-encoder，对比学习 | 精度低于 cross-encoder | 标配，性能和精度平衡最优 |

面试回答：中文场景首选 BGE，英文场景 E5 或 text-embedding-3-small。

> 参考：[[模型选型]] 中有 Baize 项目选择 Qwen3-Embedding-0.6B 和 1024 维度的具体理由

### 题目：Embedding 模型怎么选？看哪些维度？

#### 核心解析

Embedding 模型是 RAG 的地基，选错了上面建再多也歪。选型看五个维度：

**1. 领域匹配度**
最容易被忽略。通用模型在垂直领域（医疗、法律、金融）可能表现平平——模型没见过你领域的术语和表达方式，再高的 MTEB 分数也没用。

**2. 上下文长度**
模型单次能处理的最大 token 数。长文档检索（论文、法律文书）优先选 8192+ tokens 窗口的模型；短文本场景（客服对话）2048 tokens 够用。

**3. 向量维度**
维度越高，语义信息越丰富，检索精度越高——但存储成本增加、搜索变慢。
- 高维度（3072、4096）：高精度场景
- 低维度（384、512）：大规模检索、成本敏感场景
- 新趋势：Matryoshka 训练（如 OpenAI text-embedding-3、KaLM-Embedding）支持灵活截断维度，精度损失较小

**4. 语言支持**
中文场景最需要关注。OpenAI 模型对中文"可用但建议实测"，BGE 系列和 Qwen 系列专为中文深度优化。

**5. 延迟与成本**
API 按量付费 vs 本地部署（需要显卡）。轻量模型（DistilBERT、MiniLM）推理更快。

#### Embedding 评估方法

**简单评估：区分度测试**

构造正负例对，计算模型对正例和负例的相似度差距（区分度）：

```python
import numpy as np

def evaluate_embedding(get_embedding_func, test_cases):
    pos_scores, neg_scores = [], []
    for case in test_cases:
        q_vec = get_embedding_func(case["query"])
        p_vec = get_embedding_func(case["positive"])
        n_vec = get_embedding_func(case["negative"])
        pos_scores.append(cosine_similarity(q_vec, p_vec))
        neg_scores.append(cosine_similarity(q_vec, n_vec))
    return {
        "avg_positive_similarity": np.mean(pos_scores),
        "avg_negative_similarity": np.mean(neg_scores),
        "discrimination": np.mean(pos_scores) - np.mean(neg_scores)
    }

test_cases = [
    {"query": "怎么退款", "positive": "退货流程说明", "negative": "公司年假政策"},
```

### 题目：微调 Embedding 模型有用吗？什么时候做？

#### 核心解析

**作用：** 在领域特定的查询-文档对上训练 Embedding 模型，让向量空间更适配业务术语。

**解决的问题：** 通用 Embedding 不理解专业术语（医学术语、法律行话、技术缩写），相似度计算偏差大。

**对比示例：**

```
查询："什么是营运资金？"

通用嵌入：
1. "营运资金包括..." (0.72)
2. "资本市场提供..." (0.68)  ← 错误，被"资本"混淆
3. "工作条件在..." (0.65)    ← 错误，被"working"混淆

微调嵌入：
1. "营运资金包括..." (0.89)
2. "营运资金比率计算..." (0.84)
3. "有效管理营运资金..." (0.81)
```

**做法：**
- 收集 1000+ 条领域查询-文档对
- 用 `sentence-transformers` 加载基础模型（如 `all-MiniLM-L6-v2`）
- 用 `MultipleNegativesRankingLoss` 训练 3 epoch

**预期收益：** 通常提升 5-10% 准确率，较小模型可以胜过较大的通用模型。

**适用场景：** 通用嵌入表现明显不佳的专业领域（医疗、法律、财务、技术）。

**缺点：** 需要训练数据、训练时间和资源、需要定期重新训练。

#### 微调实战案例

**CustomIR 框架（合成数据微调）：**

三步流程自动生成微调数据：
1. 利用 LLM 从文档库中自动生成多样化的查询
2. 生成正例和经过 LLM 验证的难负例
3. 用合成数据对预训练 Embedding 模型进行对比学习微调

实验表明，这种方法可以让小模型在特定领域上的 Recall@10 提升 2.3 个百分点，性能接近大得多的模型，从而大幅降低部署成本。

**Cisco IT 企业实战：**

Cisco IT 团队使用 NVIDIA Nemotron RAG 微调方案，在约 925 份文档、9,200 个问答对、7,800 个训练样本的规模上，对 10 亿参数的 NV-EmbedQA 模型进行了微调。

**效果数据：**
- NDCG@1 提升了 7.1 到 7.3 个绝对百分点（相对提升 9.9% 到 11.1%）
- 整个微调流程在单张 H200 GPU 上只需几小时完成
- 无需人工标注，迭代非常快

**什么时候该微调？** 当通用模型在你的数据上检索效果明显不达预期，且你有一定规模的文档库（几百份以上），微调的投入产出比通常很高。反之，如果数据量很少或通用模型已经够用，直接使用现成模型更划算。

## 七 quater、RAG 优化策略分类与组合

> 内容来源：[2026年构建RAG系统的核心策略](https://mp.weixin.qq.com/s?__biz=Mzk0OTI1OTQ2MQ==&mid=2247597619)

### 题目：RAG 系统有哪些优化策略？如何分类？

#### 核心解析

RAG 优化策略可分为三大类：**摄取策略**（如何准备文档）、**查询策略**（如何搜索）、**后处理策略**（如何优化结果）。

#### 一、摄取策略（Ingestion Strategies）

**1. 上下文感知分块（Context-Aware Chunking）**
- **解决问题**：固定大小分块会在语义中间切断，丢失上下文
- **实现方式**：按语义边界分割，保持相关内容在一起
- **工具**：Docling HybridChunker
- **项目链接**：https://github.com/DS4SD/docling

**2. 上下文检索（Contextual Retrieval）**
- **解决问题**：分块缺乏文档级上下文（如"收入增长 40%"不知道是哪个公司、哪个季度）
- **实现方式**：为每个分块添加 1-2 句话解释其与整个文档的关系
- **工具**：使用 LLM 生成上下文前缀
- **参考**：Anthropic 上下文检索研究 https://www.anthropic.com/news/contextual-retrieval

**Contextual Retrieval 详细实现**（Anthropic 提出）：

> 内容来源：[Anthropic RAG 技术突破](https://mp.weixin.qq.com/s?__biz=MzY4NDE1MTMzNg==&mid=2247484117)

**核心问题**：文档分块破坏了上下文，导致相关信息变得不可检索。例如 chunk "公司营收相比上一季度增长了 3%" 缺少公司名和季度信息，单独检索时无法匹配查询。

**实现步骤**：
1. 文档分块
2. 为每个 chunk 用 LLM 生成上下文描述（50-100 tokens）
3. 将"上下文 + 原始 chunk"一起向量化
4. 存入向量数据库

**Prompt 模板**（简化版）：
```
<document>
{{整个文档内容}}
</document>

这是文档的一个片段：
<chunk>
{{当前 chunk}}
</chunk>

请用简洁的语言（50-100 tokens）描述这个片段的上下文，
说明它来自哪个文档、讨论什么主题。
只输出上下文描述，不要解释。
```

**三种组合方案**：
- **Contextual Embeddings**：向量检索 + 上下文增强，适合语义匹配
- **Contextual BM25**：关键词检索 + 上下文增强，适合精确匹配（如错误代码 "TS-999"）
- **混合检索（推荐）**：两者结合 + RRF 融合，覆盖更多场景

**成本优化**：使用 Prompt Caching 缓存文档内容，避免重复传输，成本约 $1.02/百万 tokens（Claude 3 Haiku + Prompt Caching）

**适用场景**：
- 知识库文档结构化程度高（财报、法律文档、技术文档）
- 查询需要精确匹配（公司名、时间、数据等）
- 检索失败率高（明明有答案却检索不到）

**不适用场景**：
- 文档本身很短（< 500 tokens）
- 查询主要是开放式问答
- 成本敏感且文档量巨大

- Anthropic 官方博客：https://www.anthropic.com/news/contextual-retrieval
- Prompt Caching 文档：https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching

- **实现方式**：提取实体和关系，构建知识图谱
- **工具**：Graphiti

#### 二、查询策略（Query Strategies）

**4. 查询扩展（Query Expansion）**
- **解决问题**：单一查询表述可能错过语义相同但表述不同的文档
- **实现方式**：生成多个查询变体（同义词、不同角度）

**5. 假设文档嵌入（HyDE - Hypothetical Document Embeddings）**
- **解决问题**：查询和文档的语义空间可能不匹配
- **实现方式**：让 LLM 生成假设答案，用假设答案的 embedding 检索

**6. 混合搜索（Hybrid Search）**
- **解决问题**：纯向量检索可能错过关键词匹配
- **实现方式**：结合向量检索（语义）+ BM25（关键词）

**7. 重排序（Reranking）**
- **解决问题**：初步检索结果可能不是最相关的
- **实现方式**：使用专门的重排序模型对结果重新打分
- **工具**：Cohere rerank-3
- **参考**：Pinecone 重排序指南 https://www.pinecone.io/learn/series/rag/rerankers/

#### 三、后处理策略（Post-Processing Strategies）

**8. 上下文压缩（Context Compression）**
- **解决问题**：检索到的内容可能包含大量无关信息
- **实现方式**：移除无关信息，保留关键内容

**9. RAG Fusion**
- **解决问题**：多个查询的结果需要合并
- **实现方式**：使用倒数排名融合（RRF）合并多个查询的结果

**10. 自适应检索（Adaptive Retrieval）**
- **解决问题**：不同查询需要不同数量的上下文
- **实现方式**：根据查询复杂度动态调整检索数量（top_k）

**11. 迭代检索（Iterative Retrieval）**
- **解决问题**：单次检索可能不够
- **实现方式**：多轮检索，每轮基于上一轮结果优化查询

### 题目：如何组合多个 RAG 策略？

#### 核心解析

单个策略只能解决特定问题，组合策略可以覆盖更多失败场景。推荐三种组合方案：

**方案 1：通用生产堆栈**
- **策略组合**：上下文感知分块 + 重排序 + 查询扩展 + 智能体 RAG
- **适用场景**：通用生产环境、客户支持、内部知识库
- **特点**：平衡效果和成本

**方案 2：高准确率堆栈**
- **策略组合**：上下文检索 + 多查询 + 重排序 + 自反思
- **适用场景**：医疗、法律、财务等错误成本高的场景
- **特点**：追求准确率，可接受更高延迟和成本

**方案 3：领域专家堆栈**
- **策略组合**：微调嵌入 + 上下文检索 + 知识图谱 + 重排序
- **适用场景**：医疗、法律、财务、技术等专业领域
- **特点**：针对特定领域深度优化

### 题目：RAG 系统应该怎么渐进式实施？

#### 核心解析

不要一次性上所有策略——系统会过于复杂、难以调试、成本失控。推荐四阶段实施：

**阶段 1：基础（第 1 周）**
- 上下文感知分块（替代固定 512 切分）
- 基础向量搜索
- 测量基准指标（建立评估数据集）

**阶段 2：快速优化（第 2-3 周）**
- 加重排序（优先实施，ROI 高）
- 加查询扩展（处理模糊查询）
- 测量改进效果

**阶段 3：高级优化（第 4-6 周）**
- 加多查询 RAG 或智能体 RAG（按用例选）
- 关键查询启用自反思
- 微调和优化参数

**阶段 4：专业化（第 2 个月以后）**
- 高价值文档加上下文检索
- 实体关系重要时引入知识图谱
- 微调 Embedding 提升领域准确率

### 题目：实施 RAG 系统时常见的错误有哪些？

#### 核心解析

| 错误 | 问题 | 解决方案 |
|------|------|---------|
| 一次用所有策略 | 系统过于复杂、难调试、成本高 | 从基础堆栈开始，按需添加 |
| 不测量基准 | 无法证明改进 | 建立评估数据集，每次改动前后测量指标 |
| 固定分块大小 | 破坏语义连贯性 | 始终用上下文感知分块 |
| 忽略重排序 | 向量相似 ≠ 语义相关 | 重排序 ROI 高，应尽早实施 |
| 没有查询预处理 | 用户查询模糊导致检索失败 | 至少加查询扩展 |
| 单一检索策略 | 不同查询需要不同策略 | 用智能体 RAG 提供灵活性 |

## 八、性能优化与工程实践

### 题目：百万级文档的 RAG 系统怎么设计？

#### 核心解析

分层来看，存储和检索分别优化。

**存储层：**
- MySQL 存文档元数据（文件名、路径、用户、分类），不存向量，百万级文档也就几 GB
- ES 存向量和关键词，HNSW 索引，冷热分层（热数据 SSD，冷数据 HDD）
- 文件本体存 MinIO 对象存储，天然支持多机共享访问

**检索层优化：**
- 先元数据过滤再向量搜索，缩小候选集
- Redis 缓存热门查询，命中率 40%+
- 多阶段级联检索：BM25 快速筛选 50 个候选 → 向量检索 + 重排精排

**异步处理流水线：**
- 文档上传立刻返回成功，后台 Kafka 异步消化
- 文档解析 → 向量化 → 入库全流程异步化
- 单日处理能力 5000+ 文档

**水平扩展关键：状态外置**
- 文件存 MinIO，任务存 Kafka，Session 存 Redis，数据存 MySQL/ES
- 检索服务、生成服务天然无状态，前面加负载均衡直接扩
- WebSocket 多机部署：粘性会话（Nginx ip_hash）或 Redis Pub/Sub 消息广播

### 题目：增量索引怎么做？

#### 核心解析

动态知识库不能每次新增文档都重建全量索引，成本太高。

**增量索引策略：**
- 仅对新增、修改、删除的文档进行索引更新
- ES 支持实时插入新文档并更新向量索引，延迟低于 1 秒
- 定期评估检索效果（人工反馈或点击率），自动调整 BM25 与向量检索的权重比例

**查询自适应权重：**
- 事实类、关键词类查询：提升稀疏检索权重
- 概念类、解释类查询：提升密集检索权重
- 实现方式：规则判断（含专有名词/编号 → 提升 BM25）或训练查询分类模型

## 九、面试答题模板

### 混合检索被追问时的完整答法

**先讲为什么（20 秒）：** "纯向量检索对专有名词、精确编号匹配弱，纯 BM25 无法理解语义。混合检索取两者之长，实验数据显示 Recall@5 比纯向量提升 21.3%。"

**再讲怎么做（40 秒）：** "两路并行——向量检索做宽召回（topK×30），BM25 做精排 rescore。融合用 RRF 算法，对每个文档按排名计算 1/(rank+60) 的倒数分，不依赖原始分数量纲，天然抗噪。加了 bge-reranker-v2-m3 做二次重排，Cross-Encoder 同时读取 query 和文档，精度比 Bi-Encoder 高，P95 延迟 50ms 以内。"

**最后讲数字（20 秒）：** "在 200 条测试集上，混合检索+重排的 Recall@5 达到 0.896，端到端准确率 0.903，响应时间 1086ms。用 RAGAS 跑自动指标，人工抽检 50 条校准，偏差在 3% 以内。"

## 十一、RAG 未来发展趋势

**1. 多模态 RAG**
不仅检索文本，还检索图片、视频、音频。用 CLIP 等模型做图片检索，多模态 LLM 综合文本和图片上下文生成答案。详见 [[RAG基础与架构#RAG-Anything（港大 HKUDS，1.7 万 Star）]]。

**2. 长上下文 + RAG 融合**
随着 LLM 上下文窗口扩大（Gemini 2.0 Flash 已达 100 万 token），RAG 的角色将转变：短上下文场景继续用 RAG 检索；长上下文场景把整个知识库放入上下文，RAG 变成"智能导航"——先粗筛 Top-50 文档，再全部放入长上下文精读。

**3. 端到端神经检索**
传统 RAG 依赖独立的 Embedding 模型，未来会出现端到端训练的检索-生成模型：RA-DIT（同时优化检索器和生成器）、DSI（把整个知识库编码到模型参数中）。

**4. 个性化与隐私保护**
企业 RAG 将更关注：基于用户历史和偏好优化检索结果、联邦学习和差分隐私保护数据安全、细粒度权限控制（不同用户看到不同检索结果）。
