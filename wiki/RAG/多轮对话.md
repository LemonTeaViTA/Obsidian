# RAG 中的多轮对话

> RAG 系统处理多轮对话有三个核心问题：==对话历史怎么存==、==上下文窗口怎么填==、==检索怎么衔接历史==。本文集中讲清楚这三个层面，把散落在各处的知识点统一起来。
>
> 去重的工程实现见 [[RAG检索策略#四、多轮对话检索去重]]——本文是它的前置和补充。

---

## 一、对话存储设计

### 1.1 双表 schema

业界通用模式是==两表分离==：会话元数据 + 具体消息。

```sql
-- 会话表（一个用户可以有多个会话）
CREATE TABLE conversations (
    id           VARCHAR(64) PRIMARY KEY,    -- 会话 ID（雪花/UUID）
    user_id      VARCHAR(64) NOT NULL,        -- 用户标识
    title        VARCHAR(255),                -- 会话标题（首次提问自动生成）
    created_at   TIMESTAMP NOT NULL,
    updated_at   TIMESTAMP NOT NULL,           -- 最后一条消息时间
    summary      TEXT,                         -- 历史摘要（达到压缩阈值后写入）
    metadata     JSON,                         -- 扩展字段：标签、模型版本、用户偏好
    INDEX idx_user_updated (user_id, updated_at DESC)
);

-- 消息表（一个会话下多条消息）
CREATE TABLE messages (
    id              BIGINT PRIMARY KEY,
    conversation_id VARCHAR(64) NOT NULL,
    role            ENUM('system','user','assistant','tool') NOT NULL,
    content         TEXT NOT NULL,
    created_at      TIMESTAMP NOT NULL,
    token_count     INT,                       -- 预先算好，省去后续计数
    metadata        JSON,                      -- 关键：检索时返回的 chunk_ids、tool_calls、置信度等
    INDEX idx_conv_time (conversation_id, created_at)
);
```

==`messages.metadata` 字段非常关键==——里面记录每轮 assistant 回答时用到的 `chunk_ids`，是后续多轮去重 [[RAG检索策略#四、多轮对话检索去重|consumed_ids]] 的来源。

### 1.2 消息格式

每条消息的标准 JSON 形态：

```json
{
  "id": 12345,
  "conversation_id": "conv_a1b2c3",
  "role": "assistant",
  "content": "重疾险的等待期通常是 90-180 天...",
  "created_at": "2026-05-25T14:30:00Z",
  "token_count": 187,
  "metadata": {
    "chunk_ids": ["chunk_887", "chunk_212", "chunk_445"],   // 用过的 chunk
    "retrieved_doc_ids": ["doc_115"],                        // 来源文档
    "model": "qwen3-72b",
    "latency_ms": 1240,
    "rerank_scores": [0.92, 0.85, 0.79]
  }
}
```

==chunk_ids 必存==，否则下一轮去重就拿不到上一轮的"已用"信息。

### 1.3 选型对比：SQLite / Redis / PostgreSQL

| 存储 | 适用 | 优势 | 劣势 |
|------|------|------|------|
| **SQLite** | 单机部署、个人助手、桌面应用 | 零依赖、嵌入式、文件即数据库 | 单机、无并发写、不适合多租户 |
| **Redis** | 高吞吐、短期对话、会话状态缓存 | 内存级延迟、原生 TTL | 重启丢数据（需配 RDB/AOF）、不适合长期归档 |
| **PostgreSQL** | 多租户 SaaS、需要分析查询 | 强一致、JSON 字段原生支持、丰富索引 | 运维成本高于 SQLite |
| **Redis + PostgreSQL** | 生产推荐 | ==Redis 当热缓存（最近 N 轮）+ PG 当冷存储==（全量历史） | 双写一致性需处理 |

==生产典型搭配==：
- Redis：当前活跃会话的最近 ~20 轮消息（TTL 1 小时滑动延期）
- PostgreSQL：全量持久化（异步写入）
- 前端拉历史先查 Redis，未命中回 PG 并回填 Redis

### 1.4 写入时机

| 模式 | 流程 | 取舍 |
|------|------|------|
| ==同步写==（推荐） | 用户消息存完才走 RAG → 助手回完存完才返回 | 数据确定性强，丢消息风险低；延迟略增（5-20ms） |
| 异步写 | 消息扔队列异步入库，不阻塞主流程 | 吞吐高；崩溃时可能丢失 |
| 批量异步 | 累积 N 条或 N 秒批量入库 | 写入性能最好；丢失窗口最大 |

==RAG 场景同步写为主==——多轮对话依赖准确的历史，异步丢消息会破坏对话连续性。

### 1.5 会话 ID 与 TTL

**会话 ID 生成**：
- ==UUID v4==：简单、全球唯一、无序（不暴露业务量）
- ==雪花 ID==：有时间序、便于按时间分片；但暴露生成节点

**TTL 策略**：
- Redis 缓存层：1 小时滑动 TTL（每次访问续期）
- PG 持久化层：不删除，按 `user_id + updated_at` 分区便于归档
- 用户主动归档/删除：会话标记 `archived` 而非物理删除（合规要求保留审计日志）

## 二、上下文窗口管理

### 2.1 上下文窗口的组成

每次调用 LLM 时，最终送进去的 prompt 由四部分组成：

```
┌─────────────────────────────────────────────────────────┐
│  System Prompt                          ~5%             │
│  ├─ 角色设定 + 任务定义                                  │
│  └─ 输出格式约束                                         │
├─────────────────────────────────────────────────────────┤
│  对话历史（History）                    ~30%            │
│  ├─ 前 N 轮消息原文                                      │
│  └─ 或：摘要 + 最近 N 轮原文（混合策略）                 │
├─────────────────────────────────────────────────────────┤
│  检索结果（Retrieval）                  ~50%            │
│  └─ Top-K chunk + 元数据                                 │
├─────────────────────────────────────────────────────────┤
│  当前 Query                             ~5%             │
├─────────────────────────────────────────────────────────┤
│  生成余量（LLM 输出）                   ~10%            │
└─────────────────────────────────────────────────────────┘
                                          总 100%
```

### 2.2 token 预算分配（128K 上下文典型示例）

```
总预算: 128,000 tokens
├─ system_prompt:    6,400  (5%)   ← 固定，写死
├─ history:         38,400  (30%)  ← 动态压缩
├─ retrieval:       64,000  (50%)  ← Top-K × chunk_size
├─ query:            6,400  (5%)   ← 通常用不完
└─ output reserve:  12,800  (10%)  ← LLM 生成留余
```

==关键原则==：
- 给生成留 ==≥10% 余量==，否则模型可能输出截断
- retrieval 占大头，因为它是 RAG 的"知识来源"
- history 不能挤占 retrieval 太多——过长的对话历史反而稀释当前问题的注意力

### 2.3 三种历史截断策略

#### 滑动窗口（Sliding Window）

只保留最近 N 轮原文，更早的丢弃。

```python
def sliding_window(messages, max_turns=10):
    return messages[-max_turns:]
```

- ✅ 实现极简，零额外延迟
- ❌ 丢失早期上下文（用户可能说"刚才那个产品" → 但已经被截掉）
- 适用：对话短、不需要回顾早期信息的场景

#### 摘要压缩（Summarization）

==历史轮次过长时用 LLM 压缩成摘要==，最近 N 轮保留原文：

```python
def summarize_history(messages, threshold_tokens=20000):
    if total_tokens(messages) < threshold_tokens:
        return messages  # 还没到压缩阈值

    early = messages[:-3]    # 早期消息要压缩
    recent = messages[-3:]   # 最近 3 轮保留原文
    summary = llm.invoke(
        f"用 200 字总结以下对话，保留实体、时间、决策：\n{early}"
    )
    return [{"role": "system", "content": f"[历史摘要] {summary}"}] + recent
```

- ✅ 大幅节省 token（10K 历史 → 200 字摘要）
- ❌ 摘要质量影响后续轮次；增加一次 LLM 调用延迟
- 适用：长对话、需要保留全局上下文的场景

#### 混合策略（推荐）

==最近 N 轮原文 + 更早的滚动摘要==：

```
[摘要: 用户在咨询重疾险，关注等待期和保额]
+ Turn N-2: 用户问 "那个保额怎么计算"
+ Turn N-1: 助手回答了保额计算公式
+ Turn N: 用户问 "如果加上免赔额呢"  ← 当前
```

最近的对话原文保留细节（指代消解需要），早期对话被摘要后只占 ~200 token。

### 2.4 怎么知道哪些消息重要？五种判断方案

==这是摘要压缩最关键的工程难点==——用户问"算法怎么知道哪些重要"，答案是：算法不知道，是 ==**人**== 通过 prompt / 规则 / 标注 告诉它的。

> [!warning] 关键认知：==99% 实际是 LLM 在做==
> 下面列五种方案，但==生产中真正用的是方案 1（LLM 判断）和方案 5（LLM 嵌套调用）==。==方案 2/3/4 在 2018-2022 传统 NLP 时代有人试过，LLM 时代基本被淘汰==——读到这里要清楚"==方案 2/3/4 是历史/学术价值,不是生产标准=="，==面试时别答"用规则打分判断重要性"，那是 BERT 之前的思路==。

#### 方案 1：让 LLM 自己判断（==生产主流==）

把判断标准明确写进 prompt：

```python
SUMMARY_PROMPT = """请总结以下对话，控制在 200 字以内。

==必须保留==的信息：
1. 实体：产品名、人名、订单号、地名等具体名词
2. 数值：金额、百分比、时间、数量
3. 用户的关键决策（"我选 A"、"取消订单"等）
4. 未解决的问题或待办事项

==可以舍弃==的信息：
- 寒暄、感谢、客套话
- 重复确认（"明白了""好的"）
- 助手反复解释同一概念
- 已彻底回答完的问题（除非有后续追问）

对话历史：
{messages}

总结："""
```

==关键==：把"什么重要"的标准写进 prompt——光说"请总结"质量很差。

- ✅ 实现简单、泛化强、加新规则只改 prompt
- ❌ 质量不稳定、增加一次 LLM 调用

#### 方案 2：消息入库时打标签（==历史方案，生产几乎不用==）

==每条消息存储时==就标注是否关键，不靠后期判断：

```python
def annotate_message(msg):
    msg["metadata"]["is_critical"] = (
        contains_decision(msg) or         # 含"选/确认/取消/同意"
        contains_entity(msg) or            # 含产品名/订单号
        has_numeric_value(msg) or          # 含金额/日期
        is_assistant_with_chunks(msg)      # assistant 用过具体知识
    )
    msg["metadata"]["intent"] = classify_intent(msg)
```

总结时优先保留 `is_critical=True` 的消息，其他丢掉。

> [!warning] 这个方案为什么生产几乎不用
> ==谁来打标签==是死穴：
> - 让==用户==每条对话标"是否重要"——==荒谬==，用户根本不会做
> - 让==Agent 自动==标——那==判断"是否重要"还是 LLM 在做==，只是把判断时机从"压缩时"提前到"入库时"，==绕回方案 1==
> - 让==预处理脚本==标（基于规则）——和方案 3 一样的脆弱性问题
>
> ==所以方案 2 不是独立方案==，而是==方案 1 的一个变体==（早一点判断）或==方案 3 的应用==（脚本判断）。生产中真正存在的"打标签"通常是 LLM 主动调 `set_critical(true)` 工具——本质还是方案 1。

- ✅ 决策一次永久有效；总结时不重判
- ❌ "谁打"无解；要么回到 LLM(方案 1)，要么回到脆弱规则(方案 3)

#### 方案 3：纯规则启发式（==只能用作"否定式过滤"，不能识别重要==）

> [!danger] 关键认知：规则识别"什么重要"==不可行==
> 中文场景规则==极脆弱==：
> - "决策关键词"词典爆炸："我选 A" / "用 A 吧" / "就 A 了" / "OK A" / "听你的" 全是决策但词面千变万化
> - 否定/转折难识别："我本来想 A，但还是 B" — 规则要识别"实际选了 B"
> - 隐式决策："这两个都不喜欢，再推荐几个" — 没有关键词但==是关键决策==
> - 跨语言/跨业务：中英混合 / 法律 / 医疗 / 电商 各自规则要分别写
>
> ==规则只能用来"==否定式过滤=="==——==确定不重要的去掉==(明显寒暄、超短消息)，==剩下的全部交给 LLM==判断。==不要让规则做"识别什么重要"==。

==正确用法==——作为方案 1 的"==预过滤=="：

```python
def compact_messages(messages):
    # ★ 第 1 步：规则过滤==明显==的垃圾（零成本快速过滤）
    filtered = [
        m for m in messages
        if len(m["content"]) > 8                    # 太短的"嗯""好的"
        and not is_pure_courtesy(m["content"])      # 纯寒暄词典命中
    ]

    # ★ 第 2 步：剩下的==全部交给 LLM==
    summary = llm_summarize(filtered, prompt=CRITICAL_PROMPT)
    return summary
```

==规则的合理打分维度==(==仅用作过滤明显垃圾==,不用作识别重要):

| 信号 | 规则 | 用法 |
|------|------|------|
| 消息长度 | < 8 字 | 过滤掉(可能是寒暄) |
| 寒暄词 | 命中寒暄词典 | 过滤掉 |
| 重复 ack | "好的""明白""谢谢" 单独成句 | 过滤掉 |

- ✅ 零延迟、零成本、可解释
- ❌ ==只能过滤垃圾,不能识别什么重要==——必须配合 LLM

#### 方案 4：基于注意力/嵌入的重要性评分（==研究前沿,生产几乎不用==）

让模型自己学哪些重要：

- **MemGPT 模式**：让 LLM 评分输出 importance: 1-10——==这其实就是方案 1 的变体==
- **LongContextLLM 类似方法**：用 attention head 注意力分布——哪些 token 被频繁 attend 就是重要的
- **嵌入聚类**：消息向量化 → 聚类 → 每簇保留簇心最近的代表性消息

> [!info] 实际生产中==几乎为零==
> attention 评分需要访问模型内部权重(API 模型不开放),嵌入聚类质量不稳定且额外延迟。==Claude Code / Cursor / ChatGPT / Anthropic / LangChain 都没用这种方法==——论文有,产品没有。

- ✅ 泛化最强(理论上)
- ❌ ==复杂、生产几乎为零==

#### 方案 5：分层摘要（==生产实际推荐==）

==离当前越远，压缩越狠==：

```
最近 5 轮：原文保留           ← 细节最完整（指代消解需要）
6-15 轮：基础摘要（每轮 1-2 句）  ← 保留核心
15+ 轮：高度浓缩（整段 1 句）     ← 只剩主线
```

##### 谁来干？三种执行者选择

| 选项 | 用什么 | 优点 | 缺点 |
|------|-------|------|------|
| ==方案 A==（推荐）：主 LLM 同步压缩 | 同一对话的主模型（如 Qwen3-72B） | 质量最好、零额外部署 | 增加用户感知延迟 1-3s、占主模型配额 |
| 方案 B：小模型异步压缩 | 专门部署 Qwen3-1.5B | 快（< 300ms）、便宜 | 质量较差、需独立服务 |
| 方案 C：主模型异步压缩 | 主模型，但用户思考时后台跑 | 不阻塞用户、质量好 | 实现复杂（后台任务队列） |

==生产推荐方案 A==：质量是关键，多 1-2 秒延迟比走错路代价低。

##### 怎么干？三种压缩组织方式

**方式 1：两步独立压缩**（==错误示范==）

```
原文 ── prompt A ──> 基础摘要
原文 ── prompt B ──> 高度浓缩
```

==问题==：两个摘要独立从原文压，可能==互相不一致==（一个说"用户选 A"另一个说"用户选 B"）。

**方式 2：递增式压缩**（推荐）

```
原文 ── 基础摘要 prompt ──> 基础摘要（10 行 × 每行 1-2 句）
                                    │
                                    └── 高度浓缩 prompt ──> 高度浓缩（整段 1 句）
```

==高度浓缩从基础摘要再压一次，不从原文==。好处：节省 token（输入只有 ~500 字而非几千字原文）+ 保证一致性（高度浓缩派生自基础摘要）。

**方式 3：滚动增量压缩**（==生产真实做法==）

==不每轮重压全部，按轮次触发，且只压新增部分==：

```python
# 缓存上次的摘要结果
cache = {
    "heavy_summary": "用户在咨询重疾险，已确认想买保额 50 万",
    "basic_summary": [   # 第 6-15 轮（共 10 条）
        "[第 6 轮][user] 问免赔额怎么算",
        "[第 7 轮][assistant] 解释了 1 万免赔额机制",
        ...
    ],
    "last_compressed_turn": 15
}

# 第 20 轮触发：每 5 轮触发一次
if turn_n - cache["last_compressed_turn"] >= 5:
    # 1. 压缩第 16-20 轮 → 5 条新的基础摘要
    new_basic = compress_to_basic(messages[15:20])

    # 2. basic 中"年纪大的"挤出去：第 6-10 轮 → 合并到 heavy
    pushed_out = cache["basic_summary"][:5]
    new_heavy = compress_to_heavy(cache["heavy_summary"] + pushed_out)

    cache["heavy_summary"] = new_heavy
    cache["basic_summary"] = cache["basic_summary"][5:] + new_basic
    cache["last_compressed_turn"] = 20
```

==每次只新压 5 条==，不是把整个历史重压一遍——成本可控。

##### 两个 prompt 怎么写？

==基础摘要和高度浓缩的 prompt 完全不同==：

**基础摘要 prompt**（每轮 1-2 句）：

```
请==逐轮==总结以下消息，每条用 1-2 句话。

==必须保留==：
- 提到的实体（产品名、人名、订单号）
- 数值（金额、百分比、日期）
- 用户的关键诉求和决策
- 助手提供的核心事实

==可舍弃==：寒暄、客套、重复确认

==格式==（必须严格）：
[第 N 轮][user/assistant] 摘要内容

对话：
{messages}
```

输出形如：
```
[第 6 轮][user] 用户问免赔额怎么计算
[第 7 轮][assistant] 解释 1 万免赔额：超过部分按比例报销
[第 8 轮][user] 选择 1 万免赔额方案
```

**高度浓缩 prompt**（整段 1 句）：

```
请用 1-2 句话总结以下早期对话主线，==不需要逐轮列出==。

==必须保留==：
- 用户咨询的核心主题
- 已确认的关键事实和决策
- 未解决的悬而未决的问题

==格式==：[早期历史] 一段连贯总结

对话：
{basic_summary}    ← 注意输入是基础摘要，不是原文
```

输出形如：
```
[早期历史] 用户在咨询重疾险，已确认选择保额 50 万 + 免赔额 1 万方案，
对等待期还有疑问待解答。
```

==两个 prompt 的关键差异==：

| 维度 | 基础摘要 | 高度浓缩 |
|------|---------|---------|
| 粒度 | 逐轮（每轮 1-2 句） | 整段（1-2 句覆盖多轮） |
| 时序 | 保留 | 弱化 |
| 输入 | 原文 | 基础摘要（==不是原文==） |
| 输出长度 | ~500 token | ~100 token |
| Prompt 焦点 | 实体、数值、决策 | 主题、悬而未决问题 |

##### 触发时机

| 触发条件 | 实现 | 适用 |
|---------|------|------|
| ==token 阈值==（推荐） | `total_tokens > 80%` 上下文时才压 | 通用，按需触发 |
| 轮次阈值 | 每 5/15 轮压一次 | 长对话场景，节奏稳定 |
| 时间阈值 | 用户停顿 > 30s 后台压缩 | 异步压缩场景 |

生产常见组合：==token 阈值 80% 主触发 + 轮次阈值兜底==。

##### 完整可执行示例

```python
BASIC_PROMPT = """请逐轮总结以下消息，每条 1-2 句。
保留：实体、数值、决策、用户诉求、助手关键信息。
舍弃：寒暄、重复确认。
格式：[第 N 轮][user/assistant] 摘要内容

对话：
{messages}"""

HEAVY_PROMPT = """请用 1-2 句话总结以下早期对话主线，不要逐轮列。
保留：核心主题、已确认事实、未解决问题。
格式：[早期历史] 一段连贯总结

对话：
{basic_summary}"""

def hierarchical_summarize(messages, llm):
    if len(messages) <= 5:
        return messages

    recent = messages[-5:]
    middle = messages[-15:-5]
    early = messages[:-15]

    # Step 1：middle → 基础摘要
    middle_summary = llm.invoke(
        BASIC_PROMPT.format(messages=format_messages(middle)),
        max_tokens=500
    )

    if early:
        # Step 2a：early 先压成基础摘要
        early_basic = llm.invoke(
            BASIC_PROMPT.format(messages=format_messages(early)),
            max_tokens=300
        )
        # Step 2b：再从基础摘要压成高度浓缩
        early_heavy = llm.invoke(
            HEAVY_PROMPT.format(basic_summary=early_basic),
            max_tokens=150
        )
        result = [
            {"role": "system", "content": f"[早期历史] {early_heavy}"},
            {"role": "system", "content": f"[最近历史摘要]\n{middle_summary}"},
            *recent
        ]
    else:
        result = [
            {"role": "system", "content": f"[历史摘要]\n{middle_summary}"},
            *recent
        ]
    return result
```

##### 摘要不可逆——生产必须留原文

==摘要压完原文就丢了==，万一摘要错了或丢关键信息，==恢复不了==。

解决方案：
1. ==始终保留原文在持久化存储==（PostgreSQL 全量历史），仅在送 LLM 的 prompt 中用摘要
2. 用户追问"刚才你说什么来着"时，==绕过摘要从持久化层拉原文==
3. 长期会话回顾时，==基于原文重新生成更精确的摘要==

==摘要是 prompt 优化，不是数据替代==。详见 [[#一、对话存储设计]] 的双层架构（Redis 缓存 + PG 持久化）。

#### 市面主流产品的实际做法

==2025-2026 年==，主流 AI 产品的对话压缩/记忆提取方案==高度趋同==——==都是 LLM 调用 + 精心设计的 prompt + 一个工具==。横向对比:

| 产品 | 触发时机 | 谁判断重要性 | 输出去向 |
|------|---------|------------|---------|
| ==Claude Code `/compact`== | context 接近上限 / 用户敲 `/compact` / 会话结束 | LLM(精心 prompt) | 替换原 messages 前部 + 最近几轮原文保留 |
| ==Claude Memory tool==(Anthropic) | LLM 在每轮 ReAct 循环中自主判断 | LLM(看 system prompt 引导) | 调 `save_memory` / `update_memory` / `delete_memory` 工具写入 |
| ==ChatGPT `bio()`== | 对话中 LLM 检测到"值得记的事实"自主调用 | LLM(看 system prompt 引导) | 调 `bio(content)` 工具写入用户 Memory |
| ==Cursor Memory== | 用户 `/save` 主动 / Composer 模式下 LLM 自动 | LLM(自主判断或用户指定) | 写入 Notepads |
| ==LangChain ConversationSummaryMemory== | 每 N 轮触发 | LLM(框架内置 summarize prompt) | 替换旧 messages |
| ==LangGraph Summarizer node== | 状态图中显式定义触发条件 | LLM(用户写 prompt) | 写入 State 字段 |

==共同点==(==这就是行业事实标准==):

1. ==判断重要性==: ==全部用 LLM==——没有一个用规则或打标签
2. ==交互方式==: ==给 LLM 一个工具==(`save_memory` / `bio` / `compact`),==让 LLM 自主决定何时调==
3. ==prompt 引导==: 在 system prompt 里==明确告诉 LLM=="什么值得记 / 什么不值得"
4. ==原始数据保留==: ==都不删==原始对话日志(JSONL/SQLite),压缩只是 prompt 优化

==Claude Code `/compact` 的 prompt 摘要==(从开源逆向看):
```
"分析以下对话历史,生成精炼摘要。==必须保留==:
- 用户的具体目标和约束
- 已完成的关键步骤(文件路径/命令/decisions)
- 未完成的任务和下一步计划
- 出现的错误和解决方案
==可以省略==:
- 寒暄、确认词
- 重复的解释
- 已彻底完成且无关的子任务"
```

==关键洞察==:这些 prompt ==99% 都长得很像==——==因为"什么值得记"这件事==,主流认知==高度收敛==。==面试时能讲出几家产品的具体 prompt 套路==(Claude Code "保留约束/已完成/下一步/错误" 四要素 + ChatGPT bio() 工具触发)==比死记五种方案有深度==。

#### 推荐组合

==与"市面主流产品的实际做法"对齐——LLM 主导,规则只做预过滤==:

```
预过滤(方案 3 否定式):去掉明显寒暄/超短消息(零成本)
   ↓
触发(2.5 节):达到 80% 上下文阈值 / 用户主动 / 会话结束
   ↓
主体(方案 5 分层):最近 5 轮原文 + 中段基础摘要 + 远段高度浓缩
   ↓
压缩(方案 1 LLM):每段都用 LLM,prompt 明确列"必须保留 vs 可以舍弃"
   ↓
持久化:摘要写入 MEMORY.md / 当前 prompt;==原始 JSONL 永久保留==
```

==没有方案 2 的入库标注==——因为"==谁来标==" 没有好答案,实际产品都是直接靠 prompt + LLM 即时判断。

#### 常被忽略的细节

1. **==时效性衰减==**：相同信息越早出现的相对越不重要
   - "我是张三"（5 轮前）—— 仍重要
   - "嗯"（5 秒前）—— 立刻可丢

2. **==实体共指==**：摘要时把代词替换为实体
   - 原文："他说他要买"
   - 摘要："张三说要买重疾险"
   - 否则摘要本身丢失指代锚点

3. **==决策反转保留==**：用户改主意时**两个版本都要保留**
   - "我选 A" → 后来 "改成 B"
   - 摘要：用户最初选 A，后改为 B（不能只留 B，审计/解释需要原因）

4. **==失败/未解决标记==**：助手回答失败的内容要标注
   - "用户问 X，未找到相关信息" 要保留
   - 否则下一轮重复问还会再次失败

### 2.5 触发时机

不是每轮都压缩，按阈值触发：

```python
HISTORY_BUDGET = 38_400  # token

def maybe_compress(messages):
    used = total_tokens(messages)
    if used > HISTORY_BUDGET * 0.8:   # 达到 80% 阈值才压缩
        return summarize_history(messages)
    return messages
```

==设 80% 阈值而非 100%==：留缓冲应对下一轮 query 突然变长。

## 三、多轮检索衔接

### 3.1 consumed_ids 的提取

去重需要的 `consumed_ids` 列表==怎么从对话历史拿==：

```python
def extract_consumed_ids(conversation_id, recent_turns=5):
    """从最近 N 轮 assistant 消息提取已用过的 chunk_ids"""
    messages = db.query(
        "SELECT metadata FROM messages "
        "WHERE conversation_id = ? AND role = 'assistant' "
        "ORDER BY created_at DESC LIMIT ?",
        conversation_id, recent_turns
    )

    consumed = set()
    for msg in messages:
        chunk_ids = msg.metadata.get("chunk_ids", [])
        consumed.update(chunk_ids)

    return list(consumed)
```

==关键设计点==：
- 只看最近 N 轮（默认 5 轮）——避免列表过长导致 [[RAG检索策略#四、多轮对话检索去重|HNSW 降级为暴力扫描]]
- 缓存到 Redis（key=`consumed:{conv_id}`，TTL 1 小时）减少 DB 查询
- 用户切换话题时（intent 变化）应清空 consumed_ids

### 3.2 指代消解

用户经常用代词、省略主语：

```
Turn 1: "重疾险等待期是多久？"
Turn 2: "保额呢？"          ← "保额"指什么？
Turn 3: "刚才那个产品有定期险吗？"  ← "刚才那个产品"是？
```

直接拿 "保额呢？" 去检索==几乎必然失败==——向量里只有 "保额"，缺少"重疾险"这个限定。

==解决方案==：用 LLM ==改写 query==，把指代替换为实体：

```python
def rewrite_with_history(current_query, recent_messages):
    prompt = f"""根据以下对话历史，把当前用户的问题改写成可独立理解的查询：

历史：
{format_messages(recent_messages[-3:])}

当前问题：{current_query}

改写后的查询（只输出改写结果，不要解释）："""

    return llm.invoke(prompt)
```

例子：
- 输入 query：`"保额呢？"`
- 历史：`Turn 1: "重疾险等待期"`
- 改写后：`"重疾险的保额是多少？"`

==改写后的 query 才是真正送去检索的==，原 query 只用作 LLM 生成时的"用户原话"参考。

### 3.3 查询改写依赖历史

更进阶：==查询改写不只是消解指代，还要补全语境==。

| 用户原话 | 改写后 |
|---------|-------|
| "保额呢？" | "重疾险的保额一般是多少？" |
| "再来一份类似的" | "再推荐一份和重疾险类似的保险产品" |
| "他什么时候说的" | "投保人 ACME 公司在哪一年签订的合同" |

实现：取最近 3 轮历史 + 当前 query → LLM 改写 → 改写后的 query 走检索。

==注意==：如果改写后 query 与原 query 语义偏差太大（如改写"错了"），需要==同时保留两路检索==——既检索改写后，也检索原 query，结果合并。

### 3.4 重复问题缓存快速路径

用户在同一会话里==短时重复问==的场景常见（"刚才说的等待期是多少来着"）。==不需要再走完整的检索 + LLM 生成==，缓存上次答案直接返回。

实现：

```python
def maybe_use_cache(conv_id, query):
    """先查缓存，命中则直接返回上次答案"""
    # 1. Hash 完全一致 → 立即返回
    qhash = hashlib.md5(query.encode()).hexdigest()
    cached = redis.get(f"qcache:{conv_id}:{qhash}")
    if cached:
        return json.loads(cached)

    # 2. Embedding 相似度 > 0.95 → 视为同一问题
    qvec = embedder.encode(query)
    recent_qa = redis.hgetall(f"qa_history:{conv_id}")
    for past_q, past_answer in recent_qa.items():
        past_vec = json.loads(past_answer)["query_vec"]
        if cosine(qvec, past_vec) > 0.95:
            return json.loads(past_answer)["answer"]

    return None  # 缓存未命中，走完整流程
```

==缓存写入时机==：每轮 RAG 流程结束时把 (query_hash, query_vec, answer) 存进 Redis，TTL 5 分钟。

==关键设计==：
- 命中缓存==绕过检索和 LLM==，延迟从 1.5s 降到 5ms
- TTL 5 分钟而非永久——知识库可能更新，旧答案不能长期复用
- 用户明确说"换个说法"或"再讲讲"时应主动失效缓存

## 四、与 Agent Memory 的边界

==Conversation Memory 是 RAG 应用层的"会话历史"概念==，与更宽泛的 Agent Memory 体系是不同抽象层级：

| 层级 | 含义 | 在本文中的对应 |
|------|------|---------------|
| Agent Memory L1（工作记忆） | 当前任务的临时变量、中间推理 | 不在本文范围 |
| Agent Memory L2（情景记忆） | 当前会话的历史交互 | ==本文 §一对话存储 + §二上下文窗口== |
| Agent Memory L3（长期记忆） | 跨会话的用户偏好、画像 | 部分对应 §一中的 `conversations.metadata` |

详见 [[Agent核心概念#Memory]] 的完整 Memory 架构。

==简单记忆==：
- 在本会话内问"刚才说的什么" → §三 多轮检索衔接
- 跨会话问"上次我们聊到的" → Agent L3 长期记忆，本文不展开
