# RAG 中的多轮对话

> [!info] 速览
> RAG 系统处理多轮对话有三个核心问题：对话历史怎么存、上下文窗口怎么填、检索怎么衔接历史。本文集中讲清楚这三个层面，把散落在各处的知识点统一起来。
>
> 去重的工程实现见 [[RAG检索策略#四、多轮对话检索去重]]——本文是它的前置和补充；对话压缩/记忆提取的深入实现见 [[对话压缩与记忆提取]]。

## 一、对话存储设计

### 1.1 双表 schema

业界通用模式是==两表分离==：会话元数据 + 具体消息。

```sql
-- 会话表（一个用户可以有多个会话）
CREATE TABLE conversations (
    id           VARCHAR(64) PRIMARY KEY,    -- 会话 ID（雪花/UUID）
    user_id      VARCHAR(64) NOT NULL,        -- 用户标识
    title        VARCHAR(255),                -- 会话标题（首次提问自动生成）
    created_at   TIMESTAMP NOT NULL,
    updated_at   TIMESTAMP NOT NULL,           -- 最后一条消息时间
    summary      TEXT,                         -- 历史摘要（达到压缩阈值后写入）
    metadata     JSON,                         -- 扩展字段：标签、模型版本、用户偏好
    INDEX idx_user_updated (user_id, updated_at DESC)
);

-- 消息表（一个会话下多条消息）
CREATE TABLE messages (
    id              BIGINT PRIMARY KEY,
    conversation_id VARCHAR(64) NOT NULL,
    role            ENUM('system','user','assistant','tool') NOT NULL,
    content         TEXT NOT NULL,
    created_at      TIMESTAMP NOT NULL,
    token_count     INT,                       -- 预先算好，省去后续计数
    metadata        JSON,                      -- 关键：检索时返回的 chunk_ids、tool_calls、置信度等
    INDEX idx_conv_time (conversation_id, created_at)
);
```

==`messages.metadata` 字段非常关键==——里面记录每轮 assistant 回答时用到的 `chunk_ids`，是后续多轮去重 [[RAG检索策略#四、多轮对话检索去重|consumed_ids]] 的来源。

### 1.2 消息格式

每条消息的标准 JSON 形态：

```json
{
  "id": 12345,
  "conversation_id": "conv_a1b2c3",
  "role": "assistant",
  "content": "重疾险的等待期通常是 90-180 天...",
  "created_at": "2026-05-25T14:30:00Z",
  "token_count": 187,
  "metadata": {
    "chunk_ids": ["chunk_887", "chunk_212", "chunk_445"],   // 用过的 chunk
    "retrieved_doc_ids": ["doc_115"],                        // 来源文档
    "model": "qwen3-72b",
    "latency_ms": 1240,
    "rerank_scores": [0.92, 0.85, 0.79]
  }
}
```

==chunk_ids 必存==，否则下一轮去重就拿不到上一轮的"已用"信息。

### 1.3 选型对比：SQLite / Redis / PostgreSQL

| 存储 | 适用 | 优势 | 劣势 |
|------|------|------|------|
| **SQLite** | 单机部署、个人助手、桌面应用 | 零依赖、嵌入式、文件即数据库 | 单机、无并发写、不适合多租户 |
| **Redis** | 高吞吐、短期对话、会话状态缓存 | 内存级延迟、原生 TTL | 重启丢数据（需配 RDB/AOF）、不适合长期归档 |
| **PostgreSQL** | 多租户 SaaS、需要分析查询 | 强一致、JSON 字段原生支持、丰富索引 | 运维成本高于 SQLite |
| **Redis + PostgreSQL** | 生产推荐 | ==Redis 当热缓存（最近 N 轮）+ PG 当冷存储==（全量历史） | 双写一致性需处理 |

==生产典型搭配==：
- Redis：当前活跃会话的最近 ~20 轮消息（TTL 1 小时滑动延期）
- PostgreSQL：全量持久化（异步写入）
- 前端拉历史先查 Redis，未命中回 PG 并回填 Redis

### 1.4 写入时机

| 模式 | 流程 | 取舍 |
|------|------|------|
| ==同步写==（推荐） | 用户消息存完才走 RAG → 助手回完存完才返回 | 数据确定性强，丢消息风险低；延迟略增（5-20ms） |
| 异步写 | 消息扔队列异步入库，不阻塞主流程 | 吞吐高；崩溃时可能丢失 |
| 批量异步 | 累积 N 条或 N 秒批量入库 | 写入性能最好；丢失窗口最大 |

==RAG 场景同步写为主==——多轮对话依赖准确的历史，异步丢消息会破坏对话连续性。

### 1.5 会话 ID 与 TTL

**会话 ID 生成**：
- ==UUID v4==：简单、全球唯一、无序（不暴露业务量）
- ==雪花 ID==：有时间序、便于按时间分片；但暴露生成节点

**TTL 策略**：
- Redis 缓存层：1 小时滑动 TTL（每次访问续期）
- PG 持久化层：不删除，按 `user_id + updated_at` 分区便于归档
- 用户主动归档/删除：会话标记 `archived` 而非物理删除（合规要求保留审计日志）

## 二、上下文窗口管理

### 2.1 上下文窗口的组成

每次调用 LLM 时，最终送进去的 prompt 由四部分组成：

```
┌─────────────────────────────────────────────────────────┐
│  System Prompt                          ~5%             │
│  ├─ 角色设定 + 任务定义                                  │
│  └─ 输出格式约束                                         │
├─────────────────────────────────────────────────────────┤
│  对话历史（History）                    ~30%            │
│  ├─ 前 N 轮消息原文                                      │
│  └─ 或：摘要 + 最近 N 轮原文（混合策略）                 │
├─────────────────────────────────────────────────────────┤
│  检索结果（Retrieval）                  ~50%            │
│  └─ Top-K chunk + 元数据                                 │
├─────────────────────────────────────────────────────────┤
│  当前 Query                             ~5%             │
├─────────────────────────────────────────────────────────┤
│  生成余量（LLM 输出）                   ~10%            │
└─────────────────────────────────────────────────────────┘
                                          总 100%
```

### 2.2 token 预算分配（128K 上下文典型示例）

```
总预算: 128,000 tokens
├─ system_prompt:    6,400  (5%)   ← 固定，写死
├─ history:         38,400  (30%)  ← 动态压缩
├─ retrieval:       64,000  (50%)  ← Top-K × chunk_size
├─ query:            6,400  (5%)   ← 通常用不完
└─ output reserve:  12,800  (10%)  ← LLM 生成留余
```

==关键原则==：
- 给生成留 ==≥10% 余量==，否则模型可能输出截断
- retrieval 占大头，因为它是 RAG 的"知识来源"
- history 不能挤占 retrieval 太多——过长的对话历史反而稀释当前问题的注意力

### 2.3 三种历史截断策略

#### 滑动窗口（Sliding Window）

只保留最近 N 轮原文，更早的丢弃。

```python
def sliding_window(messages, max_turns=10):
    return messages[-max_turns:]
```

- ✅ 实现极简，零额外延迟
- ❌ 丢失早期上下文（用户可能说"刚才那个产品" → 但已经被截掉）
- 适用：对话短、不需要回顾早期信息的场景

#### 摘要压缩（Summarization）

==历史轮次过长时用 LLM 压缩成摘要==，最近 N 轮保留原文：

```python
def summarize_history(messages, threshold_tokens=20000):
    if total_tokens(messages) < threshold_tokens:
        return messages  # 还没到压缩阈值

    early = messages[:-3]    # 早期消息要压缩
    recent = messages[-3:]   # 最近 3 轮保留原文
    summary = llm.invoke(
        f"用 200 字总结以下对话，保留实体、时间、决策：\n{early}"
    )
    return [{"role": "system", "content": f"[历史摘要] {summary}"}] + recent
```

- ✅ 大幅节省 token（10K 历史 → 200 字摘要）
- ❌ 摘要质量影响后续轮次；增加一次 LLM 调用延迟
- 适用：长对话、需要保留全局上下文的场景

#### 混合策略（推荐）

==最近 N 轮原文 + 更早的滚动摘要==：

```
[摘要: 用户在咨询重疾险，关注等待期和保额]
+ Turn N-2: 用户问 "那个保额怎么计算"
+ Turn N-1: 助手回答了保额计算公式
+ Turn N: 用户问 "如果加上免赔额呢"  ← 当前
```

最近的对话原文保留细节（指代消解需要），早期对话被摘要后只占 ~200 token。

### 2.4 对话压缩与记忆提取（已拆分）

> [!info] 本节已拆分为独立文档
> 「怎么知道哪些消息重要」涉及五种判断方案、分层摘要完整工程实现、市面主流产品做法与推荐组合，内容较长，已独立成篇：[[对话压缩与记忆提取]]。
>
> 一句话结论：判断重要性==全部交给 LLM==（精心 prompt + 一个工具），规则只做「否定式预过滤」；分层摘要（最近原文 + 中段基础摘要 + 远段高度浓缩）是生产推荐组合，原始日志永久保留。

### 2.5 触发时机

不是每轮都压缩，按阈值触发：

```python
HISTORY_BUDGET = 38_400  # token

def maybe_compress(messages):
    used = total_tokens(messages)
    if used > HISTORY_BUDGET * 0.8:   # 达到 80% 阈值才压缩
        return summarize_history(messages)
    return messages
```

==设 80% 阈值而非 100%==：留缓冲应对下一轮 query 突然变长。

## 三、多轮检索衔接

### 3.1 consumed_ids 的提取

去重需要的 `consumed_ids` 列表==怎么从对话历史拿==：

```python
def extract_consumed_ids(conversation_id, recent_turns=5):
    """从最近 N 轮 assistant 消息提取已用过的 chunk_ids"""
    messages = db.query(
        "SELECT metadata FROM messages "
        "WHERE conversation_id = ? AND role = 'assistant' "
        "ORDER BY created_at DESC LIMIT ?",
        conversation_id, recent_turns
    )

    consumed = set()
    for msg in messages:
        chunk_ids = msg.metadata.get("chunk_ids", [])
        consumed.update(chunk_ids)

    return list(consumed)
```

==关键设计点==：
- 只看最近 N 轮（默认 5 轮）——避免列表过长导致 [[RAG检索策略#四、多轮对话检索去重|HNSW 降级为暴力扫描]]
- 缓存到 Redis（key=`consumed:{conv_id}`，TTL 1 小时）减少 DB 查询
- 用户切换话题时（intent 变化）应清空 consumed_ids

### 3.2 指代消解

用户经常用代词、省略主语：

```
Turn 1: "重疾险等待期是多久？"
Turn 2: "保额呢？"          ← "保额"指什么？
Turn 3: "刚才那个产品有定期险吗？"  ← "刚才那个产品"是？
```

直接拿 "保额呢？" 去检索==几乎必然失败==——向量里只有 "保额"，缺少"重疾险"这个限定。

==解决方案==：用 LLM ==改写 query==，把指代替换为实体：

```python
def rewrite_with_history(current_query, recent_messages):
    prompt = f"""根据以下对话历史，把当前用户的问题改写成可独立理解的查询：

历史：
{format_messages(recent_messages[-3:])}

当前问题：{current_query}

改写后的查询（只输出改写结果，不要解释）："""

    return llm.invoke(prompt)
```

例子：
- 输入 query：`"保额呢？"`
- 历史：`Turn 1: "重疾险等待期"`
- 改写后：`"重疾险的保额是多少？"`

==改写后的 query 才是真正送去检索的==，原 query 只用作 LLM 生成时的"用户原话"参考。

### 3.3 查询改写依赖历史

更进阶：==查询改写不只是消解指代，还要补全语境==。

| 用户原话 | 改写后 |
|---------|-------|
| "保额呢？" | "重疾险的保额一般是多少？" |
| "再来一份类似的" | "再推荐一份和重疾险类似的保险产品" |
| "他什么时候说的" | "投保人 ACME 公司在哪一年签订的合同" |

实现：取最近 3 轮历史 + 当前 query → LLM 改写 → 改写后的 query 走检索。

==注意==：如果改写后 query 与原 query 语义偏差太大（如改写"错了"），需要==同时保留两路检索==——既检索改写后，也检索原 query，结果合并。

### 3.4 重复问题缓存快速路径

用户在同一会话里==短时重复问==的场景常见（"刚才说的等待期是多少来着"）。==不需要再走完整的检索 + LLM 生成==，缓存上次答案直接返回。

实现：

```python
def maybe_use_cache(conv_id, query):
    """先查缓存，命中则直接返回上次答案"""
    # 1. Hash 完全一致 → 立即返回
    qhash = hashlib.md5(query.encode()).hexdigest()
    cached = redis.get(f"qcache:{conv_id}:{qhash}")
    if cached:
        return json.loads(cached)

    # 2. Embedding 相似度 > 0.95 → 视为同一问题
    qvec = embedder.encode(query)
    recent_qa = redis.hgetall(f"qa_history:{conv_id}")
    for past_q, past_answer in recent_qa.items():
        past_vec = json.loads(past_answer)["query_vec"]
        if cosine(qvec, past_vec) > 0.95:
            return json.loads(past_answer)["answer"]

    return None  # 缓存未命中，走完整流程
```

==缓存写入时机==：每轮 RAG 流程结束时把 (query_hash, query_vec, answer) 存进 Redis，TTL 5 分钟。

==关键设计==：
- 命中缓存==绕过检索和 LLM==，延迟从 1.5s 降到 5ms
- TTL 5 分钟而非永久——知识库可能更新，旧答案不能长期复用
- 用户明确说"换个说法"或"再讲讲"时应主动失效缓存

## 四、与 Agent Memory 的边界

==Conversation Memory 是 RAG 应用层的"会话历史"概念==，与更宽泛的 Agent Memory 体系是不同抽象层级：

| 层级 | 含义 | 在本文中的对应 |
|------|------|---------------|
| Agent Memory L1（工作记忆） | 当前任务的临时变量、中间推理 | 不在本文范围 |
| Agent Memory L2（情景记忆） | 当前会话的历史交互 | ==本文 §一对话存储 + §二上下文窗口== |
| Agent Memory L3（长期记忆） | 跨会话的用户偏好、画像 | 部分对应 §一中的 `conversations.metadata` |

详见 [[Agent 核心概念#Memory]] 的完整 Memory 架构。

==简单记忆==：
- 在本会话内问"刚才说的什么" → §三 多轮检索衔接
- 跨会话问"上次我们聊到的" → Agent L3 长期记忆，本文不展开
