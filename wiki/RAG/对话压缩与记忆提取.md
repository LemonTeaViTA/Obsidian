# 对话压缩与记忆提取

> [!info] 速览
> 摘要压缩最关键的工程难点是：怎么知道哪些消息重要。本文给出五种判断方案、生产推荐的分层摘要完整实现、市面主流产品的实际做法与推荐组合。
>
> 本文是 [[多轮对话#二、上下文窗口管理]] 的展开——对话历史超长时如何压缩。上下文窗口的整体预算分配与截断策略见 [[多轮对话]]。

---

## 一、核心认知：谁判断"哪些消息重要"

用户常问"算法怎么知道哪些消息重要"，答案是：算法不知道，是 ==人== 通过 prompt / 规则 / 标注 告诉它的。

> [!warning] 99% 实际是 LLM 在做
> 下面列五种方案，但生产中真正用的是方案 1（LLM 判断）和方案 5（LLM 嵌套调用）。方案 2/3/4 在 2018-2022 传统 NLP 时代有人试过，LLM 时代基本被淘汰，属历史/学术价值，不是生产标准。面试时别答"用规则打分判断重要性"——那是 BERT 之前的思路。

---

## 二、五种判断方案

### 方案 1：让 LLM 自己判断（==生产主流==）

把判断标准明确写进 prompt：

```python
SUMMARY_PROMPT = """请总结以下对话，控制在 200 字以内。

必须保留的信息：
1. 实体：产品名、人名、订单号、地名等具体名词
2. 数值：金额、百分比、时间、数量
3. 用户的关键决策（"我选 A"、"取消订单"等）
4. 未解决的问题或待办事项

可以舍弃的信息：
- 寒暄、感谢、客套话
- 重复确认（"明白了""好的"）
- 助手反复解释同一概念
- 已彻底回答完的问题（除非有后续追问）

对话历史：
{messages}

总结："""
```

关键在于把"什么重要"的标准写进 prompt——光说"请总结"质量很差。

- ✅ 实现简单、泛化强、加新规则只改 prompt
- ❌ 质量不稳定、增加一次 LLM 调用

### 方案 2：消息入库时打标签（==历史方案，生产几乎不用==）

每条消息存储时就标注是否关键，不靠后期判断：

```python
def annotate_message(msg):
    msg["metadata"]["is_critical"] = (
        contains_decision(msg) or         # 含"选/确认/取消/同意"
        contains_entity(msg) or            # 含产品名/订单号
        has_numeric_value(msg) or          # 含金额/日期
        is_assistant_with_chunks(msg)      # assistant 用过具体知识
    )
    msg["metadata"]["intent"] = classify_intent(msg)
```

总结时优先保留 `is_critical=True` 的消息，其他丢掉。

> [!warning] 这个方案为什么生产几乎不用
> "谁来打标签"是死穴：
> - 让用户每条对话标"是否重要"——荒谬，用户根本不会做
> - 让 Agent 自动标——那判断"是否重要"还是 LLM 在做，只是把判断时机从"压缩时"提前到"入库时"，绕回方案 1
> - 让预处理脚本标（基于规则）——和方案 3 一样的脆弱性问题
>
> 所以方案 2 不是独立方案，而是方案 1 的一个变体（早一点判断）或方案 3 的应用（脚本判断）。生产中真正存在的"打标签"通常是 LLM 主动调 `set_critical(true)` 工具——本质还是方案 1。

- ✅ 决策一次永久有效；总结时不重判
- ❌ "谁打"无解；要么回到 LLM（方案 1），要么回到脆弱规则（方案 3）

### 方案 3：纯规则启发式（==只能做否定式过滤==）

> [!danger] 规则识别"什么重要"不可行
> 中文场景规则极脆弱：
> - "决策关键词"词典爆炸："我选 A" / "用 A 吧" / "就 A 了" / "OK A" / "听你的" 全是决策但词面千变万化
> - 否定/转折难识别："我本来想 A，但还是 B" — 规则要识别"实际选了 B"
> - 隐式决策："这两个都不喜欢，再推荐几个" — 没有关键词但是关键决策
> - 跨语言/跨业务：中英混合 / 法律 / 医疗 / 电商 各自规则要分别写
>
> 规则只能用来"否定式过滤"——确定不重要的去掉（明显寒暄、超短消息），剩下的全部交给 LLM 判断。不要让规则做"识别什么重要"。

正确用法是作为方案 1 的==预过滤==：

```python
def compact_messages(messages):
    # ★ 第 1 步：规则过滤明显的垃圾（零成本快速过滤）
    filtered = [
        m for m in messages
        if len(m["content"]) > 8                    # 太短的"嗯""好的"
        and not is_pure_courtesy(m["content"])      # 纯寒暄词典命中
    ]

    # ★ 第 2 步：剩下的全部交给 LLM
    summary = llm_summarize(filtered, prompt=CRITICAL_PROMPT)
    return summary
```

规则的合理打分维度（仅用作过滤明显垃圾，不用作识别重要）：

| 信号 | 规则 | 用法 |
|------|------|------|
| 消息长度 | < 8 字 | 过滤掉（可能是寒暄） |
| 寒暄词 | 命中寒暄词典 | 过滤掉 |
| 重复 ack | "好的""明白""谢谢" 单独成句 | 过滤掉 |

- ✅ 零延迟、零成本、可解释
- ❌ 只能过滤垃圾，不能识别什么重要——必须配合 LLM

### 方案 4：基于注意力/嵌入的重要性评分（==研究前沿，生产几乎不用==）

让模型自己学哪些重要：

- **MemGPT 模式**：让 LLM 评分输出 importance: 1-10——这其实就是方案 1 的变体
- **LongContextLLM 类似方法**：用 attention head 注意力分布——哪些 token 被频繁 attend 就是重要的
- **嵌入聚类**：消息向量化 → 聚类 → 每簇保留簇心最近的代表性消息

> [!info] 实际生产中几乎为零
> attention 评分需要访问模型内部权重（API 模型不开放），嵌入聚类质量不稳定且额外延迟。Claude Code / Cursor / ChatGPT / Anthropic / LangChain 都没用这种方法——论文有，产品没有。

- ✅ 泛化最强（理论上）
- ❌ 复杂、生产几乎为零

### 方案 5：分层摘要（==生产实际推荐==）

离当前越远，压缩越狠：

```text
最近 5 轮：原文保留           ← 细节最完整（指代消解需要）
6-15 轮：基础摘要（每轮 1-2 句）  ← 保留核心
15+ 轮：高度浓缩（整段 1 句）     ← 只剩主线
```

方案 5 的完整工程实现见下一节。

## 三、分层摘要工程实现

### 3.1 谁来干？三种执行者选择

| 选项 | 用什么 | 优点 | 缺点 |
|------|-------|------|------|
| ==方案 A==（推荐）：主 LLM 同步压缩 | 同一对话的主模型（如 Qwen3-72B） | 质量最好、零额外部署 | 增加用户感知延迟 1-3s、占主模型配额 |
| 方案 B：小模型异步压缩 | 专门部署 Qwen3-1.5B | 快（< 300ms）、便宜 | 质量较差、需独立服务 |
| 方案 C：主模型异步压缩 | 主模型，但用户思考时后台跑 | 不阻塞用户、质量好 | 实现复杂（后台任务队列） |

生产推荐方案 A：质量是关键，多 1-2 秒延迟比走错路代价低。

### 3.2 怎么干？三种压缩组织方式

**方式 1：两步独立压缩**（==错误示范==）

```text
原文 ── prompt A ──> 基础摘要
原文 ── prompt B ──> 高度浓缩
```

问题：两个摘要独立从原文压，可能互相不一致（一个说"用户选 A"另一个说"用户选 B"）。

**方式 2：递增式压缩**（推荐）

```text
原文 ── 基础摘要 prompt ──> 基础摘要（10 行 × 每行 1-2 句）
                                    │
                                    └── 高度浓缩 prompt ──> 高度浓缩（整段 1 句）
```

高度浓缩从基础摘要再压一次，不从原文。好处：节省 token（输入只有 ~500 字而非几千字原文）+ 保证一致性（高度浓缩派生自基础摘要）。

**方式 3：滚动增量压缩**（==生产真实做法==）

不每轮重压全部，按轮次触发，且只压新增部分：

```python
# 缓存上次的摘要结果
cache = {
    "heavy_summary": "用户在咨询重疾险，已确认想买保额 50 万",
    "basic_summary": [   # 第 6-15 轮（共 10 条）
        "[第 6 轮][user] 问免赔额怎么算",
        "[第 7 轮][assistant] 解释了 1 万免赔额机制",
        ...
    ],
    "last_compressed_turn": 15
}

# 第 20 轮触发：每 5 轮触发一次
if turn_n - cache["last_compressed_turn"] >= 5:
    # 1. 压缩第 16-20 轮 → 5 条新的基础摘要
    new_basic = compress_to_basic(messages[15:20])

    # 2. basic 中"年纪大的"挤出去：第 6-10 轮 → 合并到 heavy
    pushed_out = cache["basic_summary"][:5]
    new_heavy = compress_to_heavy(cache["heavy_summary"] + pushed_out)

    cache["heavy_summary"] = new_heavy
    cache["basic_summary"] = cache["basic_summary"][5:] + new_basic
    cache["last_compressed_turn"] = 20
```

每次只新压 5 条，不是把整个历史重压一遍——成本可控。

### 3.3 两个 prompt 怎么写？

基础摘要和高度浓缩的 prompt ==完全不同==：

**基础摘要 prompt**（每轮 1-2 句）：

```text
请逐轮总结以下消息，每条用 1-2 句话。

必须保留：
- 提到的实体（产品名、人名、订单号）
- 数值（金额、百分比、日期）
- 用户的关键诉求和决策
- 助手提供的核心事实

可舍弃：寒暄、客套、重复确认

格式（必须严格）：
[第 N 轮][user/assistant] 摘要内容

对话：
{messages}
```

输出形如：
```text
[第 6 轮][user] 用户问免赔额怎么计算
[第 7 轮][assistant] 解释 1 万免赔额：超过部分按比例报销
[第 8 轮][user] 选择 1 万免赔额方案
```

**高度浓缩 prompt**（整段 1 句）：

```text
请用 1-2 句话总结以下早期对话主线，不需要逐轮列出。

必须保留：
- 用户咨询的核心主题
- 已确认的关键事实和决策
- 未解决的悬而未决的问题

格式：[早期历史] 一段连贯总结

对话：
{basic_summary}    ← 注意输入是基础摘要，不是原文
```

输出形如：
```text
[早期历史] 用户在咨询重疾险，已确认选择保额 50 万 + 免赔额 1 万方案，
对等待期还有疑问待解答。
```

两个 prompt 的关键差异：

| 维度 | 基础摘要 | 高度浓缩 |
|------|---------|---------|
| 粒度 | 逐轮（每轮 1-2 句） | 整段（1-2 句覆盖多轮） |
| 时序 | 保留 | 弱化 |
| 输入 | 原文 | 基础摘要（==不是原文==） |
| 输出长度 | ~500 token | ~100 token |
| Prompt 焦点 | 实体、数值、决策 | 主题、悬而未决问题 |

### 3.4 触发时机

| 触发条件 | 实现 | 适用 |
|---------|------|------|
| ==token 阈值==（推荐） | `total_tokens > 80%` 上下文时才压 | 通用，按需触发 |
| 轮次阈值 | 每 5/15 轮压一次 | 长对话场景，节奏稳定 |
| 时间阈值 | 用户停顿 > 30s 后台压缩 | 异步压缩场景 |

生产常见组合：token 阈值 80% 主触发 + 轮次阈值兜底。

### 3.5 完整可执行示例

```python
BASIC_PROMPT = """请逐轮总结以下消息，每条 1-2 句。
保留：实体、数值、决策、用户诉求、助手关键信息。
舍弃：寒暄、重复确认。
格式：[第 N 轮][user/assistant] 摘要内容

对话：
{messages}"""

HEAVY_PROMPT = """请用 1-2 句话总结以下早期对话主线，不要逐轮列。
保留：核心主题、已确认事实、未解决问题。
格式：[早期历史] 一段连贯总结

对话：
{basic_summary}"""

def hierarchical_summarize(messages, llm):
    if len(messages) <= 5:
        return messages

    recent = messages[-5:]
    middle = messages[-15:-5]
    early = messages[:-15]

    # Step 1：middle → 基础摘要
    middle_summary = llm.invoke(
        BASIC_PROMPT.format(messages=format_messages(middle)),
        max_tokens=500
    )

    if early:
        # Step 2a：early 先压成基础摘要
        early_basic = llm.invoke(
            BASIC_PROMPT.format(messages=format_messages(early)),
            max_tokens=300
        )
        # Step 2b：再从基础摘要压成高度浓缩
        early_heavy = llm.invoke(
            HEAVY_PROMPT.format(basic_summary=early_basic),
            max_tokens=150
        )
        result = [
            {"role": "system", "content": f"[早期历史] {early_heavy}"},
            {"role": "system", "content": f"[最近历史摘要]\n{middle_summary}"},
            *recent
        ]
    else:
        result = [
            {"role": "system", "content": f"[历史摘要]\n{middle_summary}"},
            *recent
        ]
    return result
```

### 3.6 摘要不可逆——生产必须留原文

摘要压完原文就丢了，万一摘要错了或丢关键信息，==恢复不了==。

解决方案：
1. 始终保留原文在持久化存储（PostgreSQL 全量历史），仅在送 LLM 的 prompt 中用摘要
2. 用户追问"刚才你说什么来着"时，绕过摘要从持久化层拉原文
3. 长期会话回顾时，基于原文重新生成更精确的摘要

摘要是 prompt 优化，不是数据替代。详见 [[多轮对话#一、对话存储设计]] 的双层架构（Redis 缓存 + PG 持久化）。

## 四、市面主流产品的实际做法

2025-2026 年，主流 AI 产品的对话压缩/记忆提取方案==高度趋同==——都是 LLM 调用 + 精心设计的 prompt + 一个工具。横向对比：

| 产品 | 触发时机 | 谁判断重要性 | 输出去向 |
|------|---------|------------|---------|
| ==Claude Code `/compact`== | context 接近上限 / 用户敲 `/compact` / 会话结束 | LLM（精心 prompt） | 替换原 messages 前部 + 最近几轮原文保留 |
| ==Claude Memory tool==（Anthropic） | LLM 在每轮 ReAct 循环中自主判断 | LLM（看 system prompt 引导） | 调 `save_memory` / `update_memory` / `delete_memory` 工具写入 |
| ==ChatGPT `bio()`== | 对话中 LLM 检测到"值得记的事实"自主调用 | LLM（看 system prompt 引导） | 调 `bio(content)` 工具写入用户 Memory |
| ==Cursor Memory== | 用户 `/save` 主动 / Composer 模式下 LLM 自动 | LLM（自主判断或用户指定） | 写入 Notepads |
| ==LangChain ConversationSummaryMemory== | 每 N 轮触发 | LLM（框架内置 summarize prompt） | 替换旧 messages |
| ==LangGraph Summarizer node== | 状态图中显式定义触发条件 | LLM（用户写 prompt） | 写入 State 字段 |

共同点（这就是行业事实标准）：

1. **判断重要性**：全部用 LLM——没有一个用规则或打标签
2. **交互方式**：给 LLM 一个工具（`save_memory` / `bio` / `compact`），让 LLM 自主决定何时调
3. **prompt 引导**：在 system prompt 里明确告诉 LLM"什么值得记 / 什么不值得"
4. **原始数据保留**：都不删原始对话日志（JSONL/SQLite），压缩只是 prompt 优化

Claude Code `/compact` 的 prompt 摘要（从开源逆向看）：

```text
"分析以下对话历史，生成精炼摘要。必须保留：
- 用户的具体目标和约束
- 已完成的关键步骤（文件路径/命令/decisions）
- 未完成的任务和下一步计划
- 出现的错误和解决方案
可以省略：
- 寒暄、确认词
- 重复的解释
- 已彻底完成且无关的子任务"
```

关键洞察：这些 prompt 99% 都长得很像——因为"什么值得记"这件事主流认知高度收敛。面试时能讲出几家产品的具体 prompt 套路（Claude Code "保留约束/已完成/下一步/错误" 四要素 + ChatGPT bio() 工具触发）比死记五种方案有深度。

## 五、推荐组合

与"市面主流产品的实际做法"对齐——LLM 主导，规则只做预过滤：

```text
预过滤(方案 3 否定式)：去掉明显寒暄/超短消息（零成本）
   ↓
触发(§3.4)：达到 80% 上下文阈值 / 用户主动 / 会话结束
   ↓
主体(方案 5 分层)：最近 5 轮原文 + 中段基础摘要 + 远段高度浓缩
   ↓
压缩(方案 1 LLM)：每段都用 LLM，prompt 明确列"必须保留 vs 可以舍弃"
   ↓
持久化：摘要写入 MEMORY.md / 当前 prompt；==原始 JSONL 永久保留==
```

没有方案 2 的入库标注——因为"谁来标"没有好答案，实际产品都是直接靠 prompt + LLM 即时判断。

## 六、常被忽略的细节

1. **==时效性衰减==**：相同信息越早出现的相对越不重要
   - "我是张三"（5 轮前）—— 仍重要
   - "嗯"（5 秒前）—— 立刻可丢

2. **==实体共指==**：摘要时把代词替换为实体
   - 原文："他说他要买"
   - 摘要："张三说要买重疾险"
   - 否则摘要本身丢失指代锚点

3. **==决策反转保留==**：用户改主意时两个版本都要保留
   - "我选 A" → 后来 "改成 B"
   - 摘要：用户最初选 A，后改为 B（不能只留 B，审计/解释需要原因）

4. **==失败/未解决标记==**：助手回答失败的内容要标注
   - "用户问 X，未找到相关信息" 要保留
   - 否则下一轮重复问还会再次失败
