## 一、Contextual Retrieval（Anthropic 2024）

### 解决的问题

单独的 chunk 失去了原文档的语境。比如：

```
原 chunk: "营收同比增长 40%"
```

LLM 看到这段==根本不知道==是哪家公司、哪个季度、哪个区域的数据。==向量化时这种不完整 chunk 与查询的匹配度会大幅降低==。

### 核心做法

向量化前用 LLM 看整篇文档，==为每个 chunk 生成 50-100 token 的上下文描述==，把"上下文 + 原 chunk"一起向量化：

```
原文档（整篇）→ LLM 一次性读完
   ↓ 为每个 chunk 单独生成上下文
chunk_1 增强: "[本段来自 ACME 公司 2023 Q4 财报，讨论北美区业务] 营收同比增长 40%"
chunk_2 增强: "[本段来自 ACME 公司 2023 Q4 财报，讨论员工成本] 人力成本上升 12%"
   ↓ 向量化（用增强后的内容）
带文档级上下文的向量
```

### 关键技术：Prompt Caching

直接的实现方式是==每个 chunk 都把整篇文档作为 prompt 给 LLM==，但成本爆炸——100 个 chunk 就要把同一份文档读 100 遍。

**Anthropic 的优化**：用 Prompt Caching 把整篇文档作为缓存的 system prompt：
- ==处理 N 个 chunk 只完整读文档 1 次==
- 后续 N-1 次只是命中缓存
- 成本可降低 ~80-90%

```python
# 伪代码
for chunk in chunks:
    response = llm.invoke(
        system_prompt=full_document,    # 缓存
        cache_control={"type": "ephemeral"},  # 标记可缓存
        user_prompt=f"为以下 chunk 生成 50-100 token 的上下文：\n{chunk}"
    )
    enhanced_chunk = response + " " + chunk
```

### 实测效果

Anthropic 公布的数据：
- 单独使用：检索失败率（top-20 未命中率）降低约 35%
- 配合 BM25 混合检索：失败率降低约 49%
- ==配合 Rerank==：失败率降低约 67%（最佳组合）

### 优劣

**优势：**
- 实施简单（只在离线阶段加一步）
- 与现有检索 pipeline 完全兼容
- ==与混合检索（向量 + BM25）叠加效果最佳==
- Prompt Caching 后成本可控

**劣势：**
- 离线索引时间显著增加（每个 chunk 要过一次 LLM）
- 文档很短时收益不明显
- 文档结构松散时（如对话/转写）上下文 LLM 难总结

### 适用场景

- ==结构化的长文档==：财报、合同、技术规范、医学文献
- 检索时关键词不在 chunk 内（"40%" 不知道是谁的）的场景
- 需要精确匹配的高价值检索

> 原文：https://www.anthropic.com/news/contextual-retrieval

## 二、Late Chunking（Jina AI 2024）

### 解决的问题

==传统流程是"先切再编码"==：

```
文档 → 切成 chunk_1, chunk_2, ... → 每个 chunk 独立 embedding 编码
```

每个 chunk 编码时==看不到其他 chunk 的内容==，导致：
- chunk 内的代词指代丢失（"他认为..." 不知道"他"是谁）
- 缩写没有展开依据
- 隐含主语/上下文缺失

### 核心做法

==反过来——"先编码再切"==：

```
整篇文档（完整序列）
   ↓ 用长上下文 Embedding 模型做 token 级编码
[token_1的向量, token_2的向量, ..., token_N的向量]
   ↓ 此时每个 token 的向量都已包含全文语境（双向 attention）
   ↓ 在向量层面按 chunk 边界做 mean pooling
chunk_1 向量 = mean(chunk_1 范围内的 token 向量)
chunk_2 向量 = mean(chunk_2 范围内的 token 向量)
```

### 为什么有效

==Transformer 的 self-attention 天然能让每个 token 看到全文==。Late Chunking 利用这一点：每个 token 编码时看到的是完整文档，==向量里已经包含跨 chunk 的语境信息==。

```
传统："这家公司的营收增长 40%" 单独编码 → 不知道是哪家公司
Late Chunking："这家公司的营收增长 40%" 这段位于"ACME 财报"全文中编码
            → "这家公司"的向量已经知道指代 ACME
```

### 工程要求

- ==Embedding 模型必须支持长上下文==：Jina-embeddings-v2 (8K)、BGE-M3 (8K)、Qwen3-Embedding (32K)
- 文档超过模型上下文时仍要分段，但可以用重叠窗口缓解

### 优劣

**优势：**
- ==零额外 LLM 调用==（与 Contextual Retrieval 相比）
- 解决跨 chunk 指代/缩写问题
- 实现简单（改造现有 embedding pipeline 即可）

**劣势：**
- 受限于 Embedding 模型的上下文窗口（不能处理任意长度文档）
- 多文档批量处理时==内存占用显著高于传统方案==（要把整篇文档载入 GPU 显存）

### 适用场景

- ==代词/指代密集的文档==：法律合同、政策文件、学术论文
- 跨 chunk 主题关联强的内容
- 已有长上下文 Embedding 模型的环境

> 原文与代码：https://jina.ai/news/late-chunking-in-long-context-embedding-models/

## 三、判别式分块（金山办公 2026）

### 论文与背景

《Toward General Semantic Chunking: A Discriminative Framework for Ultra-Long Documents》（arXiv:2602.23370）

### 解决的问题

现有语义分块的两条路都不理想：
1. ==生成式语义分块==：让 LLM 输出切分点 → 速度慢、不稳定（LLM 输出的切分点可能漂移）
2. ==Embedding 相似度==：阈值难调，不同文档语义跳跃幅度差异大

### 核心做法

把主题分割转化为==二分类问题==："相邻句子之间是否存在主题边界？"

### 五阶段架构

```
1. 句子级细粒度建模
   文档 → 用规则切句 → [句子1, 句子2, ..., 句子N]

2. token 级语义编码
   用 Qwen3-0.6B 作为骨干，对全文做 token 编码
   → 每个 token 拿到带全文语境的向量

3. 块级语义聚合
   用注意力池化把每个句子内的 token 向量聚合成一个句子向量
   [句子1向量, 句子2向量, ..., 句子N向量]

4. 跨块上下文建模
   轻量 Transformer 建模句子间依赖
   → 句子向量 + 上下文增强

5. 边界判别预测
   MLP 分类头逐对预测：句子i 和 句子i+1 之间是否主题边界
   → 输出每个边界的概率值
```

### 超长文档处理

==重叠滑动窗口策略==——超过模型上下文（13K token）的文档：
1. 拆分为多个窗口
2. 相邻窗口保留 10% 重叠区域
3. 重叠区的边界概率取多窗口预测均值

==支持任意长度文档==。

### 启发式后处理

- 按预设阈值做初始分块，生成高置信度的语义块
- ==超长块==递归选取块内边界概率最高位置二次拆分
- ==超短块==对比左右边界语义连续性，合并到更连贯的一侧

### VFSC：超长块检索适配

分块后产生的超长块（超出 Embedding 模型上下文窗口）==截断会丢语义==。论文提出 VFSC 方案：

- 用==一个融合向量 + 一个标量因子==
- 严格还原超长块所有子段的平均余弦相似度
- 存储复杂度从 O(N) 降至 O(1)
- ==零语义损失==

### 优劣

**优势：**
- 推理速度比生成式分块==显著快==
- 端到端训练，不需要调阈值
- 零语义损失的超长块检索方案
- 与下游 Embedding 模型解耦

**劣势：**
- 需要训练数据
- 需要部署独立的分块模型
- 中文/英文之外的语言可能效果差（训练语料决定）

### 适用场景

- ==超长无结构文档==：会议纪要、口语转写、长篇报告
- 需要稳定可控的分块质量
- 业务量大、生成式分块成本不可控

## 四、命题化切割（Dense X Retrieval, EMNLP 2024）

### 论文与背景

Chen et al., "Dense X Retrieval: What Retrieval Granularity Should We Use?", arXiv:2312.06648

### 解决的问题

段落或句子级 chunk 经常==包含多件事==：

```
"苹果公司在 2023 年第四季度营收增长 40%，主要受 iPhone 15 系列驱动。"
↓ 包含两个独立事实：
  1. 苹果 2023 Q4 营收增长 40%
  2. iPhone 15 是营收增长的主要驱动力
```

==检索时只想要其中一件==（如查"营收增长率"），但向量是混合语义，匹配度被稀释。

### 核心做法

用 LLM 把文档分解为==命题（Proposition）==——每个命题是==自包含的原子事实陈述==：

| 原文 | 命题化 |
|------|--------|
| "苹果 2023 Q4 营收增长 40%，主要受 iPhone 15 驱动" | 命题1: "苹果 2023 Q4 营收增长 40%" |
| | 命题2: "iPhone 15 是苹果 2023 Q4 营收增长的主要驱动力" |

每个命题==独立向量化==，存入向量库。检索时直接命中具体事实。

### 模型

HuggingFace 上有官方 propositioner 模型可直接调用：

```python
from transformers import AutoTokenizer, T5ForConditionalGeneration

model = T5ForConditionalGeneration.from_pretrained(
    "chentong00/propositionizer-wiki-flan-t5-large"
)
# 输入：标题 + 段落 → 输出：JSON 格式的命题列表
```

### 实测效果

论文实验中：
- 在 EntityQuestions 等 benchmark 上，==命题级 > 句子级 > 段落级==
- 召回质量显著超过传统粒度

### 优劣

**优势：**
- ==检索粒度最细==，命题之间互不干扰
- 答案精确度高
- 适合多事实复合段落

**劣势：**
- ==LLM 调用成本高==（每个段落都要调用 propositioner）
- 命题之间的篇章关联（因果、对比、时序）会丢失
- 需要在检索后通过 ID 关联回原段落补全语境（类似父子分块的思路）
- 短文档收益不明显

### 适用场景

- ==高价值精确问答==：财报数字、合同条款、医学事实
- 多事实复合段落（学术综述、分析报告）
- 检索精度要求 >> 召回率的场景

## 五、四种方法的横向对比

| 方法 | 操作时机 | 核心思路 | 计算成本 | 适用场景 |
|------|---------|---------|---------|---------|
| **Contextual Retrieval** | 切分后、向量化前 | LLM 为每个 chunk 注入文档级上下文 | 高（可被 Prompt Caching 摊薄） | 结构化长文档 + 精确匹配 |
| **Late Chunking** | 编码后再切 | 在向量层面切，让向量自带全文语境 | 低（仅 embedding） | 代词密集 + 长上下文 embedding |
| **判别式分块** | 切分时 | 二分类预测主题边界 | 中（独立分块模型） | 超长无结构文档 |
| **命题化切割** | 切分时 | LLM 分解为原子命题 | 高（每段调 LLM） | 多事实复合段落 |

### 组合使用

==这四种方法不互斥==，可以组合：

| 组合 | 效果 |
|------|------|
| 判别式分块 + Contextual Retrieval | 边界准 + 上下文全 |
| Late Chunking + 命题化切割 | 跨指代解决 + 粒度细 |
| 命题化切割 + Contextual Retrieval | 粒度细 + 文档级上下文 |

### 与基础方法的关系

这些前沿方法==都不是替代 L1-L4 基础方法==，而是在基础切分（决定边界）之上的增强：

| L1-L4 基础（决定边界） | 前沿增强（决定内容） |
|---------------------|------------------|
| L1 规则 / L2 结构 / L3 语义 / L4 命题 | Contextual Retrieval：内容增强 |
| | Late Chunking：编码方式优化 |
| | 判别式分块：是 L3 语义切分的具体方法 |
| | 命题化切割：是 L4 命题级的具体实现 |
