# 幻觉检测与置信度

> RAG 解决了"知识从哪来"，==但没解决"LLM 一定按知识答"==。本文讲三件事：怎么检测幻觉、怎么打置信度分、什么时候让 LLM 主动说"不知道"。
>
> 系统评估方法见 [[RAG评估]]，本文是其姊妹篇——评估告诉你"出问题了没"，本文讲"出问题怎么办"。

---

## 一、什么是 RAG 幻觉

==有 RAG 不等于没幻觉==——RAG 改变了幻觉的形态：

| 类型 | 没 RAG | 有 RAG |
|------|-------|--------|
| 频率 | 高（任何超出训练的事实都可能编） | 降低（有外部知识兜底） |
| 形态 | 编虚构的人物、事件 | ==编 context 里没说的细节==、==错引用==、==推理跳跃== |
| 检测难度 | 相对容易（明显违反常识） | ==更难==（看起来有据可依） |

### 1.1 RAG 幻觉的三种类型

#### 类型 1：事实幻觉（Factual Hallucination）

==context 里有 A，LLM 输出 B==。

```
context: "重疾险等待期通常是 90 天"
LLM 输出: "重疾险等待期是 180 天"   ← 事实编了
```

最常见、危害最大、==也最好检测==（直接对比 context 和 answer）。

#### 类型 2：归因幻觉（Attribution Hallucination）

==答案对，但引用源不对==。

```
context_1: "等待期 90 天" （来自《条款 A》）
context_2: 无关内容        （来自《条款 B》）
LLM 输出: "等待期 90 天，详见《条款 B》"   ← 答对了但引用错了
```

==引用错==在合规场景（法律/医疗）后果严重。

#### 类型 3：推理幻觉（Reasoning Hallucination）

==推理链中有跳跃，结论可能对可能错==。

```
context: "重疾险保额 50 万；重疾发生时按保额 100% 赔付"
用户问: "重疾发生赔多少？"
LLM 输出: "100 万"   ← 推理错（应该是 50 万）
```

==隐式跳跃==在多步推理时尤其危险。

## 二、幻觉检测方法

### 2.1 引用对齐（==最常用==）

==answer 中每个事实都必须能在 context 中找到来源==。

```python
def alignment_check(answer: str, contexts: list[str], llm) -> dict:
    # 1. 把 answer 拆成原子陈述
    claims = llm.invoke(f"""
请把以下答案拆分成独立的事实陈述（每条一行）。

答案：{answer}

事实陈述：
""").splitlines()

    # 2. 对每条 claim 判断 context 是否支撑
    results = []
    for claim in claims:
        verdict = llm.invoke(f"""
判断以下陈述是否能从给定的上下文中推出。

上下文：
{format_contexts(contexts)}

陈述：{claim}

判断（必须是 SUPPORTED / CONTRADICTED / NO_EVIDENCE）：
""").strip()
        results.append({"claim": claim, "verdict": verdict})

    # 3. 计算忠实度
    supported = sum(1 for r in results if r["verdict"] == "SUPPORTED")
    return {
        "faithfulness": supported / len(claims) if claims else 0,
        "details": results
    }
```

==阈值==：Faithfulness < 0.95 触发警告，< 0.8 直接拒绝输出。

### 2.2 NLI 模型核查（更便宜更快）

==不用 LLM，用专门的 NLI（Natural Language Inference）模型==判断"前提是否蕴含假设"：

```python
from transformers import pipeline

nli = pipeline("text-classification", model="MoritzLaurer/DeBERTa-v3-large-mnli")

def nli_check(claim: str, context: str) -> str:
    """判断 context 是否蕴含 claim"""
    result = nli(f"premise: {context}\nhypothesis: {claim}")
    # 输出: ENTAILMENT / NEUTRAL / CONTRADICTION
    return result[0]["label"]
```

==对比==：
- LLM-as-Judge：贵（每条 claim 一次 LLM 调用）、慢、但准
- NLI 模型：便宜（毫秒级）、快、稍弱
- 生产推荐：==NLI 做粗筛 + LLM 做难 case 复查==

### 2.3 自一致性（Self-Consistency）

==让 LLM 用不同采样温度生成 N 次答案，看一致性==：

```python
def self_consistency_check(query, contexts, llm, n=5):
    answers = [
        llm.invoke(prompt(query, contexts), temperature=0.7)
        for _ in range(n)
    ]

    # 比较 N 个答案的事实层面一致性
    consistency = pairwise_similarity(answers)
    return {
        "answer": majority_answer(answers),  # 多数派
        "confidence": consistency,
        "alternatives": answers
    }
```

==原理==：模型确定的事实在多次采样下稳定；不确定的事实会"飘"。
==代价==：N 倍 LLM 成本——通常 N=3，且只对高风险查询用。

### 2.4 LLM-as-Judge（最强但最贵）

==让另一个 LLM 当裁判==评估答案：

```python
JUDGE_PROMPT = """你是一个事实核查员。给定问题、上下文、答案，判断答案的真实性。

问题：{query}

上下文：
{contexts}

答案：{answer}

请按以下维度评分（0-10）：
1. Faithfulness（答案是否基于上下文，没有编造）
2. Completeness（答案是否完整覆盖问题）
3. Citation Accuracy（如果有引用，是否准确）

输出 JSON: {{"faithfulness": ..., "completeness": ..., "citation_accuracy": ..., "issues": ["..."]}}
"""
```

==实践==：用比生成模型==更强==的模型当裁判（如 GPT-4 评估 GPT-3.5 的输出）。

### 2.5 四种方法对比

| 方法 | 成本 | 速度 | 准确度 | 适用 |
|------|------|------|-------|------|
| 引用对齐 | 中 | 慢（多次 LLM） | 高 | ==生产主选== |
| NLI 模型 | 极低 | 极快 | 中 | 高 QPS / 实时拦截 |
| 自一致性 | 高（N 倍） | 慢 | 较高 | 高风险查询、医疗/法律 |
| LLM-as-Judge | 极高 | 慢 | 最高 | 离线评估、定标准 |

==生产推荐==：==NLI 实时拦截 + 引用对齐周期校验 + LLM-as-Judge 离线评估==。

## 三、置信度评分

==幻觉检测是事后==——置信度是事前/事中。

### 3.1 三层置信度

#### Token-level（logprobs）

LLM 生成时每个 token 都有概率——==低概率 token 通常对应不确定==。

```python
def token_confidence(response):
    """从 LLM 响应的 logprobs 提取置信度"""
    logprobs = response.choices[0].logprobs.content
    avg_logprob = sum(t.logprob for t in logprobs) / len(logprobs)
    confidence = math.exp(avg_logprob)  # 几何平均概率
    return confidence
```

==局限==：高概率 ≠ 正确（模型可能"自信地编"）。

#### Chunk-level（检索 + Rerank 分数）

==检索本身有信号告诉你"是否有把握"==：

```python
def retrieval_confidence(retrieved_chunks):
    top_score = retrieved_chunks[0].score          # 最高分
    score_gap = (
        retrieved_chunks[0].score - retrieved_chunks[1].score
    )  # 第一名领先程度

    if top_score > 0.85 and score_gap > 0.1:
        return "high"     # 第一名遥遥领先
    elif top_score > 0.7:
        return "medium"   # 找到了相关的
    else:
        return "low"      # 检索本身就模糊
```

==这是免费==的信号，应该==默认就用==。

#### Answer-level（综合）

```python
def answer_confidence(query, retrieval_score, faithfulness, token_conf):
    # 加权组合
    return (
        0.4 * retrieval_score +    # 检索质量
        0.4 * faithfulness +        # 事实对齐
        0.2 * token_conf            # 模型确定性
    )
```

### 3.2 不确定性量化（Uncertainty Quantification）

更进阶：==用统计方法量化不确定性==。

#### 方法 A：贝叶斯推断

```
对答案的事实做概率建模：
P(answer correct | context, query) = ?

实施：
- 用多次采样近似分布
- 统计置信区间：均值 ± 1.96 × 标准差
```

#### 方法 B：Calibration（校准）

LLM 输出的"95% 确定"是真的 95% 吗？==通常不是==——通常 over-confident。

```python
# 收集历史数据：模型说 95% 确定时，实际正确率是多少？
calibration_table = {
    "0.9-0.95": 0.78,    # 模型说 90-95% 时实际只对 78%
    "0.95-1.0": 0.86,    # 还是 over-confident
    ...
}

# 上线后对模型置信度做校准
def calibrated_confidence(raw_conf):
    return calibration_table.lookup(raw_conf)
```

## 四、Verify-Then-Answer 流水线

不要==直接==让 LLM 答：

```
传统:    query + context → LLM → answer
改进:    query + context → LLM → answer
                                    ↓
                                 verify
                                    ↓
                          问题？ → regenerate
                          没问题 → 返回 + 置信度
```

### 4.1 实施

```python
def verify_then_answer(query, contexts, llm, max_retries=2):
    for attempt in range(max_retries + 1):
        answer = llm.generate(query, contexts)

        # 检查 1：引用对齐
        alignment = alignment_check(answer, contexts, llm)

        # 检查 2：与 query 相关性
        relevancy = check_relevancy(query, answer)

        if alignment["faithfulness"] >= 0.95 and relevancy >= 0.8:
            return {
                "answer": answer,
                "confidence": "high",
                "verified": True
            }

        # 失败 → 反思后重试
        feedback = format_issues(alignment, relevancy)
        contexts = enrich_or_change_contexts(contexts, feedback)

    # 重试都失败 → 降级
    return {
        "answer": "抱歉，我无法基于现有资料给出可靠回答",
        "confidence": "low",
        "verified": False
    }
```

### 4.2 Anthropic Constitutional AI 思路

==给 LLM 一套"宪法"==让它自己审查输出：

```
SYSTEM: 你是一个 RAG 助手。在输出答案前必须自检：
1. 我说的每个事实，context 中是否有支撑？
2. 我有没有编造 context 中不存在的细节？
3. 我的引用是否真实指向正确文档？
4. 如果存疑，我应该说"不确定"而非编造。
```

==成本极低（仅 prompt 增加），效果显著==——是个性价比高的入门方案。

## 五、拒绝回答策略

==让 LLM 学会说"不知道"比让它"看起来什么都懂"更重要==。

### 5.1 何时该拒绝

| 信号 | 应该拒绝 |
|------|---------|
| 检索 0 命中 | ✅ "知识库中未找到相关信息" |
| 最高分 < 0.5 | ✅ 检索质量太低 |
| Faithfulness < 0.7 | ✅ 答案与 context 偏差大 |
| 检索结果多个权威源冲突 | ✅ 让用户判断或人工介入 |
| query 触发敏感词（医疗诊断、法律建议） | ✅ 推到合规渠道 |

### 5.2 怎么让 LLM 拒绝

==Prompt 写法决定一切==：

```
❌ 错误 prompt:
"基于以下文档回答用户问题：{contexts}
问题：{query}"
→ LLM 倾向于硬答（即使 context 不充分）

✅ 正确 prompt:
"基于以下文档回答用户问题：{contexts}
问题：{query}

==重要==：
- 如果文档中没有足够信息，必须明确回答"知识库中未找到相关信息"
- 不要从训练数据推测，不要编造
- 如果 context 与问题不相关，也要拒绝"
→ LLM 会学会拒绝
```

### 5.3 用户体验设计

==拒绝答案不能冷冰冰==：

```
❌ 差: "无相关信息。"

✅ 好:
   "我在知识库中未找到关于 [关键词] 的明确信息。
    可能的原因：
    - 该问题超出了当前文档范围
    - 您的问题可以更具体一点

    建议：
    [转人工客服] [尝试改写问题]"
```

==包括==：明确原因 + 给出下一步行动 + 避免让用户困惑。

## 六、Citation / 引用溯源

### 6.1 强制引用机制

```python
CITE_PROMPT = """基于以下文档回答问题。==每个事实都必须标注引用==。

文档（每条有 [doc_N] 标识）：
[doc_1] 重疾险等待期通常是 90 天...
[doc_2] 理赔流程：1. 报案 ...

问题：重疾险等待期多久，怎么理赔？

格式要求：
- 答案中每个具体事实后必须带 [doc_N] 标记
- 不能从无 [doc_N] 标记的内容中说事实
"""

# 输出:
# "重疾险等待期通常是 90 天 [doc_1]。理赔流程包括报案 [doc_2]..."
```

### 6.2 引用质量评估

```python
def citation_quality(answer: str, doc_map: dict) -> dict:
    citations = extract_citations(answer)        # 解析 [doc_N]
    metrics = {
        "coverage": 0,        # 多少事实有引用
        "accuracy": 0,        # 引用是否真的支撑该事实
        "specificity": 0      # 引用是否精确（不是大段引）
    }
    # ... 评估逻辑
    return metrics
```

==生产标准==：coverage > 0.9，accuracy > 0.95。

### 6.3 端到端 Traceability

```
用户答案
  ├─ 引用 [doc_115]
  │   └─ chunk_887 in doc_115
  │       └─ 原始文件路径 / 页码 / 段落
  └─ 引用 [doc_212]
      └─ chunk_445 in doc_212
          └─ ...
```

==每个引用都能溯源到原始文件==——这是合规场景的硬要求。

## 七、生产实施推荐组合

==不要一次上所有==，按风险分级：

### 低风险场景（通用问答、客服）

```
基础 prompt（强调"不要编"）+ 检索分数判断 + 引用强制
```

成本最低，覆盖 80% 场景。

### 中风险场景（金融客服、技术支持）

```
低风险方案 + NLI 模型实时拦截 + Verify-Then-Answer
```

加约 30% 延迟，质量显著提升。

### 高风险场景（医疗、法律、合规）

```
中风险方案 + LLM-as-Judge 二次确认 + 自一致性多采样 + 强制人工 review
```

延迟 5-10s，但==几乎杜绝幻觉==。

### 实测效果（参考）

| 方案 | Faithfulness | 延迟增加 | 成本增加 |
|------|-------------|---------|---------|
| 仅 RAG | 0.78 | 基线 | 基线 |
| + 强制引用 | 0.85 | +5% | +3% |
| + NLI 拦截 | 0.91 | +20% | +5% |
| + Verify-Then-Answer | 0.96 | +80% | +50% |
| + 自一致性 N=3 | 0.98 | +200% | +200% |

==加 NLI 拦截是性价比最高的一步==。
