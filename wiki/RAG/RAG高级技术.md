# RAG 高级技术

> 本文覆盖超越标准 RAG 管道的高级模式：智能体 RAG、自反思 RAG、知识图谱增强、优化策略分类、性能工程与未来趋势。基础管道见 [[RAG基础与架构]]，检索策略见 [[RAG检索策略]]，向量与 Embedding 见 [[RAG向量与Embedding]]。

## 一、Agentic RAG（智能体 RAG）

### 核心思路

给 LLM Agent 提供多个检索工具（语义搜索、完整文档检索、SQL 查询），让它根据查询自主选择用哪个。

**解决的问题：** 不同问题需要不同检索策略——有些问题适合语义搜索，有些需要完整文档，有些需要查结构化数据库。一个固定的检索策略无法覆盖所有场景。

**示例流程：**
```
用户："完整的退款政策是什么？"
Agent 推理：
1. 调用 search_knowledge_base("退款政策") → 找到提及 refund_policy.pdf 的分块
2. 意识到分块不够完整 → 调用 retrieve_full_document("退款政策")
3. 拿到完整文档 → 生成全面答案
```

**适用场景：** 异构数据源（文档 + 数据库 + API）且查询复杂度差异大时。

> [!warning] Agentic RAG 的陷阱
> Agent 自主选择检索策略意味着行为不可预测。生产环境必须加护栏：限制最大检索轮数（防止死循环）、每步工具调用计费和超时控制、对 Agent 的检索决策做日志追踪。不要在没有可观测性的情况下上 Agentic RAG。

### BookRAG：结构感知的动态检索

传统 Agentic RAG 的 Agent 在"无结构的向量库"里做决策，BookRAG（港中文）提出了更精准的方式：先把文档还原成层级树 + 知识图谱，然后 Agent 根据查询类型动态选择检索策略：

- **单跳查询**：通过实体/关键词直接定位到对应章节子树，只搜这一小块
- **多跳查询**：先拆子问题，分别定位到不同章节子树检索，再利用章节层级关系合并跨块证据
- **全局聚合查询**：按页面范围、章节层级、模态类型（表格/图片）过滤整棵结构树，快速统计

核心优势是"导航式精搜"替代"盲搜全文"——检索范围从全文几万块缩到某一小节几十块，Token 消耗降低一个数量级。

局限：子章节内超长内容仍需分块，top-k 固定，未能根据查询意图动态调整检索粒度。

> 论文：https://arxiv.org/pdf/2512.03413

## 二、多文档并行：SPD-RAG

### 核心洞察

单智能体 Agentic RAG 面对需要综合十几份文档的问题时会崩：全局检索的跨文档噪声不断带偏检索方向，top-K 必然漏掉关键证据。

**SPD-RAG（Sub-Agent Per Document RAG）** 的核心洞察：多文档问答的瓶颈不是 top-K 多大或上下文多长，而是每份文档是否被充分、独立、深度挖掘。

### 三层架构

1. **协调层**：把用户复杂问题拆解成原子化的信息提取任务清单，广播给所有子 Agent。协调器不需要知道文档内容，只负责拆解
2. **并行检索层**：每份文档一个子 Agent，检索范围严格限制在自己的文档内（避免跨文档噪声），最多 5 轮迭代检索-推理，输出文档专属的发现报告 + 置信度分数。所有子 Agent 完全并行执行
3. **合成层**：对所有报告做向量嵌入 → 余弦相似度聚类 → 递归 map-reduce 合并，最终生成答案

SPD-RAG 用全上下文基线约 38% 的成本达到约 85% 的性能，成本效率是全上下文基线的 2.25 倍。学术论文场景最明显：传统 RAG 几乎无法处理，SPD-RAG 表现显著提升。

> 论文：https://arxiv.org/pdf/2603.08329

## 三、自反思 RAG（Self-RAG）

### 基本思路

检索后用 LLM 评估结果相关性，分数低则优化 query 重新搜索，最多迭代 2-3 次。

```
迭代 1：query="部署" → 评分 2/5（太模糊）
迭代 2：优化为 "机器学习模型部署到生产环境" → 评分 4/5（结果好，返回）
```

适用场景：答案准确性极重要、延迟可接受的场景（研究应用、复杂查询）。缺点：最高延迟（2-3 次 LLM 调用）、最贵的策略。

### Reflection Tokens 机制

Self-RAG（华盛顿大学 + IBM）通过训练 LLM 生成特殊的 Reflection Tokens 实现自主检索决策和质量评估：

| Token | 含义 | 可能值 |
|-------|------|--------|
| `[Retrieve]` | 是否需要检索 | Yes / No |
| `[IsRel]` | 检索内容是否相关 | Relevant / Irrelevant |
| `[IsSup]` | 生成内容是否有文档支撑 | Fully Supported / Partially / No Support |
| `[IsUse]` | 答案整体是否有用 | 1-5 分 |

推理时 LLM 在生成过程中预测这些 token，形成"检索 → 生成 → 评估 → 优化"的闭环。如果 `[IsRel]` 判定为 Irrelevant，自动重新检索；如果 `[IsSup]` 判定为 No Support，重新生成或添加低置信度警告。

7B 参数的 Self-RAG 在 TriviaQA 上超越 13B Llama2-chat + RAG，说明自反思机制比单纯增大模型更有效。

### 真实案例：金融科技公司三阶段演进

| 阶段 | 方案 | 做法 |
|------|------|------|
| 第一阶段 | Router Agent | 根据查询类型路由到不同知识源 |
| 第二阶段 | Multi-Agent | 每个主题域专属 Agent + Meta-Agent 综合 |
| 第三阶段 | Self-RAG | 自动评估相关性和支撑性，低置信度重新检索 |

关键收获：分阶段演进（不要一步到位）、Haiku 做路由 + Sonnet 做生成（节省 60% 成本）、基于真实用户查询日志优化。

### RAG-Gym：过程监督与策略优化

RAG-Gym（arXiv:2502.13957）提出系统化优化 Agentic RAG 的框架，核心创新是对检索过程的每一步进行细粒度监督，而非仅评估最终答案。

#### ReSearch Agent 架构

ReSearch（Reasoning-then-Search）的核心是"先推理再检索"：

```
构造初步答案 → 反思找不确定点 → 针对不确定点检索 → 整合新信息 → 再次反思 → ...
```

与传统 ReAct 的区别：ReAct 每步都检索，ReSearch 只在反思发现不确定点时才检索。==检索质量 > 检索数量==——盲目增加检索次数反而因噪声累积降低准确率。

#### 三维优化框架

| 维度 | 方法 | 效果（HotpotQA F1） |
|------|------|---------------------|
| Prompt Engineering | 反思 Prompt（零成本） | 41.09% → 44.91%（+3.8%） |
| Actor Tuning | DPO 微调检索策略（Llama-3.1-8B） | → 55.22%（+10.3%） |
| Critic Training | 训练 Critic 模型选最优检索动作 | → 60%（+4.8%） |

三层叠加效果 1+1+1 > 3，总提升近 19 个百分点。

#### 过程监督 vs 结果监督

- **结果监督**：只看最终答案对不对，所有中间步骤共享同一个奖励信号 → 信号稀疏，好坏步骤无法区分
- **过程监督**：给每个中间检索步骤独立打分 → 只奖励关键路径步骤，惩罚无效检索

过程监督的数据收集：记录 (query, 候选检索动作, 实际执行动作, 对最终答案的贡献度)，用于训练 Critic。

#### 落地三步骤

1. **加反思 Prompt**（成本最低）：在检索前加入"当前答案中哪些部分不确定？"的反思步骤
2. **收集过程数据训练 Critic**：记录多轮检索的决策轨迹，Critic 从候选查询中选最优
3. **DPO 微调 Actor**：1000 条标注数据即可见效，教会模型区分"好的检索动作"和"差的检索动作"

> [!tip] 评估集构建
> 公开 benchmark（HotpotQA）和实际业务场景可能完全不同。必须建立 100+ 条贴合业务的测试集，覆盖：高频问题、长尾问题、易混淆问题、多跳推理问题。

## 四、知识图谱增强检索

### 为什么需要知识图谱

向量搜索找到"语义相似"的文本，但错过实体间的明确关系——如 "CEO of"、"located in"、"reported revenue"。

**对比示例：**
```
查询："谁运营 ACME 公司，第二季度发生了什么变化？"

纯向量搜索：返回两段相似文本，但无法直接回答

知识图谱：
ACME 公司
  ├─ HAS_CEO → Jane Smith
  ├─ REPORTED_REVENUE → $314M
  │   └─ PERIOD → Q2 2024
  └─ LOCATED_IN → California
→ 直接答："Jane Smith 运营 ACME 公司，Q2 收入增至 3.14 亿美元"
```

### 主流方案

**Graphiti（开源）：** 自动从文本提取实体和关系，混合"语义相似 + BM25 + 图遍历 + 时间上下文"四种检索方式。

> 项目地址：https://github.com/graphiti-ai/graphiti

**Neo4j + 自建提取流程：** 手动定义实体类型，用 LLM 或 NER 模型抽取。

### Graphusion：知识图谱自动构建

Graphusion（WWW 2025）提出三步走方案：
1. **种子实体生成**：主题建模确定骨架
2. **候选三元组抽取**：检索 + LLM 提取实体关系
3. **知识图谱融合**：实体合并、冲突消解、跨子图补边

核心发现：去掉融合步骤后关系质量明显下降，"融合"才是从碎片化子图到全局可推理图谱的关键。另外 RAG 不一定总带来提升——长而噪的文档可能拖累模型，高质量短摘要更友好。

### 适用场景判断

- 实体间关系至关重要的领域（医疗网络、财务系统、研究数据库）→ 值得引入
- 简单事实问答、文档内容自包含 → 向量检索足够，不需要图谱
- 缺点：需要 Neo4j 等图数据库基础设施，设置和维护复杂，比纯向量慢且贵

## 五、优化策略分类与组合

### 三大类优化策略

#### 摄取策略（Ingestion Strategies）

**1. 上下文感知分块（Context-Aware Chunking）**
- 解决问题：固定大小分块会在语义中间切断，丢失上下文
- 实现方式：按语义边界分割，保持相关内容在一起
- 工具：Docling HybridChunker（https://github.com/DS4SD/docling）
- 详细技术见 [[分块策略]]

**2. 上下文检索（Contextual Retrieval）**
- 解决问题：分块缺乏文档级上下文（如"收入增长 40%"不知道是哪个公司、哪个季度）
- 实现方式：为每个分块添加 1-2 句话解释其与整个文档的关系

**Contextual Retrieval 实现步骤（Anthropic 提出）：**
1. 文档分块
2. 为每个 chunk 用 LLM 生成上下文描述（50-100 tokens）
3. 将"上下文 + 原始 chunk"一起向量化
4. 存入向量数据库

**Prompt 模板：**
```
<document>
{{整个文档内容}}
</document>

这是文档的一个片段：
<chunk>
{{当前 chunk}}
</chunk>

请用简洁的语言（50-100 tokens）描述这个片段的上下文，
说明它来自哪个文档、讨论什么主题。
只输出上下文描述，不要解释。
```

**三种组合方案：**
- Contextual Embeddings：向量检索 + 上下文增强，适合语义匹配
- Contextual BM25：关键词检索 + 上下文增强，适合精确匹配（如错误代码 "TS-999"）
- 混合检索（推荐）：两者结合 + RRF 融合

**成本优化：** 使用 Prompt Caching 缓存文档内容。同一文档的所有 chunk 请求复用缓存（文档内容作为 system prompt 的一部分被缓存），100 个 chunk 实际只完整处理 1 次文档，成本降低 80-90%。结合 BM25 混合检索后，检索失败率（Top-20 未命中率）降低约 49%，两者叠加降低 67%。

**适用场景：** 结构化文档（财报、法律、技术文档）且查询需要精确匹配。不适用于文档本身很短（< 500 tokens）或成本敏感且文档量巨大的场景。

> 参考：https://www.anthropic.com/news/contextual-retrieval

**3. 知识图谱增强（Knowledge Graph Enhancement）**
- 解决问题：向量检索难以捕捉实体间的复杂关系
- 实现方式：提取实体和关系，构建知识图谱
- 详见本文 §四

#### 查询策略（Query Strategies）

- **查询扩展**：生成多个查询变体（同义词、不同角度）
- **HyDE**：让 LLM 生成假设答案，用假设答案的 embedding 检索
- **混合搜索**：结合向量检索（语义）+ BM25（关键词）
- **重排序**：使用专门的重排序模型对结果重新打分

详细实现见 [[RAG检索策略#二、检索优化四层框架]]

#### 后处理策略（Post-Processing Strategies）

- **上下文压缩**：移除检索内容中的无关信息，保留关键内容
- **RAG Fusion**：使用 RRF 合并多个查询的结果
- **自适应检索**：根据查询复杂度动态调整检索数量（top_k）
- **迭代检索**：多轮检索，每轮基于上一轮结果优化查询

### 策略组合方案

**方案 1：通用生产堆栈**
- 策略：上下文感知分块 + 重排序 + 查询扩展 + 智能体 RAG
- 适用：通用生产环境、客户支持、内部知识库
- 特点：平衡效果和成本

**方案 2：高准确率堆栈**
- 策略：上下文检索 + 多查询 + 重排序 + 自反思
- 适用：医疗、法律、财务等错误成本高的场景
- 特点：追求准确率，可接受更高延迟和成本

**方案 3：领域专家堆栈**
- 策略：微调嵌入 + 上下文检索 + 知识图谱 + 重排序
- 适用：医疗、法律、财务、技术等专业领域
- 特点：针对特定领域深度优化

### 渐进式实施路径

不要一次性上所有策略——系统会过于复杂、难以调试、成本失控。

**阶段 1（第 1 周）：** 上下文感知分块 + 基础向量搜索 + 建立评估数据集

**阶段 2（第 2-3 周）：** 加重排序（ROI 高）+ 加查询扩展（处理模糊查询）

**阶段 3（第 4-6 周）：** 加多查询 RAG 或智能体 RAG + 关键查询启用自反思

**阶段 4（第 2 个月以后）：** 高价值文档加上下文检索 + 引入知识图谱 + 微调 Embedding

### 常见错误

| 错误 | 问题 | 解决方案 |
|------|------|---------|
| 一次用所有策略 | 系统过于复杂、难调试 | 从基础堆栈开始，按需添加 |
| 不测量基准 | 无法证明改进 | 建立评估数据集，每次改动前后测量 |
| 固定分块大小 | 破坏语义连贯性 | 始终用上下文感知分块 |
| 忽略重排序 | 向量相似 ≠ 语义相关 | 重排序 ROI 高，应尽早实施 |
| 没有查询预处理 | 用户查询模糊导致检索失败 | 至少加查询扩展 |
| 单一检索策略 | 不同查询需要不同策略 | 用智能体 RAG 提供灵活性 |

### 知识冲突处理

当多个检索结果包含矛盾信息时（如不同版本的文档、不同来源的数据），需要冲突解决策略：

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 置信度排序 | 按文档权威性/来源可靠度排序，优先采信高置信源 | 有明确权威层级的知识库 |
| 时效性优先 | 优先采信最新版本的文档 | 政策、法规、技术文档等频繁更新的内容 |
| 多源验证 | 多个独立来源一致时采信，不一致时标注分歧 | 事实性查询 |
| 不确定性表达 | 在回答中明确标注"存在不同说法"并列出各方观点 | 无法判断哪个正确时 |
| 冲突检测 | 入库时自动检测新文档与已有知识的矛盾，人工审核 | 知识库维护阶段 |

工程实现：在 chunk 元数据中维护 `source_authority`（来源权威等级）、`updated_at`（更新时间）、`version`（版本号）字段，Rerank 阶段结合这些元数据调整排序。

## 六、性能优化与工程实践

### 百万级文档的 RAG 系统设计

**存储层：**
- MySQL 存文档元数据（文件名、路径、用户、分类），不存向量，百万级文档也就几 GB
- ES 存向量和关键词，HNSW 索引，冷热分层（热数据 SSD，冷数据 HDD）
- 文件本体存 MinIO 对象存储，天然支持多机共享访问

**检索层优化：**
- 先元数据过滤再向量搜索，缩小候选集
- Redis 缓存热门查询，命中率 40%+
- 多阶段级联检索：BM25 快速筛选 50 个候选 → 向量检索 + 重排精排

**异步处理流水线：**
- 文档上传立刻返回成功，后台 Kafka 异步消化
- 文档解析 → 向量化 → 入库全流程异步化
- 单日处理能力 5000+ 文档

**水平扩展关键：状态外置**
- 文件存 MinIO，任务存 Kafka，Session 存 Redis，数据存 MySQL/ES
- 检索服务、生成服务天然无状态，前面加负载均衡直接扩
- WebSocket 多机部署：粘性会话（Nginx ip_hash）或 Redis Pub/Sub 消息广播

### 增量索引

动态知识库不能每次新增文档都重建全量索引，成本太高。

**增量索引策略：**
- 仅对新增、修改、删除的文档进行索引更新
- ES 支持实时插入新文档并更新向量索引，延迟低于 1 秒
- 定期评估检索效果（人工反馈或点击率），自动调整 BM25 与向量检索的权重比例

**查询自适应权重：**
- 事实类、关键词类查询：提升稀疏检索权重
- 概念类、解释类查询：提升密集检索权重
- 实现方式：规则判断（含专有名词/编号 → 提升 BM25）或训练查询分类模型

## 七、多模态 RAG

纯文本 RAG 的视觉盲区：图片直接丢弃或 OCR 乱码，表格结构被破坏。多模态 RAG 将图片、表格、PDF 等非文本内容纳入检索-生成流程。

### 三种方案对比

| 方案 | 原理 | 精度 | 成本 | 适用场景 |
|------|------|------|------|---------|
| CLIP 多模态嵌入 | 图文映射到同一向量空间，直接向量检索 | ~60% | 低 | 图像相似搜索、粗筛 |
| Caption/VLM 图生文 | VLM 先把图片翻译成文字描述，再走文本检索 | ~90% | 中-高 | 需要理解图片细节的场景 |
| ColPali 视觉检索 | 整页 PDF 当图片索引，零 OCR | 最高 | 中（需 GPU） | 视觉丰富的文档检索 |

### ColPali 架构（arXiv:2407.01449）

ColPali 基于 PaliGemma（Google）视觉语言模型 + ColBERT 的 Late Interaction 机制：

```
PDF 页面 → 渲染为图片 → 视觉 Transformer 编码 → 每个 Patch 生成一个向量
Query → 文本编码 → 每个 Token 生成一个向量
匹配 → Token 向量与 Patch 向量逐一 MaxSim → 聚合得分
```

核心优势：完全跳过 OCR/布局解析流水线，端到端可训练。在 ViDoRe benchmark 上大幅超越传统 OCR+文本检索管道。Vespa 和 Milvus 已开始原生支持 ColPali 索引。

### MultiVectorRetriever 模式

核心设计原则：==摘要用于检索，原始内容用于回答==。

```
┌─────────────────────────────────────────┐
│  Vector Store: 存储图片/表格的文字摘要    │ ← 检索时匹配
│  DocStore: 存储原始图片路径/base64       │ ← 命中后取原件给 LLM
└─────────────────────────────────────────┘
```

LangChain 的 `MultiVectorRetriever` 实现此模式：摘要（Caption）入向量库做检索索引，原始图片路径存 docStore，命中后将原图传给多模态 LLM 生成答案。

### 表格处理

表格必须与正文分离，不能混在同一个 chunk 里入库：

- **结构化 PDF 表格**：用 `unstructured` 的 `hi_res` 模式提取 HTML 表格，保留行列结构
- **扫描版表格**：截图直接送 VLM 提取 Markdown 格式
- **检索策略**：表格适合 BM25 关键词检索（精确数值匹配），图片适合 Caption 语义检索，长文档适合混合检索

### 六个生产踩坑

1. **base64 图片存向量库元数据** → 应存路径/URL，元数据膨胀会拖慢检索
2. **Caption 质量太差** → Caption 必须包含：图片类型、关键数值、趋势、结论
3. **图片分辨率压缩过度** → 合理下采样到 1024×768，过度压缩丢失细节
4. **扫描版 PDF 直接用文本解析** → 需 OCR 或 ColPali，文本解析只能拿到乱码
5. **三种内容用同一检索策略** → 表格用 BM25、图片用 Caption 语义、长文档用混合
6. **Caption 批量生成没限速** → 每批 10-20 张 + 重试机制，避免 API 限流

## 八、未来发展趋势

**1. 长上下文 + RAG 融合**
随着 LLM 上下文窗口扩大（Gemini 2.0 Flash 已达 100 万 token），RAG 的角色将转变：短上下文场景继续用 RAG 检索；长上下文场景把整个知识库放入上下文，RAG 变成"智能导航"——先粗筛 Top-50 文档，再全部放入长上下文精读。

**2. 端到端神经检索**
传统 RAG 依赖独立的 Embedding 模型，未来会出现端到端训练的检索-生成模型：RA-DIT（同时优化检索器和生成器）、DSI（把整个知识库编码到模型参数中）。

**3. 个性化与隐私保护**
企业 RAG 将更关注：基于用户历史和偏好优化检索结果、联邦学习和差分隐私保护数据安全、细粒度权限控制（不同用户看到不同检索结果）。
