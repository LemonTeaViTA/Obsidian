# 分块策略（Chunking）

> 切分不只是"设置一下 chunk_size"，它是需要分析文档特点、设计处理策略的工程模块。==切错了，后面做再多优化，全是在地基不稳的楼上加砖==。
>
> 本文是==基础与方法学==——前沿 SOTA 方法（Contextual Retrieval / Late Chunking / 判别式分块 / 命题化切割）的算法细节、论文出处、实测效果详见 [[分块策略前沿]]。

---

## 一、为什么分块是 RAG 的最大杠杆

分块决定了向量化的粒度，进而决定了检索能匹配的最小语义单位。==同一份文档、同一个 Embedding 模型、同一套检索策略，仅仅切分方式不同，召回质量可能天差地别==。

==分块设计要同时面对三对矛盾==：

| 矛盾 | 一端 | 另一端 |
|------|------|--------|
| ==检索精度 vs 上下文完整== | chunk 越小，向量越精准（语义单一）| chunk 越大，给 LLM 的上下文越完整 |
| ==粒度细 vs 成本== | 粒度越细（如命题级），匹配越准 | 切得越细，存储 / 向量化 / LLM 调用成本越高 |
| ==通用 vs 文档特化== | 一刀切的规则切分实现简单 | 结构感知/语义切分需要针对文档类型定制 |

所有分块策略本质上都在这三对矛盾里寻找平衡点。==§二的四种方法学就是按"如何决定 chunk 边界"逐步深入==。

---

## 二、切分边界的四种方法学

按"决定 chunk 边界的方式"，分块方法分为四层，==每一层比上一层更智能但代价也更高==：

```
L1 规则切分     → 按字符/token 数或分隔符切（最简单）
L2 结构感知     → 利用文档原生结构（标题层级、表格、列表）
L3 语义切分     → 模型判断主题边界
L4 命题级切分   → LLM 分解为原子事实
```

==L1-L4 在同一维度==——都是决定 chunk 边界画在哪。==与之正交==的还有两类策略，下文 §三 单独讲：
- ==chunk 内容增强==（Contextual Retrieval、Late Chunking）：不改边界，丰富每个 chunk 的语义表达
- ==在线检索增强==（父子分块、句子窗口检索）：检索阶段动态扩展上下文，详见 [[RAG检索策略#五、检索时的上下文扩展]]

### 2.1 L1 规则切分（Rule-based）

**做法：** 按固定长度（chunk_size）或固定分隔符（句号/换行/双换行）切分。最常见实现：LangChain 的 `RecursiveCharacterTextSplitter`。

**Overlap 是 L1 的标配：** 相邻 chunk 保留 10-20% 重叠区域，避免关键信息恰好落在边界处被截断。==进阶做法是基于句子边界的智能 overlap==——固定 overlap 可能切在句子中间，智能 overlap 在确定重叠区域后向后扫描到最近句子结尾才截断。

==优势==：实现简单、速度快、零成本、可预测。

==劣势==：不理解语义和结构，常在句子中间截断，表格被打散，列表项与其前导句分离。

==适用==：同质化短文档（如 FAQ、新闻段落）的 baseline 方案，且==所有上层方案都建议叠加 Overlap 兜底==。

### 2.2 L2 结构感知切分（Structure-aware）

**核心思路：** 利用文档的原生结构信号，让 chunk 边界与文档逻辑边界对齐。

#### 四类通用做法

==1. 层级结构识别==

通过多策略融合识别章节标题——数字编号（`1.1`、`1.1.1`）、中文编号（`第X章/节/条`、`（X）`）、混合编号（`a)`、`i.`）、Markdown 标题（`#`/`##`）、HTML 标签（`<h1>`-`<h6>`）。==切分原则==：**同一章节内容尽量放在同一个 chunk，跨章节绝不合并**。

==2. 超长章节递归细切==

章节超过 chunk_size 上限时，按层级降级切分：先按子小节边界切，没有子小节则按段落，最后按句子。规则切分常用 `RecursiveCharacterTextSplitter` 实现这种降级。

高价值章节（如技术规范的核心条款、合同的责任义务部分）可以适当放宽 chunk_size 上限，==避免任何截断==。

==3. 表格专项处理==

- 小表格（≤ 300 token）：整体作为一个 chunk
- 大表格（跨页或超 300 token）：按行切分，但==每个 chunk 都要复制表头==
- 不复制表头的后果：LLM 收到没有表头的数据行，不知道字段含义，==只能胡编==

==4. 列表项前导句合并==

==问题示例==：

```
以下情况不在保障范围：
（1）地震
（2）战争
```

如果把 `（1）地震` 单独成 chunk，向量里没有"不在保障范围"这个关键语义，==LLM 会误判为"地震在保障范围内"==。

==方案==：识别列表结构，把前导句和所有列表项合并为一个 chunk；如果超长，至少保留前导句到每个 chunk 的开头。

==适用==：有清晰结构信号的文档（技术手册、法律文件、规范、API 文档、Markdown 笔记）。==这是工业界 RAG 系统的主流方案==。

### 2.3 L3 语义切分（Semantic Chunking）

**核心思路：** 不依赖人工规则，让模型判断"哪里是主题边界"。

#### 路线一：Embedding 相似度突变检测

用 Embedding 模型计算相邻句子的向量相似度，==相似度突变处==（语义跳跃点）作为切分边界。LangChain `SemanticChunker` 是典型实现。

- ✅ 完全自动，不依赖文档结构
- ❌ 每对相邻句子都要算 Embedding，计算成本高；阈值难调（不同文档类型的语义跳跃幅度不同）

#### 路线二：判别式分块模型（==最新 SOTA==）

训练专用模型直接预测"相邻句子间是否存在主题边界"，==金山办公 2026==的方案是这条路线的代表。详细架构见 [[分块策略前沿#判别式分块（金山办公-2026）]]。

==适用==：文档无清晰结构（聊天记录、会议纪要、口语转写、自由写作）。

### 2.4 L4 命题级切分（Proposition-based）

**核心思路：** 用 LLM 把文档分解为==原子命题==（Proposition）——每个命题是自包含的事实陈述，不依赖上下文即可理解。

==示例==：

```
原文："苹果在 2023 年 Q4 营收增长 40%，主要受 iPhone 15 驱动。"

命题化：
- "苹果 2023 年 Q4 营收增长 40%"
- "iPhone 15 是苹果 2023 年 Q4 营收增长的主要驱动力"
```

来自 ==Dense X Retrieval==（Chen et al., EMNLP 2024，arXiv:2312.06648）。==每个命题语义密度最高，检索时精准匹配==。

==优势==：检索粒度最细，命题之间互不干扰，能解决"一段话讲多件事但只想检索其中一件"的问题。

==劣势==：LLM 调用成本高，处理速度慢；命题之间的关联信息可能丢失。

==适用==：高价值文档的精确事实问答（财报、合同、技术规范、医学文献）。

### 2.5 横向对比

| 层 | 方法 | 核心思路 | 适合场景 | 代价 |
|----|------|---------|---------|------|
| L1 | 规则切分 | 固定长度/分隔符 + Overlap 兜底 | 同质化短文档、baseline、所有方案的兜底 | 不理解结构，易截断语义 |
| L2 | 结构感知 | 利用标题/表格/列表的原生结构 | 有结构的文档（手册、法规、Markdown） | 需为不同文档类型定制规则 |
| L3 | 语义切分（Embedding） | 句间相似度突变检测 | 无结构文档（聊天、转写） | 计算成本高、阈值难调 |
| L3 | 语义切分（判别式） | 训练模型预测主题边界 | 同上，且有训练数据 | 推理快但需训练 |
| L4 | 命题级 | LLM 分解为原子命题 | 高价值精确问答 | LLM 成本高、可能丢失关联 |

> [!info] L1-L4 是==决定 chunk 边界==的四种方法学
> ==与 L1-L4 正交==的"chunk 内容增强"策略（Contextual Retrieval、Late Chunking）见下文 §三——这些不是更高一层的切分方式，而是==可以叠加在 L1-L4 任何一层之上==的内容/编码增强。
>
> ==与 L1-L4 正交==的"在线检索增强"策略（父子分块、句子窗口检索）属于检索阶段而非切分阶段，详见 [[RAG检索策略#五、检索时的上下文扩展]]。

## 三、chunk 内容增强（与切分边界正交）

==核心定位==：本节方法==不决定 chunk 边界画在哪==——边界由 L1-L4 决定。本节方法决定==每个 chunk 在向量化时如何编码==，让向量携带更多信息。

==与 L1-L4 是正交关系==：用 L1 规则切分照样可以叠加 Contextual Retrieval；用 L4 命题切分也能叠加 Late Chunking。

### 3.1 Contextual Retrieval（Anthropic 2024）

向量化前用 LLM 看整篇文档，==为每个 chunk 生成 50-100 token 的上下文描述==，把"上下文 + 原 chunk"一起向量化：

```
原 chunk: "营收同比增长 40%"
   ↓ LLM 看整篇文档生成上下文
增强后:   "[本段来自 ACME 公司 2023 Q4 财报，讨论北美区业务] 营收同比增长 40%"
   ↓ 向量化
带文档级上下文的向量
```

==关键技术==：用 ==Prompt Caching==把整篇文档作为缓存的 system prompt，处理 N 个 chunk 只完整读文档 1 次，==成本可降 80-90%==。

适用：结构化长文档（财报、合同、技术规范）+ 精确匹配场景。

### 3.2 Late Chunking（Jina AI 2024）

==反过来——"先编码再切"==。用长上下文 Embedding 模型先对整个文档做 token 级编码，每个 token 的向量已包含全文语境，再在向量层面按 chunk 边界做 mean pooling。

```
传统："这家公司的营收增长 40%" 单独编码 → 不知道是哪家公司
Late Chunking：在"ACME 财报"全文中编码 → "这家公司"的向量已经知道指代 ACME
```

适用：代词/指代密集的文档（法律、政策、学术），且 Embedding 模型支持长上下文（BGE-M3 8K、Qwen3-Embedding 32K）。

> [!tip] 算法细节、论文出处、实测效果详见 [[分块策略前沿]]
> 包含四种前沿方法的完整解读：Contextual Retrieval / Late Chunking / 判别式分块（金山办公 2026）/ 命题化切割（Dense X Retrieval, EMNLP 2024）。本节只给概念，工程实施请看前沿文档。

## 四、通用工程参数

### 4.1 chunk_size 选择

chunk_size 不是通用参数，==要结合文档类型和 Embedding 模型上下文长度==：

| 文档类型 | 句子平均长度 | 推荐 chunk_size |
|---------|-------------|----------------|
| 通用中文文档 | 20-30 字 | 512 token（约 2-3 段） |
| 英文技术文档 | 较短 | 512-768 token |
| 法律/合规/金融文档 | 长句多，每句 40-50 字 | 1024-1536 token |
| 对话/聊天/转写 | 短句多 | 256-512 token，配合句子窗口 |

==三个约束要同时满足==：
1. ==检索精度==：chunk 越长语义越分散，Embedding 区分度下降
2. ==语义完整性==：太小切断完整语句，太大包含多个主题
3. ==token 预算==：LLM 上下文有限，512 token × 5 chunk + prompt ≈ 4000 token，给生成留足空间

### 4.2 Overlap 推荐

==100 token 是性价比最优点==：召回率提升明显，存储仅增加约 10-15%。再往上收益递减。

配合==智能 overlap==（向后扫描到句子结尾才截断）效果更好，避免 overlap 区域本身又被切断。

### 4.3 chunk 元数据通用模板

==元数据是检索过滤和答案溯源的基础==，==离线阶段不打标签，在线阶段想过滤都没法过滤==。

```python
@dataclass
class ChunkMetadata:
    doc_id: str          # 文档 ID
    chunk_id: str        # chunk 唯一 ID
    section_path: str    # 层级路径，如 "第3章 架构 > 3.2 数据层"
    chunk_type: str      # "text" | "table" | "list" | "code" | "figure_caption"
    prev_chunk_id: str   # 前一个 chunk 的 ID（上下文扩展）
    next_chunk_id: str   # 后一个 chunk 的 ID
    parent_chunk_id: str # 父 chunk ID（父子分块场景）
    token_count: int
    source_path: str     # 来源文档路径
    page_range: str      # 来源页码范围（PDF 等有页码的格式）
    source_authority: int  # 来源权威等级（多源冲突解决用，详见 [[RAG高级技术#知识冲突处理]]）
    updated_at: str      # 更新时间（时效性优先用）
```

==字段用途==：
- `section_path` → 答案溯源、按章节过滤检索
- `chunk_type` → 不同类型走不同检索策略（表格用 BM25、文本用向量）
- `prev/next_chunk_id` → 检索到半截信息时拉取相邻 chunk 补全
- `parent_chunk_id` → 父子分块的关联键
- `source_authority` / `updated_at` → 多源冲突解决（权威优先、时效优先）

业务场景下可以追加领域字段（如"是否关键条款"、"产品线分类"等），==但通用模板应保持精简==。

## 五、选型决策树

按文档特征选方案（==只列离线分块策略，配套的在线检索策略——父子分块、句子窗口检索——见 [[RAG检索策略#五、检索时的上下文扩展]]==）：

```
文档是否有清晰结构（标题层级/Markdown/HTML）？
├── 有 → L2 结构感知（手册、法规、API 文档、Wiki）
│       └── 高价值精确问答场景叠加 L4 命题化 或 ==Contextual Retrieval（§三）==
│
└── 无 → 文档长度？
    ├── 中短（< 8K token） → L3 语义切分（判别式优先于 Embedding 相似度）
    │                       或 ==Late Chunking（§三）==（如有长上下文 Embedding 模型）
    └── 超长 → 金山判别式分块 + VFSC 超长块检索

通用兜底：所有方案都建议叠加 L1 Overlap（100 token + 智能边界）
```

==典型组合方案==：

| 场景 | 离线分块（本章）+ 在线检索增强（详见 [[RAG检索策略]]） |
|------|---------|
| 大规模通用知识库 | L2 结构感知 + L1 Overlap + 重排兜底 |
| 企业内部文档（Wiki/规范/手册） | L2 结构感知 + 检索时叠加==父子分块== |
| 法律/金融/医疗精确问答 | L2 结构感知 + Contextual Retrieval + BM25 混合检索 |
| 财报/合同的事实抽取 | L4 命题化 + 关联回原段落 |
| 会议纪要 / 客服对话 / 口语转写 | L3 判别式语义切分 + 检索时叠加==句子窗口== |
| 跨指代多的长文档 | Late Chunking |

所有方案都不是孤立的，==工程上常常组合使用==。最常见的组合是 ==L2 结构感知 + L1 Overlap + 检索时父子分块或 Contextual Retrieval==——结构感知保证 chunk 边界与逻辑边界对齐，Overlap 兜底边界截断，配套的检索增强提供上下文。
