# RAG 基础与架构

## 一、RAG 是什么

### RAG 的诞生背景

大语言模型在落地应用中存在三个固有缺陷：

1. **知识截止**：模型训练数据有截止时间，无法获取实时更新的行业信息与业务数据。
2. **事实幻觉**：模型生成内容易出现事实错误，在专业领域场景中输出错误信息，引发合规风险与业务损失。
3. **垂直领域理解不足**：通用大模型对垂直行业的专业术语、业务逻辑理解不足，无法满足企业级场景的精准问答需求。

RAG（Retrieval-Augmented Generation，检索增强生成）通过"检索-增强-生成"的核心架构，打破了模型固有知识的限制——将外部知识库作为事实依据，让模型生成的每一条结论都有可追溯的来源支撑，从根源上降低幻觉风险，同时支持知识库的动态更新，适配企业级业务的实时性需求。

### RAG 的核心价值

核心流程：用户提问 → 从知识库检索相关 chunk → 把 chunk 塞进 prompt → LLM 基于证据生成答案。

**解决的四个核心问题：**
- 知识截止：训练数据有时效，RAG 外挂动态知识库
- 事实幻觉：基于检索到的真实文档生成，答案可追溯
- 专业领域：垂直知识库比通用模型更精准
- 可控性：知识库独立管理，随时更新，无需重训模型

**RAG vs 微调：** 微调把知识烧进权重，更新成本高；RAG 把知识放在外部，随时可改。企业知识库首选 RAG。

## 二、系统架构：离线 + 在线两阶段

### 离线阶段（知识库构建）

```
原始文档 → 文档解析 → 文本清洗 → 分块(Chunking) → Embedding 向量化 → 索引构建 → 存储
```

五个模块缺一不可，任何一步出问题都会导致后续全崩。

### 在线阶段（查询响应）

```
用户 query → query 向量化 → 召回(Recall Top-100) → 重排(Rerank Top-5) → 拼 prompt → LLM 生成
```

关键点：Embedding 模型在两个阶段必须一致，文档和 query 要在同一语义空间里比较。

### 核心组件职责

| 组件 | 作用 | 性能参考 |
|------|------|----------|
| 文档解析 | 多格式 → 干净文本 | 决定知识质量上限 |
| Chunking | 切成可检索的语义单元 | 直接决定召回率 |
| Embedding | 文本 → 高维向量（768/1024/2048 维） | 语义理解能力基础 |
| 向量索引 | HNSW/IVF/FAISS，组织向量结构 | 暴力搜索 100 万文档需 10s，HNSW 只需 50ms |
| 召回 | 快速筛选候选集 | 毫秒级，高召回率 |
| 重排 | Cross-Encoder 精准评分 | 高精度，计算成本较高 |

## 三、文档解析

### 题目：PDF 多栏排版解析错乱怎么处理？

#### 核心解析

传统工具（PyPDF2）按行从上往下读，不理解"栏"的概念。双栏 PDF 会把左栏第一行和右栏第一行拼在一起，语义完全混乱。

**正确做法：引入版面分析（Layout Analysis）。**

先识别物理布局——哪些区域是左栏、右栏、表格、页眉页脚——再按逻辑结构提取内容。推荐工具：
- **MinerU / Marker**：内置版面分析，处理多栏、表格、复杂布局，详见 [[#MinerU 2.5-Pro（上海 AI 实验室，6 万 Star）]]
- **PaddleOCR + 区域检测**：扫描版 PDF，先检测区域类型（文字/表格/代码/图片），再分别处理
- **MarkItDown**：微软开源，覆盖格式最广（PDF/Word/Excel/PPT/图片/音频/ZIP/YouTube 等十几种），一行代码通吃，但对复杂版式深度不如 MinerU，详见 [[#MarkItDown（微软，9.3 万 Star）]]

**Baize 项目实现：** [[RAG管道设计#PDF 解析]] — 使用 Apache PDFBox 逐页提取，自动检测过滤页眉页脚（跨页重复 ≥2 次则删除），pageNumber 字段记录页码方便溯源。

### 题目：OCR 把表格和代码全毁了怎么办？

#### 核心解析

普通 OCR 对结构化内容还原能力差。表格经 OCR 后所有数据串成一行，代码缩进丢失、括号变形。

**解决方案：**
- 表格区域：专门的表格识别，按单元格顺序输出并保留结构化格式（Markdown 表格）
- 代码块：设置 OCR 保持换行和空格格式
- 整体策略：PaddleOCR 配合版面分析，先检测区域类型再分别处理

#### 高性能 OCR 方案：TurboOCR

对于需要大规模处理文档的 RAG 场景（批量扫描件录入、PDF 数字化），传统 PaddleOCR Python 版本的性能瓶颈（5 img/s）会成为系统瓶颈。

**TurboOCR** 是一个 GPU 加速的 OCR 推理引擎，通过 C++/CUDA/TensorRT 重写整个推理链路，将 PaddleOCR 的速度提升 50 倍：

**性能数据（FUNSD 数据集，RTX 5090）：**

| 方案 | 吞吐量 | P50 延迟 | F1 准确率 |
|------|--------|---------|----------|
| PaddleOCR Python | 5.1 img/s | 182ms | 86.6% |
| EasyOCR | 2.8 img/s | 312ms | 63.0% |
| **TurboOCR** | **268.4 img/s** | **11ms** | **90.2%** |

**核心优化：**
- 底层模型：PP-OCRv5（检测、识别、分类三大模型完整支持）
- 推理引擎：TensorRT 10.x FP16 精度 + CUDA 并行计算
- 图片解码：内嵌 Wuffs PNG 解码器（比常规方案更快）
- PDF 渲染：内置 PDFium（Chrome 的 PDF 引擎）
- 服务框架：Drogon 高性能异步 C++ HTTP 框架 + nginx 反向代理

**PDF 处理四种模式：**
1. **安全模式**：每页独立渲染，内存占用低，适合大批量处理
2. **快速模式**：全文档一次性渲染，速度最快但内存占用高
3. **平衡模式**：分批渲染，兼顾速度和内存
4. **流式模式**：边渲染边处理，适合超大文件

**部署便利性：**
- Docker 一行命令部署
- HTTP + gRPC 双协议支持
- Prometheus 监控集成
- MIT 开源协议

### 题目：PPT 里的图片信息怎么提取？

#### 核心解析

python-pptx 只能提取文本框里的文字，嵌在图片里的文字完全无能为力。关键信息如果做在图片里，就从知识库中彻底消失。

**解决方案：** 对 PPT 中的图片元素，提取出来做 OCR 识别，把图片中的文字纳入知识库。视频类文档先做 ASR（自动语音识别）得到字幕文本，再按内容语义分段入库。对于图文混排的复杂文档（含公式、图表、多栏），可使用 MinerU 等布局感知解析引擎完整还原版式结构，再结合 VLM 视觉模型处理图片内容，详见 [[#RAG-Anything（港大 HKUDS，1.7 万 Star）]]。

### 前沿方向：结构感知解析（Structure-Aware Parsing）

传统文档解析只提取文字内容，丢失了文档的原生结构。但现实中的手册、书籍、行业报告都有严格的层次化结构（目录、嵌套章节、多级小节、图文表格混合），这些结构本身就是重要的检索线索。

港中文 BookRAG 提出了一种新思路：**不只是提取文字，而是把文档还原成和目录一模一样的层级树**——根节点是文档标题，一级节点是各章，二级节点是各小节，叶子节点是段落/表格/图片/公式。再从树节点中提取实体建成知识图谱，通过**图-树链接（GT-Link）**把每个实体锚定回它所在的章节节点，实现结构和语义的双向关联。

这种结构感知的好处：检索时不用盲搜全文，而是通过实体/关键词直接定位到对应的章节子树，把检索范围从"全文几万块"缩到"某一小节几十块"。实验数据：M3DocVQA 召回率 71.2%，比最优基线高 26.7 个百分点，Token 消耗降低一个数量级。

详见 [[#BookRAG（港中文，结构感知 RAG）]]。

### 前沿方向：ChromaFs 虚拟文件系统（Mintlify）

**解决的核心问题：** 传统 RAG 只能"搜"不能"逛"——AI 只能拿到碎片化的 top-K 文本片段，无法像人类一样浏览目录、递归搜索、查看完整文档。而沙盒方案虽然体验完美，但启动慢（P90 延迟 46 秒）、成本高（年计算成本 7 万美元）、权限管理复杂。

**核心创新：** 用向量数据库模拟 UNIX 文件系统的交互逻辑，让 AI 以为自己在操作真实文件系统，实则所有操作都在向量数据库中完成。基于 Vercel Labs 的 just-bash（TypeScript 重写的 bash 解释器），实现可插拔的 IFileSystem 接口，将文件操作映射为 Chroma 向量数据库查询。

**四大核心技术：**

1. **内存化目录树**：将文档结构按"页面=文件、章节=目录"构建成 JSON 目录树，gzip 压缩后存入 Chroma。会话初始化时加载到内存，生成文件路径 Set 和目录-子文件 Map，`ls`/`cd`/`find` 等操作直接在内存执行，零网络开销、毫秒级响应

2. **轻量级访问控制**：目录树中为每个路径配置 `isPublic`（是否公开）、`groups`（可访问用户组）字段。会话初始化时根据用户 session token 剪枝无权路径，后续所有操作自动过滤无权内容，无需额外权限校验

3. **分块重组 + 懒加载**：
   - 执行 `cat` 时，从 Chroma 拉取文件的所有分块（按 `chunk_index` 排序）拼接成完整文档，结果缓存避免重复查询
   - 对于 S3 中的大文件，目录树仅存文件名与 S3 地址，`ls` 时仅显示文件名，`cat` 时才从 S3 拉取内容
   - 所有写操作（`touch`/`rm`/`mv`）直接抛出只读错误，保证系统无状态

4. **双层 Grep 优化**：
   - 粗过滤：将 grep 参数转为 Chroma 查询语句，快速定位所有可能匹配的文件路径
   - 细过滤：将粗过滤得到的文件批量预取到 Redis 缓存，在内存中执行精确的正则匹配
   - 将大规模递归 grep 时间从秒级压缩到毫秒级

**效果对比：**

| 指标 | 传统沙盒方案 | ChromaFs 方案 |
|------|-------------|--------------|
| P90 会话启动时间 | ~46 秒 | ~100 毫秒 |
| 单对话边际计算成本 | ~$0.0137 | $0（复用 Chroma） |
| 搜索机制 | 磁盘线性扫描 | 数据库元数据查询 |
| 基础设施依赖 | Daytona 等第三方服务商 | 已有 Chroma 数据库 |
| 权限管理复杂度 | 高（Linux 用户组 + 容器） | 低（目录剪枝） |

**技术启示：** 不要试图用传统技术模拟人类的操作环境，而要基于 AI 的能力特点设计适配的交互架构。AI 不需要真实的文件系统，只需要"文件系统的交互逻辑"。ChromaFs 通过复用现有基础设施（Chroma 向量数据库），实现零边际成本的"幻觉式交互"。

> 项目地址：https://www.mintlify.com/blog/how-we-built-a-virtual-filesystem-for-our-assistant

## 四、分块策略（Chunking）

> 切分不只是"设置一下 chunk_size"，它是需要分析文档特点、设计处理策略、用量化指标验证的工程模块。切错了，后面做再多优化，全是在地基不稳的楼上加砖。

### 题目：固定 512 token 切分有什么问题？

#### 核心解析

固定长度切分是最被低估的灾难。实测数据：三代方案对比，**只改切分方式，Recall@5 从 0.67 提升到 0.91**，没有换 Embedding 模型，没有调检索策略。

| 方案 | Recall@5 | 核心问题 |
|------|----------|----------|
| V1 固定 512 token | 0.67 | 在句子中间截断，语义残缺 |
| V2 句子级切分 | 0.74 | 不理解文档层级，表格/列表处理差 |
| V3 语义感知切分 | 0.91 | 四项核心改进（见下） |

**V3 的四项核心改进：**

1. **层级结构识别**：多策略融合识别章节标题（数字编号、中文编号、混合编号），按章节边界切而不是 token 数切，同一章节内容尽量放同一 chunk，跨章节绝不合并

2. **超长章节递归细切**：普通章节 max_size=1024，关键条款（责任/免责）用 1536，保证语义完整；超长则按子小节边界递归拆分

3. **表格专项处理**：
   - 小表格（≤300 token）：整体作为一个 chunk
   - 大表格（跨页或超过 300 token）：按行切分，但**每个 chunk 都复制表头**
   - 不复制表头的后果：LLM 收到一堆没有表头的数字，不知道对应什么字段，只能胡编。实测表格类问题正确率从 43% 提升到 78%

4. **列表项合并前导句**：`"以下情况不在承保范围：（1）核辐射（2）战争"` 必须把前导句和所有列表项合并为一个 chunk。否则"（1）核辐射"单独成 chunk，向量里没有"不在承保范围"这个语义，否定性查询召回率只有 0.58，加了前导句合并后提升到 0.83

### 题目：Overlap 设多少合适？怎么验证？

#### 核心解析

Overlap 解决的是"关键信息恰好落在两个 chunk 边界处"的问题。不能拍脑袋，要做实验。

| Overlap 大小 | Recall@5 | 存储增加 |
|-------------|----------|----------|
| 0 token | 0.81 | 基准 |
| 50 token | 0.86 | +5% |
| 100 token | 0.89 | +10% |
| 200 token | 0.90 | +20% |
| 300 token | 0.90 | +30% |

**100 token 是性价比最优点**：召回率提升显著（+10% vs 无 overlap），存储只增加 10%，再往上收益递减。

进一步优化：**基于句子边界的智能 overlap**——固定 overlap 可能恰好切在句子中间，智能 overlap 在确定重叠区域后向后扫描到最近句子结尾才截断。效果：避免了 87% 的"句子在 overlap 区域被截断"问题，召回率从 0.89 进一步提升到 0.91。

**Baize 项目实现：** [[RAG管道设计#分片策略]] — 三级降级策略（段落→句子→HanLP 分词→字符硬切），chunk size=512 字符，无 overlap（简化存储，代价是跨 chunk 语义可能断裂）。

### 题目：chunk_size 应该怎么确定？

#### 核心解析

chunk_size 不是通用参数，要和文档类型挂钩：
- 通用中文文档：平均句子 20-30 字，512 token ≈ 2-3 段，够用
- 保险条款：每句平均 40-50 字，512 token 经常只有一段话的开头和中间
- 实测：保险文档从 512 改到 1024（关键条款 1536），召回率提升 7 个百分点

三个约束要同时满足：
1. **检索精度**：chunk 越长语义越分散，embedding 区分度下降
2. **语义完整性**：太小容易切断完整语句，太大包含多个主题
3. **token 预算**：LLM 上下文窗口有限，512 token × 5 个 chunk + prompt ≈ 4000 token，给生成留足空间

## 五、Chunk 元数据设计

元数据是检索过滤和答案溯源的基础，离线阶段偷懒不打标签，在线阶段想过滤都没法过滤。

```python
@dataclass
class ChunkMetadata:
    doc_id: str          # 文档ID
    chunk_id: str        # chunk唯一ID
    section_path: str    # 层级路径："第3条 保险责任 > 3.2 责任免除"
    chunk_type: str      # "text" | "table" | "list"
    is_key_clause: bool  # 是否关键条款（责任/免责/费率）
    prev_chunk_id: str   # 前一个chunk的ID（上下文扩展用）
    next_chunk_id: str   # 后一个chunk的ID
    token_count: int
    page_range: str      # 来源页码范围
```

**各字段用途：**
- `section_path`：答案溯源，"根据第3条保险责任 > 3.2 责任免除 > (2)，核辐射不在保障范围"
- `is_key_clause`：检索时给关键条款加 1.5 倍权重，关键词匹配（"责任""免责""费率""赔付"）+ 章节标题判断，准确率 94%
- `prev/next_chunk_id`：检索到半截信息时自动拉取相邻 chunk 补全上下文

## 六、前沿分块技术

### 语义切分（Semantic Chunking）

用 Embedding 模型计算相邻句子的向量相似度，相似度突变处（语义跳跃点）作为切分边界。理论上比规则方法更精准，但工程问题：
- 计算成本高：每次切分都要跑 Embedding 推理
- 5000 份文档全量 Embedding 切分需几十分钟，规则方案切一份只需 2-3 秒，Semantic Chunking 要 30-40 秒

### 分层 RAG（父子分块）

**作用：** 创建父-子分块关系。检索时搜索小子分块（保证精度），返回大父分块（保证上下文）。

**解决的问题：** 小分块匹配精确但缺乏上下文，大分块上下文充足但匹配不准。两者矛盾，分层结构同时拿到。

**实现思路：**
- 父分块：2000 字符的大块，存内容和元数据
- 子分块：500 字符的小块，存向量和 `parent_id` 引用
- 检索时：用查询向量匹配子分块向量 → 通过 `parent_id` 拉取父分块返回给 LLM

**适用场景：** 文档具有清晰的层次结构（技术手册、法律文件、研究论文）。

> 数据来源：[2026年构建RAG系统的核心策略：从60%到94%准确率](https://mp.weixin.qq.com/s?__biz=Mzk0OTI1OTQ2MQ==&mid=2247597619&idx=1&sn=bbd3a4e8dff5692bb591f330d31f6066&chksm=c25feafa83b237e3a7dc36be16c612314d474925ff94e4f8757e24c20975f2bb4f241417ed31&mpshare=1&scene=24&srcid=0413bmDhEoywWxsGCSbztell&sharer_shareinfo=83328d3281654c297640b7b5627572cb&sharer_shareinfo_first=83328d3281654c297640b7b5627572cb#rd)

### Late Chunking（2024 年底）

先用长文本 Embedding 模型对整个文档编码，获取每个 token 在完整上下文下的向量表示，再在向量层面做切分。好处是每个 token 的向量里都带有全文上下文信息，不存在"切断了就丢失前文语义"的问题。缺点：对模型上下文长度有要求，多文档批量处理时内存占用是传统方案的 8-10 倍，目前处于研究向阶段。

### 金山办公判别式分块框架（2026）

论文：《Toward General Semantic Chunking: A Discriminative Framework for Ultra-Long Documents》

核心思路：跳出"生成式分块"，用判别式模型做主题边界二分类。

架构：Qwen3-0.6B 骨干 → token 级语义编码 → 注意力池化得句子级表示 → 轻量 Transformer 建模跨句上下文 → MLP 分类头输出边界概率

**实验结果（WIKI-727K 基准）：**

| 模型 | F1-Score | 召回率 |
|------|----------|--------|
| 最好的生成式模型（Jina/Qwen2-0.5B） | 0.5185 | 0.5388 |
| 金山判别式框架（Qwen3-0.6B） | **0.5503** | **0.7312** |

推理速度比生成式模型快 **100 倍**（处理 13k token 仅需数百毫秒，生成式需逐 token 解码）。

**超长文档处理：** 重叠滑动窗口策略——超过 13k token 的文档按固定长度拆分窗口，相邻窗口保留 10% 重叠区域，重叠区域的边界概率取多窗口预测均值，支持任意长度文档。

**启发式后处理三阶段：** ① 按阈值做初始分块 → ② 超长块递归选取块内最高概率位置二次拆分 → ③ 超短块对比左右语义连续性自动合并，避免过度碎片化。

超长块检索适配：VFSC（向量融合与标量校正）方法，用一个融合向量+一个标量因子严格还原超长块所有子段的平均余弦相似度，存储复杂度从 O(N) 降至 O(1)。

> 论文：https://arxiv.org/pdf/2602.23370

## 七、相关开源项目

### RAG-Anything（港大 HKUDS，1.7 万 Star）

**解决的问题：** 传统 RAG 只能处理纯文字，遇到图片、表格、公式、图表直接丢失——而现实文档几乎没有纯文字的。

**核心方案：**
1. **MinerU 文档解析**：完整还原 PDF 版式结构（多栏、公式、脚注、图表位置），区分不同内容区域分别处理，不是简单 OCR 抽文字。
2. **双图谱架构**（论文核心创新）：同时构建跨模态关系图谱（"图 2 描述的是第三段的内容"）和文字语义图谱，检索时两张图互相配合。
3. **VLM 视觉语言模型**：答案可能在图里时，自动调用视觉模型（如 GPT-4o）"看图说话"。
4. **三种查询模式**：Pure Text（纯文字）、VLM Enhanced（视觉增强）、Hybrid（混合检索，效果最好）。

> 项目地址：https://github.com/HKUDS/RAG-Anything
> 论文：https://arxiv.org/abs/2510.12323
> 姊妹项目 LightRAG：https://github.com/HKUDS/LightRAG

### MinerU 2.5-Pro（上海 AI 实验室，6 万 Star）

**解决的问题：** 文档解析的性能瓶颈不在模型架构，而在训练数据质量。所有主流解析模型在复杂嵌套表格、密集公式、非常规多栏排版上表现出高度一致的失败模式——说明问题出在数据，不是模型。

**核心方案：** 不改一行架构代码，1.2B 参数小模型仅靠数据工程登顶文档解析榜首，超越 Gemini 3 Pro、Qwen3-VL-235B 等 235B 参数大模型。

**数据引擎三大组件：**
1. **多样性与难度感知采样**：页面级 + 元素级（文本块/公式块/表格块）分别聚类和难度评估，训练数据从不足 1000 万页扩展到 6550 万页，长尾分布得到校正。
2. **跨模型一致性验证**：K 个异构 SOTA 模型同时解析同一样本，根据输出一致性量化样本难度，筛选高信息量样本。
3. **评判与精炼流水线**：将解析结果渲染成图像，与原文档视觉对比定位错误并迭代校正，极少数超出自动校正能力的样本交由专家标注兜底。

**三阶段训练：** 大规模预训练（数据扩大近 10 倍）→ 困难样本差异化微调 → GRPO 强化学习对齐评估指标。

**成绩：** OmniDocBench v1.6 评测 95.69 分，超越所有现有方法。

**启示：** 小模型 + 好数据 > 大模型 + 差数据。在算力有限时，数据质量深耕比盲目追大模型更务实。

> 项目地址：https://github.com/opendatalab/MinerU
> 论文：https://arxiv.org/pdf/2604.04771

### BookRAG（港中文，结构感知 RAG）

**解决的问题：** 传统 RAG 把文档切成无结构块做"盲搜"，完全丢失层级结构（目录、章节嵌套、图表归属），导致检索不精准、多跳推理断链。

**核心方案：** BookIndex 混合索引 = 层级树 T（还原文档章节结构）+ 知识图谱 G（实体关系）+ 图-树链接 M（实体锚定到章节节点）。检索时 Agent 动态分类查询（单跳/多跳/全局聚合），按结构导航精准定位，而非盲搜全文。

**关键技术：**
- 梯度式实体消歧：把 O(n^2) 的批量比对降到 O(n) 的增量查找，利用相似度梯度突变识别同义实体，实体数量减少 12%，图谱密度提升 20%
- 天际线排名（Skyline Ranking）：多目标优化，只保留帕累托前沿上的非支配节点，综合图谱重要性和文本相关性

**实验数据：** M3DocVQA 的 EM 达 61%（次优基线 43%），召回率 71.2%（比最优基线高 26.7 个百分点），Token 消耗降低一个数量级。

**局限：** 子章节内超长内容仍需分块，top-k 固定，未能动态调整检索粒度，距离真正的文档理解还有距离。

> 论文：https://arxiv.org/pdf/2512.03413

### MarkItDown（微软，9.3 万 Star）

**解决的问题：** RAG 系统要处理企业里堆积如山的各种格式文档（PDF、Word、Excel、PPT、图片、音频等），每种格式都要单独写解析逻辑，开发成本高。大模型又看不懂二进制文件，必须先转成文本。

**为什么选 Markdown 作为统一中间格式？** 主流 LLM 几乎都是用海量 Markdown 文本训练出来的（GPT-4o 经常主动用 Markdown 格式回复），喂 Markdown 相当于用模型的"母语"对话：
- **Token 效率高**：没有复杂排版标记，同内容比 PDF 省约 30% Token
- **结构清晰**：标题层级、列表、表格有明确标记，AI 读起来一目了然
- **幻觉更少**：信息越干净，AI 编造越少

**核心方案：** 一行代码把十几种格式统一转成 Markdown——机器友好的格式，LLM 可以直接消化。支持格式：
- 办公文档：PDF、Word (docx)、Excel (xlsx)、PowerPoint (pptx)
- 媒体：图片（OCR）、音频（语音转写）
- 网页：HTML、EPub、YouTube 视频链接
- 数据：CSV、JSON、XML、ZIP 压缩包

**设计哲学：** 从一开始就是"为 LLM 设计"，目标是"机器能读懂"而非"人看舒服"。README 里明确说：如果你想要高保真的人类可读转换，它不是最佳选择。

**工程考量：**
- **可选依赖组**：基础安装很轻，按需装 `[pdf]`、`[docx]`、`[audio-transcription]` 等 extras，避免镜像臃肿。一次装全用 `pip install 'markitdown[all]'`
- **图片/OCR 灵活替换**：官方默认调 OpenAI Vision API，但完全可以换成本地部署的 VLM（如 Qwen-VL、InternVL）做 OCR，既能保护隐私又能控成本
- **MCP Server 集成**：提供 MCP 服务器，可以被 Claude Desktop 等 Agent 框架自动调用，作为"文档摄取层"

**和 MinerU 的对比：** MarkItDown 覆盖格式最广但深度浅，MinerU 专注复杂版式（多栏、公式、嵌套表格）但格式少。实际工程中可以搭配使用：一般文档用 MarkItDown 快速处理，复杂 PDF 交给 MinerU。

**HN 社区的务实建议：** 不建议用它转图片（LLM 直接看图更好）；不建议用 Markdown 表格喂电子表格数据（LLM 对表格理解本来就差）。

> 项目地址：https://github.com/microsoft/markitdown
