# RAG 基础与架构

## 一、RAG 是什么

### RAG 的诞生背景

大语言模型在落地应用中存在三个固有缺陷：

1. **知识截止**：模型训练数据有截止时间，无法获取实时更新的行业信息与业务数据。
2. **事实幻觉**：模型生成内容易出现事实错误，在专业领域场景中输出错误信息，引发合规风险与业务损失。
3. **垂直领域理解不足**：通用大模型对垂直行业的专业术语、业务逻辑理解不足，无法满足企业级场景的精准问答需求。

RAG（Retrieval-Augmented Generation，检索增强生成）通过"检索-增强-生成"的核心架构，打破了模型固有知识的限制——将外部知识库作为事实依据，让模型生成的每一条结论都有可追溯的来源支撑，从根源上降低幻觉风险，同时支持知识库的动态更新，适配企业级业务的实时性需求。

### RAG 的核心价值

核心流程：用户提问 → 从知识库检索相关 chunk → 把 chunk 塞进 prompt → LLM 基于证据生成答案。

**解决的四个核心问题：**
- 知识截止：训练数据有时效，RAG 外挂动态知识库
- 事实幻觉：基于检索到的真实文档生成，答案可追溯
- 专业领域：垂直知识库比通用模型更精准
- 可控性：知识库独立管理，随时更新，无需重训模型

**RAG vs 微调：** 微调把知识烧进权重，更新成本高；RAG 把知识放在外部，随时可改。企业知识库首选 RAG。

> [!tip] RAG 不是万能的
> RAG 适合"需要事实依据的问答"，不适合"需要推理或创造的任务"。比如写诗、做数学题、设计架构方案——这些场景 RAG 检索到的文档反而会干扰模型的自由推理。选型时先问：**这个任务的答案是"在某个文档里"还是"需要模型思考出来"？**

## 二、系统架构：离线 + 在线两阶段

### 离线阶段（知识库构建）

```
原始文档 → 文档解析(§三) → 文本清洗(§四) → 分块Chunking(§五) → Embedding向量化 → 索引构建 → 存储
```

五个模块缺一不可，任何一步出问题都会导致后续全崩。

> [!warning] 分块（Chunking）是 RAG 质量的最大杠杆
> 实践中 80% 的 RAG 质量问题可以追溯到分块策略不当：chunk 太大导致噪声多、相关性差；chunk 太小导致上下文缺失、答案碎片化。优先优化分块，收益远大于换 Embedding 模型或调检索参数。

### 在线阶段（查询响应）

```
用户query → 查询理解与路由 → query向量化 → 召回(Recall) → 重排(Rerank) → Prompt拼接 → LLM生成
```

关键点：Embedding 模型在两个阶段必须一致，文档和 query 要在同一语义空间里比较。

> [!tip] Embedding 在两阶段==分别==做，==chunk 向量预先算好不重算==
> 这是 RAG 最核心的工程设计：
> - **离线阶段**：所有 chunk 通过 embedding 模型逐个编码，向量和文本一起存进向量数据库 → ==这是一次性成本==
> - **在线阶段**：每次用户提问==只对 query 做一次 embedding 编码==，然后向量数据库用 HNSW/IVF 索引找相似度最高的 Top-K chunk
>
> ==chunk 向量不会在每次查询时重算==——百万级知识库每次查询都重算 100 万次 embedding 是不可能的。==向量数据库的核心价值就是"预存向量 + ANN 加速搜索"==，类似传统数据库"建索引一次、查询多次"的思路。详见 [[RAG向量与Embedding#二、向量检索算法]]。
>
> 文档变化时只对变化的 chunk 重算（增量索引），其他向量不动，详见 [[RAG高级技术#增量索引]]。

## 三、文档解析

文档解析是离线管道的第一步，目标是将各种格式的原始文档转化为干净的结构化文本。解析质量直接决定了知识库的上限——解析丢失的信息，后续任何优化都无法弥补。

### 3.1 文档解析的两个正交维度

文档解析的复杂度来自两个相互独立的维度，混在一起讨论是大多数 RAG 入门资料的通病：

| 维度 | 取值 | 含义 |
|------|------|------|
| **容器格式** | PDF / Word / Excel / PPT / HTML / 扫描件 / 音视频 | 文件的封装形式，决定解析的起点（拿到的是 XML 树？坐标集合？还是像素？） |
| **内容元素** | 文本 / 表格 / 图片 / 公式 / 多栏排版 / OCR需求 | 文档里实际承载信息的"原子"，决定需要哪些处理能力 |

**关键认知：内容元素的处理方法是通用的，格式特性只决定解析的起点。**

举例：表格可能出现在 PDF、Word、Excel、PPT、扫描件甚至发票图片里——不同格式拿到的"表格"形态不一样（Excel 是网格、PDF 是坐标和线条、扫描件是像素），但==做表格结构识别的目标是统一的：还原行列单元格关系==。所以应该按"内容元素"组织通用方法，按"格式"只讲各自独有的难点。

### 3.2 格式光谱：结构化程度决定解析起点

不同格式拿到时的"信息保真度"不一样，这决定了解析的难度起点：

```
高结构 ──────────────────────────────────────── 无结构
  Word/PPT/Excel    HTML       PDF      扫描件/图片    音视频
  (XML 原生标签)   (DOM 树)  (坐标集合)   (像素)      (时序信号)
                                                       ↓
                                                需要 ASR 转写
```

| 结构化等级 | 格式 | 拿到的是 | 解析起点 |
|-----------|------|---------|---------|
| 高 | Word/PPT | XML 树，标签即语义（标题、列表、表格） | 直接遍历 XML 节点 |
| 高 | Excel | 网格化的单元格 + 单元格属性 | 直接读单元格 |
| 中 | HTML | DOM 树，有语义但夹杂噪声 | 正文提取 + 标签清洗 |
| 低 | PDF（数字版） | 文字 + 坐标 + 线条，无逻辑结构 | 版面分析 + 阅读顺序还原 |
| 无 | 扫描件/图片 | 像素阵列 | OCR + 版面分析 |
| 时序 | 音视频 | 音频/视频流 | ASR 转写 + 时间戳 |

==Word/PPT/Excel 容易解析不是因为内容简单，而是因为它们的 XML 原生保留了结构语义；PDF 难解析不是因为内容复杂，而是因为 PDF 是"打印格式"，所有结构信息都被丢弃了，只剩坐标。==

### 3.3 内容元素的通用解析方法

这是文档解析的核心章节。每种内容元素的处理方法独立于具体格式，是真正的通用能力。

#### 3.3.1 表格结构识别（TSR）

**问题本质：** 还原行列单元格关系。表格可能出现在任何格式里，不同格式给到的表格形态完全不同：

| 来源 | 拿到的形态 | 解析方法 |
|------|----------|---------|
| Excel/Word（XML 原生） | 单元格坐标 + 内容 + 合并属性 | 直接遍历，处理合并单元格 |
| HTML | `<table>` 标签 | DOM 解析，保留 `<thead>/<tbody>/<tr>/<td>` |
| PDF（数字版） | 文字坐标 + 线条坐标 | 几何关系推断单元格边界 |
| 扫描件/图片表格（含发票） | 像素 | TSR 模型直接从图像识别 |
| PPT 中的表格图片 | 像素 | 同扫描件处理 |

**主流方案：**

- **TableFormer（IBM）**：专用 TSR 模型，输入表格图像，输出单元格的行列坐标和内容。Docling 内置。
- **unstructured `hi_res` 模式**：保留 HTML 表格结构，行列对齐
- **VLM 直读**：复杂表格（嵌套、跨行跨列）直接用 GPT-4o/Qwen-VL 输出 Markdown 表格
- **MinerU**：内置版面分析 + TSR，端到端处理

**通用难点（不分格式）：**

- **跨页表格**：检测下一页开头无表头但有数据行，自动拼接并补全表头
- **合并单元格**：合并区域只有左上角有值，需展开填充到所有被合并的位置
- **表头绑定**：将每行数据转为"表头: 值"键值对，让 Embedding 能理解纯数字列的语义
- **LLM 辅助描述**：复杂表格生成自然语言摘要（"该表展示 2023 年 Q1-Q4 各产品线营收对比"），作为额外检索入口

> [!warning] 表格不要混入正文 chunk
> 表格应作为独立 chunk 入库，并打上 `type: table` 元数据。否则正文向量会污染表格的精确数值匹配能力。

> TableFormer / TATR 等模型架构原理详见 [[TSR]]。

#### 3.3.2 图片与图表理解

**问题本质：** 图片可能携带文字（发票、流程图标签）、数据（图表）、空间结构（架构图、思维导图），处理路径取决于"我们想要什么"。

**先识别图片性质：**

| 图片性质 | 典型例子 | 处理方法 |
|---------|---------|---------|
| 文字主导 | 发票、扫描表格、流程图标签 | OCR 提取文字 |
| 图表数据 | 柱状图、饼图、折线图 | VLM 提取数据点和趋势 |
| 空间结构 | 架构图、流程图、思维导图 | VLM 生成结构化描述 |
| 装饰性 | 配图、Logo、装饰背景 | 跳过或仅记录元数据 |

**处理路径：**

```
图片 → 性质判断 → 分支处理
                  ├─ OCR（PaddleOCR / TurboOCR）→ 提取文字
                  ├─ VLM（GPT-4o / Qwen-VL）→ 生成 Caption
                  └─ Caption + OCR 联合 → 既要文字也要语义
```

**Caption 质量要求**（多模态 RAG 章节也涉及，详见 [[RAG高级技术#七、多模态-RAG]]）：

- 必须包含：图片类型、关键数值、趋势、结论
- 反例：「这是一张图片」「图中显示了一些数据」
- 正例：「2023 年 Q4 营收柱状图，显示北美区营收 5.2 亿美元（同比 +18%），亚太区 3.8 亿（+24%），亚太增速领先」

**MultiVectorRetriever 模式：** Caption 入向量库做检索，原图路径存 docStore，命中后将原图传给多模态 LLM 生成答案。

> Caption 生成依赖的视觉语言模型架构详见 [[VLM文档解析]]。

#### 3.3.3 公式识别

**问题本质：** 公式在 PDF/Word/PPT/扫描件中都可能出现，传统 OCR 完全无法处理符号、上下标、积分、矩阵等数学结构。目标是输出 LaTeX 或 MathML 让 LLM 理解。

**主流方案：**

- **Pix2Tex / Nougat（Meta）**：专用公式识别模型，输入公式图像 → 输出 LaTeX
- **MinerU 内置**：自动检测公式区域，调用公式识别模型转 LaTeX，嵌入 Markdown 输出
- **Dolphin-v2（字节）**：端到端处理，原生支持行内公式（`$E=mc^2$`）和独立公式块
- **VLM 直读**：GPT-4o 等多模态模型可直接读公式图像输出 LaTeX

**输出格式选择：**

- LaTeX：适合学术文档，LLM 训练时见过大量 LaTeX
- 自然语言描述：适合非学术读者（"动能等于二分之一质量乘以速度的平方"）
- 双轨输出：原始 LaTeX + 自然语言描述，前者用于精确匹配，后者用于语义检索

> Pix2Tex / Nougat / im2markup 等模型架构原理详见 [[公式识别]]。

#### 3.3.4 多栏版面分析

**问题本质：** PDF、扫描件、部分 PPT 都可能是多栏排版。物理坐标从上到下逐行读取会把左栏第一行和右栏第一行拼在一起，得到的全是乱序文本。

**解决方案：版面分析（Layout Analysis）**

先识别页面的物理布局区域（标题/正文/表格/图片/页眉页脚/侧边栏），再按逻辑阅读顺序重组：

```
原始 PDF 页面
    ↓ 版面检测模型（DocLayNet / RT-DETR）
[区域1: 标题, 区域2: 左栏正文, 区域3: 右栏正文, 区域4: 表格, ...]
    ↓ 阅读顺序排序算法
[标题] → [左栏从上到下] → [右栏从上到下] → [表格] → ...
    ↓ 按顺序拼接文本
干净的逻辑顺序输出
```

**两条技术路线：**

- **Pipeline 方案**：版面检测模型 → 阅读顺序排序 → 按区域类型分别处理（表格走 TSR、图片走 OCR/VLM、文本直接提取）。代表：Docling、MinerU
- **端到端方案**：视觉语言模型直接将页面图像转为结构化输出，跳过显式区域检测。代表：Dolphin-v2（3B 参数，21 种内容类型）、SmolDocling（IBM 超紧凑 VLM）

> LayoutLMv3 / DocLayNet / RT-DETR / DETR 等模型架构原理详见 [[版面分析]]。

#### 3.3.5 OCR：从像素到文本

**问题本质：** 扫描件、图片、PDF 中的图层都需要 OCR。难点不在"识别字"，而在"识别完之后还原版面"。

**两层架构：**

```
图像预处理（去噪/纠偏/二值化）
    ↓
版面分析（区域检测：文字/表格/代码/图片/公式）
    ↓
分区域 OCR
    ├─ 文字区域 → 文本 OCR
    ├─ 表格区域 → 表格结构识别
    ├─ 公式区域 → 公式识别
    └─ 图片区域 → VLM 或保留为图片
    ↓
后处理拼接（按版面顺序）
```

**主流引擎对比：**

| 工具 | 速度 | 精度 | 部署 |
|------|------|------|------|
| PaddleOCR | 中（5 img/s） | 高，中文好 | CPU/GPU 通用 |
| TurboOCR | 极快（268 img/s，50× 提速） | 同 PaddleOCR | GPU 推荐，Docker 一行部署 |
| Tesseract | 慢，老牌 | 中文一般 | CPU |
| VLM 直读（GPT-4o/Qwen-VL） | 慢，按 token 收费 | 高，能理解版面 | API 或 GPU |

**TurboOCR 关键技术：** C++/CUDA/TensorRT 重写 PaddleOCR 推理链，PP-OCRv5 + TensorRT FP16 + 内嵌 Wuffs PNG 解码器 + PDFium 渲染引擎，提供安全/快速/平衡/流式四种模式。

> TurboOCR 项目：https://github.com/aiptimizer/TurboOCR

> DBNet / CRNN / SVTR / TrOCR 等模型架构原理详见 [[OCR]]。

### 3.4 各格式的独有特点

通用元素方法已在 3.3 讲完，本节只讲每种格式==真正独有的难点和优势==。

#### 3.4.1 PDF：坐标到逻辑结构的还原

**核心难点（区别于其他格式的本质问题）：** PDF 是"打印格式"，所有结构信息都丢失了。拿到的是文字 + 坐标 + 线条，**没有"段落"概念，没有"页"的语义概念，没有"标题层级"**。

> [!info] PDF 不是图片合集
> 数字版 PDF 内部是 PostScript 风格的绘制指令（"在坐标 (100,700) 用 12pt 字体写 'Hello'"），文字以 Unicode 存储，所以能复制能搜索。==这也是为什么 PDF 比扫描件信息多==——给了文字、字体、坐标、矢量图形，只是没有逻辑结构。但扫描版 PDF 每页就是一张图片塞进 PDF 容器，等同于图片，必须走 OCR。详见 [[PDF]]。

**独有问题：**

- 物理坐标 → 逻辑阅读顺序的还原（依赖版面分析，详见 3.3.4）
- 跨页元素的拼接（表格、段落跨页）
- 数字版 PDF vs 扫描版 PDF 的区分（前者有文字层，后者要走 OCR）

**适用工具：** MinerU / Docling（Pipeline 方案深度强）、Dolphin-v2（端到端拍照文档）

#### 3.4.2 Word（docx）：XML 是优势，独有问题在边缘

**优势：** docx 本质是 ZIP 包里的 XML 树，标题层级（Heading 1/2/3）、列表、表格结构都原生保留。直接用 python-docx 遍历 XML 即可拿到清晰结构。

**独有问题：**

- **嵌入对象（OLE）**：嵌入的 Excel 表格、Visio 图、PDF，python-docx 无法直接提取，需识别 OLE 类型分发到对应解析器
- **复杂样式继承**：用户可能用"加粗"代替"标题样式"，导致层级识别失败 → 需结合字号/字重启发式判断
- **修订痕迹与批注**：包含多版本内容，需决定保留哪个版本（一般保留接受后的最终版）
- **浮动文本框与形状**：不在主文档流中，容易遗漏 → 需单独遍历 shapes 元素

**适用工具：** python-docx（基础）、MarkItDown（直转 Markdown）、Docling（结构化 JSON 输出）

#### 3.4.3 Excel（xlsx）：网格化布局的独有挑战

**优势：** 表格结构最清晰，单元格坐标 + 合并属性原生可读。

> [!info] "原生可读"是怎么做到的？
> Excel 看似只是网格，但每个单元格在 OOXML 内部都是带属性的 XML 元素 `<c>`：`r` 属性 = 地址（A1/B5）、`t` 属性 = 类型（数字/字符串/布尔/日期）、`s` 属性 = 样式索引、`<v>` 子元素 = 值、`<f>` = 公式。
> 这些原生属性让 openpyxl 等解析器==直接读出==类型化的数据，不用从视觉上猜。详见 [[OOXML#四、xlsx 内部结构]]。

**独有问题：**

- **多 Sheet 跨表引用**：一个文件多张表，Sheet 之间可能有 `=Sheet2!A1` 引用 → 每个 Sheet 独立解析，Sheet 名作为元数据
- **公式 vs 计算值**：单元格存的是 `=SUM(A1:A10)`，要决定保留公式还是结果（一般两者都存，公式作元数据）
- **非表格化布局**：用户把 Excel 当画布——标题在 A1，数据从 C5 开始，中间空行空列做分隔 → 需启发式识别"数据区域"
- **语义缺失**：纯数字列没有上下文 → 表头绑定（详见 3.3.1）+ LLM 摘要描述
- **NL2SQL 路由**：需要精确数值查询时（"Q3 营收是多少"），不走向量检索，导入数据库走 SQL（详见 [[RAG检索策略#一、查询理解与意图识别]]）

#### 3.4.4 PPT（pptx）：信息分散与隐藏

**独有问题（其他格式没有的）：**

- **信息在图片里**：关键内容（流程图、架构图）做成整张图片，python-pptx 只能提取文本框文字 → 需对每张图片走 VLM 处理
- **动画与构建顺序**：内容按动画步骤逐步展示，解析时全部平铺丢失逻辑顺序 → 按动画时间戳排序（如果保留了）
- **备注页分离**：演讲者备注往往比正文更详细 → 必须将备注与正文合并入同一 chunk
- **母版与模板噪声**：页眉页脚、Logo 跨页重复 → 识别跨页重复文本，标记为模板内容过滤

**独有方案：幻灯片级分块** —— 每页幻灯片作为独立语义单元，标题作为 `section_path` 元数据。

#### 3.4.5 HTML：DOM 噪声与结构语义保留

**独有问题：**

- **70%+ 噪声**：导航/侧边栏/广告/Cookie 提示/页脚链接，与正文无关
- **动态渲染**：SPA 应用内容通过 JS 动态加载，静态抓取拿不到
- **结构语义丢失**：传统方案转纯文本会丢失标题层级、列表、表格

**独有方案：**

- **Trafilatura / Readability.js**：文本密度分析 + DOM 树结构识别正文区域
- **HtmlRAG（WWW 2025）**：直接保留清洗后的 HTML 喂给 LLM，HTML 标签本身就是结构语义信号
- **Defuddle / site2rag**：HTML → Markdown，保留结构同时减少 token
- **Playwright/Puppeteer**：浏览器渲染后再提取，处理 SPA

> HtmlRAG 项目：https://github.com/plageon/HtmlRAG

#### 3.4.6 音视频：时序信号

**独有问题（与所有静态文档完全不同的维度）：**

- **ASR 转写准确率**：专业术语、方言、多人对话场景错误率高
- **说话人分离**：多人会议需区分"谁说了什么"
- **时间戳对齐**：需知道某段内容对应视频的哪个时间点（用于溯源和定位）

**独有方案：**

- **Whisper（OpenAI）**：开源 ASR，多语言转写 + 时间戳输出
- **pyannote.audio**：说话人分离，输出带说话人标签的转写
- **语义分段**：ASR 输出的长文本按语义主题分段（不按固定时长），每段保留起止时间戳作为元数据
- **视频关键帧**：按场景变化提取关键帧，VLM 描述画面内容，与对应时间段语音转写合并

### 3.5 跳过解析：视觉检索（ColPali / VisionRAG）

当版面极复杂、OCR 错误率高、传统解析方案都失效时，有一条完全不同的路径：**直接对文档页面图像做 Embedding，跳过 OCR 和文本提取**。

- **ColPali（arXiv:2407.01449）**：基于 PaliGemma，每个 PDF 页面视为图像，生成 patch 级视觉 Embedding，用 Late Interaction 计算 query 与页面 patch 的细粒度相似度。保留版面、表格、图表的空间信息，OCR 错误不再是问题
- **VisionRAG**：在 ColPali 基础上构建完整 OCR-free 检索系统，三层金字塔索引（页面级 → 区域级 → patch 级），可插拔不同视觉编码器

**适用场景：** 扫描件多、版面复杂、OCR 错误率高的场景。
**代价：** 需要 GPU 做视觉编码，存储成本比纯文本向量高。

详细架构与方案对比见 [[RAG高级技术#七、多模态-RAG]]。

> ColPali 论文：https://arxiv.org/abs/2407.01449
> VisionRAG 论文：https://arxiv.org/abs/2511.21121

### 3.7 结构感知解析与创新架构

#### BookRAG：文档层级树 + 知识图谱（港中文）

传统文档解析只提取文字内容，丢失了文档的原生结构。BookRAG 的思路：把文档还原成和目录一模一样的层级树——根节点是文档标题，一级节点是各章，二级节点是各小节，叶子节点是段落/表格/图片/公式。再从树节点中提取实体建成知识图谱，通过**图-树链接（GT-Link）**把每个实体锚定回它所在的章节节点。

检索时通过实体/关键词直接定位到对应的章节子树，把检索范围从"全文几万块"缩到"某一小节几十块"。

> 论文：https://arxiv.org/pdf/2512.03413

#### ChromaFs：虚拟文件系统（Mintlify）

**解决的问题：** 传统 RAG 只能"搜"不能"逛"——AI 只能拿到碎片化的 top-K 片段，无法浏览目录、查看完整文档。

**核心方案：** 用向量数据库模拟 UNIX 文件系统交互逻辑。文档结构按"页面=文件、章节=目录"构建 JSON 目录树存入 Chroma，`ls`/`cd`/`find` 在内存执行（毫秒级），`cat` 时从 Chroma 拉取分块拼接完整文档，`grep` 用双层优化（Chroma 粗过滤 + Redis 缓存精确匹配）。所有写操作抛只读错误保证无状态。

> 项目：https://www.mintlify.com/blog/how-we-built-a-virtual-filesystem-for-our-assistant

## 四、文本清洗

文本清洗是解析后、分块前的关键步骤，目标是消除噪声、统一格式，确保进入分块阶段的文本是干净且语义连贯的。==这一步看似简单，但每个子任务背后都有具体的技术细节==。

> [!info] 完整方法论详见 [[文本清洗]]
> 包括：页眉页脚的多特征联合识别、PDF 水印的几何与颜色特征检测、==重复内容检测的三个层级（字节级/chunk级/近似级）以及为什么近似去重对正式文档常常误伤==、不去重而靠元数据 + 检索时过滤的工程模式、版本链管理。本章节给出 RAG 视角下的核心要点。

### 4.1 噪声去除

#### 页眉页脚识别（PDF/扫描件）

PDF 里没有"这是页眉"的标签，只有坐标。==判断依据是跨页重复性 + 位置==。

**方法一：跨页文本比对（最常用）**

```python
from collections import Counter

# 提取每页所有文本块（含坐标）
page_blocks = [extract_text_with_coords(page) for page in pdf]

# 统计每段文本在多少页出现过
text_frequency = Counter()
for page in page_blocks:
    for block in page:
        # 规范化后再计数（去多余空格）
        text_frequency[" ".join(block.text.split())] += 1

# 出现 ≥ N 页（通常 N=2 或 ⌈总页数 × 0.3⌉）的文本块 → 候选页眉页脚
candidates = {text for text, count in text_frequency.items() if count >= 2}
```

**方法二：位置过滤**

页眉通常在 y 坐标 < 页面高度 10%，页脚在 y > 90%。结合跨页重复，精度更高：

```python
def is_header_or_footer(block, page_height):
    is_top = block.y_top < page_height * 0.10
    is_bottom = block.y_bottom > page_height * 0.90
    return is_top or is_bottom
```

**方法三：版面分析模型（最彻底）**

[[版面分析]] 模型（DocLayNet 训练的 RT-DETR）直接把页眉页脚识别为独立的区域类型（`page-header`、`page-footer`），无需跨页比对。==Docling、MinerU 都用这种方式==。

#### PDF 水印与装饰文本识别

水印没有固定位置规律，比页眉页脚难识别。

**方法一：颜色/透明度检测**

PDF 里的水印通常是==半透明文字==或==浅灰色==。pdfminer/PyMuPDF 能拿到每个文字块的颜色属性：

```python
def is_watermark_by_color(block):
    r, g, b = block.color
    lightness = (r + g + b) / 3
    return lightness > 0.7  # 浅色文字认为是装饰
```

**方法二：旋转角度检测**

水印文字经常是斜的（45°、-45°）。PDF 里每个文字块都有变换矩阵，可以检测旋转：

```python
def is_watermark_by_rotation(block):
    return abs(block.rotation_angle) > 10  # 旋转超过 10 度
```

**方法三：关键词规则**

`机密 / 仅供内部使用 / CONFIDENTIAL / DRAFT / COPY / © Copyright` 等词出现在非正文位置 → 水印或版权声明。

**方法四：版面分析模型**

DocLayNet 有 `watermark` 类别，直接预测。

==实际工程中通常组合方法 1+3 兜底，方法 4 作为更彻底的解法==。

#### HTML 导航与广告剥离

HTML 比 PDF 容易，因为 [[DOM]] 树本身有结构语义。

- **基于 DOM 选择器**：用 `nav`/`aside`/`footer`/`<div class="ad">` 等标签或 class/id 直接过滤
- **正文提取算法**：Trafilatura、Readability.js（Mozilla）通过==文本密度分析==识别正文区域
  - 核心思想：正文段落字数密集且连续，导航/广告稀疏且分散，按文本密度排序找出主要内容块

#### 短片段过滤

长度过短的文本片段（< 50 字符）通常是按钮文字、菜单项、孤立的元数据，对检索无价值，直接丢弃。

### 4.2 编码与格式规范化

#### 字符编码统一

所有文本规范化为 ==UTF-8==，处理三类常见问题：

```python
# 1. BOM 标记（Windows 记事本生成的 UTF-8 文件常带 BOM）
text = text.lstrip("﻿")

# 2. 编码自动检测（拿到的字节流不知道编码时）
import chardet
detected = chardet.detect(byte_data)
text = byte_data.decode(detected["encoding"])

# 3. 替换字符（解码失败时的 �）
text = text.replace("�", "")
```

#### 空白符规范化

```python
import re

# 多余空格、制表符、连续换行 → 单个
text = re.sub(r'[ \t]+', ' ', text)
text = re.sub(r'\n{3,}', '\n\n', text)  # 保留段落分隔

# 全角空格 → 半角
text = text.replace('　', ' ')

# 零宽空格、软连字符等不可见字符 → 删除
text = re.sub(r'[​-\u200F- ﻿]', '', text)
```

#### 标点符号统一

中英文标点的混用要根据上下文统一。==判断依据是周围字符的语种==：

```python
def normalize_punct(text: str) -> str:
    # 中文上下文里的英文逗号 → 中文逗号
    text = re.sub(r'(?<=[一-龥]),(?=[一-龥])', '，', text)
    # 英文上下文里的中文逗号 → 英文逗号
    text = re.sub(r'(?<=[a-zA-Z])，(?=[a-zA-Z])', ',', text)
    return text
```

#### Unicode 规范化

Unicode 有四种规范化形式（NFC/NFD/NFKC/NFKD）：

```python
import unicodedata

# NFKC：兼容性分解 + 重组合，把全角数字转半角、把"㎏"展开为"kg"
text = unicodedata.normalize('NFKC', text)
```

==生产环境推荐 NFKC==，能处理大多数 Unicode 变体字符问题。

### 4.3 重复内容检测

==关键决策：在 chunk 前做，在文档/段落粒度做==。

**为什么不在 chunk 后做：**
同一段话被切成两个 chunk，每个 chunk 单独看都不重复，但语义重复——chunk 后去重会漏掉。

**两个层次：**

```
文档级去重（先做）→ 段落级去重 → 分块 →（chunk 级去重可选，通常不做）
```

#### 精确去重：哈希

完全相同的文本用哈希秒杀：

```python
import hashlib

def doc_fingerprint(text: str) -> str:
    # 先规范化（去空白、小写），再哈希
    normalized = " ".join(text.lower().split())
    return hashlib.md5(normalized.encode()).hexdigest()

seen = set()
for doc in documents:
    fp = doc_fingerprint(doc.text)
    if fp in seen:
        skip(doc)  # 完全重复，跳过
    seen.add(fp)
```

#### 近似去重：MinHash + LSH

精确哈希解决不了"同一份合同的两个版本只改了日期"——哈希不同但语义 90% 重复。

**MinHash 的核心思想**：==用 k 个哈希函数对文本的 n-gram 集合取最小值，得到一个签名向量==。两个文本的签名向量越相似，Jaccard 相似度越高。

```python
from datasketch import MinHash, MinHashLSH

def text_to_minhash(text: str, num_perm=128) -> MinHash:
    m = MinHash(num_perm=num_perm)
    # 用 3-gram 作为集合元素（中文可用字粒度）
    for i in range(len(text) - 2):
        m.update(text[i:i+3].encode())
    return m

lsh = MinHashLSH(threshold=0.8, num_perm=128)  # 相似度 > 80% 认为重复

for doc_id, doc in enumerate(documents):
    m = text_to_minhash(doc.text)
    duplicates = lsh.query(m)  # 找已有的相似文档
    if duplicates:
        mark_as_near_duplicate(doc_id, duplicates)
    else:
        lsh.insert(doc_id, m)
```

==关键优势：MinHash + LSH 的时间复杂度是 O(n)，不是 O(n²)==——百万文档级别也能跑。

**LSH 的核心：分桶**

LSH 把签名向量切成 b 个 band，每个 band 哈希到一个桶。==两个文档至少有一个 band 哈希到同一个桶才会被对比==——避免两两暴力比较。`threshold=0.8` 等参数通过调 b 和 r 控制。

#### 跨文档去重的策略选择

发现重复后保留谁：
- **保留最新**：按 `updated_at` 元数据排序，留最新版本
- **保留最完整**：按字符数排序，留最长版本
- **保留权威**：按 `source_authority` 元数据排序（多源知识库适用）

详见 [[RAG高级技术#知识冲突处理]]。

### 4.4 语言与句子切分

> [!warning] 多语言 embedding 时代，传统"语言检测 → 选分词器 → 分句"的流程基本过时
> 现代 RAG 不再走"先检测语言再路由到不同分词工具"的老路。Embedding 模型自带 tokenizer，不需要 jieba/pkuseg 这种预分词。本节给出 LLM 时代的简化做法。

#### 语言检测：何时还需要做

==RAG 入库流程不需要为了选 embedding 而做语言检测==——主流多语言 embedding（BGE-M3 / Qwen3-Embedding / E5-multilingual）训练见过 100+ 语言，中英混合段落直接编码即可，向量空间互相可比。详见 [[RAG向量与Embedding#选型决策树]]。

仅剩两个场景需要：

| 场景 | 用途 | 做法 |
|------|------|------|
| 元数据打标签 | 让用户检索时按语言过滤（"只看中文资料"） | 字符 Unicode 范围判断即可 |
| 路由到不同 LLM | 中文走 Qwen，英文走 GPT-4 | 同上 |

```python
def detect_language(text: str) -> str:
    has_chinese = any('一' <= c <= '鿿' for c in text)
    has_japanese = any('぀' <= c <= 'ヿ' for c in text)
    has_latin = any('a' <= c.lower() <= 'z' for c in text)
    if has_japanese: return "ja"
    if has_chinese: return "zh-en" if has_latin else "zh"
    return "en"
```

==业务场景下用这种简单规则就够了==，不需要 fastText 这种 176 语种识别。

#### 分词器：RAG 入库流程不需要

| 工具 | 真实用途 | 是否 RAG 必需 |
|------|---------|--------------|
| jieba / pkuseg | BM25 倒排索引、关键词提取、传统 NLP 任务 | ❌ 仅混合检索的 BM25 路用得到 |
| spaCy 分词 | 词性标注、句法分析、NER | ❌ |
| HanLP | 综合 NLP 工具包 | ❌ |
| Embedding 模型自带 tokenizer | BPE / SentencePiece，模型内部自理 | ✅ 自动完成 |

==分块阶段输入字符级文本即可==，embedding 模型会自己 tokenize。jieba 等中文分词器只在==混合检索==中给 BM25 路用（详见 [[RAG检索策略#三、混合检索原理]]）。

#### 段落识别已在文档解析阶段完成

==文档解析（PDF/Word/HTML）输出的就是段落级别文本块==，版面分析已经隐式还原了段落边界。分块阶段不需要重新识别段落，直接基于已有边界做长度切分即可。

如果原始文本是纯文本流（如 ASR 输出、聊天记录）需要识别段落，用简单规则：

```python
import re

def split_paragraphs(text: str) -> list[str]:
    # 双换行 = 段落边界（最强信号）
    return [p.strip() for p in re.split(r'\n\s*\n', text) if p.strip()]
```

#### 句子级切分（按需）

句子粒度的切分不是分块阶段的必需步骤，==只在特定场景才用到==：

- [[RAG基础与架构#5-2-分块策略全景：按方法学分五层|句子窗口检索]]（Sentence Window Retrieval）
- 句子级相似度计算
- 段落跨句子的精细处理

**现代做法：一个工具处理所有语言**

```python
# 方案 A：通用正则（90% 场景够用）
import re
sentences = re.split(r'(?<=[。！？.!?])\s*', text)

# 方案 B：spaCy 多语言句切器（一个模型处理所有语言）
import spacy
nlp = spacy.load("xx_sent_ud_sm")
sentences = [sent.text for sent in nlp(text).sents]

# 方案 C：pysbd（专门的多语言句子分割库，22 种语言）
import pysbd
seg = pysbd.Segmenter(language="zh")  # 或 "en"
sentences = seg.segment(text)
```

==都不需要"先检测语言再选工具"==，一个工具直接吃任何语言的输入。

#### LangChain 的事实标准：RecursiveCharacterTextSplitter

实际工程中绝大多数 RAG 系统直接用：

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", ".", "!", "?", "，", ",", " ", ""]
)
chunks = splitter.split_text(text)
```

==字符级递归切分，对中英混合一视同仁==。先按双换行（段落）切，切不下按句末标点，再不行按字符。==这就是 LLM 时代分块阶段的事实标准==，没有"分词 → 分句 → 拼回"的复杂流程。

### 4.5 内容增强（可选）

#### 缩写展开

将领域缩写展开为全称，提升检索时的语义匹配度：

```python
# 维护领域缩写表
abbreviations = {
    "RAG": "检索增强生成（Retrieval-Augmented Generation）",
    "LLM": "大语言模型（Large Language Model）",
}

# 替换时只替换首次出现，后续保留缩写避免膨胀
def expand_first_occurrence(text: str, abbr_dict: dict) -> str:
    seen = set()
    for abbr, full in abbr_dict.items():
        if abbr not in seen and abbr in text:
            text = text.replace(abbr, full, 1)  # 只替换第一次
            seen.add(abbr)
    return text
```

#### 停用词策略

==RAG 场景中过度移除停用词反而损害语义==。建议：
- 仅在 BM25 等关键词检索分支移除常见停用词（"的"、"了"、"the"、"a"）
- ==向量检索分支不要移除停用词==——Embedding 模型已经在带停用词的文本上训练
- 永远不做词干提取（"running" → "run"）——破坏语义信息

#### 元数据提取

为每段文本附加结构化元数据，用于检索过滤和来源追溯。详见 [[RAG基础与架构#5.4 通用工程参数]] 的元数据模板。

## 五、分块策略（Chunking）

> 切分不只是"设置一下 chunk_size"，它是需要分析文档特点、设计处理策略的工程模块。切错了，后面做再多优化，全是在地基不稳的楼上加砖。

### 5.1 为什么分块是 RAG 的最大杠杆

分块决定了向量化的粒度，进而决定了检索能匹配的最小语义单位。同一份文档、同一个 Embedding 模型、同一套检索策略，仅仅切分方式不同，召回质量可能天差地别。

**分块设计要同时面对三对矛盾：**

| 矛盾 | 一端 | 另一端 |
|------|------|--------|
| 检索精度 vs 上下文完整 | chunk 越小，向量越精准（语义单一）| chunk 越大，给 LLM 的上下文越完整 |
| 粒度细 vs 成本 | 粒度越细（如命题级），匹配越准 | 切得越细，存储 / 向量化 / LLM 调用成本越高 |
| 通用 vs 文档特化 | 一刀切的规则切分实现简单 | 结构感知/语义切分需要针对文档类型定制 |

所有分块策略本质上都在这三对矛盾里寻找平衡点。后面 5.2 的五层方法学就是按"如何决定 chunk 边界"逐步深入。

---

### 5.2 切分边界的四种方法学

按"决定 chunk 边界的方式"，分块方法分为四层，每一层比上一层更智能但代价也更高：

```
L1 规则切分     → 按字符/token 数或分隔符切（最简单）
L2 结构感知     → 利用文档原生结构（标题层级、表格、列表）
L3 语义切分     → 模型判断主题边界
L4 命题级切分   → LLM 分解为原子事实
```

==L1-L4 在同一维度==——都是决定 chunk 边界画在哪。==与之正交==的还有两类策略，下文 5.3 单独讲：
- ==chunk 内容增强==（Contextual Retrieval、Late Chunking）：不改边界，丰富每个 chunk 的语义表达
- ==在线检索增强==（父子分块、句子窗口检索）：检索阶段动态扩展上下文，详见 [[RAG检索策略#五、检索时的上下文扩展]]

#### L1 规则切分（Rule-based）

**做法：** 按固定长度（chunk_size）或固定分隔符（句号/换行/双换行）切分。最常见实现：LangChain 的 `RecursiveCharacterTextSplitter`。

**Overlap 是 L1 的标配：** 相邻 chunk 保留 10-20% 重叠区域，避免关键信息恰好落在边界处被截断。进阶做法是==基于句子边界的智能 overlap==——固定 overlap 可能切在句子中间，智能 overlap 在确定重叠区域后向后扫描到最近句子结尾才截断。

**优势：** 实现简单、速度快、零成本、可预测。

**劣势：** 不理解语义和结构，常在句子中间截断，表格被打散，列表项与其前导句分离。

**适用：** 同质化短文档（如 FAQ、新闻段落）的 baseline 方案，且==所有上层方案都建议叠加 Overlap 兜底==。

#### L2 结构感知切分（Structure-aware）

**核心思路：** 利用文档的原生结构信号，让 chunk 边界与文档逻辑边界对齐。

**四类通用做法：**

**1. 层级结构识别**

通过多策略融合识别章节标题——数字编号（`1.1`、`1.1.1`）、中文编号（`第X章/节/条`、`（X）`）、混合编号（`a)`、`i.`）、Markdown 标题（`#`/`##`）、HTML 标签（`<h1>`-`<h6>`）。切分原则：**同一章节内容尽量放在同一个 chunk，跨章节绝不合并**。

**2. 超长章节递归细切**

章节超过 chunk_size 上限时，按层级降级切分：先按子小节边界切，没有子小节则按段落，最后按句子。规则切分常用 `RecursiveCharacterTextSplitter` 实现这种降级。

高价值章节（如技术规范的核心条款、合同的责任义务部分）可以适当放宽 chunk_size 上限，避免任何截断。

**3. 表格专项处理**

- 小表格（≤ 300 token）：整体作为一个 chunk
- 大表格（跨页或超 300 token）：按行切分，但==每个 chunk 都要复制表头==
- 不复制表头的后果：LLM 收到没有表头的数据行，不知道字段含义，只能胡编

**4. 列表项前导句合并**

问题示例：
```
以下情况不在保障范围：
（1）地震
（2）战争
```

如果把 `（1）地震` 单独成 chunk，向量里没有"不在保障范围"这个关键语义，LLM 会误判为"地震在保障范围内"。

方案：识别列表结构，把前导句和所有列表项合并为一个 chunk；如果超长，至少保留前导句到每个 chunk 的开头。

**适用：** 有清晰结构信号的文档（技术手册、法律文件、规范、API 文档、Markdown 笔记）。这是工业界 RAG 系统的主流方案。

#### L3 语义切分（Semantic Chunking）

**核心思路：** 不依赖人工规则，让模型判断"哪里是主题边界"。

**两种实现路线：**

**路线一：Embedding 相似度突变检测**

用 Embedding 模型计算相邻句子的向量相似度，相似度突变处（语义跳跃点）作为切分边界。LangChain `SemanticChunker` 是典型实现。

- 优势：完全自动，不依赖文档结构
- 劣势：每对相邻句子都要算 Embedding，计算成本高；阈值难调（不同文档类型的语义跳跃幅度不同）

**路线二：判别式分块模型（最新 SOTA）**

训练专用模型直接预测"相邻句子间是否存在主题边界"，详见 5.3。

**适用：** 文档无清晰结构（聊天记录、会议纪要、口语转写、自由写作）。

#### L4 命题级切分（Proposition-based）

**核心思路：** 用 LLM 把文档分解为==原子命题==（Proposition）——每个命题是自包含的事实陈述，不依赖上下文即可理解。

**示例：**
```
原文："苹果在 2023 年 Q4 营收增长 40%，主要受 iPhone 15 驱动。"

命题化：
- "苹果 2023 年 Q4 营收增长 40%"
- "iPhone 15 是苹果 2023 年 Q4 营收增长的主要驱动力"
```

来自 Dense X Retrieval（Chen et al., EMNLP 2024，arXiv:2312.06648）。每个命题语义密度最高，检索时精准匹配。

**优势：** 检索粒度最细，命题之间互不干扰，能解决"一段话讲多件事但只想检索其中一件"的问题。

**劣势：** LLM 调用成本高，处理速度慢；命题之间的关联信息可能丢失。

**适用：** 高价值文档的精确事实问答（财报、合同、技术规范、医学文献）。

#### 横向对比

| 层 | 方法 | 核心思路 | 适合场景 | 代价 |
|----|------|---------|---------|------|
| L1 | 规则切分 | 固定长度/分隔符 + Overlap 兜底 | 同质化短文档、baseline、所有方案的兜底 | 不理解结构，易截断语义 |
| L2 | 结构感知 | 利用标题/表格/列表的原生结构 | 有结构的文档（手册、法规、Markdown） | 需为不同文档类型定制规则 |
| L3 | 语义切分（Embedding） | 句间相似度突变检测 | 无结构文档（聊天、转写） | 计算成本高、阈值难调 |
| L3 | 语义切分（判别式） | 训练模型预测主题边界 | 同上，且有训练数据 | 推理快但需训练 |
| L4 | 命题级 | LLM 分解为原子命题 | 高价值精确问答 | LLM 成本高、可能丢失关联 |

> [!info] L1-L4 是==决定 chunk 边界==的四种方法学
> ==与 L1-L4 正交==的"chunk 内容增强"策略（Contextual Retrieval、Late Chunking）见下文 5.3 节——这些不是更高一层的切分方式，而是==可以叠加在 L1-L4 任何一层之上==的内容/编码增强。
>
> ==与 L1-L4 正交==的"在线检索增强"策略（父子分块、句子窗口检索）属于检索阶段而非切分阶段，详见 [[RAG检索策略#五、检索时的上下文扩展]]。

### 5.3 chunk 内容增强（与切分边界正交）

**核心定位**：本节方法==不决定 chunk 边界画在哪==——边界由 L1-L4 决定。本节方法决定==每个 chunk 在向量化时如何编码==，让向量携带更多信息。

==与 L1-L4 是正交关系==：用 L1 规则切分照样可以叠加 Contextual Retrieval；用 L4 命题切分也能叠加 Late Chunking。

**两个主流方法（概念）：**

#### Contextual Retrieval（Anthropic 2024）

向量化前用 LLM 看整篇文档，==为每个 chunk 生成 50-100 token 的上下文描述==，把"上下文 + 原 chunk"一起向量化：

```
原 chunk: "营收同比增长 40%"
   ↓ LLM 看整篇文档生成上下文
增强后:   "[本段来自 ACME 公司 2023 Q4 财报，讨论北美区业务] 营收同比增长 40%"
   ↓ 向量化
带文档级上下文的向量
```

==关键技术==：用 Prompt Caching 把整篇文档作为缓存的 system prompt，处理 N 个 chunk 只完整读文档 1 次，成本可降 80-90%。

适用：结构化长文档（财报、合同、技术规范）+ 精确匹配场景。

#### Late Chunking（Jina AI 2024）

==反过来——"先编码再切"==。用长上下文 Embedding 模型先对整个文档做 token 级编码，每个 token 的向量已包含全文语境，再在向量层面按 chunk 边界做 mean pooling。

```
传统："这家公司的营收增长 40%" 单独编码 → 不知道是哪家公司
Late Chunking：在"ACME 财报"全文中编码 → "这家公司"的向量已经知道指代 ACME
```

适用：代词/指代密集的文档（法律、政策、学术），且 Embedding 模型支持长上下文（BGE-M3 8K、Qwen3-Embedding 32K）。

> [!tip] 算法细节、论文出处、实测效果详见 [[分块策略前沿]]
> 包含四种前沿方法的完整解读：Contextual Retrieval / Late Chunking / 判别式分块（金山办公 2026）/ 命题化切割（Dense X Retrieval, EMNLP 2024）。本节只给概念，工程实施请看前沿文档。

### 5.4 通用工程参数

#### chunk_size 选择

chunk_size 不是通用参数，要结合文档类型和 Embedding 模型上下文长度：

| 文档类型 | 句子平均长度 | 推荐 chunk_size |
|---------|-------------|----------------|
| 通用中文文档 | 20-30 字 | 512 token（约 2-3 段） |
| 英文技术文档 | 较短 | 512-768 token |
| 法律/合规/金融文档 | 长句多，每句 40-50 字 | 1024-1536 token |
| 对话/聊天/转写 | 短句多 | 256-512 token，配合句子窗口 |

三个约束要同时满足：
1. **检索精度**：chunk 越长语义越分散，Embedding 区分度下降
2. **语义完整性**：太小切断完整语句，太大包含多个主题
3. **token 预算**：LLM 上下文有限，512 token × 5 chunk + prompt ≈ 4000 token，给生成留足空间

#### Overlap 推荐

**100 token 是性价比最优点**：召回率提升明显，存储仅增加约 10-15%。再往上收益递减。

配合智能 overlap（向后扫描到句子结尾才截断）效果更好，避免 overlap 区域本身又被切断。

#### chunk 元数据通用模板

元数据是检索过滤和答案溯源的基础，离线阶段不打标签，在线阶段想过滤都没法过滤。

```python
@dataclass
class ChunkMetadata:
    doc_id: str          # 文档 ID
    chunk_id: str        # chunk 唯一 ID
    section_path: str    # 层级路径，如 "第3章 架构 > 3.2 数据层"
    chunk_type: str      # "text" | "table" | "list" | "code" | "figure_caption"
    prev_chunk_id: str   # 前一个 chunk 的 ID（上下文扩展）
    next_chunk_id: str   # 后一个 chunk 的 ID
    parent_chunk_id: str # 父 chunk ID（父子分块场景）
    token_count: int
    source_path: str     # 来源文档路径
    page_range: str      # 来源页码范围（PDF 等有页码的格式）
    source_authority: int  # 来源权威等级（多源冲突解决用，详见 [[RAG高级技术#知识冲突处理]]）
    updated_at: str      # 更新时间（时效性优先用）
```

**字段用途：**
- `section_path` → 答案溯源、按章节过滤检索
- `chunk_type` → 不同类型走不同检索策略（表格用 BM25、文本用向量）
- `prev/next_chunk_id` → 检索到半截信息时拉取相邻 chunk 补全
- `parent_chunk_id` → 父子分块的关联键
- `source_authority` / `updated_at` → 多源冲突解决（权威优先、时效优先）

业务场景下可以追加领域字段（如"是否关键条款"、"产品线分类"等），但通用模板应保持精简。

### 5.5 选型决策树

按文档特征选方案（==只列离线分块策略，配套的在线检索策略——父子分块、句子窗口检索——见 [[RAG检索策略#五、检索时的上下文扩展]]==）：

```
文档是否有清晰结构（标题层级/Markdown/HTML）？
├── 有 → L2 结构感知（手册、法规、API 文档、Wiki）
│       └── 高价值精确问答场景叠加 L4 命题化 或 ==Contextual Retrieval（5.3）==
│
└── 无 → 文档长度？
    ├── 中短（< 8K token） → L3 语义切分（判别式优先于 Embedding 相似度）
    │                       或 ==Late Chunking（5.3）==（如有长上下文 Embedding 模型）
    └── 超长 → 金山判别式分块 + VFSC 超长块检索

通用兜底：所有方案都建议叠加 L1 Overlap（100 token + 智能边界）
```

**典型组合方案：**

| 场景 | 离线分块（本章）+ 在线检索增强（详见 [[RAG检索策略]]） |
|------|---------|
| 大规模通用知识库 | L2 结构感知 + L1 Overlap + 重排兜底 |
| 企业内部文档（Wiki/规范/手册） | L2 结构感知 + 检索时叠加==父子分块== |
| 法律/金融/医疗精确问答 | L2 结构感知 + Contextual Retrieval + BM25 混合检索 |
| 财报/合同的事实抽取 | L4 命题化 + 关联回原段落 |
| 会议纪要 / 客服对话 / 口语转写 | L3 判别式语义切分 + 检索时叠加==句子窗口== |
| 跨指代多的长文档 | Late Chunking |

所有方案都不是孤立的，工程上常常组合使用。最常见的组合是 ==L2 结构感知 + L1 Overlap + 检索时父子分块或 Contextual Retrieval==——结构感知保证 chunk 边界与逻辑边界对齐，Overlap 兜底边界截断，配套的检索增强提供上下文。

## 六、Prompt 拼接与生成

检索和重排完成后，需要将检索到的 chunks 组装成 prompt 送入 LLM 生成最终答案。这一步看似简单，但设计不当会导致"检索到了正确内容，LLM 却给出错误答案"。

### 上下文窗口预算管理

LLM 的上下文窗口是有限资源，需要合理分配：

```
可用 chunk 预算 = 总窗口大小 - 系统提示 - 用户查询 - 预留生成空间
```

根据预算动态调整检索的 chunk 数量和长度。自适应压缩方法（ACC-RAG）根据查询复杂度动态调整压缩率——简单查询压缩率低（保留更多细节），复杂查询压缩率高（覆盖更多来源）。

### Lost in the Middle 问题

研究表明，LLM 对 prompt 开头和结尾的内容关注度更高，中间部分容易被忽视。

**解决方案：**
- 将最相关的 chunk 放在 prompt 的开头或结尾位置
- 使用重排序模型确保 chunk 按相关性严格排序
- 对于多个高相关 chunk，采用交错排列（最相关→次相关→...→第二相关）

### 去重与冗余处理

当多个 chunks 高度相似或包含重复信息时，应检测并移除冗余内容，保留最有信息量的版本。避免相似信息占用宝贵的 token 预算。

### 来源引用与元数据保留

在组织 chunks 时保留文档来源、页码、时间戳等元数据，在最终答案中格式化引用（如 `[来源1]`、`[页码X]`），便于用户验证信息真实性和追溯。

### Prompt 模板设计

构建标准化的 prompt 模板，结构为：

```
系统角色定义（你是XX领域的专家助手...）
→ 检索上下文（按优先级排序的 chunks，用分隔符划分）
→ 用户查询
→ 输出格式要求（引用来源、不确定时说明、不编造）
```

关键原则：
- 明确指示 LLM 只基于提供的上下文回答，不使用自身知识
- 要求标注答案来源（哪个文档、哪个章节）
- 对于无法回答的问题，要求明确说"根据现有资料无法回答"而非编造

### 上下文压缩

当检索内容超过预算时的处理策略：
- **提取式压缩**：从每个 chunk 中提取与查询最相关的关键句子
- **摘要式压缩**：用 LLM 对 chunk 生成针对当前查询的摘要
- **渐进式加载**：先用 top-3 chunk 生成初步答案，如果置信度不够再追加更多 chunk

**多文档压缩两种模式：**

| 模式 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| Map-Reduce | 并行对每个 chunk 独立摘要，再汇总 | 速度快（并行），适合大量文档 | 汇总时可能丢失跨 chunk 的关联信息 |
| Refine（串行迭代） | 先摘要第一个 chunk，再逐步将后续 chunk 融入精炼 | 能捕捉跨 chunk 的信息关联 | 串行执行，延迟高；早期 chunk 影响权重更大 |

选择依据：文档数量多且独立性强 → Map-Reduce；文档间有强关联（如同一事件的多篇报道）→ Refine。
