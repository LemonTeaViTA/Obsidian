# MySQL 锁机制

> 本文专讲 MySQL 各类锁的机制、加锁规则与死锁排查。事务、MVCC、隔离级别请见 [[事务与MVCC]]。

## 锁的分类

### MySQL 中有哪几种锁？

MySQL 中有多种类型的锁，按锁粒度划分有表锁、行锁；按加锁机制划分有乐观锁和悲观锁；按兼容性划分有共享锁和排他锁。

- 表锁：锁定整个表，资源开销小，加锁快，但并发度低，不会出现死锁；适合查询为主、少量更新的场景（如 MyISAM 引擎）。表锁细分为表共享读锁（S锁）和表独占写锁（X锁）。
- 行锁：锁定单行或多行，开销大、加锁慢，可能出现死锁，但并发度高（InnoDB 默认支持）。行锁细分为记录锁、间隙锁和临键锁。
- 共享锁（S锁/读锁）：允许多个事务同时读取数据，但阻塞写操作。语法：`SELECT ... LOCK IN SHARE MODE`
- 排他锁（X锁/写锁）：独占数据，阻塞其他事务的读写。语法：`SELECT ... FOR UPDATE`
- 乐观锁：假设冲突少，通过版本号或 CAS 机制检测冲突。
- 悲观锁：假设并发冲突频繁，先加锁再操作 `SELECT FOR UPDATE`。

> [!tip] 一图记住锁的分类
> 粒度（表/行/页）× 模式（共享/排他）× 思想（悲观/乐观）。MySQL 面试问"有哪些锁"时，按这三个维度展开回答最稳。

### 全局锁了解吗？

全局锁就是对整个数据库实例进行加锁，整个数据库将处于只读状态，所有写操作都会被阻塞。在进行全库备份或数据迁移时，可以使用全局锁来保证数据的一致性。

```sql
-- 锁定整个数据库
FLUSH TABLES WITH READ LOCK;

-- 执行备份操作
-- mysqldump -u username -p database_name > backup.sql

-- 释放全局锁
UNLOCK TABLES;
```

### 表锁了解吗？

表锁常见于 MyISAM 引擎，InnoDB 也可以手动通过 `LOCK TABLES` 加锁。适合读多写少、全表扫描或者表结构变更的场景。

```sql
-- 共享锁：允许多个事务同时读表，但不允许写操作
LOCK TABLES table_name READ;
SELECT * FROM table_name;
UNLOCK TABLES;

-- 排他锁：只允许一个事务进行写操作
LOCK TABLES table_name WRITE;
-- INSERT/UPDATE/DELETE，其他会话读写均阻塞
UNLOCK TABLES;
```

对于 InnoDB 引擎，无索引的 UPDATE/DELETE 可能会导致锁升级为表锁：

```sql
UPDATE innodb_table SET name = 'new' WHERE name = 'old'; -- 全表扫描，退化为表锁
```

> [!warning] InnoDB 行锁退化为表锁是踩坑高发区
> InnoDB 的行锁是**通过给索引项加锁来实现的**。如果 WHERE 条件没用上索引，InnoDB 会被迫对每一行都加锁，等同于表锁。线上更新一定要确认 EXPLAIN 走了索引，否则一不小心就锁全表。

### 说说 MySQL 的行锁？

行锁是 InnoDB 存储引擎中最细粒度的锁，它锁定表中的一行记录，允许其他事务访问表中的其他行。底层是通过给索引加锁实现的，只有通过索引条件检索数据时，InnoDB 才能使用行级锁，否则会退化为表锁。

```sql
-- 加排他锁
START TRANSACTION;
SELECT * FROM your_table WHERE id = 1 FOR UPDATE;
UPDATE your_table SET column1 = 'new_value' WHERE id = 1;
COMMIT;

-- 加共享锁
START TRANSACTION;
SELECT * FROM your_table WHERE id = 1 LOCK IN SHARE MODE;
COMMIT;
```

#### select for update 有什么需要注意的？

第一，必须在事务中使用，否则锁会立即释放。第二，使用时必须注意是否命中索引，否则可能锁全表：

```sql
-- name 没有索引，会退化为表锁
SELECT * FROM user WHERE name = '王二' FOR UPDATE;
```

### 说说记录锁吧？

记录锁是行锁最基本的表现形式，当使用唯一索引或者主键索引进行等值查询时，MySQL 会为该记录自动添加排他锁，禁止其他事务读取或者修改锁定记录。

```sql
SELECT * FROM table WHERE id = 1 FOR UPDATE;  -- 加X锁
UPDATE table SET name = '王二' WHERE id = 1;  -- 隐式加X锁
```

### 间隙锁了解吗？

间隙锁用于在范围查询时锁定记录之间的"间隙"，防止其他事务在该范围内插入新记录。仅在可重复读及以上的隔离级别下生效，主要用于防止幻读。

```sql
-- 事务A
BEGIN;
SELECT * FROM orders WHERE amount BETWEEN 1000 AND 2000 FOR UPDATE;

-- 事务B 尝试插入会被阻塞
INSERT INTO orders VALUES (null, 1500, 'pending'); -- 阻塞
```

假设表中有 id: 1, 6, 8, 12 四条数据，间隙锁会锁住：`(−∞, 1)`、`(1, 6)`、`(6, 8)`、`(8, 12)`、`(12, +∞)`。

#### 执行什么命令会加上间隙锁？

在可重复读隔离级别下，执行 `FOR UPDATE` / `LOCK IN SHARE MODE` 等加锁语句，且查询条件是范围查询时，就会自动加上间隙锁：

```sql
SELECT * FROM user WHERE score > 100 FOR UPDATE;
SELECT * FROM user WHERE id BETWEEN 10 AND 20 LOCK IN SHARE MODE;
DELETE FROM user WHERE score < 50;
```

### 临键锁了解吗？

临键锁是记录锁和间隙锁的结合体，锁住的是索引记录和索引记录之间的间隙，区间为左开右闭，例如 `(1,3]`。

```sql
SELECT * FROM table WHERE id BETWEEN 5 AND 10 FOR UPDATE;
```

MySQL 默认的行锁类型就是临键锁。当使用唯一索引的等值查询匹配到一条记录时，临键锁会退化成记录锁；如果没有匹配到任何记录，会退化成间隙锁。

> [!tip] 临键锁的退化规律
> - 唯一索引等值查询且命中 → 退化为**记录锁**
> - 唯一索引等值查询未命中 → 退化为**间隙锁**
> - 唯一索引范围查询、普通索引任何查询 → 保持**临键锁**
> 这条规律是面试官追问"为什么会死锁"时的答题钥匙。

### 意向锁是什么知道吗？

意向锁是一种表级锁，表示事务打算对表中的某些行数据加锁，但不会直接锁定数据行本身。由 InnoDB 自动管理，当事务需要添加行锁时，会先在表上添加意向锁，这样当要添加表锁的时候，可以通过查看表上的意向锁快速判断是否有冲突，而无需逐行检查。

| 兼容关系 | 意向共享锁 | 意向排他锁 | 共享锁(表级) | 排他锁(表级) |
|---------|----------|----------|------------|------------|
| 意向共享锁 | 兼容 | 兼容 | 兼容 | 冲突 |
| 意向排他锁 | 兼容 | 兼容 | 冲突 | 冲突 |
| S锁 | 兼容 | 冲突 | 兼容 | 冲突 |
| X锁 | 冲突 | 冲突 | 冲突 | 冲突 |

```sql
-- 事务A 获取某行的排他锁，自动加 IX 锁
BEGIN;
SELECT * FROM users WHERE id = 6 FOR UPDATE;

-- 事务B 尝试加表锁，发现表上有 IX 锁，与 S 锁冲突，直接阻塞而无需扫描全表
LOCK TABLES users READ;
```

### MySQL 的乐观锁和悲观锁了解吗？

悲观锁是一种"先上锁再操作"的保守策略，假设数据被外界访问时必然会产生冲突，因此在数据处理过程中全程加锁。MySQL 中的行锁和表锁都是悲观锁。

乐观锁假设并发操作不会总发生冲突，不会在读取数据时加锁，而是在提交更新时才检查数据是否被其他事务修改过。乐观锁并不是 MySQL 内置的锁机制，而是通过程序逻辑实现的，常见的实现方式有版本号机制和时间戳机制。

#### 如何通过悲观锁和乐观锁解决库存超卖问题？

悲观锁通过 `SELECT ... FOR UPDATE` 在查询时直接锁定记录：

```sql
BEGIN;
SELECT stock FROM products WHERE id = 1 FOR UPDATE;
INSERT INTO orders (user_id, product_id) VALUES (123, 1);
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

乐观锁通过在表中增加 version 字段作为判断条件：

```sql
SELECT stock, version FROM products WHERE id = 1;

UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 旧版本号;
```

时间戳机制：

```sql
UPDATE products SET stock = stock - 1, update_time = NOW()
WHERE id = 1 AND update_time = 旧时间戳;
```

> [!tip] 高并发秒杀为什么常用乐观锁
> 悲观锁会让请求串行排队，吞吐量极低；乐观锁失败重试只浪费 CPU 不阻塞连接。秒杀场景的常见做法是：Redis 预扣库存（分布式锁）→ MySQL 用 `UPDATE ... SET stock = stock - 1 WHERE id = ? AND stock > 0` 这种带条件的 CAS 更新落库。

## 死锁

### 死锁是怎么产生的？

死锁需要同时满足四个条件：互斥（资源同时只能被一个事务持有）、占有并等待（持有锁的同时等待其他锁）、不可剥夺（锁只能由持有者主动释放）、循环等待（多个事务形成环形等待链）。

MySQL 中常见的三种触发场景：

**场景一：加锁顺序不一致（最常见）**

```sql
-- 事务A：先锁 id=1，再锁 id=2
UPDATE account SET balance = balance - 10 WHERE id = 1;
UPDATE account SET balance = balance + 10 WHERE id = 2;

-- 事务B：先锁 id=2，再锁 id=1（顺序相反，形成循环等待）
UPDATE account SET balance = balance - 10 WHERE id = 2;
UPDATE account SET balance = balance + 10 WHERE id = 1;
```

**场景二：间隙锁冲突**

在可重复读隔离级别下，两个事务都持有同一范围的间隙锁，然后都尝试在对方的间隙内插入数据，互相阻塞形成死锁：

```sql
-- 假设表中有 id: 5, 10，两个事务都执行范围查询加了 (5,10) 的间隙锁
-- 事务A 尝试插入 id=7，事务B 也尝试插入 id=8，互相等待
```

**场景三：唯一索引并发插入**

两个事务并发插入相同的唯一键，一个等待另一个回滚，若两个事务互相等待对方释放，就会死锁。

### 遇到过 MySQL 死锁问题吗，你是如何解决的？

遇到过。通过 `SHOW ENGINE INNODB STATUS\G` 查看死锁信息，定位到是加锁顺序不一致导致的，最后通过统一所有事务的加锁顺序解决了这个问题。

InnoDB 检测到死锁后会自动回滚代价较小的事务（undo log 量少的那个），让另一个事务继续执行。除了调整加锁顺序，还可以：缩短事务长度减少持锁时间、为高并发写操作加应用层分布式锁、或适当降低隔离级别（读已提交下没有间隙锁，可消除场景二）。

### InnoDB 是如何检测和处理死锁的？

InnoDB 使用等待图（Wait-For Graph, WFG）算法来检测死锁。等待图是一个有向图，图中的节点代表事务，有向边代表"事务 A 正在等待事务 B 释放锁"。每当一个事务请求锁但被阻塞时，InnoDB 就会在等待图中添加一条边，并立即检测图中是否存在环路。如果存在环路，就说明发生了死锁。

```text
事务A --等待--> 事务B
  ^                |
  |     等待       |
  +-------<--------+
  检测到环路 → 死锁！
```

#### 等待图的检测流程是怎样的？

当事务请求的锁与其他事务持有的锁冲突时，InnoDB 会执行以下步骤：

1. 在等待图中为当前事务添加一条指向持锁事务的有向边
2. 从当前事务出发，沿着有向边进行深度优先搜索（DFS）
3. 如果搜索过程中回到了起始事务（发现环路），则判定为死锁
4. 如果没有环路，当前事务进入等待状态

这个检测过程的时间复杂度与等待图中的边数成正比，通常非常快。

#### InnoDB 检测到死锁后怎么处理？

InnoDB 会从环路中选择一个代价最小的事务作为"牺牲者"进行回滚。代价的衡量标准是事务已经产生的 undo log 量——undo log 越少，说明事务修改的数据越少，回滚的代价越低。被回滚的事务会收到一个错误（`ERROR 1213: Deadlock found`），应用层需要捕获这个错误并重试事务。

```java
// Spring 中处理死锁的标准姿势：捕获 DeadlockLoserDataAccessException 后重试
@Retryable(value = DeadlockLoserDataAccessException.class,
           maxAttempts = 3,
           backoff = @Backoff(delay = 100, multiplier = 2))
@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    accountMapper.deduct(fromId, amount);
    accountMapper.add(toId, amount);
}
```

> [!warning] 应用层必须做死锁重试
> InnoDB 帮你回滚了死锁事务并不等于"修复了"——业务方收到的是一个 `ERROR 1213` 异常，必须由应用层捕获后重试。Spring 用 `@Retryable` 或 Resilience4j 的 retry 都可以，重试时要带退避（backoff），避免重新撞车。

#### 死锁检测相关的参数有哪些？

```sql
-- 是否开启死锁检测（默认 ON）
SHOW VARIABLES LIKE 'innodb_deadlock_detect';

-- 锁等待超时时间（默认 50 秒），作为死锁检测的兜底机制
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';
```

`innodb_deadlock_detect = ON` 时，InnoDB 会主动检测死锁并立即回滚牺牲者。如果关闭死锁检测（高并发热点行场景下检测本身可能成为性能瓶颈），则只能依赖 `innodb_lock_wait_timeout` 超时机制来打破死锁，但响应会慢很多。

#### 如何预防死锁？

第一，统一加锁顺序：所有事务按照相同的顺序访问表和行，避免形成循环等待。

第二，缩短事务长度：事务越短，持锁时间越短，死锁概率越低。避免在事务中执行耗时操作（如远程调用、大量计算）。

第三，降低隔离级别：将隔离级别从可重复读降为读已提交，可以消除间隙锁，减少间隙锁冲突导致的死锁。

第四，合理使用索引：确保 UPDATE/DELETE 语句命中索引，避免锁升级为表锁，缩小锁的范围。

第五，应用层加锁：对于高并发的热点数据，可以在应用层使用分布式锁（如 Redis 锁）来串行化访问，从根源上避免数据库层面的锁冲突。

## DDL 与锁

### 哪些 ALTER 操作会导致锁表？怎么避免？

ALTER TABLE 是 DDL（数据定义语言）操作，不同类型的 ALTER 对表的锁定程度差异很大。MySQL 5.6 之后引入了 Online DDL，大部分 ALTER 操作不再需要全程锁表，但仍有一些操作会导致长时间锁表。

#### 会导致锁表的 ALTER 操作

第一，修改列的数据类型，需要重建整张表，全程持有排他锁（MDL 写锁）：

```sql
-- 锁表：修改列类型，需要重建表数据
ALTER TABLE orders MODIFY COLUMN amount DECIMAL(12,2);
```

第二，添加或删除主键，需要重建聚簇索引，全程锁表：

```sql
-- 锁表：重建聚簇索引
ALTER TABLE orders DROP PRIMARY KEY, ADD PRIMARY KEY (new_id);
```

第三，转换字符集，需要逐行重写数据：

```sql
-- 锁表：逐行转换编码
ALTER TABLE users CONVERT TO CHARACTER SET utf8mb4;
```

第四，早期版本（MySQL 5.5 及以前）的添加/删除索引操作也会锁表，但 5.6+ 已支持 Online DDL。

#### 不会长时间锁表的 ALTER 操作

以下操作在 MySQL 5.6+ 中支持 Online DDL，只在操作开始和结束时短暂获取 MDL 写锁，执行期间允许 DML 并发读写：

```sql
-- 在线添加索引（INPLACE 算法，不锁表）
ALTER TABLE orders ADD INDEX idx_user_id (user_id), ALGORITHM=INPLACE, LOCK=NONE;

-- 在线添加列（追加到末尾，不锁表）
ALTER TABLE orders ADD COLUMN remark VARCHAR(255);

-- 在线删除索引
ALTER TABLE orders DROP INDEX idx_user_id;
```

#### Online DDL 的三种算法

| 算法 | 是否重建表 | 是否允许并发 DML | 适用场景 |
|------|-----------|-----------------|---------|
| INSTANT | 否，只修改元数据 | 允许 | 追加列（MySQL 8.0.12+） |
| INPLACE | 可能重建 | 大部分允许 | 添加/删除索引、添加列等 |
| COPY | 是，创建临时表 | 不允许（锁表） | 修改列类型、转换字符集等 |

```sql
-- 查看 ALTER 操作会使用哪种算法
ALTER TABLE orders ADD COLUMN remark VARCHAR(255), ALGORITHM=INSTANT;
-- 如果不支持 INSTANT，MySQL 会报错，可以降级尝试 INPLACE
```

#### 怎么避免 ALTER 操作锁表？

第一，使用 Online DDL 语法，显式指定 `ALGORITHM=INPLACE, LOCK=NONE`，如果操作不支持在线执行，MySQL 会直接报错而不是默默锁表：

```sql
ALTER TABLE orders ADD INDEX idx_status (status), ALGORITHM=INPLACE, LOCK=NONE;
```

第二，对于必须锁表的操作（如修改列类型），使用第三方工具 `pt-online-schema-change`（Percona）或 `gh-ost`（GitHub）。它们的原理是创建一张新结构的影子表，通过触发器或 binlog 同步增量数据，最后原子性地 RENAME 替换原表：

```bash
# pt-online-schema-change 示例
pt-online-schema-change --alter "MODIFY COLUMN amount DECIMAL(12,2)" \
  D=mydb,t=orders --execute

# gh-ost 示例
gh-ost --alter "MODIFY COLUMN amount DECIMAL(12,2)" \
  --database=mydb --table=orders --execute
```

第三，在业务低峰期执行 DDL 操作，减少对在线业务的影响。

第四，执行前先用 `ALGORITHM=INPLACE` 试探，确认操作是否支持在线执行，避免意外锁表。
