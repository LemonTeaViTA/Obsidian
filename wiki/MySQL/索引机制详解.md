# 索引机制详解

## 索引基础

### 索引不适合哪些场景呢？

第一，区分度低的列，可以和其他高区分度的列组成联合索引。
第二，频繁更新的列，索引会增加更新的成本。
第三，TEXT、BLOB 等大对象类型的字段，可以使用前缀索引、全文索引替代。
第四，当表的数据量很小的时候，不超过 1000 行，全表扫描可能比使用索引更快。

当数据量很小时，全表扫描的成本很低，因为所有的数据可能都加载到内存中了，使用索引反而需要先查找索引，再通过索引去找到实际的数据行，增加了额外的 I/O 寻址时间。

### 性别字段适合建立单独索引吗？

性别字段不适合建立单独索引。因为性别字段的区分度很低。如果性别字段确实经常用于查询条件，数据规模也比较大，可以将性别字段作为联合索引的一部分，与区分度高的字段一起，效果会好很多。

### 什么是区分度？

区分度是衡量一个字段在 MySQL 表中唯一值的比例。区分度 = 字段的唯一值数量 / 字段的总记录数；越接近 1，就越适合作为索引。

可以通过 `COUNT(DISTINCT column_name)` 和 `COUNT(*)` 的比值来计算字段的区分度：

```sql
SELECT COUNT(DISTINCT gender) / COUNT(*) AS gender_selectivity
FROM users;
```

### 什么样的字段适合加索引？

主键、唯一键、以及经常作为查询条件的字段最适合加索引。字段的区分度要高，这样索引才能起到过滤作用；如果字段经常用于表连接、排序或分组，也建议加索引。如果多个字段经常一起出现在查询条件中，可以建立联合索引来提升性能。

查询条件中的高频字段：

```sql
SELECT * FROM orders WHERE status = 'PAID' AND create_time > '2023-01-01';
-- 若 status 和 create_time 常组合查询，建联合索引 (status, create_time)
```

多表连接时的关联字段：

```sql
SELECT * FROM user u JOIN order o ON u.id = o.user_id; -- user_id 需索引
```

参与排序或分组的字段，可以直接利用索引的有序性，避免文件排序：

```sql
SELECT * FROM product ORDER BY price DESC;
SELECT category, COUNT(*) FROM product GROUP BY category;
```

覆盖索引场景：

```sql
-- 创建联合索引 (user_id, create_time)
SELECT user_id, create_time FROM orders WHERE user_id = 100; -- 覆盖索引生效
```

### 索引是不是建的越多越好？

索引不是越多越好。虽然索引可以加快查询，但也会带来写入变慢、占用更多存储空间、甚至让优化器选错索引的风险。

每次数据写入（INSERT/UPDATE/DELETE）时，MySQL 都需同步更新所有相关索引，索引越多，维护成本越高。索引过多时，优化器需评估更多可能的执行路径，可能导致选错索引。

单表索引数量建议不超过 5 个，MySQL 官方建议单表索引总字段数 ≤ 表字段数的 30%。

### 说说索引优化的思路？

先通过慢查询日志找出性能瓶颈，然后用 `EXPLAIN` 分析执行计划，判断是否走了索引、是否回表、是否排序。接着根据字段特性设计合适的索引，如选择区分度高的字段，使用联合索引和覆盖索引，避免索引失效的写法，最后通过实测来验证优化效果。

## B+ 树索引原理

### 为什么 InnoDB 要使用 B+树作为索引？

因为 B+ 树是一种高度平衡的多路查找树，能有效降低磁盘的 IO 次数，并且支持有序遍历和范围查询。

- 相比哈希表：B+ 树支持范围查询和排序
- 相比二叉树和红黑树：B+ 树更"矮胖"，层级更少，磁盘 IO 次数更少
- 相比 B 树：B+ 树的非叶子节点只存储键值，叶子节点存储数据并通过链表连接，支持范围查询

B+ 树的非叶子节点只存储键值，不存储数据，而叶子节点存储了所有的数据，并且构成了一个有序链表。这样做的好处是，非叶子节点上由于没有存储数据，就可以存储更多的键值对，再加上叶子节点构成了一个有序链表，范围查询时就可以直接通过叶子节点间的指针顺序访问整个查询范围内的所有记录，而无需对树进行多次遍历。

通常 B+ 树高度为 3-4 层即可支持 TB 级数据，而每次查询只需 2-4 次磁盘 I/O，远低于二叉树或红黑树的 O(log2N) 复杂度。

### B+树的叶子节点是单向链表还是双向链表？

B+树的叶子节点是通过双向链表连接的，这样可以方便范围查询和反向遍历。

如果需要在 B+树中从大值向小值进行检索，可以先定位到最右侧节点，找到包含最大值的叶子节点，再利用叶节点间的双向链表向左遍历。

### 为什么 MongoDB 的索引用 B树，而 MySQL 用 B+ 树？

MongoDB 通常以 JSON 格式存储文档，查询以单键查询为主。B 树的"节点既存键又存数据"的特性允许查询在非叶子节点提前终止，从而减少 I/O 次数。

MySQL 的查询通常涉及范围（`WHERE id > 100`）、排序（`ORDER BY`）、连接（`JOIN`）等操作。B+ 树的叶子节点是链表结构，天然支持顺序遍历，效率远高于 B 树。

| 特性 | MongoDB (B树) | MySQL InnoDB (B+树) |
|------|--------------|---------------------|
| 数据模型 | 文档型数据库 | 关系型数据库 |
| 查询模式 | 侧重单文档查询 | 侧重范围查询和复杂连接 |
| 范围查询效率 | 需要多次树遍历 | 通过叶节点链表高效遍历 |
| 内存利用率 | 单个查询路径缓存更有效 | 适合批量扫描缓存 |

### 一棵 B+ 树能存多少数据？

在 InnoDB 中，页的默认大小为 16KB，当主键为 bigint 时，3 层 B+ 树通常可以存储约 2000 万条数据。

关键参数：
- 页大小：默认 16KB
- 主键大小：bigint 为 8 字节
- 页指针大小：InnoDB 源码中设置为 6 字节

非叶子节点可以存储 16384 / 14（键值+指针）= 1170 个单元。

- 层高为 2 时：1170 × 16 = 18720 条
- 层高为 3 时：1170 × 1170 × 16 ≈ 2190 万条

### 现在有一张表 2千万 数据，B+树的高度有几层？

对于 2000 万条数据来说，B+树的高度为 3 层就够了。

### 每个叶子节点能存放多少条数据？

如果单行数据大小为 1KB，那么每页可存储约 16 行（16KB / 1KB）数据。

假设有这样一个表结构：

```sql
CREATE TABLE `user` (
  `id` BIGINT PRIMARY KEY,
  `name` VARCHAR(255) NOT NULL,
  `age` TINYINT,
  `email` VARCHAR(255)
) ROW_FORMAT = COMPACT;
```

一行数据大小：8 + 50 + 1 + 30 = 89 字节，行格式开销 14 字节，实际每行约 103 字节。每页最多可以存储 16384 / 103 ≈ 158 行数据。

### 索引为什么用 B+树不用普通二叉树？

普通二叉树的每个节点最多有两个子节点。当数据按顺序递增插入时，二叉树会退化成链表，导致树的高度等于数据量，查找效率退化为 O(n)。

平衡二叉树虽然解决了退化问题，但每个节点最多只有两个子节点，树的高度仍然较大，磁盘 IO 次数多。

### 为什么用 B+ 树而不用 B 树呢？

B+ 树相比 B 树有 3 个显著优势：

第一，B 树的每个节点既存储键值，又存储数据和指针，导致单节点存储的键值数量较少。一个 16KB 的 InnoDB 页，B 树的非叶子节点只能容纳几十个键值，而 B+ 树的非叶子节点可以容纳上千个键值。

第二，B 树的范围查询需要通过中序遍历逐层回溯；而 B+ 树的叶子节点通过双向链表顺序连接，范围查询只需定位起始点后顺序遍历链表即可。

第三，B 树的数据可能存储在任意节点，查询时间波动较大。而 B+ 树的所有数据都存储在叶子节点，查询路径的长度是固定的，时间稳定为 O(logN)。

### B+树的空间复杂度是多少？

O(logN)。树的高度 $h = \lceil \log_m N \rceil$，其中 N 是数据总量，m 是阶数。总复杂度为 $O(\log_m N \cdot \log m) = O(\log N)$。

### 为什么用 B+树不用跳表呢？

跳表本质上还是链表结构，一条数据一个节点。如果需要存放 2000 万条数据，跳表的高度大约为 24 层，最坏情况下查找一次数据就需要 24 次磁盘 I/O。而 2000 万条数据在 B+树中只需要 3 层就可以了。

### B+树的范围查找怎么做的？

先通过索引路径定位到第一个满足条件的叶子节点，然后顺着叶子节点之间的链表向右/向左扫描，直到超过范围。

第一步，从 B+ 树的根节点开始，通过索引键值逐层向下，找到第一个满足条件的叶子节点。第二步，利用叶子节点之间的双向链表，从起始节点开始，依次向后遍历每个节点，当索引值超过查询范围时终止查询。

### B+树和 Hash 索引有什么区别？

B+ 树索引支持范围查询、有序扫描，是 InnoDB 的默认索引结构。Hash 索引只支持等值查找，速度快但功能弱，常见于 Memory 引擎。

```sql
-- B+ 树索引（InnoDB 默认）
CREATE TABLE example_btree (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255),
  INDEX name_index (name)
) ENGINE = InnoDB;

-- Hash 索引（Memory 引擎）
CREATE TABLE example_hash (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255),
  UNIQUE HASH (name)
) ENGINE = MEMORY;
```

InnoDB 内部使用了"自适应哈希索引"（Adaptive Hash Index, AHI）技术，当某些索引值频繁访问时，InnoDB 会在 B+ 树基础上自动创建哈希索引。

```sql
SHOW VARIABLES LIKE 'innodb_adaptive_hash_index';
```

## 聚簇索引与回表

### 什么是聚簇索引和非聚簇索引？

聚簇索引的叶子节点存储了完整的数据行，数据和索引是在一起的。InnoDB 的主键索引就是聚簇索引，叶子节点不仅存储了主键值，还存储了其他列的值，因此按照主键进行查询的速度会非常快。每个表只能有一个聚簇索引，通常由主键定义。如果没有显式指定主键，InnoDB 会隐式创建一个隐藏的主键索引 row_id。

非聚簇索引的叶子节点只包含了主键值，需要通过回表按照主键去聚簇索引查找其他列的值，唯一索引、普通索引等非主键索引都是非聚簇索引。

```sql
CREATE TABLE user (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  age INT
);
-- id 是聚簇索引，叶子节点直接存储 (id, name, age)

CREATE INDEX idx_name ON user (name);
CREATE INDEX idx_age ON user (age);
-- idx_name 叶子节点存的是 name -> id，查整行数据要回表
```

### 回表了解吗？

当使用非聚簇索引进行查询时，MySQL 需要先通过非聚簇索引找到主键值，然后再根据主键值回到聚簇索引中查找完整数据行，这个过程称为回表。

```sql
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(50),
  age INT,
  email VARCHAR(50),
  INDEX (name)
);

SELECT * FROM users WHERE name = '王二';
-- 第一步：使用 name 列上的非聚簇索引查找主键 id
-- 第二步：使用主键 id 到聚簇索引中查找完整记录
```

### 回表的代价是什么？

回表通常需要访问额外的数据页，如果数据不在内存中，还需要从磁盘读取，增加 I/O 开销。可通过覆盖索引或者联合索引来避免回表：

```sql
-- 优化方案：创建包含 age 的联合索引
ALTER TABLE users ADD INDEX idx_name_age (name, age);
-- 现在 SELECT name, age FROM users WHERE name = '张三' 不需要回表
```

### 什么情况下会触发回表？

第一，当查询字段不在非聚簇索引中时，必须回表到主键索引获取数据。第二，查询字段包含非索引列（如 `SELECT *`），必然触发回表。

### 了解 MRR 吗？

MRR（Multi-Range Read）是 InnoDB 为了解决回表带来的大量随机 IO 问题而引入的一种优化策略。它会先把非聚簇索引查到的主键值列表进行排序，再按顺序去主键索引中批量回表，将随机 I/O 转换为顺序 I/O，以减少磁盘寻道时间。

```sql
SHOW VARIABLES LIKE 'optimizer_switch';  -- 查看 mrr=on/off
SHOW VARIABLES LIKE 'read_rnd_buffer_size';  -- MRR 缓冲区大小，默认 256KB
```

## 联合索引与覆盖索引

### 联合索引了解吗？

联合索引就是把多个字段放在一个索引里，但必须遵守"最左前缀"原则，只有从第一个字段开始连续使用，索引才会生效。

联合索引会按字段顺序构建 B+树。例如 `(age, name)` 索引会先按照 age 排序，age 相同则按照 name 排序。创建 `(A,B,C)` 联合索引相当于同时创建了 `(A)`、`(A,B)` 和 `(A,B,C)` 三个索引。

```sql
CREATE INDEX idx_order_user_product ON orders(user_id, product_id, create_time);

SELECT * FROM orders
WHERE user_id = 1001 AND product_id = 2002
ORDER BY create_time DESC;
```

联合索引的叶子节点属于非聚簇索引，存储的是联合索引各列的值和对应行的主键值，而不是完整的数据行。

### 覆盖索引了解吗？

覆盖索引指的是：查询所需的字段全部都在索引中，不需要回表，从索引页就能直接返回结果。

```sql
CREATE INDEX idx_empname_job ON employee(empname, job);

-- 查询字段恰好是索引字段，不需要回表
SELECT empname, job FROM employee WHERE empname = '王二' AND job = '程序员';
```

普通索引只用于加速查询条件的匹配，而覆盖索引还能直接提供查询结果。

#### 一个表（name, sex, age, id），`select age, id, name from tblname where name='paicoding'` 怎么建索引？

由于查询条件有 name 字段，最少应该为 name 字段添加一个索引。查询结果中还需要 age、id 字段，可以为这三个字段创建联合索引，利用覆盖索引减少回表：

```sql
CREATE INDEX idx_name_age_id ON tblname (name, age, id);
```

### 什么是最左前缀原则？

最左前缀原则指的是：MySQL 使用联合索引时，必须从最左边的字段开始匹配，才能命中索引。

假设有一个联合索引 `(A, B, C)`：

| 查询条件 | 是否触发索引 | 说明 |
|---------|------------|------|
| WHERE A = 1 | 是 | 使用索引的第一列 |
| WHERE A = 1 AND B = 2 | 是 | 使用索引的前两列 |
| WHERE A = 1 AND B = 2 AND C = 3 | 是 | 使用索引的全部列 |
| WHERE B = 2 | 否 | 跳过左侧列 A，索引失效 |
| WHERE B = 2 AND C = 3 | 否 | 无左侧列，索引失效 |
| WHERE A = 1 AND C = 3 | 部分生效 | 仅用 A 列，C 列无法利用索引 |

如果排序或分组的列是最左前缀的一部分，索引还可以加速操作：

```sql
-- 索引 (a,b)
SELECT * FROM table WHERE a = 1 ORDER BY b; -- 可以利用索引排序
```

范围查询后的列无法使用索引：

```sql
-- 索引 (a,b,c)
SELECT * FROM table WHERE a = 1 AND b > 2 AND c = 3;
-- 只能使用 a 和 b，c 无法使用索引
```

#### 为什么不从最左开始查，就无法匹配呢？

因为联合索引在 B+ 树中是按照最左字段优先排序构建的，如果跳过最左字段，MySQL 无法判断查找范围从哪里开始，自然也就无法使用索引。

#### 联合索引 (a, b)，where a = 1 和 where b = 1，效果是一样的吗？

不一样。`WHERE a = 1` 能命中联合索引，因为 a 是联合索引的第一个字段，符合最左前缀匹配原则。而 `WHERE b = 1` 无法命中联合索引，因为缺少 a 的匹配条件，MySQL 会全表扫描。

#### 假如有联合索引 abc，下面的 SQL 怎么走联合索引？

```sql
SELECT * FROM t WHERE a = 2 AND b = 2;      -- 命中 a、b 两列
SELECT * FROM t WHERE b = 2 AND c = 2;      -- 未使用最左前缀，全表扫描
SELECT * FROM t WHERE a > 2 AND b = 2;      -- 范围条件后 b 无法利用索引
```

#### (A,B,C) 联合索引，`select * from tbn where a=? and b in (?,?) and c>?` 会走索引吗？

会命中联合索引。a 是等值匹配，b 是 IN 等值多匹配，c 是范围条件，符合最左前缀原则。EXPLAIN 中 type 为 range，Extra 显示 `Using index condition`（ICP 生效）。

#### 联合索引 abc，(b,c) 是否会走索引？

不会。根据最左前缀原则，查询条件中没有包含 a，MySQL 无法利用这个索引。

#### 建立联合索引 (a,b,c)，where c = 5 是否会用到索引？

不会。只有索引的第三列 c 被用作查询条件，而前两列 a 和 b 都没有被使用，不符合最左前缀原则。

#### SQL 中使用 like，如果遵循最左前缀匹配，查询是不是一定会用到索引？

如果查询模式是后缀通配符 `LIKE 'prefix%'`，且该字段有索引，优化器通常会使用索引。否则即便遵循最左前缀匹配，`LIKE '%xxx'` 也无法命中索引。

#### 联合索引 abc，a=1,c=1 / b=1,c=1 / a=1,c=1,b=1 走不走索引？

- `a=1,c=1`：能用上索引，a=1 符合最左前缀，c=1 通过索引下推在存储引擎层过滤
- `b=1,c=1`：无法使用索引，全表扫描
- `a=1,c=1,b=1`：MySQL 优化器会自动重排为 a=1 AND b=1 AND c=1，能命中索引

### 什么是索引下推？

索引下推（ICP）是指：MySQL 把 WHERE 条件尽可能"下推"到索引扫描阶段，在存储引擎层提前过滤掉不符合条件的记录。

传统查询流程：存储引擎通过联合索引定位到符合最左前缀条件的主键 ID → 回表读取完整数据行 → Server 层对所有返回的行进行 WHERE 条件过滤。

有了 ICP 后：存储引擎在索引层直接过滤可下推的条件，仅对符合索引条件的记录回表读取数据。

例如有一张 user 表，建了联合索引 `(name, age)`，查询 `select * from user where name like '张%' and age=10`：

- 没有 ICP：先找到所有 `name like '张%'` 的主键（假设 10000 行），全部回表，Server 层再过滤 age=10
- 有 ICP：在索引层同时过滤 `name like '张%' and age=10`，只回表符合条件的行（假设 10 行）

```sql
SET optimizer_switch = 'index_condition_pushdown=on';
```

EXPLAIN 中 Extra 显示 `Using index condition` 表示 ICP 生效。

### 如何查看是否用到了索引？

可以通过 `EXPLAIN` 关键字来查看是否使用了索引：

```sql
EXPLAIN SELECT * FROM table WHERE column = 'value';
```

如果使用了索引，结果中的 `key` 值会显示索引的名称；`key` 为 NULL 则表示未使用索引。

## 索引失效场景

### 索引失效的常见场景有哪些？

索引失效是指 SQL 查询没有按预期使用索引，而是走了全表扫描。常见的失效场景有以下几种：

第一，对索引列使用函数或表达式，会导致索引失效：

```sql
-- 失效：对索引列使用函数
SELECT * FROM orders WHERE YEAR(create_time) = 2023;
-- 优化：改写为范围查询
SELECT * FROM orders WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01';

-- 失效：对索引列做运算
SELECT * FROM user WHERE id + 1 = 5;
-- 优化：把运算移到等号右边
SELECT * FROM user WHERE id = 4;
```

第二，隐式类型转换，当字段类型和查询值类型不一致时，MySQL 会对索引列做隐式转换，相当于加了函数：

```sql
-- phone 是 VARCHAR 类型，传入数字会触发隐式转换，索引失效
SELECT * FROM user WHERE phone = 13800138000;
-- 优化：传入字符串
SELECT * FROM user WHERE phone = '13800138000';
```

第三，`LIKE` 以通配符开头：

```sql
-- 失效：左模糊
SELECT * FROM user WHERE name LIKE '%王';
-- 生效：右模糊
SELECT * FROM user WHERE name LIKE '王%';
```

第四，违反最左前缀原则，联合索引 `(a, b, c)` 中跳过左侧列直接查询后面的列：

```sql
-- 失效：跳过 a
SELECT * FROM t WHERE b = 2 AND c = 3;
```

第五，使用 `OR` 连接条件时，如果 OR 两边的列不是都有索引，会导致整个查询不走索引：

```sql
-- 如果 age 没有索引，整个查询不走索引
SELECT * FROM user WHERE name = '王二' OR age = 18;
```

第六，使用 `!=`、`<>`、`NOT IN` 等否定条件，优化器通常认为需要扫描大量数据，倾向于全表扫描：

```sql
SELECT * FROM user WHERE status != 1;
SELECT * FROM user WHERE id NOT IN (1, 2, 3);
```

第七，`IS NULL` / `IS NOT NULL` 在某些情况下可能导致索引失效，取决于优化器对 NULL 值比例的判断。如果 NULL 值占比很高，`IS NOT NULL` 可能走索引；反之 `IS NULL` 可能走索引。
