# 优化与实战

## SQL 实战

### 一张表：id，name，age，sex，class，写出以下查询

```sql
CREATE TABLE students (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(50),
  age INT,
  sex CHAR(1),
  class VARCHAR(50)
);
```

#### 所有年龄为 18 的人的名字？

```sql
SELECT name FROM students WHERE age = 18;

-- 可以给 age 字段加上索引
ALTER TABLE students ADD INDEX age_index (age);
```

#### 找到每个班年龄大于 18 有多少人？

```sql
SELECT class, COUNT(*) AS number_of_students
FROM students
WHERE age > 18
GROUP BY class;
```

#### 找到每个班年龄排前两名的人？

```sql
SELECT a.class, a.name, a.age
FROM students a
WHERE (
  SELECT COUNT(DISTINCT b.age)
  FROM students b
  WHERE b.class = a.class AND b.age > a.age
) < 2
ORDER BY a.class, a.age DESC;
```

### 有一个查询需求，MySQL 中有两个表，一个表 1000W 数据，另一个表只有几千数据，要做一个关联查询，如何优化？

第一步，为关联字段建立索引，确保 ON 连接的字段都有索引：

```sql
ALTER TABLE big_table ADD INDEX idx_small_id(small_id);
```

第二步，小表驱动大表，将小表放在 JOIN 的左边（驱动表），大表放在右边：

```sql
SELECT ... FROM small_table s
JOIN big_table b ON s.id = b.small_id;
```

### 先建索引再插入数据，还是先插入数据再建索引，哪个效率高？

在大数据量导入场景下，先导入数据、后建索引的效率显著高于先建索引、后导入数据。

- 先插入数据再创建索引：在没有索引的情况下插入数据，数据库不需要在每次插入时更新索引。
- 先创建索引再插入数据：数据库需要在每次插入新记录时维护索引结构，随着数据量的增加，索引的维护会导致额外的性能开销，频繁触发页分裂和索引结构调整。

如果是小批量插入，可以先建索引；但在大数据量数据导入场景下，推荐先插入数据再建索引。

### 什么是深分页？

深分页是指在 MySQL 中获取比较靠后的数据页，比如第 1000 页、第 10000 页等。使用 `LIMIT offset, count` 这种方式，当 offset 特别大时，会带来严重的性能问题。

#### offset 太大时 MySQL 内部具体做了什么？

以 `SELECT * FROM orders LIMIT 1000000, 10` 为例，MySQL 的执行过程是：

1. 从聚簇索引（主键索引）的第一条记录开始，逐行扫描
2. 对每一行读取完整的数据（如果有非聚簇索引还需要回表）
3. 将前 1000000 行全部读出来，然后丢弃
4. 只返回第 1000001 到 1000010 这 10 行数据

也就是说，MySQL 并不能直接"跳到"第 1000000 行，它必须从头一行一行地扫过去，把前面 100 万行全部读进来再扔掉。offset 越大，扫描和丢弃的行数越多，I/O 和 CPU 开销越大。

#### 书签分页（游标分页）是为了解决什么问题？

书签分页就是利用上一页的最后一条记录的主键值作为"书签"，下次查询直接从这个位置往后取，跳过了"从头扫描再丢弃"的过程。

```sql
-- 传统 offset 分页：必须扫描前 100 万行再丢弃
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;

-- 书签分页：直接通过主键索引定位到起点，只扫描 10 行
SELECT * FROM orders WHERE id > 1000000 LIMIT 10;
```

书签分页之所以快，是因为 `WHERE id > 1000000` 可以直接利用主键 B+ 树的有序性，从叶子节点精确定位到 id = 1000000 的位置，然后顺着链表往后读 10 条就结束了，不需要扫描和丢弃任何数据。

#### 书签分页会引入什么新问题？

书签分页依赖上一页的最大 ID 作为起点，所以无法直接跳转到任意页码（比如直接查第 50 页）。它只适合"上一页/下一页"的连续翻页场景。

另外，如果主键 ID 不连续（有删除操作），`WHERE id > 1000` 拿到的不一定是第 1001 条数据，而是 id 大于 1000 的第一条，所以不能用 `id > (page - 1) * pageSize` 来跳页。

如果业务确实需要跳页，可以用子查询先在主键索引上定位，再回表取数据：

```sql
SELECT * FROM orders
WHERE id > (SELECT id FROM orders ORDER BY id LIMIT 1000000, 1)
LIMIT 10;
```

子查询只在主键索引上扫描（不回表），定位到第 1000000 个 id 后，外层查询再从这个位置取 10 条完整数据。虽然子查询内部仍有 offset 扫描，但因为只走覆盖索引不回表，性能远好于直接 `SELECT * ... LIMIT 1000000, 10`。

### SQL 题：一个学生成绩表，字段有学生姓名、班级、成绩，求各班前十名

```sql
CREATE TABLE student_scores (
  student_name VARCHAR(100),
  class VARCHAR(50),
  score INT
);
```

MySQL 8.0+ 使用窗口函数：

```sql
SELECT student_name, class, score
FROM (
  SELECT
    student_name,
    class,
    score,
    ROW_NUMBER() OVER (PARTITION BY class ORDER BY score DESC) AS rn
  FROM student_scores
) AS tmp
WHERE rn <= 10;
```

MySQL 8.0 以下使用变量：

```sql
SET @cur_class = NULL, @cur_rank = 0;

SELECT student_name, class, score
FROM (
  SELECT
    student_name,
    class,
    score,
    @cur_rank := IF(@cur_class = class, @cur_rank + 1, 1) AS rank,
    @cur_class := class
  FROM student_scores
  ORDER BY class, score DESC
) AS ranked
WHERE ranked.rank <= 10;
```

| 步骤 | 解释 |
|------|------|
| @cur_class 变量 | 记录当前正在处理的班级 |
| @cur_rank 变量 | 记录当前班级的排名，默认 0 |
| IF(@cur_class = class, @cur_rank + 1, 1) | 如果班级没变，就排名 +1；如果换了新班级，排名从 1 重新开始 |
| ORDER BY class, score DESC | 必须先按班级升序、成绩降序排好，才能保证变量正确打排名 |
| 外层 WHERE rank <= 10 | 只取每班前十名 |

## 慢查询分析与优化

### 怎么分析慢查询？

分析慢查询主要分三步：定位、分析、验证。

第一步，开启慢查询日志，定位慢 SQL：

```sql
-- 查看慢查询日志是否开启
SHOW VARIABLES LIKE 'slow_query_log';

-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;

-- 设置慢查询阈值（默认 10 秒，建议设为 1 秒）
SET GLOBAL long_query_time = 1;

-- 查看慢查询日志文件位置
SHOW VARIABLES LIKE 'slow_query_log_file';
```

也可以通过 `SHOW PROCESSLIST` 实时查看当前正在执行的慢 SQL：

```sql
SHOW FULL PROCESSLIST;
-- 关注 Time 列较大的查询，State 为 Sending data / Sorting result 等
```

第二步，用 `EXPLAIN` 分析执行计划，判断 SQL 为什么慢：

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1001;
```

重点关注以下字段：

| 字段 | 关注点 |
|------|--------|
| type | 访问类型，从好到差：const > eq_ref > ref > range > index > ALL（全表扫描） |
| key | 实际使用的索引，NULL 表示没走索引 |
| rows | 预估扫描行数，越大越慢 |
| Extra | `Using filesort`（额外排序）、`Using temporary`（临时表）都是性能警告 |

第三步，针对分析结果进行优化，优化后再用 `EXPLAIN` 验证执行计划是否改善。

### 怎么避免慢查询？

第一，合理使用索引：为高频查询条件、JOIN 关联字段、ORDER BY / GROUP BY 字段建立索引，优先使用联合索引和覆盖索引减少回表。

第二，避免索引失效的写法：不要对索引列使用函数或运算，不要用 `LIKE '%xxx'` 左模糊，注意隐式类型转换，避免 `SELECT *`。

```sql
-- 反例：索引失效
SELECT * FROM user WHERE YEAR(create_time) = 2023;
-- 正例：改写为范围查询
SELECT * FROM user WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01';
```

第三，避免深分页，使用游标分页或子查询优化：

```sql
-- 反例：深分页
SELECT * FROM orders LIMIT 1000000, 10;
-- 正例：基于主键游标
SELECT * FROM orders WHERE id > last_max_id LIMIT 10;
```

第四，小表驱动大表，JOIN 时将小表放在驱动表位置，确保被驱动表的关联字段有索引。

第五，避免在事务中执行耗时操作，长事务会持有锁导致其他查询阻塞。

第六，对于复杂查询，考虑拆分为多个简单查询，避免产生临时表和文件排序。
