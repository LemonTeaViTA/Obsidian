# 计算机网络篇

## 一、传输层协议：TCP 面试痛点核心

### TCP三次握手与四次挥手

#### 核心缩写词速记（小白防懵雷达）
在背诵握手流程前，必须要知道这几个全大写英文到底是个啥意思，否则只会死记硬背：
- **SYN (Synchronize - 同步)**：意思是“发起一个新连接”。就像打电话时你说的一句：“**喂？你能听到我说话吗？**”
- **ACK (Acknowledge - 确认)**：意思是“我收到你的消息了”。就像打电话时对方回的一句：“**听得到！/ 收到！**”
- **FIN (Finish - 结束)**：意思是“我的话全说完了，我要挂了”。就像打电话时的那句：“**我这里没啥事了，再见。**”

#### 什么是三次握手（建立连接）
TCP（Transmission Control Protocol，传输控制协议）是面向连接的、可靠的传输层协议。三次握手就是为了在客户端和服务端之间建立一个可靠的通信通道。

1. **第一次握手（SYN）**：客户端发送连接请求报文段（SYN=1, seq=x），进入 `SYN_SENT` 状态。此时客户端问：“大佬，你在吗？我要连你了。”
2. **第二次握手（SYN + ACK）**：服务端收到请求，分配资源，并向客户端返回确认报文段（SYN=1, ACK=1, seq=y, ack=x+1），进入 `SYN_RCVD` 状态。服务端答：“我在，我准备好了，你真要连吗？”
3. **第三次握手（ACK）**：客户端收到确认，向服务端再次发送确认报文段（ACK=1, seq=x+1, ack=y+1），进入 `ESTABLISHED` 状态。服务端收到后也进入 `ESTABLISHED` 状态。客户端答：“真连，开搞！”

#### 为什么必须是三次，不是两次？（核心必考与物理隐喻）
如果是两次握手：客户端喊“我们建立连接吧”（SYN），服务端回“好的”（ACK），然后服务端就傻傻地把资源开辟好了。但这在**“网络可能延迟、迷路”**的物理世界是个灾难！

- **服务端单相思（资源耗尽防范）**：假设客户端第一次发出的 SYN 并没有丢，只是在某个网络节点滞留了。客户端迟迟收不到回信，触发超时重传，重新发了一个 SYN 建立了连接并完成了数据传输。此时，**那个最开始迷路的死 SYN 突然飘到了服务端**！
- 如果只有两次握手，服务端一收到这个“穿越时空”的旧 SYN，就会以为客户端又要建立新连接，立刻回发 ACK 并**单方面分配好内存和端口**开始死等客户端发数据。但客户端这边早就完事儿了，根本不理它。这就导致了服务端的资源被白白挂起浪费掉。
- **为了防止历史失效连接请求突然到达带来的灾难**，必须有第三次握手——客户端收到服务端的 ACK 后，亲自回一个：“对，刚才那个是我**现在的真实请求**，开始通信吧！”（又或者回一个 RST 拒绝掉旧的超时连接）。这样就避免了单边资源的盲目开销。

#### 为什么要有四次挥手（断开连接）
建立连接只需要 3 次，因为建立时是可以把同意连接（ACK）和告诉对方自己也准备好了（SYN）**合并在一起打包发送**的。

但在断开连接时，TCP 是**全双工**的（双方都能同时发数据）：
1. 客户端发 FIN（“我说完了”）。
2. 服务端发 ACK（“我知道你说完了”，但此时服务端可能**还有数据没发完**）。
3. 等服务端把最后的数据全发完，它才发 FIN（“我也说完了”）。
4. 客户端发 ACK（“收到，拜拜”）。

因为发完 ACK 还要等自己的剩余数据发送完毕才能发 FIN，所以这俩包**没法合并**，所以断开连接硬生生多了一次，变成了 4 次。

## 四、传输层核心对决：TCP 与 UDP

### TCP 和 UDP 的区别是什么？（传输速度谁更快？）
这也是面试中必问的经典对比题。理解了它们各自的“性格”，就不需要死记硬背它们复杂的报文结构头了。我们可以用**“打电话”**和**“用大喇叭喊话”**的物理隐喻来彻底区分它们。

#### 1. 性格对比（物理隐喻）
- **TCP（打电话 / 寄挂号信）**：
  - **可靠、面向连接**。就像打电话，你必须先拨号，等对方接起（**三次握手**），然后你才能开始说事。你说一句，对方必须回一句“嗯，听清了”（**ACK 确认**）。
  - 如果中间电话信号不好卡了，你会立刻停下来问：“喂？那句没听清，我再说一遍？”（**超时重传**）。
  - 如果对方是个结巴，他听得慢，你也会主动讲慢点（**流量控制与拥塞控制**）。
  

- **UDP（用大喇叭喊话 / 发无回执明信片）**：
  - **不可靠、无连接**。就像你在操场上拿着高音大喇叭喊广播，你根本不管底下的同学有没有捂住耳朵（**没有握手建连过程**）。
  - 你吼完一长串就下班了，到底有多少个字飘进了别人的耳朵里你根本不关心（**不保证送达、没有重传机制**）。
  - 如果风太大把声音吹跑了，丢了就彻底丢了（**允许丢包**）。

#### 2. 谁的传输速度更快？
**毫无悬念，绝对是 UDP 的速度快得多！**

如果你刚才看懂了那个大喇叭的比喻，就能立刻理解 TCP 为了它引以为傲的“绝对可靠性”，付出了极其惨痛的**性能代价（Overhead）**：
- **TCP 慢在“前戏多”**：每次发数据前，必须硬生生等 3 次握手的网络延迟飞一遍才能开工。
- **TCP 慢在“包袱重”**：因为要记录每一句话对应的编号（Seq）以防乱序，TCP 的标准头部就有满满的 20 个字节。
- **TCP 慢在“畏首畏尾”**：为了不把自己挤爆，它还得时刻计算各种滑动窗口、拥塞窗口，束手束脚。

而 **UDP 快得飞起**，因为它就是个极致轻量级的“土匪”：
- **UDP 快在“没有前戏”**：拿起来就直接往目的 IP 疯狂喷包，绝不浪费时间打招呼。
- **UDP 快在“浑身轻松”**：头部极度简陋，连序列号和确认号全丢掉，只有 8 个字节，开销极小。
- **UDP 快在“不计后果”**：不管网络有多堵，只要网线的物理带宽还在，它就以最高速度倾泻，丢掉拉倒。

#### 3. 它们分别用在什么场景？（面试必杀技）
- **必须用 TCP 的场景（绝不能丢哪怕一个字节的数据）**：
  - 加载网页（HTTP/HTTPS）：一旦由于丢包少了一个 `<html>` 标签，整个网页直接排版崩塌（白屏）。
  - 发送邮件（SMTP）、传输文件（FTP）：下载苍井空老师的电影只要因为UDP丢了一个字节，整部两小时的片子直接变成 `文件损坏无法播放`。
- **必须用 UDP 的场景（宁可丢包，也绝对不能卡顿延迟）**：
  - 直播、在线竞技游戏（王者荣耀、LOL）、视频语音通话（Zoom）。
  - 在打王者开团时，你的攻击动作如果是用 TCP 发送的，哪怕就丢了 1 毫秒的一个包，TCP 马上会在原地停下死等“重发”，导致整个画面疯狂卡死重连，那简直是噩梦。
  - 而使用 UDP，就算那一毫秒的动作飞丢了，它根本不管不顾继续接收下一毫秒的动作帧，你最多只是觉得画面有一丝极其轻微的“掉帧”，瞬间又流畅了。在竞技游戏和直播里，**“低延迟的新鲜数据，远比完整却迟到了三秒的老旧数据更有价值”**。

### 键入网址到网页显示，期间发生了什么？（全链路宏观视角）
这是计算机网络面试中的“八股之王”，它几乎串联了从应用层到底层的全部网络知识。为了避免流水账背诵，我们可以用**“去饭店点外卖吃”**的物理隐喻来贯穿这个过程。

#### 1. URL 解析与 DNS 域名解析（分辨需求与查黄页找地址）
- **现象**：你在地址栏敲下一串字符，浏览器会先做一个“安检”：**你输入的是一个合法的 URL (网址) 还是一个搜索关键字？**
  - **👉 如果是关键字（比如你搜了“外卖”）**：浏览器发现这不是个网址（没有 `http` 开头，也没有 `.com` 这类后缀），它就会**默认使用你设置的搜索引擎**（比如百度、Google），将你的关键字拼接到搜索引擎的专门地址里。例如它会自动把你重定向到 `https://www.baidu.com/s?wd=外卖`。此时，请求对象就变成了“百度服务器”。
  - **👉 如果是合法 URL（比如输入了 `www.ruanyifeng.com`）**：浏览器认为你目标明确，接下来就要去寻找这家“饭店”的真实物理坐标。我们要知道计算机是看不懂纯英文域名的，它只认 IP 地址（经纬度坐标）。
- **动作**：浏览器开始查**DNS（域名系统）**寻找该域名背后的 IP。查询顺序是有优先级的：
  1. 浏览器自己的 DNS 缓存
  2. 操作系统的 DNS 缓存（比如电脑里的 `hosts` 文件）
  3. 本地的 DNS 服务器（通常是你家路由器或宽带运营商提供的）
  4. 根域名服务器（最高层级的互联网黄页）
- **结果**：一番折腾后，老天保佑，我们拿到了目标服务器的 IP 地址（拿到了饭店的绝对经纬度坐标）。

#### 2. 建立 TCP 连接（打通订餐电话，拿稳号牌）
- **现象**：知道了饭店在哪，你不能直接把外卖单甩过去，你要先确定那边是不是开门营业。
- **动作**：此时浏览器会通过 HTTP 协议向服务器发起连接请求，而 HTTP 的底层是基于 TCP 的，所以这里必须触发大名鼎鼎的 **[[#TCP三次握手与四次挥手|TCP 三次握手]]**。
- **结果**：客户端发 SYN，服务端回 SYN+ACK，客户端再回 ACK。确认双方都能听见并且愿意干活后，这条可靠的数据传输通道就正式搭建完毕了。如果是 HTTPS，在这里还会多出一个 **TLS/SSL 握手**（互相交换加密秘钥，防止别人偷听点餐内容）。

#### 3. 发送 HTTP 请求（把写好的外卖单递过去）
- **现象**：电话接通了，现在可以光明正大地要求发菜过来了。
- **动作**：浏览器依照 HTTP 协议的标准，把要求打包成一个“请求报文”发过去。报文包含了**请求行**（我要 `GET` 拿主页）、**请求头**（我是 Chrome 浏览器、你给我返回 HTML 格式）、以及**请求体**（POST 时会带的表单数据）。

#### 4. 服务器处理请求并返回响应（后厨炒菜并装盒送出）
- **现象**：饭店后厨（服务器的 Tomcat/Nginx、后端的 Spring Boot 等）收到了这张单子。
- **动作**：服务器后端程序根据你的请求去数据库查信息，经过运算后，生成了一份 HTML 文件（也就是你要的网页），再把它塞进 HTTP “响应报文”里。
- **结果**：响应报文里会带有状态码（比如 `200 OK` 代表炒菜成功，`404 Not Found` 代表菜卖光了没找到，`500` 代表厨师把锅炸了）。

#### 5. 浏览器解析渲染页面（拆外卖包装、组装材料）
这是一个纯粹的前端工程过程。浏览器收到代码后，它拿到手的只不过是一连串**毫无生机的、干瘪的“纯英文字符串”**，它必须要像拼装宜家家具一样把它翻译出来：
- **什么是 HTML 与 HTML 解析？**：**HTML（HyperText Markup Language，超文本标记语言）** 就是带有一堆尖括号标签的纯文本图纸（比如 `<h1>标题</h1>`）。**HTML 解析**，就是浏览器里的引擎开始从上到下、逐行逐字地阅读这些枯燥的文本符号，把它们像“翻译文言文”一样转换成计算机能轻松管理的数据结构。
- **什么是 DOM 树？**：**DOM（Document Object Model，文档对象模型）**。经过了刚刚的翻译解析后，浏览器在内存里建起了一棵倒挂的“家族大树”。比如 `<html>` 是最顶层的太爷爷，它的下面细分出 `<head>` 和 `<body>` 两个爷爷，然后再往下长出了一堆 `<div>` 和 `<p>` （大标题、段落文本）的子子孙孙。这就是把那一长串死文本，变成了内存里活生生的、方便代码增删改查的**“网页立体骨架”**。
- **什么是 CSS 解析与 CSSOM 树？**：如果说 HTML 是网页的裸体骨架，那 **CSS（层叠样式表）** 就是用来做装潢和买衣服的规则（比如规定“所有的标题必须是红色的、字号为 24px”）。浏览器阅读并且翻译这些样式表规则的过程就是 **CSS解析**，解析完后，同样会在内存里盖起一棵与骨架完美贴合的 **CSSOM 树（样式规则树 / 衣服裁剪版型）**。
- **合并渲染树（Render Tree）**：把骨架（DOM 树）和量身定制的衣服（CSSOM 树）穿在一起，计算每一个按钮、每一行字在屏幕上的确切物理坐标（Layout 布局计算），最后调用显卡把它涂上颜料“画”（Paint）在屏幕上。
- **JS 处理**：在此期间如果遇到了 `<script>` 标签里的 JavaScript 代码，引擎会暂停渲染去执行 JS，JS 就是用来给刚才的静止画面加上“行为”的（比如点个按钮会弹出一个警告框，这就是改变了 DOM 树或者 CSSOM 树）。

#### 6. 断开连接（吃完抹嘴走人）
- **现象**：网页完全展示在了你面前，数据传输结清了。
- **动作**：如果 HTTP 头部里没有要求 `keep-alive`（保持连接续杯），为了不占用别人的电话线，TCP 会启动著名的 **[[#TCP三次握手与四次挥手|四次挥手]]** 把这个连接通道优雅地切断。释放双方系统的内存和端口资源。

## 三、应用层协议：HTTP 与 HTTPS 面试核心

### 什么是 HTTP 与 HTTPS？（“裸奔”与“武装”的快递员）
很多时候我们天天在地址栏敲这两个东西，却不知道它们有生与死的安全级别差异。

#### 什么是 HTTP？（明文裸奔的快递员）
**HTTP（HyperText Transfer Protocol，超文本传输协议）**，是互联网上应用最为广泛的一种网络协议。它规定了浏览器和服务器之间怎么传递那个“宜家图纸（HTML）”。
- **致命弱点**：HTTP 的数据是**完全明文传输**的。这就好比你在寄快递的时候，把一叠写着你银行卡密码的 A4 纸直接光溜溜地塞给顺丰小哥。网管、黑客、甚至是咖啡厅的免费 WiFi 路由器，只要在这个传输链路上的任何人，都可以轻而易举地把纸抽出来看（窃听），甚至把你纸上的卡号偷偷用橡皮擦掉换成他自己的（篡改）。

#### 什么是 HTTPS？（套上黑客无法破解的密码箱）
**HTTPS（HyperText Transfer Protocol Secure，超文本传输安全协议）**，那个多出来的 `S` 就是 Secure（安全）。
- **物理隐喻**：HTTPS 本质上依然是让那个快递员去送信，但这次，你在寄信之前，**把信锁进了一个极其坚固的钛合金密码箱里**。快递员、中间路由节点、黑客即使抱住了这个箱子，因为没有钥匙，他们看到的就是一堆无规则的乱码。就算他们暴力把箱子砸烂，内容也直接销毁了。

#### TLS 和 SSL 又是个啥？（那个密码箱的学名）
你经常听到的 SSL/TLS，其实就是 HTTPS 里面用来加密的那把**“锁头（加解密协议）”**。
- **SSL (Secure Sockets Layer，安全套接层)**：设计于 1990 年代的老古董防盗门，现在早就被黑客找出了大量漏洞，已经被全网淘汰弃用。
- **TLS (Transport Layer Security，传输层安全协议)**：它是 SSL 的**直系亲属和威力加强版升级换代产品**（就像是钛合金密码箱）。因为 SSL 名字太火了，所以现在大家平常嘴里习惯性喊的“配置 SSL 证书”，其实底层跑的早就全都是更安全的 TLS 协议了（目前主流是 TLS 1.2 和 TLS 1.3）。
- **HTTPS 的真正身份**：HTTPS 根本不是一个全新的协议，它等于 **HTTP + TLS/SSL**。也就是“跑在密码箱之内的 HTTP”。

#### 核心区别大总结（面试必杀技）
1. **安全性（本质区别）**：HTTP 是明文裸奔传输，极度危险；HTTPS 是装在 TLS/SSL 密码箱里的加密传输，防窃听、防篡改。
2. **默认端口不同**：HTTP 默认敲门的是服务器的 **80** 端口；HTTPS 默认敲门的是 **443** 端口。
3. **成本与门槛**：HTTP 是个土大款，随便搭；HTTPS 需要向专业的 CA（证书颁发机构）申请**数字证书**（用来向浏览器证明“我是个正经的好人，不是钓鱼伪造的网站”）。
4. **连接耗时**：HTTP 只需要经过 **TCP 三次握手** 就能直接抛数据；而 HTTPS 在 TCP 三次握手打通底层信道之后，还要接着进行非常繁琐的 **TLS 四次握手**（通过极其复杂的非对称加密去偷偷交换密码箱的钥匙），所以 HTTPS 的首屏握手加载通常会比 HTTP 慢那么一两百毫秒。
