# 简历写法指南

> 简历三大核心模块（专业技能 / 项目经历 / 实习经历）的写法规范。好的面试官通常从这三块展开提问，篇幅占比也最高。
>
> 配套范例见 [[RAG项目简历范例]]。

> [!tip] 速览（一分钟读完）
> - **专业技能**：大小写规范 + 主动带出核心八股（"想问就问这些"）
> - **项目经历黄金公式**：用了什么技术栈 → 解决了什么问题 → 实现了什么业务 → 有哪些量化数据
> - **项目四注意**：描述简洁、技术栈不重复列、淘汰技术别写、突出"解决问题"而非堆业务
> - **实习经历**：本质是"披着公司 title 的项目经历"

## 一、专业技能

### 反面写法

```text
【数据库】熟悉关系型数据库 mysql、redis；
```

两个问题：

1. **大小写不规范**——`MySQL`、`Redis` 才是专业写法（`mysql`/`redis` 显得不熟）。
2. **没带出核心八股**——只说"熟悉"，不告诉面试官该问什么。

### 正面写法

主动把每个技术的**核心考点列出来**，等于告诉面试官"想问八股就问这些"——把面试引导到自己的舒适区：

```text
- MySQL：熟练使用 MySQL，掌握索引、事务、锁、日志等核心概念，具备 SQL 性能优化的能力。
- Redis：对 Redis 的底层数据结构、缓存策略及持久化方法有深入了解，能有效解决缓存穿透、缓存击穿和缓存雪崩问题。
```

三个加分细节：

- 中英文之间**加空格**（`使用 MySQL` 而非 `使用MySQL`）
- 标点用**中文标点**
- 关键字**大小写完全正确**

### 一页简历的精简版

如果只有一页、空间紧张，可以删掉 `【分类】` 前缀，直接平铺：

```text
- 熟悉 Redis 的常见数据类型和应用场景，以及持久化、过期淘汰策略、分布式锁等特性。
- 熟练使用 MySQL，掌握索引、事务、锁、binlog/redo log/undo log 等。
```

## 二、项目经历

> 项目经历通常是简历中**第一/第二重要**的内容。

### 黄金公式

> **用了什么技术栈，解决了什么问题，实现了什么业务，有哪些量化数据。**

反面示例的通病：

- 项目名太直白、太"捡漏"
- 没有周期（起止时间）
- 没有技术栈、没有项目描述
- 全是无关紧要的业务描述堆砌

### 标准结构

```text
项目名 ｜ 角色（2024.09 - 2025.03）

项目描述：一段话讲清做什么、给谁用、核心价值（结合大模型/向量检索等技术亮点）。

技术栈：Spring Boot、MySQL、Redis、Kafka、Elasticsearch ...

核心职责：
- 每条 bullet = 技术手段 + 解决的问题 + 量化结果
```

具体 bullet 范例见 [[RAG项目简历范例]]。

### 四个注意事项

| # | 注意点 | 说明 |
|---|--------|------|
| 1 | **项目描述简洁** | 别动辄 300-400 字。简洁明了即可 |
| 2 | **技术栈不重复列** | 两个项目技术栈相同，第二个就不必再列。避免"六页简历技术栈列不停" |
| 3 | **淘汰技术别写** | JSP、jQuery 等过时技术不要写；落后就主动改进优化 |
| 4 | **突出"解决问题"** | 重点是技术栈解决了什么问题、有什么性能提升/亮眼表现，**不是业务堆砌** |

## 三、实习经历

> **实习经历 = 披着公司 title 的项目经历**——本质就是多了个公司名的项目经历。

- 写法与项目经历**完全一致**（技术栈通用，只是业务不同）。
- 如果觉得实习"很水、乏善可陈"，就想办法**润色技术细节**——因为底层技术栈是通用的。

## 关键认知

1. **专业技能要"引导提问"**——主动列核心考点，把面试引到自己准备充分的领域。
2. **项目经历套黄金公式**——技术栈 + 问题 + 业务 + 量化，缺一不可，尤其量化数据是亮点。
3. **业务堆砌是大忌**——面试官要看的是"你用技术解决了什么"，不是"这个项目有哪些功能"。
4. **实习 ≈ 项目**——用同一套公式写。
