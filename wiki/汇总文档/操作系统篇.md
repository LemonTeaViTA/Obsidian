# 操作系统篇

## 一、内存管理

### 为什么要用虚拟内存而不是直接使用物理内存？

这道题的本质是：如果让每个进程直接操作物理内存地址，会出什么灾难？

#### 没有虚拟内存的世界（三大致命问题）

用"合租房"的隐喻来理解。假设物理内存是一间大通铺，所有进程（租客）直接睡在同一张床上，各自记住自己睡在"第几米到第几米"：

1. **安全隔离崩塌**：进程 A 可以随意读写进程 B 的内存区域。一个有 bug 的程序（甚至恶意程序）能直接篡改别人的数据，甚至改掉操作系统内核的数据，整个系统随时崩溃。就像合租室友可以随便翻你的抽屉。

2. **内存使用效率极低**：每个进程必须一次性获得连续的物理内存块。当多个进程反复启动和退出后，物理内存中会出现大量不连续的碎片空洞（外部碎片），明明总剩余内存够用，但因为不连续，新进程就是放不进去。

3. **程序地址不确定**：程序每次运行时被加载到哪段物理地址是不确定的，程序内部的跳转指令、全局变量地址都得在加载时重新计算（重定位），极其麻烦且容易出错。

#### 虚拟内存怎么解决这些问题？

虚拟内存的核心思想：给每个进程一个**独立的、完整的、假的地址空间（虚拟地址空间）**，让它以为自己独占了整台机器的内存。操作系统和硬件（MMU）在背后默默地把虚拟地址翻译成真实的物理地址。

- **隔离性**：每个进程只能看到自己的虚拟地址空间，根本摸不到别人的物理内存。就像每个租客有了独立的房间和门锁。
- **消除碎片**：虚拟地址空间是连续的，但背后映射的物理内存页可以散落在任意位置。操作系统以"页"为单位管理内存（通常 4KB 一页），不需要连续的物理块。
- **地址确定**：每个进程的虚拟地址空间布局是固定的（代码段从 0x400000 开始、栈从高地址向下长），编译器生成的地址不需要运行时重定位。
- **超卖能力**：虚拟地址空间可以比物理内存大得多。暂时不用的页可以被换出（swap）到磁盘，需要时再换回来，实现"用 8GB 物理内存跑 16GB 的程序"。

### 虚拟地址到物理地址是怎么映射的？

这是上一题的必然追问。面试官想看你是否理解页表和 TLB 的工作机制。

#### 1. 分页机制（Paging）

操作系统把虚拟地址空间和物理内存都切成固定大小的块：虚拟空间的块叫**虚拟页（Page）**，物理内存的块叫**物理页帧（Frame）**，大小通常都是 **4KB**。

一个虚拟地址被拆成两部分：

```
虚拟地址 = [ 虚拟页号 (VPN) | 页内偏移 (Offset) ]
```

- **虚拟页号（VPN）**：用来查页表，找到对应的物理页帧号。
- **页内偏移（Offset）**：页内的第几个字节，直接拼到物理页帧号后面。

翻译过程：`物理地址 = 页表[VPN] 得到的物理页帧号 + Offset`。

#### 2. 页表（Page Table）

每个进程有自己的页表，本质上就是一个映射数组：下标是虚拟页号，值是物理页帧号 + 权限位（可读/可写/是否在内存中）。

问题是：32 位系统的虚拟地址空间有 4GB，按 4KB 分页就有 100 万个页表项，每项 4 字节，单个进程的页表就要占 4MB。如果有 100 个进程，光页表就吃掉 400MB 内存。

#### 3. 多级页表（解决页表太大的问题）

实际中大部分虚拟页是没有被使用的（进程不可能用满 4GB）。多级页表的思路是：**只为实际使用的虚拟地址区域分配页表**，没用到的区域连页表都不创建。

以二级页表为例，虚拟地址被拆成三段：

```
虚拟地址 = [ 一级页号 | 二级页号 | 页内偏移 ]
```

- 一级页表（页目录）只有 1024 项，指向二级页表。
- 只有被使用的一级页表项才会分配对应的二级页表。
- 大量未使用的虚拟地址区域不会产生任何页表开销。

64 位系统（如 x86-64）使用**四级页表**，Linux 5.x 甚至引入了五级页表来支持更大的地址空间。

#### 4. TLB：页表的高速缓存

每次访问内存都要查多级页表（二级页表要查 2 次内存，四级要查 4 次），太慢了。CPU 内部有一个专门的硬件缓存叫 **TLB（Translation Lookaside Buffer，地址转换后备缓冲）**，缓存最近使用过的"虚拟页号 → 物理页帧号"映射。

- **TLB 命中**：直接拿到物理地址，不需要查页表，速度和访问 L1 缓存一样快。
- **TLB 未命中**：走多级页表查询，查到后把结果写入 TLB 供下次使用。
- **进程切换时**：因为每个进程的页表不同，TLB 中的旧映射全部失效，需要刷新（flush）。这就是为什么进程切换比线程切换代价更大——线程共享地址空间，不需要刷 TLB。

#### 5. 缺页中断（Page Fault）

当 CPU 访问一个虚拟页，但页表显示该页不在物理内存中（可能还没加载，或者被换出到磁盘了），就会触发**缺页中断**：

1. CPU 暂停当前指令，陷入内核态。
2. 操作系统找到该页在磁盘上的位置（swap 分区或可执行文件）。
3. 如果物理内存满了，用页面置换算法（LRU 等）淘汰一个旧页。
4. 把需要的页从磁盘读入物理内存，更新页表和 TLB。
5. 重新执行刚才那条被打断的指令。

这就是虚拟内存"超卖"能力的底层支撑——不是所有页都需要同时驻留在物理内存中。

## 二、进程与线程

### 进程和线程的区别是什么？

#### 核心解析

**进程**是操作系统资源分配的基本单位，拥有独立的地址空间、文件描述符表、信号处理器等。**线程**是 CPU 调度的基本单位，同一进程内的线程共享地址空间和资源，但各自拥有独立的栈、寄存器和程序计数器。

| 维度 | 进程 | 线程 |
|------|------|------|
| 地址空间 | 独立，互相隔离 | 共享所属进程的地址空间 |
| 创建开销 | 大（需要分配独立资源） | 小（只需分配栈和寄存器） |
| 切换开销 | 大（需要切换页表、刷新 TLB） | 小（共享页表，不需要刷 TLB） |
| 通信方式 | IPC（管道、消息队列、共享内存、Socket） | 直接读写共享变量（需要同步） |
| 崩溃影响 | 一个进程崩溃不影响其他进程 | 一个线程崩溃会导致整个进程终止 |

面试一句话：进程是资源容器，线程是执行单元。

### 协程和线程有什么区别？

#### 核心解析

线程由操作系统内核调度，切换时需要陷入内核态，开销在微秒级。协程由用户态程序自己调度，切换只需要保存和恢复少量寄存器，开销在纳秒级。

| 维度 | 线程 | 协程 |
|------|------|------|
| 调度者 | 操作系统内核 | 用户态调度器 |
| 切换开销 | ~1-10μs（内核态切换） | ~100ns（用户态切换） |
| 栈大小 | 默认 1-8MB | 通常 2-8KB，可动态增长 |
| 并发模型 | 抢占式 | 协作式（主动让出） |
| 典型实现 | pthread、Java Thread | Go goroutine、Python asyncio、Kotlin coroutine |

Go 的 goroutine 是协程的典型代表：M:N 模型，M 个 goroutine 映射到 N 个 OS 线程上，由 Go runtime 调度。

## 三、进程调度

### 常见的进程调度算法有哪些？

#### 核心解析

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| FCFS（先来先服务） | 按到达顺序排队 | 简单公平 | 短作业等太久（护航效应） |
| SJF（最短作业优先） | 优先执行预计时间最短的 | 平均等待时间最优 | 长作业可能饿死；难以预估执行时间 |
| RR（时间片轮转） | 每个进程分配固定时间片，用完就排队尾 | 响应快，适合交互式系统 | 时间片太小→切换开销大；太大→退化为 FCFS |
| 优先级调度 | 按优先级执行 | 重要任务优先处理 | 低优先级可能饿死（需要老化机制） |
| 多级反馈队列（MLFQ） | 多个优先级队列，新进程从最高优先级开始，用完时间片降级 | 兼顾响应和吞吐 | 实现复杂 |

Linux 的 CFS（Completely Fair Scheduler）用红黑树维护所有可运行进程，按虚拟运行时间（vruntime）排序，每次选 vruntime 最小的进程执行，保证每个进程获得公平的 CPU 时间。

## 四、同步与锁

### 什么是死锁？产生条件和解决方法？

#### 核心解析

死锁是两个或多个进程互相等待对方持有的资源，导致所有进程都无法继续执行。

**四个必要条件（同时满足才会死锁）：**
1. **互斥**：资源同一时刻只能被一个进程持有
2. **持有并等待**：进程持有至少一个资源，同时等待获取其他资源
3. **不可抢占**：资源只能由持有者主动释放，不能被强制剥夺
4. **循环等待**：存在进程的循环等待链

**解决方法：** 破坏任意一个条件即可。
- 破坏持有并等待：一次性申请所有资源
- 破坏不可抢占：申请不到就释放已持有的资源
- 破坏循环等待：对资源编号，按序申请

**检测与恢复：** 允许死锁发生，用资源分配图（Wait-For Graph）检测环路，发现后终止环中某个进程或回滚。

## 五、文件系统

### Linux 文件系统的基本结构是什么？

#### 核心解析

Linux 一切皆文件。文件系统的核心抽象是 **inode + 数据块**：

- **inode（索引节点）**：存储文件的元数据（大小、权限、时间戳、数据块指针），不存文件名。每个文件对应一个唯一的 inode 号。
- **数据块（Data Block）**：存储文件的实际内容，通常 4KB 一块。
- **目录**：本质上也是一个文件，内容是"文件名 → inode 号"的映射表。

查找 `/home/user/test.txt` 的过程：根目录 inode → 找到 `home` 的 inode → 找到 `user` 的 inode → 找到 `test.txt` 的 inode → 读取数据块。

**硬链接 vs 软链接：**
- 硬链接：多个文件名指向同一个 inode，删除一个文件名不影响其他的，引用计数归零才真正删除
- 软链接（符号链接）：一个独立的文件，内容是目标文件的路径字符串，类似快捷方式

## 六、I/O 模型

### IO 多路复用

#### 核心解析

传统阻塞 IO 模型中，一个线程只能等待一个连接的数据，高并发场景下需要大量线程，开销巨大。IO 多路复用让**一个线程同时监听多个文件描述符**，哪个就绪就处理哪个。

#### select / poll / epoll 对比

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| 数据结构 | 固定大小的 bitmap（fd_set） | 动态数组（pollfd） | 内核红黑树 + 就绪链表 |
| 最大连接数 | 1024（FD_SETSIZE 硬编码） | 无上限（数组动态扩展） | 无上限 |
| 内核实现 | 每次调用都遍历所有 fd | 每次调用都遍历所有 fd | 事件驱动，回调机制，只返回就绪的 fd |
| 时间复杂度 | O(n) | O(n) | O(1)（就绪事件获取） |
| fd 拷贝 | 每次调用都要把 fd 集合从用户态拷贝到内核态 | 同 select | fd 只需注册一次（epoll_ctl），不需要反复拷贝 |
| 触发模式 | 水平触发（LT） | 水平触发（LT） | 支持水平触发（LT）和边缘触发（ET） |

**为什么 epoll 是 Linux 高并发的标配？**

select/poll 的致命问题：每次调用都要把所有 fd 拷贝到内核，内核再逐个检查是否就绪，O(n) 扫描。1 万个连接中只有 10 个活跃，也要扫描 1 万次。

epoll 的三个系统调用：
1. `epoll_create`：创建 epoll 实例（内核中的红黑树 + 就绪链表）
2. `epoll_ctl`：注册/修改/删除要监听的 fd（只需操作一次）
3. `epoll_wait`：阻塞等待，只返回就绪的 fd 列表

当网卡收到数据时，内核通过回调函数把就绪的 fd 加入就绪链表，`epoll_wait` 直接从链表取，不需要遍历。

**水平触发（LT）vs 边缘触发（ET）：**
- LT：只要 fd 的缓冲区有数据可读，每次 `epoll_wait` 都会返回它。编程简单，但可能重复通知。
- ET：只在 fd 状态变化时通知一次（从无数据变为有数据）。必须一次性读完所有数据（循环读到 EAGAIN），否则不会再通知。性能更高但编程复杂。

**工程应用：** Nginx、Redis、Netty 底层都用 epoll。Java NIO 的 Selector 在 Linux 上就是 epoll 的封装。
