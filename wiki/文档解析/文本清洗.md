# 文本清洗

> 文档解析输出原始文本之后、分块入库之前的关键工程步骤。==清洗不是越多越好==——过度清洗（特别是粗暴去重）会丢失业务关键信息。本文讲清楚清洗要做什么、怎么做，以及哪些"清洗"反而会害事。

---

## 一、文本清洗要做的三件事

| 目标 | 说明 |
|------|------|
| 消除噪声 | 去掉与正文无关的元素：页眉页脚、水印、导航、广告 |
| 规范化格式 | 编码统一、空白字符规范、不可见字符去除 |
| 处理重复 | ==不是简单"去重"==，而是判断哪些重复是噪声、哪些是数据本身的特征 |

==所有清洗操作都遵循同一个原则==：宁可保留可疑内容，不要误删业务信息。==误删比误留代价高得多==——保留的噪声只是稀释，但删错的数据没法找回。

---

## 二、噪声去除

### 2.1 页眉页脚怎么判断

PDF / Word / 扫描件里页眉页脚没有显式标签（PDF 完全没有"页眉"概念）。识别靠==多个特征联合判断==：

| 特征 | 含义 |
|------|------|
| ==位置== | 出现在每页固定的顶部/底部区域（页面高度的前 8% 或后 8%） |
| ==跨页重复== | 多个页面在相同位置出现高度相似的文本（公司名、文件标题、页脚版权声明） |
| 字号字重 | 通常比正文小 1-2 号，或者字重不同 |
| 独立文本块 | 与正文之间有明显空白间距 |
| 模式特征 | 包含页码（如 `第 X 页 / 共 Y 页`、`Page X of Y`）、日期、版本号 |

### 实际检测算法

```
1. 把多页文档每页按 y 坐标分成"顶部 / 中部 / 底部"
2. 对顶部文本块做跨页比对：
   - 完全相同 → 一定是页眉
   - 90%+ 相似（仅页码不同）→ 是页眉
3. 同样处理底部
4. 工具自动加 type=header/footer 元数据
```

==开源实现==：Docling、MinerU、unstructured 内部都做了页眉页脚检测，PyMuPDF 也有 `page.get_text("dict")` 提供 bbox 用于位置判断。

> [!tip] 不要直接删，应该打标签
> 推荐做法：识别出页眉页脚后==打 `type=header` / `type=footer` 元数据==而不是物理删除。==因为有时候页眉里有有用信息==（如章节标题作为页眉），后续 chunk 可能需要从页眉补全 section_path。

### 2.2 PDF 水印和装饰文本

水印没有"我是水印"标签，识别靠几个特征：

| 特征 | 说明 |
|------|------|
| ==半透明 / 灰度低== | PDF 中水印对象的 alpha 通道或颜色亮度通常偏低 |
| ==旋转角度非零== | "机密"、"DRAFT"、"内部资料"通常 30-60 度对角线放置 |
| ==字号异常大== | 水印往往跨整页，字号远超正文 |
| ==重复同一字符串== | 整篇文档每页都是同一行字 |
| 位于内容层下方 | PDF 渲染顺序中水印通常先绘制在底部 |
| 文本无逻辑结构 | 不参与正文段落流，独立坐标 |

实际检测：

```
PDF 解析时拿到每个文本块的 bbox + 颜色 + 旋转角度 + 字号
   ↓
特征联合判断：
  rotation != 0 ?
  font_size 异常大 ?
  alpha < 0.5 ?
  文本在多页中重复出现 ?
   ↓
  打 type=watermark 标签
```

==装饰文本==（如分隔符 `========`、`★★★★★`、ASCII art）通常用正则识别——连续重复的同一字符就是装饰。

### 2.3 HTML 导航/广告/Cookie 提示

HTML 因为有标签信息，相对好处理。主流工具：
- **Trafilatura**：基于文本密度和 DOM 树结构识别正文区域
- **Readability.js**：Mozilla 出的正文提取算法
- **HtmlRAG**：保留有语义的 HTML 标签（标题/列表/表格），过滤无语义容器

详见 [[RAG基础与架构#3.4.5 HTML：DOM 噪声与结构语义保留]]。

## 三、编码与格式规范化

这一步相对机械，主要是各种==字符替换==：

| 问题 | 处理 |
|------|------|
| 全角/半角不一致 | 统一转半角（数字、字母、标点） |
| 各种引号 | `""''"'` → `"'` |
| 不可见字符 | 零宽空格 `​`、零宽连接符 `‍`、BOM `﻿` 直接删 |
| 多余空白 | 连续空格压缩为一个，连续换行压缩为最多两个 |
| Unicode 标准化 | NFC / NFKC 规范化，统一字符的多种编码形式 |
| 编码错误（乱码） | 检测 UTF-8 / GB18030 / Latin-1 后转码；OCR 出来的乱字符标记为待人工 |

==这一步要保守==：不要轻易做"简繁转换"或"中英标点统一"，业务上可能两种都要保留。

## 四、重复内容检测的三个层级

去重是文本清洗里==最容易出错==的环节。把"重复"分成三层来理解：

| 层级 | 判定方式 | 该不该删 |
|------|---------|---------|
| L1 字节级重复 | 整个文档 SHA-256 一致 | ==删==（同一文件被多次摄入，常见于爬虫/上传重试） |
| L2 chunk 级碎片重复 | 多个文档都包含同一段（如免责声明、版权声明） | ==独立成 chunk + 引用==，不是删 |
| L3 近似重复 | MinHash / SimHash 相似度高 | ==看场景，对正式文档通常不删== |

### 4.1 字节级去重（L1，最安全）

入库前对原始文档算 SHA-256：

```python
import hashlib

def file_hash(path):
    with open(path, "rb") as f:
        return hashlib.sha256(f.read()).hexdigest()

# 维护一张已摄入哈希表，新文件哈希命中即跳过
```

==代价为零，应该做==。覆盖"同一份 PDF 被上传两次"、"爬虫重复抓取"等场景。

### 4.2 chunk 级碎片重复（L2，独立成 chunk）

很多文档包含相同片段：
- 公司每份合同都有同一段免责声明
- 每份手册都有同一份版权声明
- 每份产品规格都有同一段安全提示

==处理方式不是删，而是去重存储==：
1. 检测高频重复 chunk（出现在 N% 以上文档中）
2. 提取为独立 chunk，赋一个公共 ID
3. 各个文档通过 ID 引用，避免向量库里同一段被存上百次

==这样既节省存储和向量计算，又保留了"这个文档包含这段声明"的关联==。

### 4.3 近似重复（L3，==慎用==）

近似去重的标准技术：
- **SimHash**：把文本哈希到固定长度的二进制，海明距离衡量相似度
- **MinHash**：把文本表示为 shingle 集合，用 Jaccard 相似度
- **LSH**（Locality Sensitive Hashing）：在大规模数据下加速近似搜索

> [!warning] ==近似去重对正式文档常常误伤==
>
> "形态相似" ≠ "语义重复"。在企业知识库场景，形态相似常常意味着合法的数据特征：

| 场景 | 文档间差异 | 简单近似去重的后果 |
|------|----------|------------------|
| 历年财报 | 只是数据和日期变了，结构 95% 一样 | 只留最新一年，丢失历史对比 |
| 合同模板的多份实例 | 甲方/乙方/金额/日期不同 | 只留一份，其他客户的合同都查不到 |
| 法规版本迭代 | 某条修订，新旧版差几个字 | "2023 版" vs "2024 版"被合并 |
| 产品 SKU 说明书 | 同系列产品规格表，差几个参数 | 所有 SKU 文档归并为一份 |
| 政策文件的批注转发 | 同一文件 + 各部门批注 | 丢失批注信息 |

==这些场景里"相似"恰恰是数据本身的特征==，把它们当噪声删掉就是丢业务数据。

**近似去重适用的场景**：
- 网络爬虫数据（去重转载和镜像）
- 训练语料预处理
- 论坛/UGC 数据（重复发帖）

**近似去重不适用的场景**：
- 企业知识库（财报、合同、规章、手册等正式文档）
- 任何"版本迭代是数据本身"的场景

## 五、不去重而靠元数据 + 检索时过滤

工业界 RAG 处理"重复"的标准模式：==入库阶段保留所有版本，记录足够元数据，靠检索时过滤来消解==。

### 5.1 保留所有版本，但记录元数据

```python
chunk_metadata = {
    "doc_id": "财报_2023Q4",         # 文档级唯一 ID
    "version": "v2",                 # 版本号
    "issue_date": "2024-01-15",      # 签发/更新时间
    "doc_type": "quarterly_report",  # 文档类型（让检索可按类型过滤）
    "entity": "ACME公司",            # 业务实体
    "supersedes": "财报_2023Q4_v1",  # 此版本替代了哪个旧版本
    "is_template_instance": True,    # 是否模板实例
    "source_authority": 1,           # 来源权威等级
}
```

### 5.2 检索时按用户意图过滤

不在入库时做语义判断，==在检索时通过策略避免误返==：

| 用户意图 | 检索策略 |
|---------|---------|
| 只想要最新的 | `WHERE doc_type=X ORDER BY issue_date DESC LIMIT 1` |
| 想要历年对比 | 全部召回，按时间排序展示 |
| 想要某个客户的合同 | 按 `entity` 元数据过滤 |
| 想要看条款变化 | 全部召回，让 LLM 对比差异 |

==这样"重复"在用户实际查询时才被消解，而不是在入库时被简单删除==。

### 5.3 版本链管理（可选，进阶）

维护文档之间的"替代关系"图：

```
财报_2023Q4_v1 ──supersedes──→ 财报_2023Q4_v2 ──supersedes──→ 财报_2023Q4_v3
                                                                      ↑ 当前最新
```

检索时根据用户意图选取整条链或最末节点。这个机制配合==知识冲突处理==使用，详见 [[RAG高级技术#知识冲突处理]]。

## 六、清洗的取舍原则

文本清洗本质是==在"信噪比"和"信息完整性"之间做取舍==：

| 倾向 | 适用 |
|------|------|
| ==更激进的清洗== | 大规模通用语料、训练数据预处理 |
| ==更保守的清洗== | 企业知识库、合规文档、检索系统 |

后者宁可保留可疑内容，靠==元数据标签 + 检索时过滤==来处理"噪声 vs 信号"的不确定性。==因为清洗是不可逆的，标签是可逆的==。
