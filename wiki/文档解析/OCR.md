# OCR：光学字符识别

> **OCR = Optical Character Recognition（光学字符识别）**，从图像中提取文字。难点不在"认出一个字"，而在"==处理任意场景下的文字=="——多语言、艺术字、手写、低分辨率、复杂背景、歪斜、密集排版。

---

## 一、OCR 解决什么问题

**输入**：包含文字的图像（扫描件、拍照、屏幕截图、视频帧、自然场景照片）
**输出**：图像中所有文字的内容 + 位置（文本框坐标）

```
输入：[一张扫描的合同图片]
输出：[
  {text: "合同编号: HT-2026-0042", bbox: [120, 80, 580, 110]},
  {text: "甲方：北京XXX科技有限公司", bbox: [120, 130, 480, 160]},
  ...
]
```

### 现代 OCR 不只是"识字"

==当代 OCR 实际上是两个独立任务的组合==：

```
图像
  ↓
① 文本检测（Text Detection）：找出图中所有文本所在的区域
  ↓
② 文本识别（Text Recognition）：识别每个区域里的文字内容
  ↓
带坐标的文本结果
```

为什么要拆成两步：早期 OCR 只能处理"文档扫描件"这种规整场景（先二值化、找行、切字、识别）。但拍照/自然场景里文字是任意角度、任意位置的，必须先==检测==再==识别==。

---

## 二、文本检测的关键模型

文本检测本质是==特殊的目标检测==——目标是"文字行"或"文字块"。难点是文字形状任意（曲线、倾斜）、长宽比悬殊（一行字可能是 1000:30 的细条）。

### DBNet：可微分二值化（Differentiable Binarization）

**核心思想**：传统方法先用模型预测"概率图"（每个像素是文字的概率），然后阈值化得到二值图（是/否文字）。==阈值化是不可微的==，导致模型训练时这一步无法端到端优化。

DBNet 的创新：把二值化函数本身做成可微分的版本，让阈值也成为模型学到的参数。

```
图像 → CNN（FPN）→ 概率图 P + 阈值图 T → 可微二值化 → 二值结果
                                               ↑
                                         B = 1 / (1 + exp(-k(P - T)))
                                         k 控制陡峭程度
```

**优点**：精度高、推理速度快、对曲文/弯文鲁棒。==是 PaddleOCR 默认的检测模型==。

### 其他主流检测模型

| 模型 | 思路 | 特点 |
|------|------|------|
| EAST | 直接回归文字框四边形 | 快，但对曲文支持差 |
| PSENet | 渐进式扩展核（解决密集文字） | 精度高，速度慢 |
| TextSnake | 把文字建模为蛇形管道 | 适合任意形状 |

## 三、文本识别的关键模型

文本识别的输入是==裁剪好的文字行图像==，输出是文字字符串。本质是 image-to-sequence 任务。

### CRNN：卷积 + 循环 + CTC

==经典架构==，至今仍在生产环境广泛使用。

```
文字行图像（如 32×280 像素）
   ↓
CNN（VGG）：提取特征图
   ↓
特征序列（如 70 个时间步，每步 512 维）
   ↓
双向 LSTM：建模字符间上下文
   ↓
全连接 → 字符概率分布（每个时间步预测一个字符）
   ↓
CTC 解码：去重、去空白
   ↓
"合同编号: HT-2026-0042"
```

**关键技术：CTC（Connectionist Temporal Classification）**

CTC 解决"输出序列长度 < 输入时间步数"的问题。模型输出 70 个字符（包括空白和重复），CTC 通过动态规划计算"所有能合并出目标序列的路径"的总概率。

例：模型输出 `H-H-T-T-空-H` 经过 CTC 解码（合并相邻重复 + 去掉空白）→ `HTH`。

### SVTR：纯 Transformer 文本识别

==PaddleOCR v3+ 的识别模型==，在 CRNN 基础上去掉了 RNN，改为纯 Transformer。

**关键设计**：把图像 patch 化（类似 ViT），用 Transformer 直接建模 patch 之间的关系，识别速度比 CRNN 快很多。

### TrOCR：端到端 Transformer

Microsoft 提出。==彻底端到端==——视觉 Transformer Encoder + 文本 Transformer Decoder。

```
图像 → ViT Encoder → 特征 → Text Decoder（自回归生成）→ 字符串
```

适合多语言、长文本、艺术字。代价是计算量大。

## 四、PaddleOCR 与 PP-OCR 系列

==中文 OCR 事实标准==。PP 系列是百度针对工业场景的优化版：

```
PP-OCRv1（2020）→ v2 → v3 → v4 → v5（2024）
```

**PP-OCR 流水线：**
```
图像 → 文档方向分类（PP-LCNet）  ← 90度倍数旋转矫正
     → 文本检测（DBNet）
     → 文本方向分类（PP-LCNet）   ← 180度旋转矫正
     → 文本识别（SVTR / CRNN）
     → 后处理（拼接、格式化）
```

**v3 之后的关键改进**：
- 检测：DBNet++ 加入自适应特征融合
- 识别：SVTR 替换 CRNN+LSTM
- 模型小型化：MobileNetV3 backbone 让模型可以 CPU 部署

PP-OCRv5 在中文识别上达到工业级精度，模型体积约 10 MB（极致版本），单核 CPU 可跑。

## 五、端到端 OCR：VLM 直读

近两年趋势：==大模型直接看图说话==，跳过显式的检测+识别两阶段。

```
图像 → 视觉编码器 → 大语言模型 → "请识别图中所有文字" → 输出
```

代表：GPT-4V、Qwen-VL、InternVL。优势：
- 鲁棒性强（手写、艺术字、复杂背景都能处理）
- 不只识字，还能==理解版面语义==（"这是表格"、"这是图表标签"）
- 复杂版面 Pipeline 容易翻车的场景，VLM 反而稳

代价：==慢、贵、不能边端部署==。详见 [[VLM文档解析]]。

## 六、为什么扫描件 OCR 要配版面分析

直接对整张扫描件做 OCR 容易出问题：
- 多栏排版被读串行
- 表格被打散成普通文字
- 公式被识别成乱码
- 页眉页脚混入正文

正确做法：==先用[[版面分析]]模型识别每个区域的类型==（文字/表格/图片/公式/页眉页脚），再对不同区域用不同处理：
- 文字区域 → 文本 OCR
- 表格区域 → [[TSR]]
- 公式区域 → [[公式识别]]
- 页眉页脚 → 标记后过滤
