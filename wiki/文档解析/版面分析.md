# 版面分析（Layout Analysis）

> 版面分析是 Document AI 中最==基础==也最关键的一步——它告诉系统"页面里==哪一块是什么=="。是文字？标题？表格？图片？公式？页眉？侧边栏？没有这一步，后面的 [[OCR]]、[[TSR]]、[[公式识别]] 都不知道要送给什么模型处理。

---

## 一、版面分析解决什么问题

**输入**：页面图像（PDF 一页、扫描件一页、PPT 截图）
**输出**：页面内每个区域的边界框 + 类型标签 + ==阅读顺序==

```text
输入：[一张论文页面]

输出：
[
  {bbox: [...], type: "title",     order: 0},  ← 标题
  {bbox: [...], type: "author",    order: 1},  ← 作者信息
  {bbox: [...], type: "abstract",  order: 2},
  {bbox: [...], type: "text",      order: 3},  ← 左栏正文
  {bbox: [...], type: "text",      order: 4},  ← 右栏正文（按读序排在左栏后面）
  {bbox: [...], type: "table",     order: 5},
  {bbox: [...], type: "figure",    order: 6},
  {bbox: [...], type: "caption",   order: 7},
  {bbox: [...], type: "footnote",  order: 8},
  {bbox: [...], type: "page-num",  order: -1}, ← 页码（标记为不进入正文）
]
```

### 为什么这一步必须做

直接把整页图像送 [[OCR]] 会出问题：
- 多栏排版被读串行（左栏第一行 + 右栏第一行 → 拼成一行废话）
- 表格被打散成普通文字（行列结构丢失）
- 公式被识别成乱码
- 页眉页脚混入正文

==正确流程：先用版面分析模型把页面切成"已分类的区域"，再分类型送给对应模型==。

---

## 二、本质：目标检测 + 关系推理

版面分析可以拆成两个子任务：

```text
页面图像
   ↓
① 区域检测（Layout Detection）：标注每个区域的边界框 + 类型
   ↓
② 阅读顺序推理（Reading Order）：决定区域之间的先后关系
   ↓
带读序的结构化区域列表
```

==区域检测本质就是目标检测==——和 YOLO 检测人脸、检测车辆是同一类问题，只是目标变成了"标题/正文/表格/图片"等版面元素。

==阅读顺序==更微妙——多栏文档不能简单按 y 坐标排序，需要先识别栏，再栏内按 y、栏间按 x。

## 三、关键模型架构原理

### LayoutLM 系列（Microsoft）：多模态预训练

==版面分析的里程碑==。核心思想：把版面理解当作 BERT 风格的预训练任务，但==同时利用三种信号==：

```text
LayoutLM 输入 = 文本 + 2D 位置（bbox）+ 视觉特征
```

**LayoutLMv1** 在 BERT 基础上加 2D 位置编码（每个 token 有 x、y、w、h）。

**LayoutLMv3**：完全多模态预训练，把图像也 patch 化作为输入。三种预训练目标：
- Masked Language Modeling（恢复被遮盖的词）
- Masked Image Modeling（恢复被遮盖的图像 patch）
- Word-Patch Alignment（学习词与图像 patch 的对齐）

下游任务（如 DocVQA、版面元素分类）只要在预训练模型上微调一个分类头。==LayoutLMv3 在多个版面理解 benchmark 上是 SOTA==。

### DETR 系列：端到端目标检测

DETR（DEtection TRansformer，Facebook）把目标检测从"先生成候选框再分类"改为"直接生成检测结果"，消除了传统检测器（Faster R-CNN）的 anchor、NMS 等手工设计。

```text
图像 → CNN backbone → 特征图 → Transformer Encoder-Decoder → N 个目标查询
                                                              ↓
                                               每个查询输出: (类别, bbox)
```

**关键设计**：
- 用一组可学习的"目标查询"（object queries），每个查询负责预测一个目标
- 训练用==二分图匹配==（Hungarian matching）找预测和真值的最佳配对
- 完全端到端，不需要 NMS 后处理

### RT-DETR / DFINE：实时 DETR

DETR 精度高但慢。**RT-DETR（百度）**和 **DFINE** 通过架构优化把 DETR 加速到实时级别：
- 多尺度特征融合替代标准 Transformer 编码
- 解码器优化，减少层数
- IoU-aware 查询初始化

==Docling 用 RT-DETR/DFINE 训练的 DocLayNet 模型做版面检测==。

### PubLayNet / DocLayNet 数据集

模型再好，没数据也是空谈。版面分析的核心数据集：

| 数据集 | 来源 | 规模 | 类别 |
|-------|------|------|------|
| PubLayNet | IBM，PubMed 论文 | 36 万页 | 5 类（text/title/list/table/figure） |
| DocLayNet | IBM，多源 | 8 万页 | 11 类（含 caption、footnote、page-header 等） |
| M6Doc | 多模态文档 | 6 万页 | 74 类（极细粒度） |

==DocLayNet 是当前主流 Pipeline 方案的事实标准训练集==。

## 四、阅读顺序推理

检测到所有区域后，怎么决定"先读哪个再读哪个"？

### 基于规则的方法

朴素版：按 y 坐标从上到下、相同 y 按 x 从左到右。

但多栏排版会出错——两栏的同一水平位置都靠上。改进规则：
1. 先识别栏（用 X 投影直方图找栏间空白）
2. 同一栏内按 y 排序
3. 栏间按 x 排序（左栏全部 → 右栏全部）

==Docling、MinerU 等 Pipeline 方案大多用这种规则方法==——简单可靠，不需要训练。

### 基于学习的方法

把阅读顺序建模为==图上的拓扑排序==：每个区域是一个节点，模型预测节点之间的"先于/后于"关系，然后做拓扑排序。

代表：LayoutReader（Microsoft），用 BERT 风格模型预测顺序对。

适合复杂排版（不规则期刊、网页）。但训练数据少，泛化能力是挑战。

---

## 五、Pipeline 方案 vs 端到端 VLM

### Pipeline 方案（DocLayNet + RT-DETR + 阅读顺序规则）
- ==精度高、可解释、可调优==
- 速度快（CPU 也能跑）
- Docling、MinerU 走这条路

### 端到端 VLM 方案（Dolphin-v2、SmolDocling）
- 一个模型直接吐出版面元素序列（按读序）
- ==鲁棒性强==，对各种边缘场景（手写、复杂排版）更好
- 代价：GPU、慢、可解释性差
- 详见 [[VLM文档解析]]
