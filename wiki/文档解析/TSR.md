# TSR：表格结构识别

> **TSR = Table Structure Recognition（表格结构识别）**，从图像还原表格的行列单元格关系，把"一堆文字 + 横竖线"变回"结构化网格"。这是 Document AI 中最难的子任务之一。

---

## 一、TSR 解决什么问题

**输入**：表格图像（来自 PDF 截图、扫描件、PPT 中的表格图等）
**输出**：单元格的行列坐标 + 内容 + 合并关系（通常以 HTML `<table>` 或 Markdown 表格形式）

```
输入:  ┌─────┬─────┬─────┐
       │ 姓名 │ 部门 │ 工资 │
       ├─────┼─────┼─────┤
       │ 张三 │ 研发 │ 12000│
       └─────┴─────┴─────┘   ← 像素图

输出:  <table>
         <tr><th>姓名</th><th>部门</th><th>工资</th></tr>
         <tr><td>张三</td><td>研发</td><td>12000</td></tr>
       </table>           ← 结构化数据
```

==注意 TSR 不只是 OCR==。OCR 只能拿到"姓名/部门/工资/张三/研发/12000"这串文字，不知道哪个是表头、哪个对应哪个、行列怎么对齐。==TSR 是在 OCR 之上还原表格的几何结构和语义==。

### 为什么单独成为一个任务

表格在文档中无处不在但==格式千差万别==：
- 全边框表 / 半边框 / 无边框（仅靠空白对齐）
- 合并单元格（横跨多列、纵跨多行）
- 嵌套表格（单元格里又是一张表）
- 跨页表格（表头只在第一页）
- 表格里夹着图片或公式

通用 OCR 解决不了这些==几何结构问题==，需要专门的模型。

## 二、TSR 的任务可以拆成三个子任务

```
表格图像
   ↓
① 单元格检测：找出每个单元格的边界框（坐标）
   ↓
② 行列结构识别：判断单元格属于第几行第几列、是否合并
   ↓
③ 单元格内容识别：对每个单元格做 OCR
   ↓
结构化输出
```

不同的 TSR 方案对这三个子任务的处理方式不同：
- **Pipeline 方案**：三步分开做，每步用不同模型
- **端到端方案**：一个模型直接输出 HTML 序列

## 三、关键模型架构原理

### TableFormer（IBM Research）

==当前 Pipeline 方案的代表，Docling 内置使用==。

**架构：Encoder-Decoder Transformer**

```
表格图像
   ↓
CNN Backbone（ResNet-18）  ← 提取视觉特征
   ↓
Transformer Encoder         ← 编码空间关系
   ↓
   ┌──────────┴──────────┐
   ↓                     ↓
结构 Decoder              单元格 Decoder
（输出 HTML 标签序列）     （输出每个单元格的 bbox）
   ↓                     ↓
   <table><tr><td>...    [x1,y1,x2,y2]
```

**关键设计**：
- 双 Decoder 同时生成 HTML 结构标签（`<tr>`、`<td>`、`<td colspan=2>`）和对应单元格的边界框
- 训练时用 PubTabNet（IBM 发布的 50万 张科学论文表格图像数据集）
- 输出的是 HTML 序列，==天然支持合并单元格和复杂结构==

**为什么是序列生成**：
表格结构本质是树（或可视为序列化的 HTML），用 Transformer 逐 token 生成 HTML 标签是最直接的建模方式。

### TATR（Table Transformer，Microsoft）

**架构：DETR 改造**

把 TSR 当作==目标检测==问题：DETR 是 Facebook 的端到端目标检测架构，TATR 把 DETR 的"检测物体"改为"检测表格元素"——把表格的"行"、"列"、"单元格"、"表头"都当作不同类别的目标来检测。

```
表格图像 → CNN Backbone → DETR Encoder-Decoder → 目标列表
                                                  [行: bbox]
                                                  [列: bbox]
                                                  [单元格: bbox + 行索引 + 列索引]
                                                  [表头: bbox]
```

**优点**：训练简单（标准目标检测）、可解释性强（每个元素都有明确语义）
**缺点**：合并单元格的处理不如 TableFormer 自然

### 端到端 VLM 方案（Dolphin-v2、SmolDocling）

最新趋势：==用多模态大模型一次性输出表格的 Markdown/HTML==，跳过单元格检测和行列推理的显式中间步骤。

```
表格图像 → 视觉编码器 → 语言模型 Decoder → "| 姓名 | 部门 | 工资 |\n|---|---|---|\n..."
```

模型架构本质是 Vision-Language Model（详见 [[VLM文档解析]]）。优势是==鲁棒性强==（对各种变体的表格都能处理），代价是计算量大、需要 GPU。

## 四、训练数据集

TSR 模型的能力上限主要取决于训练数据：

| 数据集 | 来源 | 规模 | 特点 |
|-------|------|------|------|
| PubTabNet | IBM，PubMed 论文 | ~50 万张 | 科学论文表格，HTML 标注 |
| FinTabNet | IBM，财报 | ~11 万张 | 复杂财务表格，多合并单元格 |
| TableBank | Microsoft | ~41 万张 | 多领域表格 |
| ICDAR 系列 | 学术会议 | 较小 | 评测基准 |

==中文表格数据集仍然稀缺==，是中文 TSR 效果不如英文的主要原因。

## 五、TSR 的难点回顾

理解了模型架构后再看难点，更容易明白为什么这些场景特别难：

| 难点 | 为什么难 |
|------|---------|
| 无边框表格 | 没有视觉线条，只能从文本对齐推断行列，模型容易切错 |
| 合并单元格 | 需要预测 colspan/rowspan，端到端方案处理更自然 |
| 跨页表格 | 单页模型看不到下一页，需要在解析后做拼接 |
| 嵌套表格 | 一个单元格里又是 HTML，递归生成对端到端模型挑战大 |
| 手写或歪斜表格 | 几何结构不规则，传统 Pipeline 方案失效 |
