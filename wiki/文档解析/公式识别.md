# 公式识别（Math OCR）

> 公式识别是从图像还原数学公式的 LaTeX 或 MathML 表达。==普通 [[OCR]] 完全无法处理==——分数线、上下标、积分号、矩阵这些数学结构是二维的，标准 OCR 假设的"从左到右一行字"模型根本不适用。

---

## 一、公式识别解决什么问题

**输入**：包含数学公式的图像
**输出**：公式的 LaTeX 字符串（或 MathML / 自然语言描述）

```text
输入：[公式图像 E = mc²]
输出：E = mc^2

输入：[公式图像 ∫₀^∞ e^(-x²) dx = √π/2]
输出：\int_0^{\infty} e^{-x^2} dx = \frac{\sqrt{\pi}}{2}

输入：[公式图像 矩阵 [[a,b],[c,d]]]
输出：\begin{pmatrix} a & b \\ c & d \end{pmatrix}
```

### 为什么普通 OCR 不行

普通 OCR 假设"==文字是一维序列=="：图像里有一行字，从左到右识别。但数学公式是==二维结构==：

```text
2-D 视觉关系     →   一维 LaTeX 序列
                  
   a                   \frac{a}{b}
   ─                   
   b
                  
   x²                  x^2
                  
   ∫₀^∞                \int_0^{\infty}
```

==模型必须理解空间布局==——这个符号是在分数线上面（分子）还是下面（分母）？这个数字是上标（指数）还是下标？积分号上面的"∞"是积分上限。这些都是空间关系。

---

## 二、本质：image-to-sequence 任务

公式识别是==图像到序列==的转换问题，和机器翻译同构（图像 → LaTeX 类似中文 → 英文）。

```text
公式图像
   ↓
视觉编码器（CNN 或 ViT）：提取空间特征
   ↓
特征序列
   ↓
序列解码器（RNN 或 Transformer）：自回归生成 LaTeX 字符
   ↓
"\int_0^{\infty} e^{-x^2} dx"
```

==这就是为什么所有公式识别模型都是 encoder-decoder 架构==——和神经机器翻译的范式一脉相承。

## 三、关键模型架构原理

### im2markup（早期奠基工作）

2017 年 Harvard 的论文《Image-to-Markup Generation with Coarse-to-Fine Attention》。==第一个把神经机器翻译思路用到公式识别==的工作。

**架构：CNN + RNN + Attention**

```text
公式图像
   ↓
CNN：得到 H×W 的视觉特征图
   ↓
按行扁平化 → 序列特征
   ↓
注意力 RNN Decoder：每生成一个 token 都对视觉特征做注意力
   ↓
LaTeX 字符串
```

**关键洞察**：用注意力机制让模型在生成每个 LaTeX 符号时"==看回==图像中对应的位置"——比如生成 `\frac` 时关注分数线，生成分子时关注分子区域。

这套范式确立了后续所有公式识别模型的骨架。

### Pix2Tex（开源主流方案）

==当前最常用的开源公式识别模型==。

**架构：ResNet + Transformer**

```text
公式图像 (灰度)
   ↓
ResNet backbone：提取特征
   ↓
Transformer Encoder：编码空间关系
   ↓
Transformer Decoder（自回归）：生成 LaTeX token
   ↓
LaTeX 字符串
```

相比 im2markup 的改进：
- 用 Transformer 替代 RNN（更好建模长距离依赖）
- 在大规模 LaTeX 公式数据集上预训练
- 推理用 beam search 提升质量

==有 pip 包 pix2tex，开箱即用==。

### Nougat（Meta 2023）：端到端学术文档解析

Nougat 不只识别公式——它直接==处理整页学术 PDF==，输出 Markdown（包含正文、公式、表格、图片说明）。

**架构：Donut 视觉编码器 + mBART 解码器**

```text
PDF 页面图像 → Swin Transformer Encoder → 特征
                                          ↓
                                   mBART Decoder（自回归）
                                          ↓
"# Section 1\n\n正文 ... $E=mc^2$ ..."
```

**关键创新**：==没有显式的检测/分割步骤==——一个模型从整页图像直接吐出结构化 Markdown。这是端到端文档解析的早期成功范例。

**训练数据**：800 万页 arXiv 论文（PDF + 对应的 LaTeX 源码经过对齐）。

**优势**：
- 公式、文字、结构一体化生成
- 对扫描质量较好的学术 PDF 效果惊艳

**劣势**：
- 极慢（一张 A100 处理一页要几秒）
- 仅限学术文档（训练数据特化）
- 在通用文档（财报、合同、说明书）上表现差

### MinerU 内置公式识别

[[文档解析综述#六、文档解析工具全景]] 提到的 MinerU 内置了公式识别管线：

1. 用 [[版面分析]] 检测页面里的公式区域
2. 切出公式图像送公式识别模型（基于 UniMERNet，团队自研）
3. 输出 LaTeX 嵌入到 Markdown 输出里

==这种"先检测公式区域，再专门识别"的 Pipeline 是当前生产环境的主流==。

### Dolphin-v2 / VLM 方案

最新趋势：==多模态大模型直接读公式==。Dolphin-v2 这种端到端文档解析模型把公式识别作为一个"内容类型"原生支持，不需要单独的公式识别模型。

详见 [[VLM文档解析]]。

## 四、训练数据集

| 数据集 | 来源 | 规模 | 用途 |
|-------|------|------|------|
| im2latex-100k | Harvard | 10 万张 | 基础公式识别 |
| im2latex-230k | 衍生扩充 | 23 万张 | Pix2Tex 训练 |
| arXiv 配对数据 | arXiv 论文 + LaTeX 源码 | 800 万页 | Nougat |
| UniMER-1M | 上海 AI 实验室 | 100 万张 | 中英文混合公式 |

==中文公式（含中文符号、混合中英文上下文）==的数据集仍然稀缺，是开源中文公式识别效果不如英文的主因。

## 五、为什么端到端是趋势

公式识别和[[版面分析]]、[[OCR]]、[[TSR]]存在边界模糊的问题——比如：

- 一段正文里夹着行内公式 `$x^2$`：到底切到正文还是切到公式？
- 公式里的注释文字：是公式的一部分还是单独的文字？
- 公式表格（multi-equation 对齐）：是公式还是表格？

Pipeline 方案的边界划分容易出错。==端到端 VLM 把"识别什么"和"识别为什么"统一在同一个模型里==，避免边界问题。这是 Nougat、Dolphin-v2 这类方案兴起的根本动力。

## 六、输出格式选择

| 格式 | 适用场景 | 例 |
|------|---------|-----|
| LaTeX | 学术、科技文档 | `\frac{a}{b}` |
| MathML | Web 渲染（标准 W3C） | `<mfrac><mi>a</mi><mi>b</mi></mfrac>` |
| AsciiMath | 简化输入 | `a/b` |
| 自然语言 | 给非数学背景的 LLM | "a 除以 b" |

LaTeX 是事实标准——LLM 训练时见过大量 LaTeX，所以==输出 LaTeX 的公式可以直接被 LLM 理解和处理==。
