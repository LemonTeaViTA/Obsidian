# 面向对象
## 面向对象基础

### 面向对象和面向过程的区别？

**答案：**

- 面向过程：以步骤/函数为中心，关注“怎么做”。
- 面向对象：以对象/类为中心，关注“谁来做”。

面向对象更利于封装、复用和扩展。

---

### 面向对象编程有哪些特性？

**答案：**

三大特性：封装、继承、多态。

#### 封装

封装是把对象的**状态（字段）和行为（方法）**包装在一起，对外隐藏内部实现细节，只暴露必要的接口。

**怎么实现的：**
- 用 `private` 修饰字段，禁止外部直接访问
- 通过 `public` 的 getter/setter 方法控制读写
- 方法内部的实现逻辑对调用方不可见

```java
public class BankAccount {
    private double balance;  // 外部不能直接改余额

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("金额必须为正");
        this.balance += amount;
    }

    public boolean withdraw(double amount) {
        if (amount > balance) return false;  // 业务校验
        this.balance -= amount;
        return true;
    }

    public double getBalance() {
        return balance;  // 只读，不提供 setBalance
    }
}
```

这里 `balance` 被封装起来了，外部只能通过 `deposit/withdraw` 操作余额，不能直接 `account.balance = -100` 搞出非法状态。

**好处：**
- 保护数据完整性（setter 里可以加校验逻辑）
- 降低耦合（调用方不依赖内部实现，内部改了外面不用动）
- 提高可维护性（修改内部逻辑不影响外部代码）

#### 继承

子类通过 `extends` 获得父类的字段和方法，在此基础上扩展或重写。

```java
public class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public void eat() {
        System.out.println(name + “ is eating”);
    }
}

public class Dog extends Animal {
    private String breed;

    public Dog(String name, String breed) {
        super(name);          // 必须调用父类构造器
        this.breed = breed;
    }

    @Override
    public void eat() {
        System.out.println(name + “ is eating dog food”);  // 重写父类方法
    }

    public void fetch() {    // 子类扩展的新方法
        System.out.println(name + “ is fetching the ball”);
    }
}
```

**Java 的单继承限制：**
- 一个类只能 `extends` 一个父类，不支持多继承
- 但可以通过实现多个接口来弥补

**为什么不支持多继承（菱形继承问题）：**

```
      A (method())
     / \
    B   C       // B 和 C 都重写了 method()
     \ /
      D         // D.method() 该调谁的？——歧义！
```

假设类 B 和类 C 都继承了类 A，类 D 同时继承 B 和 C。如果 B 和 C 都重写了 A 的某个方法，D 调用这个方法时就不知道该用 B 的还是 C 的——这就是菱形继承（Diamond Problem）。Java 通过禁止类的多继承从根本上避免了这个歧义。

> 注意：接口的 default 方法也可能产生类似冲突，此时编译器会强制要求实现类显式重写该方法来消除歧义。

**继承的缺点：**
- 破坏封装性：子类依赖父类的实现细节，父类改了子类可能跟着出问题
- 紧耦合：继承是编译期绑定的，不如组合灵活
- 层次过深时代码难以理解和维护

**组合优于继承的示例：**
```java
// 继承方式：紧耦合，如果 ArrayList 内部实现变了，子类可能出问题
public class MyStack extends ArrayList<Integer> { ... }

// 组合方式：松耦合，内部实现可以随时替换
public class MyStack {
    private List<Integer> list = new ArrayList<>();  // has-a
    public void push(int val) { list.add(val); }
    public int pop() { return list.remove(list.size() - 1); }
}
```

**设计原则**：优先使用组合（has-a）而非继承（is-a），除非确实存在”是一种”的关系。

#### 多态

同一个方法调用，作用在不同对象上产生不同的行为。

**多态的三个必要条件：**
1. 继承或实现接口
2. 方法重写（子类覆盖父类方法）
3. 父类引用指向子类对象（向上转型）

**完整示例：**
```java
public abstract class Shape {
    public abstract double area();
}

public class Circle extends Shape {
    private double radius;
    public Circle(double radius) { this.radius = radius; }

    @Override
    public double area() { return Math.PI * radius * radius; }
}

public class Rectangle extends Shape {
    private double width, height;
    public Rectangle(double w, double h) { this.width = w; this.height = h; }

    @Override
    public double area() { return width * height; }
}

// 使用多态：方法不关心具体是什么形状
public void printArea(Shape shape) {
    // 运行时根据实际类型调用对应的 area()
    System.out.println("面积: " + shape.area());
}

// 调用
printArea(new Circle(5));      // 输出圆的面积
printArea(new Rectangle(3, 4)); // 输出矩形的面积
// 新增 Triangle 只需加一个子类，printArea 方法完全不用改
```

**两种多态：**
- **编译时多态（静态绑定）**：方法重载，编译器根据参数类型决定调用哪个方法
- **运行时多态（动态绑定）**：方法重写，JVM 在运行时根据对象的实际类型决定调用哪个实现

```java
// 编译时多态（重载）
public class Printer {
    public void print(String s) { System.out.println("String: " + s); }
    public void print(int i)    { System.out.println("Int: " + i); }
    // 编译器在编译期就确定了调用哪个 print
}

// 运行时多态（重写）
Shape s = new Circle(5);  // 编译期类型是 Shape，运行期类型是 Circle
s.area();                 // 运行时才确定调用 Circle.area()
```

**向上转型与向下转型：**
```java
Animal a = new Dog("旺财", "柴犬");  // 向上转型：自动的，安全的
a.eat();     // OK，调用 Dog 重写的 eat()
// a.fetch(); // 编译错误！Animal 没有 fetch 方法

// 向下转型：需要强转，有风险
if (a instanceof Dog) {       // 先判断再转，避免 ClassCastException
    Dog d = (Dog) a;
    d.fetch();                // OK
}
```

**多态解决了什么问题：**
让业务代码依赖抽象而不是具体实现。新增一种实现只需要加一个子类，不用修改已有代码（开闭原则）。典型场景：策略模式、工厂模式、Spring 的依赖注入。

## 方法与重载重写

### 重载和重写的区别？

**答案：**

- 重载（Overload）：同一个类中，方法名相同、参数列表不同。
- 重写（Override）：子类对父类同签名方法提供新实现。

```java
// 重载：同一个类，方法名相同，参数不同
public class Calculator {
    public int add(int a, int b) { return a + b; }
    public double add(double a, double b) { return a + b; }  // 参数类型不同
    public int add(int a, int b, int c) { return a + b + c; } // 参数个数不同
    // 注意：仅返回值不同不算重载，编译报错
}

// 重写：子类覆盖父类方法
public class Animal {
    public void speak() { System.out.println(“...”); }
}
public class Cat extends Animal {
    @Override  // 加注解让编译器帮你检查签名是否一致
    public void speak() { System.out.println(“喵”); }
}
```

**重写的规则（两小一大）：**
- 子类方法的访问权限 ≥ 父类（不能缩小）
- 子类方法的返回类型 ≤ 父类（可以是子类型，即协变返回）
- 子类方法抛出的异常 ≤ 父类（不能抛更宽的受检异常）

补充要点：
- 重载看”参数”；重写看”继承关系 + 方法签名一致”
- 重写要遵循里氏替换原则，子类应可替代父类
- `private`、`static`、`final` 方法不能被重写

## 访问控制与关键字

### 访问修饰符 public、private、protected、默认 的区别？

**答案：**

Java 的 4 种访问级别：

- `public`：任意位置可见。
- `protected`：同包可见，不同包仅子类可见。
- 默认（不写）：仅同包可见。
- `private`：仅当前类内部可见。

补充：`private/protected` 不能修饰外部顶级类。

---

### this 关键字有什么作用？

**答案：**

`this` 代表当前对象引用，常见用途：

1. 访问当前对象成员：`this.field`。
2. 解决形参与成员变量同名冲突。
3. 调用当前类的其他构造方法：`this(...)`。

## 抽象类与接口

### 抽象类和接口有什么区别？

**答案：**

- 抽象类：描述”is-a”，可有状态、构造器、抽象/非抽象方法。
- 接口：描述能力约束，更偏”can-do / has-a”，面向规范。
- 一个类只能继承一个抽象类，但可实现多个接口。

```java
// 抽象类：有状态、有构造器、有具体方法
public abstract class AbstractPayment {
    protected String orderId;  // 可以有字段

    public AbstractPayment(String orderId) {  // 可以有构造器
        this.orderId = orderId;
    }

    public abstract boolean pay(double amount);  // 抽象方法，子类必须实现

    public void log() {  // 具体方法，子类可以直接用
        System.out.println(“Processing order: “ + orderId);
    }
}

// 接口：定义能力规范
public interface Serializable { }  // 标记接口
public interface Comparable<T> {
    int compareTo(T o);  // 抽象方法
}
public interface Collection<E> {
    default boolean isEmpty() {  // Java 8+ default 方法
        return size() == 0;
    }
    int size();
}

// 一个类可以继承一个抽象类 + 实现多个接口
public class WechatPayment extends AbstractPayment
        implements Serializable, Comparable<WechatPayment> {
    // ...
}
```

**对比表：**

| 维度 | 抽象类 | 接口 |
|------|--------|------|
| 关键字 | `abstract class` | `interface` |
| 继承/实现 | 单继承 | 多实现 |
| 构造器 | 可以有 | 不能有 |
| 字段 | 任意字段 | 只能 `public static final` |
| 方法 | 抽象 + 具体都行 | 抽象 + default + static（Java 8+） |
| 设计语义 | “是什么”（is-a） | “能做什么”（can-do） |

常见补充：
- Java 类不支持多继承；接口支持多继承
- Java 8 之后接口可以有 default 方法和 static 方法
- Java 9 之后接口可以有 private 方法（用于 default 方法间复用逻辑）

## 变量与 static/final

### 成员变量与局部变量的区别有哪些？

**答案：**

| 维度 | 成员变量（实例变量） | 局部变量 |
|------|---------------------|----------|
| 定义位置 | 类中，方法外 | 方法、构造器、代码块内 |
| 存储位置 | ==堆内存==（随对象分配在堆上） | ==栈内存==（随栈帧分配在线程栈上） |
| 生命周期 | 随对象创建而存在，GC 回收对象时销毁 | 随方法调用创建，方法结束即销毁 |
| 默认值 | 有默认值（int→0, 引用→null, boolean→false） | 无默认值，必须显式赋值后才能使用 |
| 修饰符 | 可用 `private`/`protected`/`public`/`static`/`final`/`transient`/`volatile` | 只能用 `final` |
| 线程安全 | 多线程共享对象时可能有并发问题 | 线程私有，天然线程安全 |

```java
public class Example {
    // 成员变量：存在堆上（对象的一部分）
    private int count;           // 默认值 0
    private String name;         // 默认值 null
    private static int total;    // 静态成员变量：存在方法区（元空间）

    public void doSomething() {
        // 局部变量：存在栈上（当前线程的栈帧中）
        int temp = 10;           // 必须赋值才能用
        // System.out.println(temp2); // 编译错误！局部变量未赋值
        int temp2;
        temp2 = 20;             // OK，使用前赋值了

        // 局部变量如果是引用类型，引用在栈上，对象本身仍在堆上
        String s = new String("hello");  // s 在栈上，"hello"对象在堆上
    }
}
```

> [!tip] 面试追问：static 成员变量存在哪？
> 静态成员变量不属于某个对象实例，存储在==方法区（JDK 8+ 的元空间）==，随类加载而创建，随类卸载而销毁。生命周期比实例变量更长。

补充要点：
- 局部变量表是栈帧的一部分，大小在编译期就确定了
- 成员变量的访问需要通过对象引用（或类名访问静态变量），局部变量直接通过栈帧偏移量访问，速度更快
- 方法参数也是局部变量的一种

### static 关键字了解吗？

**答案：**

`static` 可修饰变量、方法、代码块、内部类，核心语义是"属于类，而非属于某个实例"。

#### 四种用法

```java
public class Counter {
    // 1. 静态变量：所有实例共享同一份数据
    private static int count = 0;

    // 2. 静态方法：通过类名直接调用，不能访问 this 和实例成员
    public static int getCount() {
        return count;
    }

    // 3. 静态代码块：类加载时执行一次，常用于初始化静态资源
    static {
        System.out.println("Counter 类被加载");
    }

    // 4. 静态内部类：不持有外部类引用，见下方"内部类"章节
    public static class Config { }

    public Counter() {
        count++;
    }
}
```

#### 静态方法为什么不能被重写？

这是面试高频追问。原因在于方法分派机制不同：

- **实例方法**：通过 `invokevirtual` 指令调用，运行时根据对象实际类型做动态分派（虚方法表）→ 支持多态
- **静态方法**：通过 `invokestatic` 指令调用，编译期就确定了调用目标（静态绑定）→ 不存在"重写"

```java
public class Parent {
    public static void greet() { System.out.println("Parent"); }
}
public class Child extends Parent {
    public static void greet() { System.out.println("Child"); }  // 这是"隐藏"，不是重写
}

Parent p = new Child();
p.greet();  // 输出 "Parent"！编译期绑定到 Parent.greet()
```

子类定义同签名静态方法叫做**方法隐藏（Method Hiding）**，不是重写。引用类型是谁就调谁的，不看实际对象类型。

#### 静态代码块的执行时机

与[[JVM 类加载机制|类加载]]过程绑定：

```
类加载 → 连接(验证/准备/解析) → 初始化（执行 <clinit>）
                                         ↑
                              静态变量赋值 + 静态代码块（按代码顺序合并执行）
```

- 一个类的 `<clinit>` 只执行一次（JVM 保证线程安全）
- 父类的 `<clinit>` 先于子类执行
- 仅"主动使用"才触发初始化（new、访问静态字段、反射等）

```java
public class Init {
    static int x = 10;
    static { x = 20; }
    static int y = x;  // y = 20，按顺序执行
}
```

#### static 的常见陷阱

| 陷阱 | 说明 |
|------|------|
| 静态方法中用 `this` | 编译错误，static 上下文没有 this |
| 静态变量的内存泄漏 | 静态集合持有对象引用，生命周期与类一样长，不会被 GC |
| 静态方法与多态 | 静态方法不参与多态，容易写出"以为重写了实际没有"的 bug |

### final 关键字有什么作用？

**答案：**

`final` 表示"不可变"，可修饰类、方法、变量，含义各不相同：

#### 三种用法

```java
// 1. 修饰类：不能被继承
final class String { }  // String 不能被继承，保证不可变性

// 2. 修饰方法：不能被重写
class Parent {
    public final void template() { }  // 模板方法，子类不能覆盖
}

// 3. 修饰变量：只能赋值一次
final int MAX = 100;  // 基本类型：值不能改

final List<String> list = new ArrayList<>();
list.add("hello");    // OK！可以修改对象内容
// list = new ArrayList<>();  // 编译错误！不能改变引用指向
```

> [!warning] 常见误区
> `final` 修饰引用变量时，不能改指向，但引用对象的内部状态仍然可变。`final List` 不等于不可变 List。

#### final 变量的分类

| 类型 | 说明 | 赋值时机 |
|------|------|----------|
| 编译期常量 | `static final` + 字面量/常量表达式 | 编译期确定，内联到调用处 |
| 运行期常量 | `static final` + 运行时计算 | 类加载时确定 |
| 实例 final 字段 | 非 static 的 final 字段 | 构造器结束前必须赋值 |
| 局部 final 变量 | 方法内的 final 变量 | 使用前赋值一次 |

```java
public class FinalDemo {
    static final int A = 10;              // 编译期常量，会被内联
    static final int B = new Random().nextInt();  // 运行期常量，不会内联
    final int c;                          // 实例 final，必须在构造器中赋值

    public FinalDemo(int c) {
        this.c = c;
    }
}
```

#### final 对 JIT 的优化意义

JVM 的 JIT 编译器看到 `final` 方法时，可以放心做**方法内联（Inlining）**——把方法体直接嵌入调用处，省去方法调用开销。

- 非 final 方法：JIT 需要做"去虚化"分析，确认没有子类重写才能内联（CHA 分析）
- final 方法：编译器直接确定不会被重写，内联更激进

> 实际上现代 JVM（HotSpot）即使没有 final 也能做去虚化优化，但 `final` 给了编译器更强的保证，在复杂继承体系中优化效果更明显。

#### effectively final（Java 8+）

Lambda 和匿名内部类中引用的局部变量必须是 final 或"事实上 final"的：

```java
public void process() {
    int threshold = 10;  // 没写 final，但后续没有修改过 → effectively final
    // threshold = 20;   // 如果取消注释，下面的 lambda 编译报错

    List<Integer> nums = List.of(1, 5, 12, 8, 20);
    nums.stream()
        .filter(n -> n > threshold)  // OK，threshold 是 effectively final
        .forEach(System.out::println);
}
```

原因：Lambda 捕获的是变量值的副本（类似闭包），如果允许修改会导致语义混乱——你改了局部变量，Lambda 里的副本不会跟着变。

### final、finally、finalize 的区别？

> 基础对比见 [[基础语法#final、finally、finalize 的区别？]]，这里补充深层理解。

**答案：**

| 关键字 | 本质 | 核心用途 |
|--------|------|----------|
| `final` | 语言修饰符 | 不可变约束（类/方法/变量） |
| `finally` | 异常处理结构 | 资源释放，保证一定执行 |
| `finalize` | Object 的方法 | GC 前回调，**已废弃**（Java 9+ @Deprecated） |

`finalize` 为什么不推荐：
- 执行时机不确定（GC 时机不可控）
- 可能导致对象"复活"（在 finalize 中重新建立引用）
- 严重影响 GC 性能（对象需要经历两次 GC 才能真正回收）
- 替代方案：`try-with-resources` + `AutoCloseable`，或 `Cleaner`（Java 9+）

## 内部类

### Java 有哪几种内部类？各有什么特点？

**答案：**

Java 有 4 种内部类，使用场景和限制各不相同：

| 类型 | 持有外部类引用 | 能否有 static 成员 | 典型场景 |
|------|:-:|:-:|------|
| 成员内部类 | 是 | 否 | 需要访问外部类实例状态 |
| 静态内部类 | 否 | 是 | 与外部类逻辑相关但不依赖实例（Builder、Node） |
| 局部内部类 | 是 | 否 | 方法内临时使用（极少见） |
| 匿名内部类 | 是 | 否 | 一次性实现接口/抽象类（回调、监听器） |

#### 成员内部类 vs 静态内部类

```java
public class Outer {
    private int x = 10;
    private static int y = 20;

    // 成员内部类：隐式持有 Outer.this 引用
    class Inner {
        void show() {
            System.out.println(x);  // OK，可以访问外部实例变量
        }
    }

    // 静态内部类：不持有外部类引用
    static class StaticInner {
        void show() {
            // System.out.println(x);  // 编译错误！不能访问外部实例变量
            System.out.println(y);     // OK，可以访问外部静态变量
        }
    }
}

// 创建方式不同
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();           // 需要外部类实例
Outer.StaticInner sInner = new Outer.StaticInner();  // 不需要外部类实例
```

#### 匿名内部类

```java
// Java 8 之前：匿名内部类实现回调
button.setOnClickListener(new View.OnClickListener() {
    @Override
    public void onClick(View v) {
        // 处理点击
    }
});

// Java 8 之后：如果是函数式接口，用 Lambda 替代
button.setOnClickListener(v -> { /* 处理点击 */ });
```

#### 面试高频问题

**1. 为什么静态内部类不会导致内存泄漏，而成员内部类可能会？**

成员内部类隐式持有外部类的引用（`Outer.this`）。如果内部类对象的生命周期比外部类长（比如被缓存、被其他线程持有），外部类对象就无法被 GC 回收。

```java
// 经典内存泄漏场景：Handler 持有 Activity 引用
public class MyActivity extends Activity {
    // 错误：非静态内部类持有 Activity 引用
    private Handler handler = new Handler() {
        @Override
        public void handleMessage(Message msg) { ... }
    };

    // 正确：使用静态内部类 + WeakReference
    private static class SafeHandler extends Handler {
        private final WeakReference<MyActivity> ref;
        SafeHandler(MyActivity activity) { ref = new WeakReference<>(activity); }
    }
}
```

**2. 静态内部类的典型应用？**

- **Builder 模式**：`Person.Builder`
- **数据结构节点**：`HashMap.Node`、`LinkedList.Node`
- **单例模式（静态内部类实现）**：利用类加载机制保证线程安全

```java
// 静态内部类实现单例（推荐写法之一）
public class Singleton {
    private Singleton() {}

    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;  // 第一次调用时才触发 Holder 类加载
    }
}
```

## equals 与 hashCode

### == 和 equals 的区别？

> 基础对比见 [[基础语法#== 和 equals 的区别？]]，这里侧重深入原理和陷阱。

**答案：**

- `==`：
    - 基本类型比较值。
    - 引用类型比较是否同一对象地址。
- `equals`：比较内容相等性（默认与 `==` 一样，很多类如 `String` 已重写）。

```java
String s1 = new String("hello");
String s2 = new String("hello");
String s3 = "hello";
String s4 = "hello";

s1 == s2;      // false，两个不同的堆对象
s1.equals(s2); // true，内容相同

s3 == s4;      // true，字符串常量池中是同一个对象
s3.equals(s4); // true

Integer a = 127;
Integer b = 127;
a == b;        // true，Integer 缓存了 -128~127

Integer c = 128;
Integer d = 128;
c == d;        // false，超出缓存范围，是两个不同对象
c.equals(d);   // true
```

**结论**：比较对象内容永远用 `equals`，`==` 只用于判断是否是同一个对象或比较基本类型。

---

### 为什么重写 equals 时必须重写 hashCode？

**答案：**

哈希容器（如 `HashMap/HashSet`）先按 `hashCode` 定位桶，再用 `equals` 判等。

如果 `equals` 相等但 `hashCode` 不同，会导致：

- 逻辑相等对象分散到不同桶。
- 查询、去重、覆盖行为异常。

契约要求：`equals` 相等的两个对象，`hashCode` 必须相等。

## 参数传递与拷贝

### Java 是值传递还是引用传递？

**答案：**

Java 只有值传递。

- 基本类型：传递值副本。
- 引用类型：传递”引用值（地址）的副本”。

```java
public class PassByValueDemo {
    public static void main(String[] args) {
        // 1. 基本类型：方法内修改不影响外部
        int num = 10;
        changeValue(num);
        System.out.println(num);  // 仍然是 10

        // 2. 引用类型：可以修改对象内容
        StringBuilder sb = new StringBuilder(“hello”);
        appendStr(sb);
        System.out.println(sb);   // “hello world”，对象内容被改了

        // 3. 引用类型：不能改变外部引用的指向
        String str = “original”;
        changeRef(str);
        System.out.println(str);  // 仍然是 “original”
    }

    static void changeValue(int x) { x = 99; }
    static void appendStr(StringBuilder s) { s.append(“ world”); }
    static void changeRef(String s) { s = “changed”; }  // 只改了副本的指向
}
```

关键理解：方法接收的是引用的副本（一个新的指针变量，指向同一个对象）。你可以通过这个副本操作对象，但让副本指向新对象不会影响原来的引用变量。

### 说说深拷贝和浅拷贝的区别？

**答案：**

- 浅拷贝：复制对象本身，引用类型字段只复制地址，新旧对象共享内部引用对象。
- 深拷贝：递归复制所有引用对象，新旧对象完全独立。

```java
public class Address {
    public String city;
    public Address(String city) { this.city = city; }
}

public class Person implements Cloneable {
    public String name;
    public Address address;  // 引用类型字段

    // 浅拷贝：address 字段只复制了引用
    @Override
    public Person clone() throws CloneNotSupportedException {
        return (Person) super.clone();
    }

    // 深拷贝：手动递归复制引用对象
    public Person deepClone() throws CloneNotSupportedException {
        Person copy = (Person) super.clone();
        copy.address = new Address(this.address.city);  // 新建 Address
        return copy;
    }
}

// 浅拷贝的问题
Person p1 = new Person("张三", new Address("北京"));
Person p2 = p1.clone();
p2.address.city = "上海";
System.out.println(p1.address.city);  // "上海"！p1 也被改了

// 深拷贝没有这个问题
Person p3 = p1.deepClone();
p3.address.city = "广州";
System.out.println(p1.address.city);  // "上海"，p1 不受影响
```

常见实现方式：
- 浅拷贝：`Object.clone()`（默认语义）
- 深拷贝：手动逐层拷贝 / 序列化再反序列化（简单但性能差）

## 对象创建方式

### Java 创建对象有哪几种方式？

**答案：**

共 5 种方式，面试中能答出前 4 种即可：

```java
public class User implements Cloneable, Serializable {
    private String name;
    public User() { this.name = "default"; }
    public User(String name) { this.name = name; }

    @Override
    protected User clone() throws CloneNotSupportedException {
        return (User) super.clone();
    }

    @Override
    public String toString() { return "User{name='" + name + "'}"; }
}
```

#### 1. new 关键字（最常用）

```java
User user = new User("张三");
```

JVM 执行过程：类加载检查 → 分配内存 → 初始化零值 → 设置对象头 → 执行 `<init>`（构造器）。

#### 2. 反射创建

```java
// 方式一：Class.newInstance()（已废弃，只能调无参构造器）
User user1 = User.class.newInstance();

// 方式二：Constructor.newInstance()（推荐，可调任意构造器）
Constructor<User> ctor = User.class.getDeclaredConstructor(String.class);
User user2 = ctor.newInstance("李四");
```

反射创建是 Spring IoC 容器的核心手段。详见 [[注解、反射与 Java 8|反射]]。

#### 3. clone() 创建

```java
User original = new User("王五");
User copy = original.clone();  // 浅拷贝，不走构造器
```

注意：`clone()` 不会调用构造器，直接复制内存中的对象数据。

#### 4. 反序列化创建

```java
// 序列化
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("user.dat"));
oos.writeObject(new User("赵六"));
oos.close();

// 反序列化：不走构造器，直接从字节流恢复对象
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("user.dat"));
User user = (User) ois.readObject();
ois.close();
```

#### 5. Unsafe.allocateInstance()（了解即可）

```java
// 绕过构造器直接分配对象内存，字段全是默认值
User user = (User) Unsafe.getUnsafe().allocateInstance(User.class);
// user.name == null，构造器没有执行
```

这是 JDK 内部 API，正常业务代码不应使用。一些序列化框架（如 Objenesis、Kryo）底层会用到。

#### 对象创建时的初始化顺序

```
父类静态变量 + 静态代码块（按代码顺序）  ← 只在类首次加载时执行一次
    ↓
子类静态变量 + 静态代码块
    ↓
父类实例变量 + 实例代码块
    ↓
父类构造器
    ↓
子类实例变量 + 实例代码块
    ↓
子类构造器
```
