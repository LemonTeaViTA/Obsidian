# Agent 框架选型

> 本文对比 OpenClaw 和 Hermes 两个 Agent 框架，帮助在不同场景下做出合适的选型决策。

---

## 一、为什么需要 Agent 框架？

直接用 LLM API 构建 Agent 需要自己处理：循环控制、工具调用、上下文管理、错误处理、状态持久化、安全隔离……每个都是工程难题。

Agent 框架把这些通用能力封装好，让开发者专注于业务逻辑。

---

## 二、OpenClaw vs Hermes 核心对比

| 维度 | OpenClaw | Hermes |
|------|---------|--------|
| **安全机制** | 依赖大模型自己判断风险 | 框架层面强制授权和审批，不依赖 LLM |
| **技能沉淀** | 手动编写和维护 Skills，质量可控 | 自动从任务中提炼 Skill，贴近实际但需人工审核 |
| **记忆检索** | SQLite + FTS5 + sqlite-vec 向量检索 | SQLite + FTS5 + 大模型摘要，无向量检索 |
| **模型支持** | 需要手动配置 LLM 接口 | 200+ 模型开箱即用（LiteLLM） |
| **多端接入** | 主要面向飞书/企微等企业 IM | Telegram/Discord/Slack 等消费级平台 |
| **路由机制** | bindings 精准路由，支持多 Agent | 单 Agent 为主 |
| **适用场景** | 企业内部工具，需要精确控制行为 | 个人助手，需要自适应学习 |

---

## 三、选型建议

**选 OpenClaw 的场景**：
- 企业内部部署，数据安全要求高
- 需要精确控制 Agent 行为（哪些工具可用、哪些操作需要审批）
- Skills 需要精确定义，不能有随机性
- 需要接入飞书、企微等企业 IM

**选 Hermes 的场景**：
- 个人使用，希望 Agent 自动学习和优化
- 需要接入 Telegram、Discord 等消费级平台
- 希望快速接入多种 LLM，不想手动配置
- 对 Skills 质量要求不那么严格，接受自动生成

**两者都支持 MCP 协议**，工具层可以复用，不存在生态锁定问题。
