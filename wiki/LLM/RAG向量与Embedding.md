# RAG 向量与 Embedding

> 本文覆盖向量基础设施全链路：Embedding 模型演进与选型、向量数据库与检索算法、生产级优化、召回率评估。检索策略见 [[RAG检索策略]]，高级技术见 [[RAG高级技术]]。

## 一、向量数据库基础

### 为什么 LLM 需要向量数据库

传统关系型数据库（MySQL）通过精确匹配检索结构化数据。但 AI 时代面临的是非结构化数据（图片、视频、文本），这些数据难以用键值对描述。向量数据库通过计算高维空间中的距离来表征相似度，让机器能理解数据的"上下文"和"语义"。

在 RAG 系统中，两者通常并存：
- **向量数据库：** 负责大规模"初筛"，从百万级文档中找出最相关的段落
- **传统数据库：** 负责"元数据"管理，记录文档的作者、发布时间、权限等确切信息

### 向量数据库 vs 传统数据库

| 特性 | 关系型数据库 (SQL) | 向量数据库 (Vector DB) |
|------|-------------------|----------------------|
| **存储数据类型** | 结构化数据（数字、字符串、日期） | 非结构化数据的向量表示 (Embedding) |
| **检索方式** | 精确匹配：基于条件（如 `WHERE ID=1`） | 近似匹配：基于距离（如余弦相似度） |
| **查询结果** | 是或否（匹配或不匹配） | 相似度分值（从最相似到最不相似） |
| **扩展性** | 擅长处理数百万级的行记录 | 专为处理数十亿级的高维向量设计 |
| **典型代表** | MySQL, PostgreSQL | Milvus, Zilliz, Pinecone |

### KNN vs ANN

| 维度 | KNN (精确搜索) | ANN (近似搜索) |
|------|---------------|---------------|
| **计算方式** | 暴力计算：遍历全库，计算所有距离 | 索引检索：只在候选子集中计算距离 |
| **准确率** | 100%（理论最优解） | < 100%（存在微小召回损失） |
| **查询速度** | 随数据量线性增长 O(N) | 极快 O(log N) 或 O(1)，支持亿级数据 |
| **适用场景** | 小数据集、对精度要求极高的科研 | 生产环境、RAG、推荐系统、海量检索 |

## 二、向量检索算法

### 算法总览

| 算法类型 | 代表算法 | 核心原理 | 优点 | 缺点 |
|---------|---------|---------|------|------|
| **基于图** | HNSW, NSG | 构建分层小世界图，通过节点跳转快速逼近目标 | 性能最强，召回率极高 | 索引构建慢，内存占用大 |
| **基于聚类** | IVF, IVFPQ | 将空间划分为簇，查询时仅搜索相邻簇 | 内存占用低，压缩率高 | 召回率受聚类质量影响 |
| **基于哈希** | LSH | 局部敏感哈希将相似点映射到同个桶 | 计算简单，空间复杂度低 | 高维数据下性能下降 |
| **基于树** | Annoy, KD-Tree | 超平面切分空间，构建二叉树 | 适合中低维度 | 高维"维度灾难" |

### HNSW（分层导航小世界图）

构建多层图结构，类似不同缩放级别的地图。底层包含所有数据点，上层由从底层采样的数据点子集组成。

**搜索流程：**
1. 从顶层的固定入口点开始
2. 在当前层贪婪地移动到近邻，直到达到局部最小值
3. 跳转到下层，重复贪婪搜索
4. 在最底层进行最后的细化步骤

**关键参数：**
- `m`：每层的邻居数，越大索引越准但内存占用越多
- `ef_search`：检索时的候选数，越大越准但越慢
- 实测：`m=16, ef_search=100` 在准确率和延迟之间是较好的平衡

HNSW 是目前性能最强的向量检索算法，但索引构建慢、内存占用大。

### IVF 系列

**IVF_FLAT（反转文件 + 扁平存储）：** 先用 K-means 聚类将向量空间划分为区域，每个聚类有一个中心点。查询时先比较聚类中心点，选择最有希望的簇，再在簇内搜索。适合需要快速查询和高精确度的大规模数据集。

**IVF_SQ8（+ 标量量化）：** 在 IVF 基础上用 8 位整数替代 32 位浮点数存储每个维度值，大幅减少内存。适合内存受限但需要较高精度的场景。

**IVF_PQ（+ 乘积量化）：** 将每个高维向量分解为 `m` 个子向量，每个子空间内用 k-means 学习编码本，用中心点索引替代原始值。存储需求从 D × 32 位减少到 m × nbits 位。内存占用比 HNSW 少得多，是大型数据集的实用选择。

**IVF_RABITQ（+ 二进制量化）：** 将 FP32 向量量化为二进制表示，压缩比 1:32。步骤：归一化到单位超球面 → 随机正交旋转 → 只保留符号位二值化。存储效率极高。

### DISKANN（基于磁盘的向量检索）

当数据集超过可用 RAM 时（数十亿向量），结合 Vamana 图形（基于磁盘的图形索引）和乘积量化（内存中的压缩方法）。适合超大规模数据集、内存受限的生产环境。

### ScaNN（Google）

三阶段：分区（K-means 划分子集）→ 量化（各向异性向量量化压缩）→ 重新排序（对候选做精确内积计算）。

## 三、向量数据库选型

### 主流方案对比

| 方案 | 适用场景 | 优势 | 劣势 |
|------|----------|------|------|
| **ElasticSearch 8.x** | 混合检索（向量+关键词） | 天然支持 BM25+向量双路，运维成熟 | 内存占用大，向量性能非最优 |
| **Milvus** | 纯向量，超大规模（亿级） | 专为向量设计，分布式架构，支持 GPU 加速 | 运维复杂（etcd/MinIO/Pulsar） |
| **Qdrant** | 中小规模，混合检索 | Rust 实现，单机性能强，原生支持过滤+向量 | 分布式能力弱于 Milvus |
| **Pinecone** | 云托管，快速上线 | 零运维，自动扩缩容 | 成本高，数据出境风险 |
| **Weaviate** | 混合检索，GraphQL 生态 | 内置 BM25+向量，模块化架构 | 社区相对小 |
| **pgvector** | 小规模（<100 万），已有 PG | 复用现有数据库，零额外运维 | 大规模性能差 |
| **Chroma** | 原型验证，本地开发 | Python 原生，嵌入式，5 分钟上手 | 不适合生产环境 |

> [!tip] 选型决策
> 已有 ES 集群 + 数据量 < 5000 万 → ES 8.x 最省事；纯向量 + 亿级数据 → Milvus；中小团队 + 高性能要求 → Qdrant；验证阶段 → Chroma/pgvector。大多数 RAG 项目用 ES 8.x 就够了。

### Milvus 架构特点

通过 Growing Segment（实时数据，快速索引）和 Sealed Segment（历史数据，高性能 Immutable 索引）的划分，配合异步 Compaction 机制，解决了"大数据量下索引构建慢"与"实时搜索需求"之间的矛盾。

**混合检索能力：**
- Sparse Vector（稀疏向量）：结合 BM25 算法与 BERT 语义能力
- 多列检索：支持 Dense + Dense（如人脸+步态）或 Scalar + Dense（如证件号+人脸）

**丰富的查询语义：**
- Range Search：寻找"足够相近"的结果，避免无关干扰（不只是 Top K）
- Groupby：解决结果多样性问题（如搜索"狗"时确保包含不同品种）

### ES HNSW 参数调优

ES 8.x 支持 HNSW 索引，向量检索复杂度 O(log N)。

**三个检索优化思路：**
1. **先过滤再向量搜索**：用 term 查询把候选集缩小（按文档类型、用户权限、时间范围），再做向量相似度计算
2. **冷热分层**：热数据放 SSD，冷数据放 HDD，兼顾性能和成本
3. **语义缓存**：80% 的查询集中在 20% 的热门问题，把检索结果缓存在 Redis，命中率能到 40% 以上

## 四、Embedding 三代演进

### 第一代：静态词向量（Word2Vec / GloVe / FastText）

核心思路：用一个词周围的词来预测这个词。训练后每个词对应一个固定向量。

- **Word2Vec**：CBOW 和 Skip-gram 两种训练方式，「国王 - 男人 + 女人 ≈ 女王」
- **GloVe**：对整个语料库的词共现矩阵做分解，对全局统计信息利用更充分
- **FastText**：把词拆成字符级 n-gram 子词，解决未登录词问题

**致命缺陷：** 静态。「我吃了苹果」和「苹果手机发布了」里的「苹果」向量完全相同，处理不了多义词。

### 第二代：上下文相关向量（ELMo / BERT）

同一个词在不同语境下有不同向量表示。

- **ELMo**：双向 LSTM，从左往右和从右往左扫描句子
- **BERT**：Transformer 替代 LSTM，Masked Language Model 预训练，用 [CLS] token 表示整个句子

**致命缺陷：** 检索时必须把两个句子拼在一起跑 BERT，百万级知识库要跑百万次。BERT 一次前向传播几十毫秒，百万次就是好几个小时，完全不可接受。

### 第三代：句子级对比学习 Embedding（SBERT / SimCSE / BGE）——RAG 标配

每个句子独立生成一个向量，直接用余弦相似度比较。知识库所有文档向量提前算好存起来，检索时只算一次查询向量，毫秒级返回。

- **SBERT（2019）**：bi-encoder 结构，两个句子分别独立过 BERT，各自得到句子级向量。精度比 cross-encoder 低（两个句子不再交互），但速度提升几个数量级
- **SimCSE**：对比学习，同一句话做两次 dropout 得到正样本对，解决 BERT 向量「各向异性」问题
- **BGE（北京智源）**：中文 RAG 场景最常用的开源模型，同时支持 bi-encoder 和 reranker

**三代对比：**

| 世代 | 代表模型 | 核心特点 | 主要局限 | RAG 适用性 |
|------|----------|----------|----------|-----------|
| 第一代 | Word2Vec、GloVe、FastText | 词级静态向量 | 无法处理多义词 | 不适用 |
| 第二代 | ELMo、BERT | 上下文动态向量 | 检索时需两两拼接，速度极慢 | 不适用于实时检索 |
| 第三代 | SBERT、SimCSE、BGE、E5 | 句子级 bi-encoder，对比学习 | 精度低于 cross-encoder | 标配，性能和精度平衡最优 |

## 五、Embedding 模型选型

### 选型五维度

| 维度 | 说明 | 关键考量 |
|------|------|---------|
| **领域匹配度** | 通用模型在垂直领域可能表现差 | 医疗、法律、金融等需要在自己数据上评估，不要迷信 MTEB 分数 |
| **上下文长度** | 单次能处理的最大 token 数 | 长文档选 8192+ tokens（Qwen3 支持 32K），短文本 2048 够用 |
| **向量维度** | 维度越高精度越高，但存储和检索成本也越高 | 高精度选 3072/4096，大规模选 384/512；Matryoshka 训练支持灵活截断 |
| **语言支持** | 中文场景需要特别关注 | BGE、Qwen 系列对中文深度优化，OpenAI 对中文"可用但建议实测" |
| **延迟与成本** | API 按量付费 vs 本地部署 | 轻量模型（DistilBERT、MiniLM）延迟更低 |

### Embedding 评估方法

**核心原则：不要迷信排行榜，在自己数据上测最靠谱。**

**简单评估：区分度测试**

构造正负例对，计算模型对正例和负例的相似度差距：

```python
import numpy as np

def evaluate_embedding(get_embedding_func, test_cases):
    pos_scores, neg_scores = [], []
    for case in test_cases:
        q_vec = get_embedding_func(case["query"])
        p_vec = get_embedding_func(case["positive"])
        n_vec = get_embedding_func(case["negative"])
        pos_scores.append(cosine_similarity(q_vec, p_vec))
        neg_scores.append(cosine_similarity(q_vec, n_vec))
    return {
        "avg_positive_similarity": np.mean(pos_scores),
        "avg_negative_similarity": np.mean(neg_scores),
        "discrimination": np.mean(pos_scores) - np.mean(neg_scores)
    }

test_cases = [
    {"query": "怎么退款", "positive": "退货流程说明", "negative": "公司年假政策"},
    {"query": "如何请假", "positive": "带薪假期申请指南", "negative": "工资发放时间"},
]
```

区分度越高，模型越能准确区分相关和不相关内容。

**进阶评估：** 使用 RAGAS 框架评估上下文精度、召回率、忠实度等多维度指标。结合 DeepEval、TruLens-Eval 等工具统一评估流程。

### 前沿趋势

1. **超大参数模型崛起：** KaLM-Embedding（12B）登顶 MTEB，Embedding 也在走向"大模型化"
2. **多模态统一嵌入：** Amazon Nova 首次实现单模型支持文本/图像/视频/音频，简化多模态 RAG 架构
3. **知识蒸馏让小模型也能打：** LEAF 等框架让小型模型达到接近大型模型的效果
4. **混合检索成为标配：** BGE-M3 同时生成稠密向量和稀疏向量，兼顾关键词精确性和语义理解

## 六、Embedding 微调

### 什么时候需要微调

通用 Embedding 不理解专业术语（医学术语、法律行话、技术缩写），相似度计算偏差大。当通用模型在你的数据上检索效果明显不达预期，且有一定规模的文档库（几百份以上），微调的投入产出比通常很高。

**对比示例：**
```
查询："什么是营运资金？"

通用嵌入：
1. "营运资金包括..." (0.72)
2. "资本市场提供..." (0.68)  ← 被"资本"混淆
3. "工作条件在..." (0.65)    ← 被"working"混淆

微调嵌入：
1. "营运资金包括..." (0.89)
2. "营运资金比率计算..." (0.84)
3. "有效管理营运资金..." (0.81)
```

### CustomIR 框架（合成数据微调）

三步流程自动生成微调数据，无需人工标注：
1. 利用 LLM 从文档库中自动生成多样化的查询
2. 生成正例和经过 LLM 验证的难负例
3. 用合成数据对预训练 Embedding 模型进行对比学习微调

优势：单张 GPU 几小时完成，迭代快。让小模型在特定领域上的性能接近大得多的模型，大幅降低部署成本。

### Cisco IT 企业实战

Cisco IT 团队使用 NVIDIA Nemotron RAG 微调方案，在约 925 份文档、9,200 个问答对、7,800 个训练样本的规模上，对 10 亿参数的 NV-EmbedQA 模型进行了微调。整个微调流程在单张 H200 GPU 上只需几小时完成，无需人工标注。

### 微调做法

```python
from sentence_transformers import SentenceTransformer, losses

model = SentenceTransformer("BAAI/bge-large-zh-v1.5")
train_loss = losses.MultipleNegativesRankingLoss(model)
# 用 1000+ 条领域查询-文档对训练 3 epoch
```

**不需要微调的情况：** 数据量很少或通用模型已经够用，直接使用现成模型更划算。

## 七、生产级检索优化

### 延迟优化策略

- **语义缓存**：Redis 存储高频 query 的检索结果，query 相似度 > 0.95 视为缓存命中，命中时延迟 < 5ms
- **预过滤**：先用 metadata 过滤（文档类型、权限、时间），再做向量检索，候选集缩小 10-100x
- **冷热分层**：30 天内文档放 SSD + 内存索引，历史文档放 HDD + 磁盘索引，冷数据成本降 70%
- **异步预计算**：文档入库时预计算 Embedding + 预构建索引，查询时直接用

### 并发优化

- 检索和重排可以流水线化：第一批 Top-20 结果立即送重排，同时继续检索后续批次
- 多路检索（BM25 + 向量 + 知识图谱）并行执行，用 RRF 融合结果
- 长文档分片后的多 chunk 检索天然适合并行，利用线程池提升吞吐

### 百万级向量存储规模估算

- 100 万文档 × 平均 50 个 chunk = 5000 万条 chunk
- 每条 chunk 存 2048 维向量：5000 万 × 2048 × 4 字节 ≈ **400GB**
- MySQL 存文档元数据（文件名、路径、用户、分类），百万级文档也就几 GB

## 八、召回率评估体系

### 测试集构建六步流程

1. **确定评估范围**：覆盖哪些领域和文档类型，每个场景至少 50 条 query
2. **设计评估问题**：LLM 生成 500 条初始问题 → 人工筛选保留 200 条（标准：能在知识库找到明确答案、表述自然、不重复）
3. **标注参考答案和支持证据**：每个问题标注正确答案 + 答案来自哪个文档的哪个 chunk（文档 ID、Chunk ID、页码）
4. **覆盖多样性和难度**：
   - 简单事实题：答案在单个 chunk 里
   - 多跳推理题：需要综合多个 chunk
   - 否定性查询："核辐射能赔吗"（答案在免责条款，容易召回错误的承保条款）
   - 口语化查询："孩子在学校摔了能报不"（表述和知识库差异大）
5. **质量审查**：抽样 20% 做交叉验证
6. **持续迭代**：线上 badcase 定期加入测试集

### 核心指标

**检索阶段：**
- **Recall@K**：前 K 个结果中包含真实相关文档的比例，是检索系统的核心指标
- **Precision@K**：前 K 个结果中有多少是相关的
- **MRR（平均倒数排名）**：第一个正确结果排名的倒数均值
- **NDCG@K**：综合考虑相关性等级和排名位置

**生成阶段：**
- **忠实度（Faithfulness）**：回答中每句话是否有检索证据支撑
- **答案相关性**：回答是否真正解决了用户的问题
- **端到端准确率**：生成答案的事实正确率

### 评估工具选型

| 工具 | 适用场景 | 特点 |
|------|----------|------|
| RAGAS | 快速评估，无标注数据 | 不需要 ground truth，LLM 自动判断 |
| LangSmith | 已用 LangChain，有参考答案 | LLM-as-a-Judge，批量对比 |
| TruLens | 系统已上线，需持续监控 | 支持线上实时监控，有 Dashboard |
| DeepEval | 团队有单测习惯 | 类 Pytest，可集成 CI/CD |

**推荐组合：** 离线评估用 RAGAS + 人工抽检（50 条校准自动指标），线上监控用 TruLens Dashboard 看每日质量指标。

**分维度拆解的价值：** 按问题类型分别统计召回率（文本类/表格类/否定性查询/口语化查询），能直接指导优化方向——先解决最差的那个维度。
