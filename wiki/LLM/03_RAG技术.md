## 一、RAG 在 LLM 技术栈中的位置

RAG（Retrieval-Augmented Generation）解决的是 LLM 的两个核心局限：

1. **知识截止**：LLM 的训练数据有截止日期，无法回答最新信息
2. **幻觉**：LLM 对不确定的问题会"编造"答案，RAG 通过注入真实文档来锚定回答

RAG 是 Agent 系统的标配组件——Agent 需要访问外部知识时，通过 RAG 检索相关文档注入上下文，而不是依赖模型的参数记忆。

## 二、GraphRAG

### 是什么

Microsoft 开源的知识图谱增强 RAG 框架（GitHub 30k+ stars）。在传统向量检索的基础上，先从文档中提取实体和关系，���建知识图谱，再基于图结构做检索。

### 解决了传统 RAG 的什么问题

传统 RAG 是"局部检索"——找到和 query 最相似的几个文档片段。但有些问题需要"全局理解"：

- "这份报告的整体结论是什么？"
- "这些文档里提到的所有风险因素有哪些？"
- "A 和 B 之间有什么关联？"

这类问题需要跨文档、多跳推理，传统 RAG 无法处理。

### 核心机制

**第一步：构建知识图谱**
1. 用 LLM 从文档中提取实体（人物、组织、概念）和关系（A 属于 B、A 导致 B）
2. 构建实体-关系图

**第二步：社区检测**
1. 用图算法（如 Leiden 算法）把图中的实体聚类成"社区"（主题相关的实体群）
2. 每个社区代表文档集中的一个主题

**第三步：层次摘要**
1. 对每个社区��成摘要（Community Summary）
2. 多个社区摘要再聚合成更高层次的摘要
3. 形成层次化的知识结构

**第四步：检索**
- 局部查询：直接在知识图谱中找相关实体和关系
- 全局查询：用社区摘要回答需要全局理解的问题

### GraphRAG vs 传统 RAG

| 维度 | 传统 RAG | GraphRAG |
|------|---------|---------|
| 检索方式 | 向量相似度（局部） | 图遍历 + 社区摘要（全局） |
| 适合的问题 | 具体事实查询 | 全局理解、多跳推理 |
| 构建成本 | 低（只需向量化） | 高（需要 LLM 提取实体关系） |
| 查询成本 | 低 | 高（全局查询需要处理大量摘要） |
| 适用场景 | 文档问答、知识库检索 | 大型文档集分析、跨文档推理 |

### 适用场景

- 分析大量报告/论文，需要提炼全局结论
- 企业知识库，需要理解不同文档之间的关联
- 需要回答"整体上..."、"所有...的共同点是..."类型的问题
