# Agent 核心概念

## 一、Agent 定义与基本架构

### Agent 到底是什么？

Agent 不是某种更高级的模型，模型还是那个模型，参数没变，结构也没变，变的是外面那一圈工程系统。

Agent = LLM + Memory + Search + MCP + Skills

拆开来看每个都不新，拼起来看着就很新。就像早期的电脑，CPU、内存、��盘都早就有了，但装进一个盒子里，你才觉得它是"一个完整机器"。

### Agent 的四个核心模块

| 模块 | 职责 | 类比 |
|------|------|------|
| LLM | 推理和决策，接收输入、理解意图、输出下一步计划 | Agent 的 CPU |
| Memory | 短期记忆（对话上下文）+ 长期记忆（跨会话持久化） | Agent 的内存+硬盘 |
| Tools | 外部能力：搜索、代码执行、数据库查询、API 调用 | Agent 的四肢 |
| Action Executor | 真正执行 LLM 决定的操作，把结果反馈回去 | Agent 的执行器 |

### Agent 和传统 LLM Chain 的区别

先搞清楚三个层次：

**普通 LLM 调用**：一问一答，你发一条消息，模型回一条，结束。流程完全由你控制，模型只负责生成文字。

**LLM Chain（链式调用）**：把多个 LLM 调用串成固定流水线，前一步的输出自动作为后一步的输入。比如"先翻译→再摘要→再润色"，每一步干什么、顺序怎么排，都是开发者写死的。LangChain 里的 SequentialChain 就是典型实现。

**Agent**：动态决策循环。模型不只是回答，还会主动决定"下一步该干什么"——调哪个工具、传什么参数、结果不满意要不要重试、要不要换个方向。整个过程是模型在驱动，而不是你在一步步指挥。

核心区别在于**谁控制流程**：

| 维度 | LLM Chain | Agent |
|------|-----------|-------|
| 流程控制 | 开发者预定义，固定的 DAG 或线性链 | 模型动态决策，每步根据上一步结果决定下一步 |
| 分支逻辑 | 需要开发者提前写好所有 if-else 分支 | 模型自主判断走哪条路，能处理预料之外的情况 |
| 工具调用 | 哪一步调什么工具是写死的 | 模型自己选工具、选参数、决定调不调 |
| 错误处理 | 开发者预设 retry/fallback 逻辑 | 模型观察到错误后自主调整策略 |
| 适用场景 | 步骤固定、边界清晰的流水线任务 | 任务开放、需要大量判断和探索的场景 |

一句话总结：LLM Chain 是你画好路线图让模型走，Agent 是你给模型一个目的地让它自己找路。

> [!tip] 面试区分度
> 很多候选人把 LangChain 和 Agent 混为一谈。关键区分：LangChain 是工具库（提供 Chain 和 Agent 的实现），Agent 是架构模式（模型驱动决策）。LangChain 里既能写固定的 Chain，也能写动态的 Agent。

选型原则：任务步骤固定、边界清晰，用 Chain；任务开放、需要动态判断，用 Agent。两者不是替代关系——Agent 内部的每个子任务执行，本质上也可以是一段 Chain。

### 各组件之间的关系

| 层次 | 机制 | 解决的问题 | 类比 |
|------|------|-----------|------|
| 基础能力层 | Function Call | 模型和函数之间如何沟通 | 嘴巴（说出指令） |
| 协议层 | MCP | 工具怎么标准化接入 | USB 接口（统一插口） |
| 经验层 | Skills | 如何复用过去的成功经验 | 使用说明书（肌肉记忆） |

一个完整的 Agent 系统通常三层都有：用 Function Call 触发工具、���过 MCP 管理工具注册、用 Skills 积累执行经验。

## 二、ReAct 框架与推理模式

Agent 的核心差异在于"怎么思考、怎么行动"。目前主流有三种推理框架：ReAct、Plan-and-Execute、Reflection。它们解决的问题不同，适用场景也不同，理解清楚才能在工程里做对选型。

### 2.1 三种推理框架

#### ReAct（Reasoning + Acting）

ReAct 的核心思想是**推理和行动交替进行**，每一步都是"想一下→做一下→看结果→再想"。

工作流程是不断循环的「思考-行动-观察」三步：

1. **Thought（思考）**：LLM 分析当前情况，决定下一步该干什么。这一步是内部推理，不对外输出
2. **Action（行动）**：LLM 决定调用哪个工具、传什么参数，以结构化 JSON 格式输出调用指令
3. **Observation（观察）**：外部系统执行 Action，把工具返回的结果注入上下文

LLM 拿到 Observation 后更新上下文，进入下一轮 Thought，判断任务是否完成。没完成继续循环，完成了输出 Final Answer。

具体例子——让 Agent 查 RAG 优化论文：
- 第一轮：Thought→需要搜索；Action→调用 search 工具；Observation→返回 10 篇摘要
- 第二轮：Thought→需要筛选；Action→调用 filter 工具；Observation→三篇精选
- 第三轮：Thought→信息足够；Final Answer→整理输出

**关键设计点**：
- LLM 每次只生成「下一步指令」，循环控制权在外部 Harness 程序，不在模型自身。这也是为什么实现 Agent 框架时，循环逻辑和上下文拼接非常重要
- 每一步都能根据上一步的实际结果动态调整策略，天然具备纠错能力
- 缺点是**缺乏全局规划**，容易陷入局部最优——比如搜到一篇相关论文就开始深挖，忘了还有其他方向没探索

#### Plan-and-Execute（先规划后执行）

Plan-and-Execute 的核心思想是**规划和执行分离**：先让 LLM 一口气生成完整计划，再逐步执行每个子任务。

工作流程分两个阶段：

**规划阶段（Planner）**：
1. LLM 接收用户任务，分析任务目标和约束条件
2. 将复杂任务拆解为有序的子任务列表（Step 1, Step 2, ...）
3. 输出完整的执行计划，每个步骤有明确的输入、���出和依赖关系

**执行阶段（Executor）**：
1. 按计划顺序逐步执行每个子任务
2. 每个子任务可以调用工具、生成内容
3. 前一步的输出作为后一步的输入

具体例子——让 Agent 写一份竞品分析报告：
- Plan：①搜索竞品列表 → ②逐个收集产品信息 → ③整理对比维度 → ④生成分析表格 → ⑤撰写结论
- Execute：按 ①→⑤ 顺序执行，每步完成后进入下一步

**关键设计点**：
- 规划阶段有全局视野，能合理分配资源和安排步骤顺序，避免 ReAct 的"走一步看一步"问题
- 缺点是**计划一旦生成就相对固定**，如果中间某步结果出乎意料（比如某个 API 挂了），整个计划可能需要推翻重来
- 改进方案：加入 Replan 机制——执行过程中如果发现偏差超过阈值，回到 Planner 重新规划剩余步骤。LangGraph 的 Plan-and-Execute 实现就支持这种动态重规划

**适用场景**：任务步骤相对明确、子任务之间有清晰的依赖关系、不太需要中途大幅调整方向的场景。比如数据处理流水线、报告生成、多步骤部署操作。

#### Reflection（反思/自我修正）

Reflection 的核心思想是**生成→评估→改进的迭代循环**，通过对自身输出的批判性审视来持续提升质量。

工作流程是一个质量驱动的循环：

1. **Generate（生成）**：LLM 根据任务要求生成初始结果
2. **Evaluate（评估）**：对生成结果进行质量评估，找出问题和不足。评估可以是 LLM 自评，也可以是外部工具（如代码编译器、测试用例、lint 工具）
3. **Refine（改进）**：根据评估反馈，针对性地修正和优化结果
4. **循环判断**：检查是否满足质量标准或达到最大迭代次数，满足则输出，否则回到步骤 2

具体例子——让 Agent 生成一段排序算法：
- 第一轮：Generate→写出快排代码；Evaluate→跑测试用例，发现边界条件没处理（空数组报错）；Refine→加上空数组判断
- 第二轮：Evaluate→跑测试全过，但 lint 报变量命名不规范；Refine→修正命名
- 第三轮：Evaluate→测试通过 + lint 通过 + 复杂度合理；输出最终结果

**两个主要变体**：
- **自我反思（Self-Reflection）**：同一个 LLM 既当生成者又当评审者。成本低、延迟小，但存在盲点——模型容易对自己的错误"视而不见"，因为生成和评估用的是同一套知识和偏见
- **对等评审（Critic Model）**：用一个专门的 Critic 模型（可以是不同的 LLM 或同一模型的不同 Prompt）来评估。质量更高，能发现自我反思发现不了的问题，但成本翻倍且增加延迟

**关键设计点**：
- Reflection 不是独立的执行框架，更像是一个**质量保障层**，可以叠加在 ReAct 或 Plan-and-Execute 之上
- 生产中一般用自我反思做快速迭代（日常代码生成、文案撰写），高精度场景（合同审核、医疗建议、金融报告）才上专门的 Critic 模型
- 需要设置最大迭代次数（通常 3-5 次），防止无限循环。如果多轮反思后质量仍不达标，应该上报给用户而不是继续空转

### 2.2 执行、规划与反思：三种核心能力

三个框架看起来各不相同，但拆开来看，它们其实是三种核心能力的不同组合：

| 核心能力 | 含义 | 对应的工程实现 |
|---------|------|--------------|
| **执行（Acting）** | 调用工具、与外部世界交互、产生实际结果 | Function Call、MCP 工具调用 |
| **规划（Planning）** | 将复杂任务拆解为子任务，安排执行顺序和依赖 | Task Decomposition、DAG 编排 |
| **反思（Reflection）** | 对已有结果进行评估和改进 | Self-Critique、外部验证器 |

- **ReAct** 侧重执行，规划是隐式的（每步的 Thought 只规划下一步），反思也是隐式的（通过 Observation 间接纠错）
- **Plan-and-Execute** 侧重规划，执行是按计划推进的，反思体现在 Replan 机制中
- **Reflection** 侧重反思，执行是生成结果的过程，规划体现在改进策略的制定��

实际生产中，成熟的 Agent 系统往往**混合使用**这三种能力：先用 Plan-and-Execute 做全局规划，每个子任务用 ReAct 模式执行，关键节点加 Reflection 做质量把关。

### 2.3 三种框架对比

| 维度 | ReAct | Plan-and-Execute | Reflection |
|------|-------|-----------------|------------|
| 核心思路 | 推理与行动交替，边想边做 | 先全局规划，再按计划执行 | 生成→评估→改进循环 |
| 规划能力 | 弱，只看下一步 | 强，有全局视野 | 无独立规划，依附于其他框架 |
| 纠错能力 | 中等，靠 Observation 间接纠错 | 弱，需要 Replan 机制补救 | 强，专门设计用于纠错 |
| 灵活性 | 高，每步都能动态调整 | 低，计划变更成本高 | 中等，在固定结构内迭代 |
| Token 消耗 | 中等，每轮都有 Thought 开销 | 规划阶段集中消耗，执行阶段较低 | 高，多轮迭代累积消耗 |
| 适用场景 | 探索性任务、信息检索、通用 Agent | 步骤明确的多步任务、流水线 | 代码生成、长文写作、高精度场景 |
| 典型实现 | LangChain ReAct Agent、Claude Code | LangGraph Plan-and-Execute | Reflexion、SELF-REFINE |

**选型建议**：
- 任务开放、需要大量探索和工具调用 → ReAct
- 任务步骤清晰、子任务有明确依赖 → Plan-and-Execute
- 对输出质量要求极高、允许多轮迭代 → 在基础框架上叠加 Reflection
- 复杂生产系统 → 三者混合使用

## 三、Function Call

### Function Calling 是什么？

Function Calling 是模型侧的原生能力。厂商通过微调，让 LLM 能够输出符合特定 JSON Schema 的结构化调用指令，而不是在自然语言里猜工具。

比如你问「今天北京天气怎么样」，模型会输出：
```json
{"tool": "weather_query", "params": {"city": "北京"}}
```

它解决的核心问题：如何让模型准确地触发工具调用。

### 如何保证 Function Calling 的可靠性？

不可靠来自三个层面：

**模型层**：LLM 生成的调用参数可能格式错误、字段缺失。应对方式是在 Prompt 里明确每个工具的 JSON Schema，调用前做参数校验，格式不对打回重新生成，最多重试 3 次。

**执行层**：工具本身可能超时、返回异常。标准做法是设超时上限，加 retry 逻辑（指数退避），对返回结果做合法性检查，异常情况向 Agent 上报让它自己决定下一步。

**系统层**：高并发下多个 Agent 同时调同一个外部服务可能打挂下游。引入限流和熔断机制，触发熔断后返回降级结果。

经常被忽视的细节：工具描述的质量直接影响调用准确率。工具的 name、description 写得越清晰，模型选错工具的概率就越低。

> [!warning] Function Calling 不等于 100% 可靠
> 即使是 GPT-4 级别的模型，Function Calling 的参数正确率也不是 100%。生产环境必须做参数校验 + 重试 + 降级三层防护。不要因为"模型支持 Function Calling"就假设调用一定正确。

### Agent 如何处理工具调用失败？

上面讲的是工程侧怎么保证可靠性（参数校验、超时重试、熔断降级），这里从 Agent 决策的角度讲——工具调用失败后，Agent 自己会怎么做。

**Agent 的四层应对策略**：

| 层级 | 策略 | 触发条件 | 示例 |
|------|------|---------|------|
| L1 重试 | 相同工具、相同或修正后的参数再试一次 | 临时性错误（超时、网络抖动） | search 工具超时 → 等 2 秒重试 |
| L2 参数修正 | 分析错误信息，调整参数后重新调用 | 参数错误（格式不对、值越界） | 日期格式传错 → Thought 分析报错 → 修正格式重调 |
| L3 换工具 | 放弃当前工具，选择替代工具完成同一目标 | 工具不可用或持续失败 | Google Search 挂了 → 换 Bing Search |
| L4 换策略 | 重新规划，绕过这个步骤或改变整体方案 | 没有替代工具，或目标本身需要调整 | 数据库查询权限不足 → 改为从缓存文件读取 |

**关键机制——错误信息回注上下文**：

工具调用失败后，Harness 不是简单返回"失败"，而是把完整的错误信息（错误码、错误描述、堆栈摘要）作为 Observation 注入上下文。LLM 在下一轮 Thought 中分析这些信息，决定走哪一层策略。这就是 ReAct 模式天然支持错误处理的原因——Observation 不区分成功结果和失败结果，都是 LLM 的决策输入。

**工程实现要点**：

1. **最大重试次数**：每个工具设上限（通常 2-3 次），防止 Agent 陷入无限重试循环
2. **错误分类**：区分可重试错误（超时、限流）和不可重试错误（权限不足、资源不存在），不可重试的直接跳到 L3/L4
3. **全局兜底**：设置整个任务的最大工具调用次数（比如 20 次），超过后强制终止并向用户汇报当前进度和失败原因
4. **优雅降级**：如果核心步骤失败无法绕过，Agent 应该输出"我完成了 X 和 Y，但 Z 步骤因为某原因失败，需要你介入"，而不是默默返回一个不完整的结果

### 如何设计安全的工具权限控制机制？

工具权限控制的核心原则是**最小权限**：Agent 只能调用完成当前任务所必需的工具，不能多。

**三个控制维度**：

1. **工具粒度**：哪些工具可以被调用
   - 白名单模式：明确列出允许调用的工具列表，默认拒绝其他所有工具
   - 黑名单模式：列出禁止调用的工具，其余允许（不推荐，容易遗漏）
   - 生产环境一律用白名单，宁可功能受限也不能权限过宽

2. **参数粒度**：工具的哪些参数值合法
   - 路径限制：文件读写工具只允许访问指定目录（如 `/workspace/`），禁止 `../` 路径穿越
   - 值域限制：数据库查询工具只允许 SELECT，禁止 DROP/DELETE
   - 格式校验：所有参数在执行前做 Schema 校验，不合法直接拒绝

3. **频率粒度**：调用频率上限
   - 单工具调用次数上限（防止某个工具被反复调用）
   - 全局工具调用次数上限（防止 Agent 失控）
   - 时间窗口内的调用频率限制（防止打挂下游服务）

**声明式权限 vs 运行时检查**：

| 方式 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| 声明式权限 | 在 Agent 配置文件里预先声明允许的工具和参数范围 | 可审计、可版本化、启动时就能发现配置错误 | 灵活性低，动态场景难处理 |
| 运行时检查 | 每次工具调用前由 Harness 拦截并校验 | 灵活，可根据上下文动态判断 | 逻辑分散，容易遗漏 |

生产中两者结合：声明式权限做粗粒度控制（哪些工具可用），运行时检查做细粒度校验（参数是否合法）。

**危险操作的人工审批工作流**：

对于不可逆操作（删除文件、发送邮件、执行 SQL 写操作、调用付费 API），不能让 Agent 自主执行，需要插入人工确认环节：

1. Agent 生成操作意图，暂停执行
2. 向用户展示"即将执行：删除 /data/prod/users.csv，是否确认？"
3. 用户确认后才真正执行
4. 操作结果记录审计日志

这个模式叫 **Human-in-the-Loop**，是生产级 Agent 的标配安全机制。

## 四、MCP 协议

### MCP（Model Context Protocol）是什么？

MCP 是工具接入的标准化协议，相当于 AI 世界的「USB 接口」。

**解决的问题**：没有 MCP 之前，每个工具都要为每个模型单独适配，M 个模型 × N 个工具 = M×N 个集成工作。MCP 把这个变成 M+N：工具只需实现一次 MCP 接口，所有支持 MCP 的模型都能调用。

**架构**：Host（AI 应用）- Client（MCP 客户端）- Server（工具服务），基于 JSON-RPC 2.0 通信。

**和 Function Call 的区别**：Function Call 解决"模型怎么触发工具调用"，MCP 解决"工具怎么标准化暴露给模型"。两者是不同层次的问题。

> [!info] MCP 的行业采用现状
> 截至 2026 年，MCP 已被 Claude、Cursor、Windsurf、VS Code 等主流 AI 工具原生支持，成为事实标准。OpenAI 也在 2025 年宣布支持 MCP。面试中提到 MCP 时，强调它是"已经落地的行业标准"而非"实验性协议"。

### MCP 和 A2A 的区别

MCP 和 A2A 解决的是两个不同层次的问题：

- **MCP**：Agent 对工具的协议。解决"一个 Agent 如何调用外部工具"的问题——文件系统、数据库、API、搜索引擎都通过 MCP 标准化接入
- **A2A（Agent-to-Agent）**：Agent 对 Agent 的协议，由 Google 提出。解决"多个 Agent 之间如何互相调用和协作"的问题，尤其是跨团队、跨组织的场景

**A2A 的工作机制**：

每个 Agent 发布一张"能力名片"（Agent Card），是一个 JSON 文件，声明这个 Agent 能做什么、接受什么输入、返回什么输出、支持哪些认证方式。其他 Agent 通过标准 HTTP API 发现并调用它。

```json
{
  "name": "CodeReviewAgent",
  "description": "审查代码，输出问题列表和改进建议",
  "capabilities": ["code_review", "security_scan"],
  "endpoint": "https://agents.company.com/code-review",
  "inputSchema": { "type": "object", "properties": { "code": { "type": "string" } } }
}
```

**A2A 和单 Agent 的核心区别**：

| 维度 | 单 Agent | A2A 多 Agent |
|------|---------|-------------|
| 能力边界 | 一个 Agent 包揽所有工具和技能 | 每个 Agent 专注一个领域，通过 A2A 组合 |
| 团队协作 | 单团队维护 | 不同团队各自维护自己的 Agent，通过 A2A 互操作 |
| 扩展方式 | 给 Agent 加工具 | 接入新的专业 Agent |
| 故障隔离 | 一个工��挂了影响整体 | 一个 Agent 挂了其他 Agent 不受影响 |
| 适用规模 | 中小型任务 | 企业级、跨组织的复杂任务 |

**MCP 和 A2A 的互补关系**：

在一个完整的 Agent 系统里，两者通常同时存在：
- 每个 Agent 内部用 MCP 接入工具（文件、数据库、API）
- Agent 之间用 A2A 互相调用（编排者 Agent 调用执行者 Agent）

类比：MCP 是 Agent 的"手"（操作工具），A2A 是 Agent 的"嘴"（和其他 Agent 沟通）。

## 五、Memory 系统

### 短期记忆、长期记忆、工作记忆分别对应什么技术实现？

Agent 的记忆系统分三层，每层解决不同的问题：

| 层次 | 名称 | 生命周期 | 技术实现 | 存储位置 |
|------|------|---------|---------|---------|
| L1 | 工作记忆（Working Memory） | 单次推理步骤内 | LLM 的 KV Cache + 当前 Prompt | 内存，不持久化 |
| L2 | 短期记忆（Short-term Memory） | 当前会话 | 上下文窗口（Context Window） | JSONL 会话日志文件 |
| L3 | 长期记忆（Long-term Memory） | 跨会话持久化 | 向量检索 + FTS 全文索引 | MEMORY.md + SQLite |

**工作记忆（L1）**：存在于单次 ReAct 循环内部，是 LLM 当前正在"思考"的内容——当前的 Thought、正在构造的 Action 参数、刚收到的 Observation。这一层完全在 LLM 的推理过程中，不需要显式管理，���环结束就消失。

**短期记忆（L2）**：当前会话的完整对话历史，包括所有轮次的 Thought/Action/Observation。存放在上下文窗口里，会话结束就消失。容量受 token 限制，接近上限时触发 Compaction 压缩。

**长期记忆（L3）**：跨会话持久化存储，存放用户偏好、历史决策、重要事实。不是全量注入上下文，而是通过检索按需取出。

**短期→长期的转换机制**：
1. **session-memory Hook**：用户执行 `/new` 重置会话时，自动将上一个会话的关键内容转换为 Markdown
2. **Memory Flush**：Session 接近 context 上限时触发 Compaction，提取重要信息写入 Memory

### Memory 的两层架构

**短期记忆（Working Memory）**：当前会话的上下文窗口，存放本轮对话历史。会话结束就消失，容量受 token 限制。存储在 JSONL 会话日志文件中，自动记录每次对话内容，是最原始的未经处理的记忆。

**长期记忆（Long-term Memory）**：跨会话持久化存储，存储在 MEMORY.md 和 memory/*.md 文件中。用户偏好���历史决策、重要事实都放这里。需要时通过检索按需取出，不是全量注入上下文。

**短期→长期的转换机制**：
1. **session-memory Hook**：用户执行 `/new` 重置会话时，自动将上一个会话的关键内容转换为 Markdown
2. **Memory Flush**：Session 接近 context 上限时触发 Compaction，提取重要信息写入 Memory

### Memory 和 RAG 的区别

以 OpenClaw 为例：

| 维度 | Memory | RAG |
|------|--------|-----|
| 目的 | 记住用户偏好、历史决策、个性化信息 | 检索外部知识库回答专业问题 |
| 数据来源 | 对话历史自动沉淀 | 外部文档、知识库主动导入 |
| 存储 | SQLite（轻量本地） | ElasticSearch / 向量数据库（重量级） |
| 检索方式 | FTS5+BM25 关键词 + sqlite-vec 向量混合检索 | 向量检索 + BM25 混合检索 + Rerank |
| 注入方式 | 按需检索，不全量注入上下文 | 召回 Top-K 片段拼入 Prompt |

### Memory 的索引建立四步

**关键架构观点：Markdown 是记忆本体（source of truth），SQLite 只是加速层。**

1. **发现**：监控 MEMORY.md 和 memory/*.md 的变化，标记 dirty
2. **切块**：把 Markdown 切成多个 chunk，让一个块只表达一小段完整的意思
3. **索引**：每个 chunk 同时走向量索引（语义召回）和 FTS 全文索引（关键词命中）
4. **落库**：chunk、元信息、全文索引、向量索引都放到本地 SQLite

### 混合检索为什么比纯向量靠谱？

搜 `nomic-embed-text` 这种精确关键词→FTS5+BM25 命中。搜「上次说过的那个写作偏好」这种模糊语义→向量检索召回。两边结果融合返回，互补短板。

### Memory 怎么被 Agent 使用？

**核心原则：按需检索而非全量注入。** Memory 文件不是每回合全量注入上下文窗口，而是通过两步按需读取，这就是"能越记越多但又不撑爆上下文"的根本原因。

OpenClaw 给 Agent 暴露了两个工具：
- **`memory_search(query)`**：返回最相关的 snippet + 文件路径 + 行号范围（不是整篇内容）。先粗召回找到"值得展开读"的位置，同时控制 token。
- **`memory_get(path, line_range)`**：根据 `memory_search` 返回的路径和行段精读具体内容。

**工程启示：** 不需要部署重量级向量数据库，一个 SQLite 文件 + FTS5 + sqlite-vec 扩展就够了。本地 Agent 场景下，这种"最简方案解决最复杂问题"的思路比堆技术栈更有价值。

### 上下文溢出的四种处理方案

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| 滑动窗口 | 只保留最近 N 轮对话 | 简单粗暴，会丢早期重要信息 |
| 摘要压缩 | LLM 把旧对话压缩成摘要 | 保留大意，细节丢失 |
| 向量检索增强 | 历史对话存向量库，每轮检索最相关片段注入 | 精准但增加延迟 |
| 关键事实提取 | 维护「重要事实列表」，每轮注入 | 最精准，维护成本高 |
