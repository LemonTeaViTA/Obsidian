# Java 内存模型

> 来源： [并发编程篇](../并发编程篇.md)
> 关联： [volatile](volatile.md), [synchronized](synchronized.md)
## 核心结论
Java 内存模型描述了线程之间共享变量的可见性、有序性和原子性问题。
## 入口
- 主内存与工作内存
- happens-before
- as-if-serial
- volatile
- synchronized
- 指令重排
## 基本概念
JMM 不是一套具体的物理内存结构，而是一套抽象规范。它定义了线程如何读写共享变量，以及这些读写在什么条件下对其他线程可见。

主内存保存共享变量的真实值，线程会把变量拷贝到自己的工作内存中使用。线程修改变量后，需要在合适的同步动作下把结果刷新回主内存。

可以把整个过程理解成两步：先从主内存把变量读到自己的工作内存里，再在完成修改后把结果同步回主内存。线程之间不是直接交换寄存器里的值，而是通过主内存这个公共区域来完成通信。
### 为什么线程要用自己的内存？

1. **性能优化**：线程从主内存拷贝变量到工作内存，可以减少 CPU 频繁访问 RAM 的开销，提高执行效率。
2. **避免冲突**：每个线程都有自己的变量副本，可以避免多个线程同时修改共享变量导致的数据冲突。
## 可见性
可见性问题的核心是，一个线程修改了变量，另一个线程能否及时看到这个变化。volatile 和 synchronized 都可以解决这个问题，只是实现手段不同。详见 [[并发编程/锁#volatile 关键字的作用？|volatile 关键字的作用]]。

## 原子性
原子性表示一个操作要么全部完成，要么完全不做。像 count++、i = i + 1 这种复合操作就不是原子操作，需要用锁或原子类来保护。
## 有序性
编译器和 CPU 会为了性能做指令重排。JMM 通过 happens-before 规则和内存屏障约束这种重排，保证在并发场景下结果仍然符合预期。
## happens-before
happens-before 不是时间上的绝对先后，而是可见性和有序性的保证关系。只要 A happens-before B，就意味着 A 的结果对 B 可见，并且 A 的执行顺序不能被重排到 B 之后。
常见规则包括：程序顺序规则、监视器锁规则、volatile 规则、传递性、线程启动规则和线程终止规则。
## as-if-serial
as-if-serial 允许编译器和 CPU 优化单线程代码，只要最终执行结果不变即可。它不保证多线程场景下的执行顺序，所以多线程还需要 volatile 或 synchronized 来配合。
