# 聊天助手模块

## 一、模块定位

聊天助手模块是派聪明系统的**核心交互组件**，承载用户与系统之间的主要对话能力。

**核心能力**：
1. **实时通信**：WebSocket 双向通信，支持流式返回
2. **上下文管理**：Redis 存储会话历史，支持多轮对话
3. **知识检索**���深度集成 Elasticsearch 混合检索
4. **Prompt 构建**：动态生成高质量 Prompt，引导模型生成

## 二、技术选型

| 功能模块 | 技术选型 | 备注 |
| --- | --- | --- |
| 实时通信 | WebSocket（Spring WebSocket） | 支持 STOMP 子协议 |
| 对话上下文存储 | Redis（Spring Data Redis） | 高性能缓存，支持 TTL |
| 本地知识库（当前） | Elasticsearch | 支持混合检索 |
| 本地知识库（规划） | Faiss | 提升向量检索性能 |
| 语言模型调用 | DeepSeek API | 通过 WebClient 调用 |
| Prompt 管理 | 自研模板引擎 | 支持动态模板和变量替换 |
| 异步处理 | Spring WebFlux | 支持响应式编程 |
| 安全认证 | JWT | 确保 WebSocket 连接安全 |

## 三、关键流程

### 1. 用户发起对话流程

```
1. 客户端发起 WebSocket 连接请求（携带 JWT Token）
2. 服务端验证用户身份和权限
3. 建立 WebSocket 长连接
4. 用户发送问题 → 服务端接收并解析
5. 获取或创建会话 ID（conversationId）
6. 启动知识库检索流程：
   - 调用 /api/search/hybrid 接口
   - 执行混合检索（KNN 向量 + BM25 关键词）
   - 筛选、排序、提取关键内容和出处
7. 构建 Prompt：
   - 选择合适的 Prompt 模板
   - 填充检索结果
   - 添加系统级指令
   - 管理上下文长度
8. 调用大模型 API（DeepSeek）
9. 流式接收模型生成内容
10. 通过 WebSocket 实时推送给客户端
11. 更新 Redis 中的对话历史
12. 刷新会话过期时间
```

**流程图**：

```

  ↓

  ↓

  ↓

  ↓

  ├─ 选择模板
  ├─ 填充检索结果

  ↓
调用 DeepSeek API
  ↓

  ↓

  ↓

```

### 2. 新建会话流程

```
1. 客户端发送"创建会话"请求（REST 接口）
2. 服务端验证用户身份
3. 生成全局唯一的 conversationId
4. 准备空的历史记录结构
5. 在 Redis 中建立用户和会话 ID 的映射关系
6. 设置会话过期时间（如 24 小时或 7 天）
7. 返回 conversationId 给客户端
```

**Redis 数据结构**：

```
# 会话元信息
conversation:{conversationId} → {
  "userId": 123,
  "title": "新对话",
  "createdAt": "2025-01-01T10:00:00Z",
  "expiresAt": "2025-01-08T10:00:00Z"
}

# 对话历史
conversation:{conversationId}:history → [
  {"role": "user", "content": "什么是 RAG？"},
  {"role": "assistant", "content": "RAG 是检索增强生成..."}
]

# 用户会话列表
user:{userId}:conversations → [conversationId1, conversationId2, ...]
```

### 3. 上下文管理流程

**问题**：多轮对话时，如何让模型"记住"前文？

**解决方案**：
1. **Redis 存储历史**：每轮对话后，将 user 和 assistant 消息存入 Redis
2. **动态注入上下文**：构建 Prompt 时，从 Redis 读取最近 N 轮对话
3. **长度控制**：设置轮数上限（如最近 5 轮），避免超出模型上下文限制
4. **TTL 管理**：会话过期后自动清理，节省存储

### 4. 流式返回流程

**为什么需要流式返回？**
- 大模型生成耗时长（5-10 秒），用户体验差
- 流式返回可以"边生成边展示"，提升交互流畅度

**实现方式**：

```java
// 调用 DeepSeek API，启用流式模式
WebClient.create()
    .post()
    .uri("https://api.deepseek.com/v1/chat/completions")
    .bodyValue(Map.of(
        "model", "deepseek-chat",
        "messages", messages,
        "stream", true  // 启用流式
    ))
    .retrieve()
    .bodyToFlux(String.class)  // 响应式流
    .subscribe(chunk -> {
        // 解析 SSE 格式的 chunk
        String content = parseChunk(chunk);
        
        // 通过 WebSocket 推送给客户端
        webSocketSession.sendMessage(new TextMessage(content));
    });
```

**客户端处理**：

```javascript
const ws = new WebSocket('ws://localhost:8080/chat');

ws.onmessage = (event) => {
  const chunk = event.data;
  // 逐段追加到页面
  document.getElementById('response').innerText += chunk;
};
```

```java
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    }
    
    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/chat")
            .setAllowedOrigins("*")
            .withSockJS();
    }

```

### 2. 消息处理器

```java
@Controller
public class ChatController {
    
    @MessageMapping("/chat")
    @SendToUser("/queue/messages")
    public void handleChatMessage(ChatRequest request, Principal principal) {
        String username = principal.getName();
        String conversationId = request.getConversationId();
        String userMessage = request.getMessage();
        
        // 1. 检索知识库
        List<Document> docs = searchService.hybridSearch(userMessage);
        
        // 2. 构建 Prompt
        List<Message> messages = promptService.buildPrompt(
            conversationId, userMessage, docs
        );
        
        // 3. 调用大模型（流式）
        deepSeekService.streamChat(messages)
            .subscribe(chunk -> {
                // 4. 推送给客户端
                messagingTemplate.convertAndSendToUser(
                    username, 
                    "/queue/messages", 
                    new ChatResponse("chunk", chunk, conversationId)
                );
            });
        
        // 5. 更新 Redis 历史
        conversationService.appendHistory(conversationId, userMessage, response);
    }
}
```

### 3. 异步处理

```java
@Async
public CompletableFuture<List<Document>> hybridSearch(String query) {
    // 并行执行 KNN 和 BM25
    CompletableFuture<List<Document>> knnFuture = 
        CompletableFuture.supplyAsync(() -> knnSearch(query));
    
    CompletableFuture<List<Document>> bm25Future = 
        CompletableFuture.supplyAsync(() -> bm25Search(query));
    
    return CompletableFuture.allOf(knnFuture, bm25Future)
        .thenApply(v -> mergeResults(knnFuture.join(), bm25Future.join()));
}
```

## 七、面试要点

### Q1：为什么选择 WebSocket 而不是 HTTP 轮询？

**A**：
1. **实时性**：WebSocket 双向通信，服务端可主动推送
2. **性能**：长连接复用，避免频繁建立 TCP 连接
3. **流式友好**：适合大模型流式返回场景

**缺点**：
- 连接管理复杂（需要心跳、重连机制）
- 负载均衡需要支持 WebSocket（如 Nginx sticky session）

### Q2：Redis 存储对话历史，如何控制内存？

**A**：
1. **TTL 过期**：会话 24 小时后自动清理
2. **轮数限制**：只保留最近 5 轮对话
3. **LRU 淘汰**：Redis 配置 `maxmemory-policy allkeys-lru`
4. **归档机制**：重要对话持久化到 MySQL

### Q3：如何处理 WebSocket 连接断开？

**A**：
1. **客户端重连**：指数退避重连（1s → 2s → 4s → 8s）
2. **会话恢复**：重连后通过 conversationId 恢复上下文
3. **心跳检测**：客户端每 30 秒发送 ping，服务端响应 pong
4. **超时清理**：服务端检测到连接空闲 5 分钟后主动关闭

### Q4：流式返回如何处理异常？

**A**：
1. **超时处理**：设置 30 秒超时，超时返回"生成超时"
2. **内容为空**：检测到空响应，返回"模型未生成内容"
3. **格式错误**：解析失败时，记录日志并返回友好提示
4. **降级策略**：DeepSeek 失败时，切换到备用模型（如 Qwen）
