# 百万级文档的存储与检索压力应对方案

> 问题：文档数量百万级怎么办？MySQL 怎么存？存储压力大怎么办？检索压力大怎么办？（不换 ES）
> 理论基础：[[检索召回与优化#五、向量数据库选型]]（向量数据库选型与 HNSW 调参） · [[Baize项目/RAG管道设计|RAG管道设计]]（基线架构）

---

## 一、当前存储架构

```
MySQL                          Elasticsearch
├── file_upload（文件元数据）    └── knowledge_base 索引
├── document_vectors（chunk原文）     ├── fileMd5
├── chunk_info（分片上传信息）        ├── chunkId
├── users                            ├── textContent
├── conversations                    ├── pageNumber
├── organization_tags                ├── anchorText
├── invite_codes                     ├── vector (dense_vector)
├── model_provider_configs           ├── userId
├── recharge_orders                  ├── orgTag
├── recharge_packages                └── isPublic
├── user_daily_chat_count
└── user_token_record
```

关键点：chunk 数据双存储——MySQL `document_vectors` 存原文备份，ES `knowledge_base` 存检索副本+向量。

---

## 二、百万级文档的数据量估算

假设 100 万个文档，平均每个文档切 50 个 chunk：

| 数据 | 数量 | 说明 |
|------|------|------|
| 文件元数据 | 100 万行 | MySQL `file_upload` |
| chunk 记录 | 5000 万行 | MySQL `document_vectors` + ES |
| 向量数据 | 5000 万 × 1024 维 × 4 字节 | ≈ 200GB 纯向量 |

---

## 三、MySQL 压力与应对

### 问题
- `document_vectors` 表 5000 万行，单表查询变慢
- `conversations` 表随用户增长持续膨胀
- 全表扫描、JOIN 查询性能急剧下降

### 应对方案

**1. 分表（最关键）**

按 `fileMd5` 哈希分表，把 `document_vectors` 拆成 16 或 64 张子表：

```sql
-- 分表策略：fileMd5 的哈希值 % 64
document_vectors_00
document_vectors_01
...
document_vectors_63
```

可以用 ShardingSphere 或 MyBatis 插件实现透明分表，业务代码不用改。

**2. 索引优化**

```sql
-- document_vectors 必须有的索引
CREATE INDEX idx_file_md5 ON document_vectors(fileMd5);
CREATE INDEX idx_user_id ON document_vectors(userId);
CREATE INDEX idx_org_tag ON document_vectors(orgTag);

-- file_upload 必须有的索引
CREATE INDEX idx_user_status ON file_upload(userId, status);
CREATE INDEX idx_org_tag ON file_upload(orgTag);
```

**3. 冷热分离**

- 热数据：最近 3 个月上传的文档，保留在主库
- 冷数据：3 个月前的文档，归档到只读从库或压缩表
- `conversations` 表定期归档，只保留最近 N 天

**4. 读写分离**

- 主库：写入（上传、解析）
- 从库：读取（文件列表、历史记录）
- Spring Boot 配置多数据源，用 `@Transactional(readOnly=true)` 自动路由

**5. 考虑去掉 MySQL 中的 chunk 冗余**

当前 chunk 在 MySQL 和 ES 都存了一份。百万级时 MySQL 的 `document_vectors` 表会非常大，而它的主要用途是重新向量化时的数据源。可以考虑：
- 只在 ES 存 chunk + 向量
- MySQL 只存文件元数据
- 需要重新向量化时从 MinIO 重新解析

## 四、ES 检索压力与应对（不换 ES）

### 问题
- 5000 万条 dense_vector，KNN 搜索变慢
- 向量索引占用大量内存（200GB+）
- BM25 rescore 窗口大（topK×30=150）时延迟高

### 应对方案

**1. 分索引（按组织/租户）**

```
knowledge_base_org_default
knowledge_base_org_finance
knowledge_base_org_tech
...
```

每个组织一个索引，检索时只查对应索引，大幅缩小搜索范围。

**2. 索引分片（Shard）调优**

```json
{
  "settings": {
    "number_of_shards": 5,
    "number_of_replicas": 1
  }
}
```

- 单分片建议不超过 1000 万条文档
- 5000 万条 → 至少 5 个主分片
- 多节点部署时分片分布到不同机器，并行检索

**3. 向量量化（减少内存占用）**

ES 8.x 支持向量量化，把 float32 压缩为 int8：

```json
{
  "mappings": {
    "properties": {
      "vector": {
        "type": "dense_vector",
        "dims": 1024,
        "index": true,
        "similarity": "cosine",
        "index_options": {
          "type": "int8_hnsw"
        }
      }
    }
  }
}
```

内存占用降为原来的 1/4（200GB → 50GB），检索速度也更快，精度损失很小。

**4. 预过滤优化**

当前是 KNN + must match 同时执行。百万级时可以改为先 filter 再 KNN：

```java
// 先用 filter 缩小范围（orgTag、userId），再在子集上做 KNN
s.knn(kn -> kn
    .field("vector")
    .queryVector(queryVector)
    .k(recallK)
    .numCandidates(recallK)
    .filter(f -> f.term(t -> t.field("orgTag").value(userOrg)))
);
```

这样 KNN 只在该组织的文档里搜索，而不是全量 5000 万条。

**5. 缩小召回窗口**

当前 `recallK = topK * 30 = 150`，百万级时可以调小：

```java
int recallK = topK * 10;  // 50 条候选足够
```

召回窗口越大越慢，50 条候选配合 BM25 rescore 已经足够精准。

**6. 缓存热门查询**

用 Redis 缓存高频查询的检索结果：

```java
String cacheKey = "search:" + md5(query + userId);
List<SearchResult> cached = redis.get(cacheKey);
if (cached != null) return cached;
// ... 执行检索
redis.set(cacheKey, results, Duration.ofMinutes(10));
```

## 五、存储压力应对

### MinIO（文件存储）
- 百万文件 × 平均 5MB = 5TB
- MinIO 天然支持分布式，加节点扩容即可
- 开启纠删码（Erasure Coding）保证数据安全

### ES（向量+文本）
- 向量量化（int8_hnsw）减少 75% 内存
- 冷热架构：热节点用 SSD 存近期数据，温节点用 HDD 存历史数据
- `_source` 排除 vector 字段（只在需要时获取），减少磁盘 IO

### MySQL
- 分表 + 读写分离
- 定期归档冷数据
- 考虑去掉 `document_vectors` 冗余

## 六、总结：优先级排序

| 优先级 | 措施 | 效果 |
|--------|------|------|
| P0 | ES 分索引（按组织） | 检索范围缩小 N 倍 |
| P0 | ES 向量量化（int8_hnsw） | 内存降 75%，速度更快 |
| P1 | ES KNN 预过滤（先 filter 再 KNN） | 避免全量扫描 |
| P1 | MySQL 分表（document_vectors） | 单表不超过千万行 |
| P1 | MySQL 索引优化 | 查询性能保障 |
| P2 | 缩小召回窗口（150→50） | 减少 rescore 延迟 |
| P2 | Redis 缓存热门查询 | 减少重复检索 |
| P2 | MySQL 读写分离 | 读性能翻倍 |
| P3 | 冷热分离（ES + MySQL） | 长期存储成本优化 |
| P3 | 去掉 MySQL chunk 冗余 | 减少存储浪费 |

## 七、面试怎么说

> 百万级文档大概 5000 万个 chunk，核心压力在 ES 的向量检索和 MySQL 的 chunk 存储。
>
> ES 方面，首先按租户分索引缩小检索范围，然后用 int8_hnsw 向量量化把内存降到 1/4，KNN 查询加 filter 预过滤避免全量扫描，召回窗口从 150 缩到 50。
>
> MySQL 方面，document_vectors 表按 fileMd5 哈希分 64 张子表，加上读写分离和索引优化。长期看可以去掉 MySQL 的 chunk 冗余，只在 ES 存检索数据，需要重新向量化时从 MinIO 重新解析。
>
> 存储方面，MinIO 天然支持分布式扩容，ES 用冷热架构把历史数据放 HDD 节点，热门查询用 Redis 缓存。
