# 用户管理与权限

## 一、模块目标

用户管理模块负责处理用户的**注册、登录和权限控制**功能，核心目标：

1. 确保用户身份的安全性
2. 提供灵活的权限管理机制（RBAC + 组织标签）
3. 为其他模块提供用户信息支持

## 二、技术选型

| 功能模块 | 技术选型 | 备注 |
| --- | --- | --- |
| 认证框架 | Spring Security | 基于 Filter 的安全框架 |
| 密码加密 | BCrypt | 单向加密，防止密码泄露 |
| 身份令牌 | JWT | 无状态认证，支持分布式 |
| 权限模型 | RBAC + 组织标签 | 功能权限 + 数据权限 |
| 用户存储 | MySQL | 用户表、组织标签表 |

## 三、RBAC 权限模型

### 1. 核心概念

RBAC（Role-Based Access Control）通过将用户分配到不同角色，并为角色分配权限，实现对资源的访问控制。

**派聪明的角色定义**：

| 角色 | 权限范围 |
| --- | --- |
| USER（普通用户） | 上传文件、进行对话、查看自己的历史记录 |
| ADMIN（管理员） | 普通用户所有权限 + 管理所有用户 + 查看系统状态 + 管理知识库 |

### 2. 角色继承模型

派聪明采用**角色继承的 RBAC 模型**：上层角色继承下层角色的所有权限，并可额外拥有其他权限。

```
ADMIN（管理员）
  ├─ 继承 USER 的所有权限
  └─ 额外权限：用户管理、系统配置、知识库管理
```

### 3. URL 级别的权限控制

```java
// 任何人都能访问的接口（如登录、注册）
.requestMatchers("/api/v1/users/register", "/api/v1/users/login").permitAll()

// 普通用户和管理员都能访问的接口
.requestMatchers("/api/v1/upload/**").hasAnyRole("USER", "ADMIN")

// 只有管理员能访问的接口
.requestMatchers("/api/v1/admin/**").hasRole("ADMIN")
```

### 4. 组织标签权限控制

除了角色权限（功能权限），派聪明还通过**组织标签**实现数据权限隔离：

| 资源类型 | 访问规则 |
| --- | --- |
| 私人空间 | 以 "PRIVATE_" 开头的资源，只有创建者能访问 |
| 组织资源 | 根据用户的 org_tags 控制访问权限 |
| 公开资源 | 所有认证用户都能访问 |

## 四、关键流程

### 1. 用户注册流程

```
1. 接收用户注册请求，验证用户名和密码
2. 检查用户名是否已存在
3. 使用 BCrypt 加密密码
4. 创建用户记录，设置默认角色为 USER
5. 创建用户私人组织标签（PRIVATE_username）
6. 将私人组织标签设置为用户的主组织标签
7. 返回注册成功响应
```

**关键代码**：

```java
// 密码加密
String encodedPassword = passwordEncoder.encode(password);

// 创建用户
User user = new User();
user.setUsername(username);
user.setPassword(encodedPassword);
user.setRole(Role.USER);

// 创建私人组织标签
String privateTag = "PRIVATE_" + username;
OrganizationTag tag = new OrganizationTag();
tag.setTagId(privateTag);
tag.setName("私人空间");
tag.setCreatedBy(user.getId());

// 设置为主组织
user.setPrimaryOrg(privateTag);
user.setOrgTags(privateTag);
```

### 2. 用户登录流程

```
1. 接收用户登录请求，获取用户名和密码
2. 查询用户记录并验证密码
3. 加载用户组织标签信息
4. 生成包含用户信息和组织标签的 JWT Token
5. 返回登录成功响应和 Token
```

**JWT Token 结构**：

```json
{
  "sub": "username",
  "userId": 123,
  "role": "USER",
  "orgTags": ["PRIVATE_username", "TECH_DEPT"],
  "primaryOrg": "PRIVATE_username",
  "iat": 1609459200,
  "exp": 1609545600
}
```

### 3. 组织标签管理流程

```
1. 管理员创建组织标签，设置标签名称和描述
2. 可选择设置父级组织标签（支持层级）
3. 管理员为用户分配组织标签
4. 系统自动保留用户的私人组织标签，确保其不被移除
5. 用户查看自己的组织标签
```

**层级示例**：

```
公司（COMPANY）
  ├─ 技术部（TECH_DEPT）
  │   ├─ 后端组（BACKEND_TEAM）
  │   └─ 前端组（FRONTEND_TEAM）
  └─ 产品部（PRODUCT_DEPT）
```

### 4. 权限验证流程

```
1. 解析请求头中的 JWT Token，验证有效性
2. 提取用户 ID、角色和组织标签信息
3. 对功能权限请求，根据用户角色判断是否允许访问
4. 对数据权限请求，根据用户组织标签判断是否可以访问特定资源
5. 允许或拒绝请求访问
```

**Spring Security 过滤器链**：

```
请求 → SecurityContextPersistenceFilter
    → JwtAuthenticationFilter（自定义）
    → UsernamePasswordAuthenticationFilter
    → FilterSecurityInterceptor
    → Controller
```

```
│   ├── UserRepository.java              # 用户数据访问
│   ├── OrganizationTagRepository.java   # 组织标签数据访问
├── security/
│   ├── JwtUtil.java                     # JWT 工具类
│   ├── CustomUserDetails.java          # 用户详情实现
```

### 2. SecurityConfig 核心配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf().disable()
            .authorizeHttpRequests(auth -> auth
                // 公开接口
                .requestMatchers("/api/v1/users/register", "/api/v1/users/login")
                    .permitAll()
                // 普通用户接口
                .requestMatchers("/api/v1/upload/**", "/api/v1/chat/**")
                    .hasAnyRole("USER", "ADMIN")
                // 管理员接口
                .requestMatchers("/api/v1/admin/**")
                    .hasRole("ADMIN")
                // 其他请求需要认证
                .anyRequest().authenticated()
            )
            .sessionManagement()
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            .and()
            .addFilterBefore(jwtAuthenticationFilter(), 
                UsernamePasswordAuthenticationFilter.class);
        
        return http.build();
    }
    
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

### 3. JWT 认证过滤器

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    
    @Override
    protected void doFilterInternal(HttpServletRequest request, 
                                    HttpServletResponse response, 
                                    FilterChain filterChain) {
        // 1. 从请求头获取 JWT Token
        String token = extractToken(request);
        
        if (token != null && jwtUtil.validateToken(token)) {
            // 2. 解析 Token，提取用户信息
            String username = jwtUtil.extractUsername(token);
            List<String> orgTags = jwtUtil.extractOrgTags(token);
            String role = jwtUtil.extractRole(token);
            
            // 3. 构建 Authentication 对象
            CustomUserDetails userDetails = new CustomUserDetails(
                username, role, orgTags
            );
            
            UsernamePasswordAuthenticationToken authentication = 
                new UsernamePasswordAuthenticationToken(
                    userDetails, null, userDetails.getAuthorities()
                );
            
            // 4. 设置到 SecurityContext
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }
        
        filterChain.doFilter(request, response);
    }
}
```

## 六、接口设计

### 1. 用户注册

- **URL**: `/api/v1/users/register`
- **Method**: POST
- **Request Body**:
```json
{
  "username": "string",
  "password": "string"
}
```
- **Response**:
```json
{
  "code": 200,
  "message": "User registered successfully"
}
```

### 2. 用户登录

- **URL**: `/api/v1/users/login`
- **Method**: POST
- **Request Body**:
```json
{
  "username": "string",
  "password": "string"
}
```
- **Response**:
```json
{
  "code": 200,
  "message": "Login successful",
  "data": {
    "token": "JWT_TOKEN_STRING"
  }
}
```

### 3. 获取用户信息

- **URL**: `/api/v1/users/me`
- **Method**: GET
- **Headers**: `Authorization: Bearer JWT_TOKEN_STRING`
- **Response**:
```json
{
  "code": 200,
  "data": {
    "id": 123,
    "username": "user1",
    "role": "USER",
    "orgTags": ["PRIVATE_user1", "TECH_DEPT"],
    "primaryOrg": "PRIVATE_user1"
  }
}
```

## 七、面试要点

### Q1：为什么选择 JWT 而不是 Session？

**A**：
1. **无状态**：JWT 自包含用户信息，服务端不需要存储 Session，适合分布式系统
2. **可扩展**：可以在 Token 中携带自定义信息（如组织标签）
3. **跨域友好**：不依赖 Cookie，适合前后端分离架构

**缺点**：
- Token 无法主动失效（需要配合黑名单或短过期时间）
- Token 体积较大（相比 Session ID）

### Q2：BCrypt 为什么比 MD5 安全？

**A**：
1. **加盐**：BCrypt 自动生成随机盐值，防止彩虹表攻击
2. **慢哈希**：计算成本高，防止暴力破解
3. **自适应**：可调整计算复杂度（work factor），随硬件升级保持安全性

### Q3：组织标签和角色的区别？

**A**：
- **角色（Role）**：控制功能权限（能做什么），如 ADMIN 可以管理用户
- **组织标签（Org Tags）**：控制数据权限（能看什么），如技术部只能看技术部的文档

两者结合实现**功能权限 + 数据权限**的双重隔离。

### Q4：如何防止 JWT 被盗用？

**A**：
1. **HTTPS**：防止传输过程中被窃听
2. **短过期时间**：减少被盗用后的影响时间窗口
3. **Refresh Token**：Access Token 短期（15 分钟），Refresh Token 长期（7 天）
4. **IP 绑定**：Token 中记录 IP，验证时检查
5. **黑名单**：用户登出时将 Token 加入 Redis 黑名单
