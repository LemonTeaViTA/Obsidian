## 一、BM25 算法

### 是什么
BM25（Best Matching 25）是经典的关键词匹配算法，基于词频统计给文档打分。核心思想：**一个词在文档中出现越多、在整个语料库中越稀有、文档越短，这个文档就越相关。**

### 公式

```
score(q, d) = Σ IDF(qi) × (tf(qi, d) × (k1 + 1)) / (tf(qi, d) + k1 × (1 - b + b × |d| / avgdl))
```

三个因子：
- **IDF（逆文档频率）**：词越稀有，权重越高。"的" 出现在所有文档里 → 权重低；"向量化" 只出现在少数文档 → 权重高
- **TF（词频）**：词在文档中出现次数越多，分数越高，但有饱和效应（出现 10 次不会比 5 次高很多）
- **文档长度归一化**：短文档中出现一次比长文档中出现一次更有意义

### 在项目中的实现
不需要自己实现，Elasticsearch 内置 BM25 作为默认评分算法。我们用 `match` 查询时 ES 自动用 BM25 打分：

```java
s.rescore(r -> r
    .query(rq -> rq
        .rescoreQueryWeight(1.0d)
        .query(rqq -> rqq.match(m -> m
            .field("textContent")
            .query(query)
            .operator(Operator.And)
        ))
    )
);
```

### 优缺点
- 优点：精确关键词匹配强，专有名词（"派聪明"）不会漏
- 缺点：不理解语义，"上传文件" 和 "文档导入" 匹配不上

## 二、KNN 算法

### 是什么
KNN（K-Nearest Neighbors）在这里指的是向量近似最近邻搜索。把文本转成向量后，在向量空间中找距离最近的 K 个邻居。

### 核心思路

```
用户问题 "怎么上传文件"
    ↓ Embedding 模型
向量 [0.12, -0.34, 0.56, ...]  (1024维)
    ↓ 在 ES 中找最近的向量
命中 "文档导入功能说明" 的向量（余弦相似度 0.87）
```

### 距离度量
我们用 **cosine 相似度**（Qwen3-Embedding 配置的就是 cosine）：

```
cosine_similarity(A, B) = (A · B) / (|A| × |B|)
```
值域 [-1, 1]，越接近 1 越相似。

### ES 中的实现
ES 8.x 用 HNSW（Hierarchical Navigable Small World）算法做近似最近邻，不是暴力遍历所有向量：

```java
s.knn(kn -> kn
    .field("vector")
    .queryVector(queryVector)
    .k(recallK)              // 返回 top-K
    .numCandidates(recallK)  // HNSW 搜索候选数，越大越准但越慢
);
```

HNSW 是一种图索引结构，把向量组织成多层图，搜索时从顶层粗定位逐层细化，时间复杂度 O(log N)，比暴力搜索 O(N) 快几个数量级。

### 优缺点
- 优点：理解语义，"上传文件" 能匹配 "文档导入"
- 缺点：对精确关键词弱，专有名词可能漏掉

### 为什么两者结合
KNN 宽召回（语义不漏）+ BM25 精排（关键词准），取两者之长。

## 三、文档分片存储怎么考虑的

这里有两个"分片"概念，不要混淆：

### 上传分片（Upload Chunk）
- 大小：5MB
- 目的：大文件拆小传输，支持断点续传
- 存储：MinIO `chunks/{fileMd5}/{index}`，合并后清理
- 详见：[分片上传状态追踪](2026-04-22_分片上传状态追踪_Redis_Bitmap.md)

### 文本分片（Text Chunk）
- 大小：512 字符
- 目的：把文档内容切成适合 embedding 和检索的小段
- 存储：MySQL `document_vectors` + ES `knowledge_base`
- 策略：段落 → 句子 → 分词 三级降级，无 overlap
- 详见：[RAG 管道设计](2026-04-22_RAG管道设计_分片解析向量化检索.md)

### 存储分布

```
上传分片 → MinIO（临时，合并后删除）
完整文件 → MinIO（永久）
文本分片原文 → MySQL document_vectors（持久化备份）
文本分片 + 向量 → ES knowledge_base（检索用）
```

## 四、对话上下文过长怎么办

### 当前实现：保留最近 20 条消息（10 轮对话）

```java
// ChatHandler.updateConversationHistory()
// 限制历史记录长度，保留最近的20条消息
if (history.size() > 20) {
    history = history.subList(history.size() - 20, history.size());
}
```

20 条消息 = 10 轮对话（每轮 1 条 user + 1 条 assistant）。

### 怎么保存的

存在 Redis 里，key 是 `conversation:{conversationId}`，value 是 JSON 数组，TTL 7 天：

```java
String key = "conversation:" + conversationId;
String json = objectMapper.writeValueAsString(history);
redisTemplate.opsForValue().set(key, json, Duration.ofDays(7));
```

每次对话完成后，把 user 消息和 assistant 回复追加到数组末尾，超过 20 条就截断只保留最新的 20 条。

### 为什么用 Redis 不用 MySQL

- 对话历史读写频繁（每次对话都要读+写）
- Redis 读写速度远快于 MySQL
- 对话历史不需要永久保存（7 天 TTL 自动过期）

## 五、除了取最近 N 轮，还有什么方法？

### 方法 1：滑动窗口 + Token 计数（最实用）

不按轮数截断，按 token 数截断。从最新的消息往前累加 token，超过预算就停：

```
token 预算 = max_tokens - system_prompt_tokens - 预留回复空间
从最新一轮往前加，加到超预算为止
```

好处：充分利用上下文窗口，不会浪费也不会超限。

### 方法 2：摘要压缩（Summary）

对话超过 N 轮后，用 LLM 把早期对话压缩成一段摘要，放在 system prompt 里：

```
system: 之前的对话摘要：用户询问了文件上传功能，你解释了分片上传机制...
user: （最近 5 轮的完整对话）
user: 新问题
```

好处：保留了早期对话的关键信息，不会完全丢失。
缺点：需要额外调用一次 LLM 做摘要，增加延迟和成本。

### 方法 3：RAG 式对话检索

把历史对话也存入向量库，每次新问题先检索相关的历史对话片段，只把相关的放进上下文：

```
用户问 "上次说的那个 bug 修了吗"
→ 检索历史对话，找到 3 天前关于 bug 的对话
→ 只把那段对话放进上下文
```

好处：能跨越很长的对话历史找到相关内容。
缺点：实现复杂，检索质量影响对话质量。

### 方法 4：重要消息标记

让 LLM 判断每轮对话是否"重要"（包含决策、结论、关键信息），重要的永远保留，不重要的优先丢弃：

```
[重要] 用户确认了用 Qwen3.5-9B 作为 LLM → 保留
[普通] 用户说了 "好的" → 可丢弃
```

### 对比

| 方法 | 实现难度 | 效果 | 适合场景 |
|------|----------|------|----------|
| 最近 N 轮（当前） | 最简单 | 够用 | 短对话、知识问答 |
| Token 计数截断 | 简单 | 更精确 | 上下文窗口有限时 |
| 摘要压缩 | 中等 | 保留早期信息 | 长对话、多轮讨论 |
| RAG 式检索 | 复杂 | 跨历史关联 | 客服、长期项目对话 |
| 重要消息标记 | 复杂 | 信息密度最高 | 决策类对话 |

## 面试怎么说

> 我们当前用最简单的方案——保留最近 20 条消息（10 轮对话），存在 Redis 里，JSON 格式，TTL 7 天。每次对话完成后追加到数组末尾，超过 20 条就截断。
>
> 如果要优化，首选 token 计数截断（按 token 预算而不是固定轮数），其次是摘要压缩（用 LLM 把早期对话压缩成摘要保留关键信息）。更复杂的场景可以用 RAG 式对话检索，把历史对话也向量化，每次只检索相关的历史片段放进上下文。
