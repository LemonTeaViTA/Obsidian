# 缓存经典问题

## 缓存三大问题

### 🌟什么是缓存击穿？

缓存击穿是指某个热点数据缓存过期时，大量请求就会穿透缓存直接访问数据库，导致数据库瞬间承受的压力巨大。

解决缓存击穿有两种常用的策略：

第一种是加互斥锁。当缓存失效时，第一个访问的线程先获取锁并负责重建缓存，其他线程等待或重试：

```java
String cacheKey = "product::" + productId;
RLock lock = redissonClient.getLock("lock::" + productId);
if (lock.tryLock(10, TimeUnit.SECONDS)) {
    try {
        String result = cache.get(cacheKey);
        if (result == null) {
            result = database.queryProductById(productId);
            cache.set(cacheKey, result, 60 * 1000);
        }
    } finally {
        lock.unlock();
    }
}
```

第二种是永不过期策略。缓存项本身不设置过期时间，但在缓存值中维护一个逻辑过期时间。当缓存逻辑上过期时，返回旧值的同时，异步启动一个线程去更新缓存：

```java
public String getData(String key) {
    CacheItem item = cache.get(key);
    if (item == null) {
        String data = db.query(key);
        cache.set(key, new CacheItem(data, System.currentTimeMillis() + expireTime));
        return data;
    } else if (item.isLogicalExpired()) {
        asyncRefresh(key);
        return item.getData();
    }
    return item.getData();
}
```

### 什么是缓存穿透？

缓存穿透是指查询的数据在缓存中没有命中，因为数据压根不存在，所以请求会直接落到数据库上。如果这种查询非常频繁，就会给数据库造成很大的压力。

缓存击穿是因为单个热点数据缓存失效导致的，而缓存穿透是因为查询的数据不存在，原因可能是自身的业务代码有问题，或者是恶意攻击造成的。

常用的解决方案有两种：

第一种是布隆过滤器，将所有可能存在的数据哈希到布隆过滤器中，查询时先检查布隆过滤器，如果布隆过滤器认为该数据不存在，就直接返回空：

```java
public String getData(String key) {
    String cacheResult = cache.get(key);
    if (cacheResult != null) {
        return cacheResult;
    }
    if (!bloomFilter.mightContain(key)) {
        return null;
    }
    String dbResult = db.query(key);
    cache.set(key, dbResult != null ? dbResult : "", expireTime);
    return dbResult;
}
```

第二种是缓存空值。对于不存在的数据，将空值写入缓存，并设置一个合理的过期时间：

```java
public String getData(String key) {
    String cacheResult = cache.get(key);
    if (cacheResult != null) {
        return cacheResult.equals("") ? null : cacheResult;
    }
    String dbResult = db.query(key);
    int expireTime = dbResult == null ? EMPTY_EXPIRE_TIME : NORMAL_EXPIRE_TIME;
    cache.set(key, dbResult != null ? dbResult : "", expireTime);
    return dbResult;
}
```

### 什么是缓存雪崩？

缓存雪崩是指在某一时间段，大量缓存同时失效或者缓存服务突然宕机了，导致大量请求直接涌向数据库，引发系统崩溃的现象。

缓存雪崩主要有三种成因和应对策略：

第一种，大量缓存同时过期，解决方法是添加随机过期时间：

```java
public void setCache(String key, String value) {
    int baseExpireSeconds = 1800;
    int randomSeconds = new Random().nextInt(300);
    cache.set(key, value, baseExpireSeconds + randomSeconds);
}
```

第二种，缓存服务崩溃，解决方法是使用高可用的缓存集群，并配置本地缓存作为二级缓存（缓存降级）：

```java
LoadingCache<String, UserPermissions> permissionsCache = Caffeine.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(this::loadPermissionsFromRedis);

public UserPermissions loadPermissionsFromRedis(String userId) {
    try {
        return redisClient.getPermissions(userId);
    } catch (Exception ex) {
        return permissionsCache.getIfPresent(userId);
    }
}
```

第三种，并发请求量超过了缓存服务的承载能力，采用限流和降级措施：

```java
public String getData(String key) {
    try {
        return cache.get(key);
    } catch (Exception e) {
        if (circuitBreaker.shouldTrip()) {
            circuitBreaker.trip();
            return getFromDbDirectly(key);
        }
        throw e;
    }
}
```

## 布隆过滤器

### 🌟能说说布隆过滤器吗？

布隆过滤器是一种空间效率极高的概率性数据结构，用于快速判断一个元素是否在一个集合中。它的特点是能够以极小的内存消耗，判断一个元素"一定不在集合中"或"可能在集合中"，常用来解决 Redis 缓存穿透的问题。

布隆过滤器的核心由一个很长的二进制向量和一系列哈希函数组成：

- 初始化时，创建一个长度为 m 的位数组，初始值全为 0，同时选择 k 个不同的哈希函数。
- 添加元素时，用 k 个哈希函数计算出 k 个位置，将这些位置的二进制位都设为 1。
- 查询元素时，同样用 k 个哈希函数计算出 k 个位置，如果这些位置有任何一个为 0，该元素一定不在集合中；如果全部为 1，则该元素可能在集合中。

#### 布隆过滤器存在误判吗？

是的，布隆过滤器存在误判。它可能会错误地认为某个元素在集合中，而元素实际上并不在集合中。但如果布隆过滤器认为某个元素不存在于集合中，那么它一定不存在。

误判产生的原因是哈希冲突。随着向布隆过滤器中添加的元素越来越多，位数组中的 1 也越来越多，发生哈希冲突的概率随之增加，误判率也就随之上升。

误判率取决于三个因素：位数组的大小（m）、哈希函数的数量（k）、存入的元素数量（n）。

要降低误判率，可以增加位数组的大小或者减少插入的元素数量。要彻底解决布隆过滤器的误判问题，可以在布隆过滤器返回"可能存在"时，再通过数据库进行二次确认。

#### 布隆过滤器支持删除吗？

布隆过滤器并不支持删除操作。当我们添加一个元素时，会将位数组中的 k 个位置设置为 1。由于多个不同元素可能共享相同的位，如果尝试删除一个元素，将其对应的 k 个位重置为 0，可能会错误地影响到其他元素的判断结果。

如果想要实现删除操作，可以使用计数布隆过滤器，它在每个位置上存储一个计数器而不是单一的位，通过减少计数器的值来实现删除操作，但会增加内存开销。

#### 为什么不能用哈希表而是用布隆过滤器？

布隆过滤器最突出的优势是内存效率。假如要判断 10 亿个用户 ID 是否曾经访问过特定页面，使用哈希表至少需要 10GB 内存（每个 ID 至少需要 8 字节），而使用布隆过滤器只需要约 1.2GB 内存。

## 缓存一致性

### 🌟如何保证缓存和数据库的数据一致性？

对于允许短暂不一致的数据，采用 Cache Aside + TTL 过期机制：读取时先查 Redis，未命中再查 MySQL，同时为缓存设置合理的过期时间；更新时先更新 MySQL，再删除 Redis。

```java
// 读取逻辑
public UserInfo getUser(String userId) {
    UserInfo user = cache.get("user:" + userId);
    if (user != null) return user;
    user = database.selectUser(userId);
    if (user != null) cache.set("user:" + userId, user, 3600);
    return user;
}

// 更新逻辑
public void updateUser(UserInfo user) {
    database.updateUser(user);
    cache.delete("user:" + user.getId());
}
```

#### 为什么要删除缓存而不是更新缓存？

在并发环境下，如果采用更新缓存的策略，可能出现时序问题：操作 A 先更新 MySQL 将值改为 10，操作 B 后更新 MySQL 将值改为 11，但在缓存更新时，B 先执行将缓存设为 11，然后 A 才执行将缓存设为 10，造成 MySQL 是 11 但 Redis 是 10 的不一致状态。

而采用删除策略，无论 A 和 B 谁先删除缓存，后续的读取操作都会从 MySQL 获取最新值。另外，删除操作只是简单的 DEL 命令，比更新缓存（需要重新序列化整个对象再写入）速度快得多。

#### 为什么要先更新数据库，再删除缓存？

如果先删缓存再更新数据库，在高并发场景下可能出现：线程 A 删除了缓存，线程 B 此时查询发现缓存为空，于是查询数据库（此时还是旧值）并将旧值放入缓存，线程 A 完成数据库更新。结果是数据库是新值，但缓存中还是旧值。

而先更新数据库再删缓存，最坏的情况也只是短暂地从缓存中读取到了旧值，缓存删除后的请求会直接从数据库中获取最新值。另外，如果数据库更新失败，缓存保持原状，系统仍然能继续正常提供服务。

#### 对缓存数据库一致性要求很高，该怎么办？

第一种，引入消息队列保证缓存最终被删除：

```java
@Transactional
public void updateUser(UserInfo user) {
    database.updateUser(user);
    LocalMessage message = new LocalMessage("CACHE_DELETE", "user:" + user.getId());
    database.insertLocalMessage(message);
    eventPublisher.publishEvent(new UserUpdateEvent(this, "user:" + user.getId()));
}

@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void sendCacheDeleteMessage(UserUpdateEvent event) {
    messageQueue.send("cache-delete-topic", event.getCacheKey());
}
```

第二种，使用 Canal 监听 MySQL 的 binlog，在数据更新时将数据变更记录到消息队列中，消费者监听到变更后去删除缓存。这种方案的优势是完全解耦了业务代码和缓存维护逻辑。

第三种，延迟双删策略，在更新数据库前后各删一次缓存，并在第二次删除后延迟再删一次：

```java
public void updateUser(UserInfo user) {
    cache.delete("user:" + user.getId());
    database.updateUser(user);
    cache.delete("user:" + user.getId());
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(1000);
            cache.delete("user:" + user.getId());
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}
```

无论采用哪种策略，最好为缓存设置一个合理的过期时间作为最后的保障。

### 如何保证本地缓存和分布式缓存的一致？

当数据更新时，通过 Redis 的 pub/sub 机制向所有应用实例发送缓存更新通知，收到通知后的实例立即更新或者删除本地缓存：

```java
public void updateData(String key, Object value) {
    database.update(key, value);
    redisTemplate.opsForValue().set(key, value, 30, TimeUnit.MINUTES);
    CacheUpdateMessage message = new CacheUpdateMessage(key, "UPDATE", value);
    redisTemplate.convertAndSend("cache-update-channel", message);
}

@EventListener
public void handleCacheUpdate(CacheUpdateMessage message) {
    if ("UPDATE".equals(message.getAction())) {
        localCache.put(message.getKey(), message.getValue());
    } else if ("DELETE".equals(message.getAction())) {
        localCache.invalidate(message.getKey());
    }
}
```

考虑到消息可能丢失，还可以引入版本号机制作为补充：每次从 Redis 获取数据时添加一个最新的版本号，从本地缓存获取数据前先检查版本号是否是最新的，如果发现版本落后，就主动从 Redis 中获取最新数据。

#### 本地缓存和 Redis 的区别了解吗？

Redis 可以部署在多个节点上，支持数据分片、主从复制和集群；而本地缓存只能在单个服务器上使用。

对于读取频率极高、数据相对稳定、允许短暂不一致的数据，优先选择本地缓存，比如系统配置信息、用户权限数据、商品分类信息等。

对于需要实时同步、数据变化频繁、多个服务需要共享的数据，选择 Redis，比如用户会话信息、购物车数据、实时统计信息等。

#### 如果在项目中多个地方都要使用到二级缓存的逻辑，如何设计这一块？

我的思路是将二级缓存抽象成一个统一的组件。设计一个 CacheManager 作为核心入口，提供 get、put、evict 等基本操作，执行先查本地缓存，再查分布式缓存，最后查数据库的完整流程。

```java
public class CacheManager {

    private final LocalCache localCache;
    private final RedisCache redisCache;
    private final Database database;

    public CacheManager(LocalCache localCache, RedisCache redisCache, Database database) {
        this.localCache = localCache;
        this.redisCache = redisCache;
        this.database = database;
    }

    public Object get(String key) {
        // 先查本地缓存
        Object value = localCache.get(key);
        if (value != null) {
            return value;
        }

        // 再查分布式缓存
        value = redisCache.get(key);
        if (value != null) {
            // 更新本地缓存
            localCache.put(key, value);
            return value;
        }

        // 最后查数据库
        value = database.get(key);
        if (value != null) {
            // 更新分布式缓存和本地缓存
            redisCache.put(key, value);
            localCache.put(key, value);
        }

        return value;
    }
}
```

## 热 Key 与大 Key

### 什么是热 Key？

所谓的热 Key，就是指在很短时间内被频繁访问的键。比如电商大促期间爆款商品的详情信息、流量明星爆瓜时的个人资料、热门话题等。

由于 Redis 是单线程模型，大量请求集中到同一个键会导致该 Redis 节点的 CPU 使用率飙升，响应时间变长。在 Redis 集群环境下，热 Key 还会导致数据分布不均衡。更严重的情况是，当热 Key 过期或被误删时，会引发缓存击穿问题。

#### 那怎么监控热 Key 呢？

可以使用 `redis-cli --hotkeys` 命令来监控 Redis 中的热 Key：

```bash
redis-cli -h <address> -p <port> -a <password> --hotkeys
```

或者在访问缓存时，在本地维护一个计数器，当某个键的访问次数在一分钟内超过设定阈值，就将其标记为热 Key：

```java
@Component
public class HotKeyDetector {
    private final ConcurrentHashMap<String, AtomicLong> accessCounter = new ConcurrentHashMap<>();
    private final int HOT_KEY_THRESHOLD = 1000;

    public boolean isHotKey(String key) {
        long count = accessCounter.computeIfAbsent(key, k -> new AtomicLong(0))
            .incrementAndGet();
        return count > HOT_KEY_THRESHOLD;
    }
}
```

### 那怎么处理热 Key 呢？

最有效的解决方法是增加本地缓存，将热 Key 缓存到本地内存中，这样请求就不需要访问 Redis 了。

对于一些特别热的 Key，可以将其拆分成多个子 Key，随机分布到不同的 Redis 节点上：

```java
public String getHotData(String key) {
    if (isHotKey(key)) {
        int replica = ThreadLocalRandom.current().nextInt(HOT_KEY_REPLICAS);
        return redis.get(key + ":" + replica);
    }
    return redis.get(key);
}
```

### 怎么处理大 Key 呢？

大 Key 是指占用内存空间较大的缓存键，比如超过 10MB 的键值对。大 Key 会导致 Redis 内存不足、主从复制同步延迟，甚至引发网络拥塞。

可以通过 `redis-cli --bigkeys` 命令来监控 Redis 中的大 Key。

对于大 Key 问题，最根本的解决方案是拆分大 Key，将其拆分成多个小 Key 存储：

```java
public void splitBigKey(String bigKey) {
    Map<String, String> bigData = redisTemplate.opsForHash().entries(bigKey);
    for (Map.Entry<String, String> entry : bigData.entrySet()) {
        String smallKey = bigKey + ":" + entry.getKey();
        redisTemplate.opsForValue().set(smallKey, entry.getValue());
    }
    redisTemplate.delete(bigKey);
}
```

另外，对于 JSON 数据，可以进行 Gzip 压缩后再存储，虽然会增加一些 CPU 开销，但在内存敏感的场景下是值得的。

## 其他

### 缓存预热怎么做呢？

缓存预热是指在系统启动或者特定时间点，提前将热点数据加载到缓存中，避免冷启动时大量请求直接打到数据库。

可以在项目启动时将热门文章提前加载到 Redis 中，在每天凌晨定时将最新的站点地图更新到 Redis 中：

```java
@Scheduled(cron = "0 15 5 * * ?")
public void autoRefreshCache() {
    refreshSitemap();
}
```

### 无底洞问题听说过吗？如何解决？

无底洞问题的核心在于，随着缓存节点数量的增加，虽然总的存储容量和理论吞吐量都在增长，但是单个请求的响应时间反而变长了。根本原因是网络通信开销的增加，以及数据分布的碎片化。

解决方案：

第一，将同一节点的多个请求合并成一个批量请求，减少网络往返次数：

```java
public Map<String, Object> batchGet(List<String> keys) {
    Map<String, List<String>> nodeKeysMap = groupKeysByNode(keys);
    Map<String, Object> results = new ConcurrentHashMap<>();
    List<CompletableFuture<Void>> futures = nodeKeysMap.entrySet().stream()
        .map(entry -> CompletableFuture.runAsync(() -> {
            Map<String, Object> nodeResults = getFromNode(entry.getKey(), entry.getValue());
            results.putAll(nodeResults);
        }))
        .collect(Collectors.toList());
    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
    return results;
}
```

第二，使用一致性哈希算法来优化数据分布，将相同类别的数据尽量分配到相同节点，减少数据迁移和重分布的开销。

### Redis 报内存不足怎么处理？

第一，使用 `INFO memory` 命令查看 Redis 的内存使用情况：

```bash
redis-cli INFO memory
```

第二，如果服务器还有可用内存，修改 `redis.conf` 中的 `maxmemory` 参数，增加 Redis 的最大内存限制：

```ini
maxmemory 8gb
```

第三，修改 `maxmemory-policy` 参数来调整内存淘汰策略：

```ini
maxmemory-policy allkeys-lru
```

### Redis key 过期策略有哪些？

Redis 主要采用了两种过期删除策略：

**惰性删除**：当客户端访问一个 key 时，Redis 会检查该 key 是否已过期，如果过期就会立即删除并返回 nil。优点是不会有额外的 CPU 开销，但如果一个过期的 key 永远不被访问，它就会一直占用内存。

**定期删除**：Redis 会定期随机选择一些设置了过期时间的 key 进行检查，删除其中已过期的 key。这个过程默认每秒执行 10 次，每次随机选择 20 个 key 进行检查。

### 🌟Redis 有哪些内存淘汰策略？

当内存使用接近 maxmemory 限制时，Redis 会依据内存淘汰策略来决定删除哪些 key：

- `noeviction`：默认策略，内存不足时不会删除任何 key，直接返回错误信息，生产环境基本不使用。
- `allkeys-lru`：删除最近最少使用的 key，在纯缓存场景中最常用，能自动保留热点数据。
- `allkeys-lfu`：删除访问频率最低的 key，更适合长期运行的系统。
- `allkeys-random`：随机删除一些 key，一般不推荐使用。
- `volatile-lru`：在设置了过期时间的 key 中，删除最近最少使用的，在混合存储场景中经常使用。
- `volatile-lfu`：在设置了过期时间的 key 中，删除访问频率最低的。
- `volatile-ttl`：优先删除即将过期的 key，在用户会话管理系统中推荐使用。
- `volatile-random`：在设置了过期时间的 key 中，随机删除一些 key。

### LRU 和 LFU 的区别是什么？

LRU（Least Recently Used）基于时间维度，淘汰最近最少访问的键。LFU（Least Frequently Used）基于次数维度，淘汰访问频率最低的键。

假设缓存中有三个数据 A、B、C，在 LRU 场景下，如果访问顺序是 A→B→C→A，那么此时的 LRU 顺序是 B→C→A，如果需要淘汰，会先删除 B。但在 LFU 场景下，如果 A 被访问了 5 次，B 被访问了 2 次，C 被访问了 1 次，那么无论最近的访问顺序如何，都会优先淘汰 C。

LRU 更适合有明显时间局部性的场景，比如新闻网站中用户更关心最新的新闻。LFU 则更适合有长期访问模式的场景，比如电商平台中某些商品长期保持热销状态。
