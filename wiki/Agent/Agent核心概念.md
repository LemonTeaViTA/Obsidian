# Agent 核心概念

## 一、Agent 定义与基本架构

### Agent 到底是什么？

Agent 不是某种更高级的模型，模型还是那个模型，参数没变，结构也没变，变的是外面那一圈工程系统。

Agent = LLM + Memory + Search + MCP + Skills

拆开来看每个都不新，拼起来看着就很新。就像早期的电脑，CPU、内存、硬盘都早就有了，但装进一个盒子里，你才觉得它是"一个完整机器"。

### Agent 的四个核心模块

#### 核心解析

| 模块 | 职责 | 类比 |
|------|------|------|
| LLM | 推理和决策，接收输入、理解意图、输出下一步计划 | Agent 的 CPU |
| Memory | 短期记忆（对话上下文）+ 长期记忆（跨会话持久化） | Agent 的内存+硬盘 |
| Tools | 外部能力：搜索、代码执行、数据库查询、API 调用 | Agent 的四肢 |
| Action Executor | 真正执行 LLM 决定的操作，把结果反馈回去 | Agent 的执行器 |

### Agent vs LLM Chain

#### 核心解析

LLM Chain 是固定流程，预先定义好每一步，流程是静态的，不会根据中间结果调整。

Agent 是动态决策，每一步都是 LLM 根据当前状态重新决定的，「下一步该干什么」是实时推理出来的。

面试一句话：LLM Chain 是剧本，Agent 是即兴表演。

选型原则：任务边界清晰、步骤可枚举，选工作流；任务开放、需要大量判断，选智能体。更常见的做法是混合架构——外层用工作流控制主流程，内层某些节点换成智能体处理开放性子任务。

## 二、ReAct 框架

### ReAct 模式是怎么工作的？

#### 核心解析

ReAct = Reasoning + Acting，工作流程是不断循环��「思考-行动-观察」：

1. **Thought（思考）**：LLM 分析当前情况，决定下一步。这一步不对外输出，只是内部推理
2. **Action（行动）**：LLM 决定调用哪个工具、传什么参数，以结构化 JSON 格式输出
3. **Observation（观察）**：系统执行 Action，把结果返回给 LLM

LLM 拿到结果后更新上下文，进入下一轮思考，判断任务是否完成。没完成继续循环，完成了输出最终答案。

关键点：ReAct 里的 LLM 每次只生成「下一步指令」，循环控制是外部程序负责的。这也是为什么实现 Agent 框架时，循环逻辑和上下文拼接非常重要。

具体例子——让 Agent 查 RAG 优化论文：
- 第一轮：Thought→需要搜��；Action→调用 search 工具；Observation→返回 10 篇摘要
- 第二轮：Thought→需要筛选；Action→调用 filter 工具；Observation→三篇精选
- 第三轮：Thought→信息足够；Final Answer→整理输出

### ReAct vs Plan-and-Execute vs Reflection

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| ReAct | 交替推理和执行，边想边做 | 通用 Agent 场景 |
| Plan-and-Execute | 先一口气规划完，再按计划执行 | 步骤明确、不容易走弯路的任务 |
| Reflection | 生成→评估→改进循环 | 代码生成、长文写作等高精度场景 |

## 三、Function Call

### Function Calling 是什么？

#### 核心解析

Function Calling 是模型侧的原生能力。厂商通过微调，让 LLM 能够输出符合特定 JSON Schema 的结构化调用指令，而不是在自然语言里猜工具。

比如你问「今天北京天气怎么样」，模型会输出：
```json
{"tool": "weather_query", "params": {"city": "北京"}}
```

它解决的核心问题：如何让模型准确地触发工具调用。

### 如何保证 Function Calling 的可靠性？

#### 核心解析

不可靠来自三个层面：

**模型层**：LLM 生成的调用参数可能格式错误、字段缺失。应对方式是在 Prompt 里明确每个工具的 JSON Schema，调用前做参数校验，格式不对打回重新生成，最多重试 3 次。

**执行层**：工具本身可能超时、返回异常。标准做法是设超时上限，加 retry 逻辑（指数退避），对返回结果做合法性检查，异常情况向 Agent 上报让它自己决定下一步。

**系统层**：高并发下多个 Agent 同时调同一个外部服务可能打挂下游。引入限流和熔断机制，触发熔断后返回降级结果。

经常被忽视的细节：工具描述的质量直接影响调用准确率。工具的 name、description 写得越清晰，模型选错工具的概率就越低。

## 四、MCP 协议

### MCP（Model Context Protocol）是什么？

#### 核心解析

MCP 是工具接入的标准化协议，相当于 AI 世界的「USB 接口」。

它解决的问题不是「模型怎么调工具」，而是「不同的工具怎么用统一的协议暴露给不同的模型」。

架构：Host-Client-Server，基于 JSON-RPC 2.0 通信。写一次工具实现，就能被所有支持 MCP 的模型调用，不用为每个模型单独适配。

### Function Call vs MCP vs Skills 三者关系

#### 核心解析

| 层次 | 机制 | 解决的问题 | 类比 |
|------|------|-----------|------|
| 基础能力层 | Function Call | 模型和函数之间如何沟通 | 嘴巴（说出指令） |
| 协议层 | MCP | 工具怎么标准化接入 | USB 接口（统一插口） |
| 经验层 | Skills | 如何复用过去的成功经验 | 使用说明书（肌肉记忆） |

一个完整的 Agent 系统通常三层都有：用 Function Call 触发工具、通过 MCP 管理工具注册、用 Skills 积累执行经验。

### MCP 和 A2A 的区别

MCP 是 Agent 对工具，解决 Agent 能力扩展问题。A2A（Agent-to-Agent）是 Google 提出的 Agent 对 Agent 协议，解决跨团队、跨组织的 Agent 互操作问题。每个 Agent 有一个"能力名片"（Agent Card），通过标准 HTTP API 通信。

## 五、Skills

### Skills 是什么？为什么会出现？

#### 核心解析

**Skills 是把一类问题的解决方案固化下来，变成可反复调用的模块。** 就像写代码时封装的工具类，一次开发、到处复用。

**为什么会出现？** 传统 Prompt 有三个硬伤：
- **散落难管理**：Prompt 散落在各个同事的笔记、聊天记录里，你改了你的版本，别人还在用旧版本，团队执行标准不统一。
- **静态不进化**：规范更新了、发现了新漏洞，每个人都要手动同步自己的 Prompt。
- **无法团队协作**：资深工程师的经验只能靠老带新口口相传，传播慢、容易失真。

Skills 把重复出现的 AI 调用模式封装成可版本化、可复用、可迭代的模块，解决了这三个问题。

#### 给谁用？

**文档给人看，Skills 给 AI 用——这是本质区别。**

| 维度 | 文档 | Skills |
|------|------|--------|
| 使用对象 | 人 | AI |
| 执行确定性 | 理解偏差、记忆遗漏、应用变形，每个人执行效果都不一样 | 定义什么就做什么，不会走样 |
| 更新传播 | 需要重新学习、培训 | 更新 Skill 后所有人立刻生效 |

### Skills 和 Prompt 的本质区别

#### 核心解析

Prompt 是一次性的对话指令，说完就完了。Skills 是把一类问题的解决方案固化下来，变成可反复调用的模块。

Skills 的核心价值三个词：**沉淀、复用、进化**。

- 沉淀：把个人经验变成团队资产
- 复用：让好的实践快速传播
- 进化：让能力持续迭代升级

**进化示例：** 小组代码审查 Prompt 早期只是"关注安全性和性能"，后来发现漏掉并发问题加了并发检查，再后来规范要求 JavaDoc 又改一次。用 Prompt 时这些更新散落各处，有人用旧版本有人用新版本；用 Skill 时，更新一次所有人立刻生效。

### Skills 的工作原理——渐进式披露

#### 核心解析

不是让 AI 知道更多，而是让 AI 在恰当的时间知道恰当的事：

1. **发现阶段**：AI 启动时只加载所有 Skills 的元数据（名称、描述、触发关键词），像图书馆目录
2. **语义匹配**：用户输入问题时，基于向量相似度匹配相关 Skills
3. **执行阶段**：匹配成功后才加载完整内容

这就是为什么可以有几十上百个 Skills 却不用担心上下文爆炸。

### Skill 创建的六步流程

1. **Understanding**：搞清楚要解决什么问题、使用场景、触发条件
2. **Planning**：拆解成可复用的资源类型（scripts/、references/）
3. **Initializing**：调用脚本自动生成标准目录结构和模板
4. **Editing**：填充具体内容，description 写清触发条件，SKILL.md 不超过 500 行
5. **Packaging**：六重验证（文件存在性、Frontmatter 格式、YAML 解析、必填字段、命名规范 hyphen-case、长度限制 1024 字符）
6. **Iteration**：根据实际使用反馈持续优化

### Skill 的自由度分级

- **高自由度**：纯文本指令，适用于有多种有效方法的场景
- **中自由度**：伪代码或带参数的脚本，有推荐模式但允许变化
- **低自由度**：特定脚本，少量参数，适用于需要严格一致性的操作

### Skills 和 MCP 的关系：互补不是竞争

| | MCP | Skills |
|---|-----|--------|
| 解决什么 | AI 与外部系统的**连接**问题（调 GitHub API、查数据库、操作文件、打开浏览器） | AI 的**能力封装**问题（代码审查能力、文档生成能力、技术方案评审能力） |
| 类比 | 给 AI 装上机械臂，让它能接触外部世界 | 给 AI 使用说明书，让它知道怎么做事 |

两者结合才能真正成为生产力工具：MCP 提供"手"，Skills 提供"怎么做"。

## 六、Memory vs RAG

### Memory 和 RAG 的区别

#### 核心解析

以 OpenClaw 为例：

| 维度 | Memory | RAG |
|------|--------|-----|
| 目的 | 记住用户偏好、历史决策、个性化信息 | 检索外部知识库回答专业问题 |
| 数据来源 | 对话历史自动沉淀 | 外部文档、知识库主动导入 |
| 存储 | SQLite（轻量本地） | ElasticSearch / 向量数据库（重量级） |
| 检索方式 | FTS5+BM25 关键词 + sqlite-vec 向量混合检索 | 向量检索 + BM25 混合检索 + Rerank |
| 注入方式 | 按需检索，不全量注入上下文 | 召回 Top-K 片段拼入 Prompt |

### Memory 的两层架构

**短期记忆**：存储在 JSONL 会话日志文件中，自动记录每次对话内容，是最原始的未经处理的记忆。

**长期记忆**：存储在 MEMORY.md 和 memory/*.md 文件中。从短期记忆中提炼出需要重点记住的内容（用户偏好、身份信息、回答偏好等）。

转换机制：
1. **session-memory Hook**：用户执行 `/new` 重置会话时，自动将上一个会话的关键内容转换为 Markdown
2. **Memory Flush**：Session 接近 context 上限时触发 Compaction，提取重要信息写入 Memory

### Memory 的索引建立四步

**关键架构观点：Markdown 是记忆本体（source of truth），SQLite 只是加速层。** 很多人误以为 SQLite 就是 Memory 本身，其实不是——Markdown 文件才是记忆实体，SQLite 负责把它们变成"可检索"的东西。

1. **发现**：监控 MEMORY.md 和 memory/*.md 的变化，标记 dirty
2. **切块**：把 Markdown 切成多个 chunk，让一个块只表达一小段完整的意思
3. **索引**：每个 chunk 同时走向量索引（语义召回）和 FTS 全文索引（关键词命中）
4. **落库**：chunk、元信息、全文索引、向量索引都放到本地 SQLite

### 混合检索为什么比纯向量靠谱？

搜 `nomic-embed-text` 这种精确关键词→FTS5+BM25 命中。搜「上次说过的那个写作偏好」这种模糊语义→向量检索召回。两边结果融合返回，互补短板。

### Memory 怎么被 Agent 使用？按需检索而非全量注入

很多人以为 Memory 系统的终点是"查到了"，其实不是。真正关键的是**怎么用**。OpenClaw 给 Agent 暴露了两个工具：

- **`memory_search(query)`**：返回最相关的 snippet + 文件路径 + 行号范围（不是整篇内容）。先粗召回找到"值得展开读"的位置，同时控制 token。
- **`memory_get(path, line_range)`**：根据 `memory_search` 返回的路径和行段精读具体内容。

**关键设计：Memory 文件不是每回合全量注入上下文窗口。** `memory/*.md` 这种 daily 文件默认不会塞进 prompt，而是通过 search + get 两步按需读取。这就是"能越记越多但又不撑爆上下文"的根本原因。

**工程启示：** 不需要部署重量级向量数据库，一个 SQLite 文件 + FTS5 + sqlite-vec 扩展就够了。本地 Agent 场景下，这种"最简方案解决最复杂问题"的思路比堆技术栈更有价值。

### 上下文溢出的四种处理方案

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| 滑动窗口 | 只保留最近 N 轮对话 | 简单粗暴，会丢早期重要信息 |
| 摘要压缩 | LLM 把旧对话压缩成摘要 | 保留大意，细节丢失 |
| 向量检索增强 | 历史对话存向量库，每轮检索最相关片段注入 | 精准但增加延迟 |
| 关键事实提取 | 维护「重要事实列表」，每轮注入 | 最精准，维护成本高 |

## 七、RAG 核心要点

### 如何提升 RAG 问答准确度？

#### 核心解析

分两个环节诊断：

**召回侧**：
- 混合检索：向量检索（语义）+ BM25（关键词）双路并行，用 RRF 倒数排名融合，比单路召回率高 15%~30%
- 文档切片：建议 512~1024 token 一个片段，前后各留 50~100 token 重叠
- 重排序：先粗召回 20 条，再用 Reranker 精排���前 5 条，性价比最高的提升手段

**生成侧**：
- 提示词约束：明确告诉模型「只根据以下文档回答，没有相关信息直接说不知道，不要编造」
- 幻觉率能显著下降

> 参考：[[RAG管道设计]] 中有 Baize 项目的分片、解析、向量化、检索具体实现

### 知识库快速更新怎么应对？

#### 核心解析

三层方案：
1. **增量更新**：监听数据源变更事件，只对新增/修改内容重新向量化
2. **时效性过滤**：检索时加时间维度过滤，给文档打创建时间和有效期字段
3. **实时工具兜底**：更新极快的内容不存向量库，直接挂实时查询工具

### Reflection / Self-Correction 模式

#### 核心解析

Agent 生成初始结果→启动评估步骤→LLM 对结果打分找问题→输出改进建议→反馈给 Agent 重新生成→循环直到满足质量标准或达到最大迭代次数。

两个变体：
- **自我反思**：同一个 LLM 评估自己的输出，成本低但盲点相同
- **对等评审**：专门的 Critic 模型评估，质量更高但成本也更高

生产里一般用自我反思做快速迭代，高精度场景（合同审核、医疗建议）才上专门的 Critic 模型。
