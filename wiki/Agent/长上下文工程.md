# 长上下文工程（Long Context Engineering）

> ==2025-2026 Agent 工程的核心方向之一==——模型 window 从 32k → 200k → 1M 之后,问题不再是"塞不下",而是"==怎么塞才省钱、不丢失中间信息、跨会话复用=="。
>
> 涉及两个独立但深度耦合的话题:==Prompt Caching==(API 层的成本杀手锏)+ ==Context Mode 切换策略==(Harness 层根据模型能力动态调整行为)。
>
> 与 [[Harness Engineering#上下文资源化]] / [[多轮对话#二、上下文窗口管理]] / [[模型路由策略]] 形成完整 Context Engineering 视图。

## 一、Prompt Caching:Coding Agent 的成本第一杠杆

### 1.1 是什么 + 为什么是头号优化点

==Prompt Caching== = LLM provider 把请求的==前缀部分==缓存到服务端,下次请求==同样前缀==时不用重新计算 KV,==直接复用==。

==为什么 Coding Agent 必须开 Prompt Caching==(否则字面意义烧钱):

| 部分 | 典型 token 量 | 是否每轮重复 |
|------|-------------|------------|
| ==System prompt==(身份 + 行为指南) | 2k - 10k | ==每轮一模一样== |
| ==工具描述==(tools schema) | 5k - 30k | ==每轮一模一样== |
| ==项目上下文==(CLAUDE.md / 规则文件) | 1k - 5k | ==每轮一模一样== |
| ==历史对话== | 不定 | ==前缀部分一模一样== |
| 当前用户消息 | 50 - 500 | 每轮变化 |

==没缓存==:每轮都要把 8k-50k 的固定前缀==重新算 KV==——延迟 + 成本 N 轮叠加。

==开缓存==:固定前缀==算一次缓存住==,后续 N 轮只算增量(几百 token)——==省 80%-90% 成本 + 延迟降到 1/3==。

==Anthropic 公开数据==:Claude Code 用户的 cache hit rate 中位数 ==85%+==,这就是为什么 Claude Code 长会话也不贵。

### 1.2 三家主流 API 的实现差异

| 厂商 | 触发方式 | TTL | 命中省钱 | 命中省时 |
|------|---------|-----|---------|---------|
| ==Anthropic== | ==手动 `cache_control` 标记缓存点== | ==5 分钟==(可买扩展到 1 小时) | ==90%==(命中部分按 10% 计费) | ==~80%==(无 prefill 计算) |
| ==DeepSeek== | ==自动==(任何重复前缀) | ~小时级(磁盘) | ==90%==(0.014 元 / 0.14 元 per M tokens) | ~70% |
| ==OpenAI== | ==自动==(>1024 token 前缀,完全相同才命中) | ~5-10 分钟 | ==50%==(命中部分半价) | ~50% |
| ==Gemini== | 显式 `CachedContent` API | 1 小时(默认) | 25%(命中部分 0.25x) | ~50% |
| ==Qwen / GLM / Kimi== | 自动(类 DeepSeek 模式) | 小时级 | 80%-90% | 70%+ |

==关键差异==:
- ==Anthropic 是手动==——你必须主动标记"==这里建缓存点==",更精准但要设计
- ==DeepSeek / 国产== 自动——开箱省钱,但缓存命中粒度由 server 决定
- ==OpenAI 自动==但命中条件最严(完全相同前缀 + >1024 token),代码场景 hit rate 低

### 1.3 Anthropic `cache_control` 用法详解

==四个关键设计==:

```python
# Anthropic API 示例
messages = [
    {"role": "user", "content": [
        # === 缓存点 1:System prompt + 工具描述 ===
        {
            "type": "text",
            "text": SYSTEM_PROMPT + TOOLS_SCHEMA_TEXT,
            "cache_control": {"type": "ephemeral"}  # ★ 标记此处建缓存
        },
        # === 缓存点 2:项目上下文(CLAUDE.md) ===
        {
            "type": "text",
            "text": project_context,
            "cache_control": {"type": "ephemeral"}
        },
        # === 缓存点 3:历史对话(前 N 轮) ===
        {
            "type": "text",
            "text": history_summary,
            "cache_control": {"type": "ephemeral"}
        },
        # === 不加 cache_control:当前用户消息(每轮变化) ===
        {"type": "text", "text": current_user_message}
    ]}
]
```

==四条规则==:

| 规则 | 说明 |
|------|------|
| ==最多 4 个缓存点== | API 限制——超过会报错;一般够用(system / tools / project / history) |
| ==前缀必须完全一致== | 缓存点之前的内容==哪怕变 1 个 token==,缓存就失效 |
| ==顺序敏感== | tools 在 system 之前 vs 之后 = 完全不同的缓存 |
| ==最小长度 1024 token== | 短于这个不缓存(避免缓存抖动) |

==踩坑==:
- ==时间戳进 prompt== → 每秒变,缓存==每次都失效==。==解决==:时间戳放最末段(用户消息部分),前缀保持稳定
- ==tools schema 顺序变化== → 用 `dict.items()` 不固定顺序时,工具列表顺序每次不同,缓存失效。==解决==:`sorted(tools, key=lambda t: t["name"])`
- ==消息合并方式不一致== → 一会儿合并多条 tool_result 一会儿不合,前缀变化。==解决==:统一合并策略

### 1.4 缓存命中率经济学

==实际成本计算==(Claude Sonnet 4.5):
- 普通 input:$3 / M token
- ==Cache write==(首次写入):$3.75 / M token(==贵 25%==)
- ==Cache read==(后续命中):$0.30 / M token(==便宜 90%==)

==举例==:Coding Agent 一次会话 20 轮,前缀 30k token

```
不开缓存:
   20 轮 × 30k × $3 / M = $1.8

开缓存:
   首轮:    30k × $3.75 / M = $0.1125  (cache write)
   后 19 轮: 19 × 30k × $0.30 / M = $0.171  (cache read)
   总计                          = $0.28

省 84% 成本(¥1.52 → ¥0.28)
```

==Coding Agent 实战==:Claude Code / Cursor / Cline / PaiCli ==默认开 cache==——这是==生产级 Coding Agent 的标配==,关掉就是字面烧钱。

### 1.5 缓存点设计的最佳实践

==分层缓存策略==(Coding Agent 通用模式):

```
┌──────────────────────────────────────────────┐
│ 缓存点 1: System Prompt + 工具描述           │  ←━ 最稳定,生命周期最长
│         (会话级别甚至跨会话不变)             │
├──────────────────────────────────────────────┤
│ 缓存点 2: 项目上下文 (CLAUDE.md / 规则)      │  ←━ 项目级稳定
│         (单个项目内不变)                     │
├──────────────────────────────────────────────┤
│ 缓存点 3: 历史摘要 / 早期对话                │  ←━ 会话内增量
│         (新增轮次时才扩展)                   │
├──────────────────────────────────────────────┤
│ (无缓存) 当前用户消息 + tool_result           │  ←━ 每轮变化
└──────────────────────────────────────────────┘
```

==面试时讲清==:"==Prompt Caching 的命中率取决于前缀稳定性,设计原则是把变化的内容放最末段==,稳定的(system/tools/project)放前面并加缓存点——这就是为什么 Coding Agent 把 system prompt 设计得很长也不贵。"

## 二、模型能力声明与动态预算

### 2.1 LLM Client 能力 self-aware

==生产级 Agent 的隐藏标配==——LlmClient 抽象层==声明每个模型的能力==,Harness 根据能力动态调行为:

```java
public interface LlmClient {
    int maxContextWindow();                  // 200_000 / 1_000_000 / ...
    boolean supportsPromptCaching();         // Anthropic/DeepSeek/Qwen/Kimi: true
    PromptCacheMode promptCacheMode();       // MANUAL / AUTOMATIC
    int recommendedOutputTokens();           // 一般 4096 / 8192
    boolean supportsParallelToolCalls();     // 是否支持并行工具调用
}
```

==为什么必须 self-aware==:
- ==同一个 Harness 适配多模型==——切到 GLM 还是 DeepSeek,Harness 不需要改代码
- ==不同模型 window 差 5 倍==(200k vs 1M),策略必须不同
- ==Prompt cache 模式不同==——Anthropic 要主动标缓存点,DeepSeek 自动,代码要分支处理

### 2.2 2026 主流模型 context window 对照

| 模型 | 标称 window | 实际可靠范围 | Prompt Cache |
|------|-----------|------------|-------------|
| ==Claude Sonnet 4.5== | 200K(可扩 1M) | ~150K | ==手动 `cache_control`== |
| ==Claude Opus 4.7== | 200K / ==1M==(扩展模式) | ~150K / 700K | 手动 |
| ==GPT-5== | 200K | ~80K | 自动(>1024) |
| ==DeepSeek V4== | ==1M== | ~500K | ==自动 prefix== |
| ==Gemini 2.5 Pro== | 1M+ | 长文最强 | `CachedContent` API |
| ==GLM-5.1== | ==200K== | ~150K | 自动 |
| ==StepFun Step-3== | ==256K== | ~200K | 自动 |
| ==Kimi K2.6== | ==256K== | ~200K | 自动 |
| ==Qwen3-Max== | 256K | ~200K | 自动 |

==实际可靠范围 < 标称 window==——参考 [[LLM 基础与训练#上下文窗口管理]] 解释了为什么(Lost in the Middle / 注意力稀释 / RoPE 外推)。

### 2.3 80% Rule:动态预算计算

==生产规则==:`AgentBudget = 0.8 × maxContextWindow`

==为什么留 20%==:
- ==Output tokens 不算 input,但占 window==——常用 4-8k
- ==tool_result 突增==——一次 grep 大文件可能返回几万 token
- ==工具描述本身占 token==——内置 9 个工具 ~1.5k,挂 3 个 MCP server 可达 12k;工具集越大预算越紧,详见 [[Coding Agent 工具集#66-工具集-token-管理工具描述是隐形的-prompt-成本]]
- ==Safety margin==——避免 boundary 报错(模型 419 报错很难恢复)

```python
def compute_budget(client: LlmClient) -> int:
    window = client.max_context_window()
    output_reserve = client.recommended_output_tokens()  # 一般 4-8k
    safety_margin = int(window * 0.05)                   # 5%

    # 80% rule(默认)
    budget = int(window * 0.8) - output_reserve - safety_margin

    # 但允许系统属性覆盖(企业用户可能要更保守)
    override = os.getenv("AGENT_CONTEXT_BUDGET")
    if override:
        budget = int(override)

    return budget
```

==典型预算结果==:
- Claude Sonnet(200k):==~152k input budget==
- DeepSeek V4(1M):==~792k input budget==
- GLM-5.1(200k):==~152k input budget==

==关键点==:==预算不能写死==——同一份 Harness 代码跑在不同模型上,budget 自动调整。

## 三、Context Mode 切换策略

### 3.1 三种模式:short / balanced / long

==生产级 Coding Agent 都有 context mode 切换==(Cursor 的 "Max" / Cline 的 context optimization / PaiCli 的 short/balanced/long),==因为没有"一刀切最优"==:

| 模式 | 适用 | 历史压缩 | RAG topK | Resources 索引 | 触发条件 |
|------|------|---------|---------|---------------|---------|
| ==short== | 快问快答 / 简单任务 / 成本敏感 | ==积极压缩==(摘要 + 滑窗) | ==5== | ==不注入== | window < 50k 或用户选 |
| ==balanced== | 默认模式 / 中等任务 | ==中等压缩==(关键信息保留) | ==10== | ==不注入== | 默认 |
| ==long== | 大型代码库 / 跨文件分析 / 1M window | ==不压缩==(原文保留) | ==20== | ==URI + 描述索引注入== | window ≥ 200k 或用户选 |

==为什么不统一用 long==:
- 历史不压缩 → 历史轮次多了之后==前置内容覆盖大量 token==,反而把当前任务挤出去
- topK 拉到 20 → ==每次检索 token 翻倍==,且大量低相关结果稀释注意力
- Resources 索引注入 → ==system prompt 多 5-10k token==,缓存命中率下降

==long mode 是"换一种代价"==(更多 token / 更慢 / 更贵)换"==更全的可见信息=="——只有真正需要时才开。

### 3.2 模式驱动的下游策略矩阵

==同一个 Agent 行为根据 mode 切换==——典型联动:

```python
class ContextMode(Enum):
    SHORT = "short"
    BALANCED = "balanced"
    LONG = "long"

def apply_mode(mode: ContextMode) -> AgentConfig:
    if mode == ContextMode.SHORT:
        return AgentConfig(
            history_strategy="summarize_aggressive",  # 摘要压缩
            rag_top_k=5,
            inject_mcp_resources_index=False,
            max_tool_result_size=5_000,               # 工具返回截断
            sliding_window_turns=10,                  # 只保留最近 10 轮
        )
    elif mode == ContextMode.BALANCED:
        return AgentConfig(
            history_strategy="summarize_balanced",
            rag_top_k=10,
            inject_mcp_resources_index=False,
            max_tool_result_size=20_000,
            sliding_window_turns=30,
        )
    elif mode == ContextMode.LONG:
        return AgentConfig(
            history_strategy="keep_raw",              # ★ 不压缩
            rag_top_k=20,                             # ★ 拉满
            inject_mcp_resources_index=True,          # ★ 注入索引
            max_tool_result_size=100_000,
            sliding_window_turns=None,                # ★ 不滑窗
        )
```

==关键设计==:==mode 是 1 个 enum,联动 5+ 个策略==——别让用户单独调每个开关,==复杂度爆炸==。

### 3.3 模式选择:自动 vs 手动

==生产做法==:==默认 balanced + 用户可手动切== + ==自动建议==:

```python
def auto_suggest_mode(client: LlmClient, task: str) -> ContextMode:
    window = client.max_context_window()

    # 模型 window 太小 → 强制 short
    if window < 50_000:
        return ContextMode.SHORT

    # 模型 window 巨大且任务关键词命中 → 建议 long
    if window >= 500_000 and any(kw in task for kw in ["全项目", "整个代码库", "跨文件"]):
        return ContextMode.LONG

    return ContextMode.BALANCED
```

==用户体验==:`/context` 命令显示当前 mode + 提示"==当前模型 window 1M,你可以试试 long mode 处理大型分析任务=="。

## 四、Long Context 下的 RAG topK 自适应

### 4.1 问题:topK 不能写死

==传统 RAG 默认 topK = 5/10==——这是 32k window 时代的合理值。==long context 下==:

| Window | 合理 topK | 理由 |
|--------|---------|------|
| 32K | 3-5 | 检索结果占太多 prompt 空间 |
| 200K | 10 | 平衡覆盖率和噪声 |
| 1M | ==20-50== | 召回率优先,模型能消化 |

==没自适应的代价==:
- ==topK = 5 跑在 1M 模型上== → 召回率太低,关键文档被漏
- ==topK = 50 跑在 32k 模型上== → 检索结果直接挤爆 prompt

### 4.2 实现:跟着 context mode 走

```python
def search_code(query: str, top_k: Optional[int] = None) -> list[Doc]:
    # 显式指定优先
    if top_k is not None:
        return semantic_search(query, k=top_k)

    # 默认:根据当前 context mode 自适应
    mode = current_context_mode()
    auto_k = {
        ContextMode.SHORT: 5,
        ContextMode.BALANCED: 10,
        ContextMode.LONG: 20,
    }[mode]
    return semantic_search(query, k=auto_k)
```

==踩过的坑==:用户切到 long mode 期望"看到更多结果",但工具签名 `search_code(query, top_k=10)` 把 top_k 写死了——==自适应逻辑被覆盖==。==修复==:让 top_k 默认 None,只在显式传值时覆盖。

### 4.3 实时 grep 优先 vs RAG

==生产经验==(Claude Code 是反例):==代码定位优先实时 grep + read==,==RAG 是补充==。

==为什么==:
- grep ==确定性高==——给定文件名/函数名,grep 一定找到
- RAG ==基于语义==——可能召回相关但不准确的文件
- ==Coding Agent 的工具集设计==:`grep` / `read_file` 第一优先级,`search_code`(语义检索) 第二优先级

==long mode 下==:topK 拉高(20)是==增强 search_code 的兜底能力==,不是把它当主要手段——主路径仍然是 LLM 用 grep + read 自己定位。详见 [[Coding Agent 工具集#5.1 Tool / Command / Skill 三者的关系]]。

## 五、MCP Resources 在 Long Context 下的索引注入

### 5.1 问题:LLM 不知道有哪些 Resource 可读

==MCP Resources== 是只读数据(参考 [[MCP 协议#43-resources-怎么暴露给-llm自动注册为虚拟工具]])——但默认实现下:
- LLM 只看到 `list_resources` / `read_resource` 两个虚拟工具
- ==LLM 不知道实际有哪些 Resource 可读==——必须先调 `list_resources` 才知道
- 多走一轮 → 延迟 + token 浪费

### 5.2 long mode 下的索引注入策略

==long mode 下,window 富余==——可以==预先把 Resource 索引塞到 system prompt==:

```python
def build_system_prompt(mode: ContextMode, mcp_resources: dict) -> str:
    base = LOAD_BASE_SYSTEM_PROMPT()

    if mode == ContextMode.LONG:
        # ★ 注入 URI + description,不注入正文
        index_section = "## 可用 MCP Resources\n\n"
        for server, resources in mcp_resources.items():
            for r in resources:
                index_section += f"- `{r.uri}` — {r.description}\n"
        base += "\n\n" + index_section

    return base
```

==关键==:==只注入 URI + 描述,不注入正文==——

| 注入方式 | Token 量 | 何时用 |
|---------|---------|------|
| ==只 URI + 描述==(索引) | ~10 token / resource | ==long mode 默认== |
| ==URI + 描述 + 摘要== | ~50 token / resource | 极长 window 且 resources 少 |
| ==URI + 描述 + 完整正文== | ~1000+ token / resource | ==几乎不用==——直接塞死 prompt |

==典型规模==:50 个 resources × 10 token = 500 token——==一个长 system prompt 多 500 token,换来 LLM 直接知道有哪些资源可读,省一轮 list_resources 调用==。

### 5.3 short / balanced mode 下不注入

==为什么 short/balanced 不注入==:
- ==window 紧张==——多 500 token 影响其他重要内容
- ==Prompt cache 命中率下降==——resources 列表会变,前缀稳定性破坏
- ==大部分任务用不到 Resources==——LLM 需要时再调 list_resources 即可

==面试时讲清==:"==long mode 下注入 Resource 索引==是 token-friendly 的优化,核心是==只注入元信息(URI + 描述)不注入正文==,让 LLM 知道有什么可读但不预先加载——避免 50 个 resource 全部塞 prompt 的灾难。"

## 六、Token 与成本可观测性

### 6.1 必须暴露的 5 个指标

==生产 Coding Agent 必须实时显示==:

| 指标 | 含义 | 为什么重要 |
|------|------|---------|
| ==Total input tokens== | 本轮 input 总数 | 看 prompt 是否快爆 budget |
| ==Cached input tokens== | 命中缓存的 input | ==判断 cache 有没有正常工作== |
| ==Cache hit rate== | cached / total input | ==<50% 就要查为什么没命中== |
| ==Output tokens== | 本轮 output | 看 LLM 输出长度 |
| ==估算成本== | 实时累加(input + cached + output) | ==让用户对成本有感知== |

### 6.2 显示位置:状态栏 vs 正文

==关键设计==:==token / cost 信息进状态栏,不进正文输出区==——

==错误做法==:每轮 LLM 输出后打印
```
[Token: input=23456, cached=20123, output=512, cost=$0.0089]
```

==问题==:污染正文,用户复制代码时把 token 信息也复制了。

==正确做法==:==底部状态栏==(类似 vim 的 status line):

```
工作区: ~/code/myapp     模型: claude-sonnet-4-5    Mode: balanced
─────────────────────────────────────────────────────────────────
[正文输出区]
> 我帮你修改了 src/auth.ts,把 JWT 验证改为...
─────────────────────────────────────────────────────────────────
Tokens: 23.4k (cached 20.1k / 86%)  Cost: $0.012  Last: 2.3s
```

==Coding Agent 的常见做法==:
- ==Claude Code==:CLI 模式底部 dim 文字显示
- ==Cursor==:状态栏右下角
- ==PaiCli==:inline 模式底部状态栏

### 6.3 `/context` 命令:暴露 Harness 内部状态

==生产 Coding Agent 都有 `/context` 命令==,显示完整 Harness 状态:

```
$ /context

模型:         claude-sonnet-4-5
Window:       200,000 tokens
Budget:       152,000 tokens (80% rule)
当前占用:     18,453 tokens (12%)

Mode:         balanced
Prompt Cache: enabled (manual cache_control)
```

## 七、关键认知与面试要点

### 认知 1:Prompt Caching 是省钱第一杠杆

==Coding Agent 长会话不贵的根本原因==——cache hit rate 85%+,后续轮次成本只有首轮的 10-20%。==面试时讲==:"我们 Coding Agent 默认开 prompt caching,长 system prompt + 工具描述 + 项目上下文都加 `cache_control`,实测命中率 80%+,成本省 80%+——==关掉 cache 是字面意义烧钱=="。

### 认知 2:模型能力 self-aware 是多模型 Agent 的基础

==Harness 不能写死参数==——同一份代码跑 Claude / DeepSeek / GLM,window 差 5 倍,cache 模式不同。==LlmClient 抽象层声明能力,Harness 动态调行为==。

### 认知 3:Context Mode 是"换一种代价"

==long mode 不是"更高级"==——是用 5x token 换更全可见信息,80% 任务 balanced 够用。==Context Mode 切换是 Cursor / Cline / PaiCli 都有的标配==,没切换就是"一刀切"。

### 认知 4:Long Context 下 RAG topK 必须自适应

==topK = 5 跑 1M 模型 = 召回率太低==,==topK = 50 跑 32k 模型 = 挤爆 prompt==——topK 必须跟着 context mode / window 走。

### 认知 5:Token 可观测性必须实时暴露

==Coding Agent 必显的 5 个指标==:total input / cached / hit rate / output / 估算成本——==状态栏不污染正文==。`/context` 命令暴露 Harness 内部状态,问题诊断必备。

### 面试时的高质量回答

==问==:"你们 Agent 怎么处理长上下文?"

==❌ 平庸答案==:"用 RAG + 摘要压缩。"

==✅ 高质量答案==:"==Long Context Engineering 是个系统工程==——

1. ==Prompt Caching==:Anthropic 加 `cache_control`,DeepSeek/Qwen 自动 prefix,Coding Agent 长会话省 80%+ 成本的核心
2. ==模型能力 self-aware==:LlmClient 声明 maxContextWindow / supportsPromptCaching / promptCacheMode,Harness 自适应不同模型
3. ==80% Rule 动态预算==:budget = 0.8 × window,留 20% 给 output + safety
4. ==Context Mode 三档==:short(摘要 + topK 5) / balanced(默认) / long(原文 + topK 20 + Resources 索引)
5. ==MCP Resources 长 mode 下注入 URI + 描述==(不注入正文,500 token 换省一轮 list_resources)
6. ==Token / cost 可观测==:状态栏显示 cached input / hit rate / 实时成本,`/context` 命令暴露 Harness 状态

==关键认知==:Prompt Caching 是省钱第一杠杆,Context Mode 是"换一种代价"——不是越长越好,而是按需切。"
