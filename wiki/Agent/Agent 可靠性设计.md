# Agent 可靠性设计

> ==Agent 是非确定性系统==——LLM 抽风、工具失败、上下文污染都可能导致任务出错。==可靠性设计==是把非确定性的模型放在确定性的兜底机制里。
>
> 与 [[Agent 安全模型]] 的区别：可靠性=系统正常运行不出错；安全=对抗恶意输入和误操作。两者互补。

---

## 一、Agent 死循环检测与处理

### 死循环的两种形态

1. **重复型**：反复调用同一个工具、传同样的参数（如一直搜索同一个关键词）
2. **震荡型**：在两个状态间来回切换（如"需要更多信息→调用搜索→信息不够→需要更多信息"）

### 检测机制（三层）

| 层次 | 检测方法 | 实现方式 |
|------|---------|---------|
| ==硬限制== | 全局步骤计数器 | 超过 max_steps（通常 20-50）强制终止 |
| ==重复检测== | Action 指纹哈希 | 对 (tool_name, params_hash) 做滑动窗口去重，连续 3 次相同判定为循环 |
| ==语义检测== | 状态向量相似度 | 对最近 N 步的 Observation 做 embedding，余弦相似度 > 0.95 判定为震荡 |

### 处理策略（不是简单终止）

1. ==升级策略==：检测到循环后，不是直接终止，而是在 Prompt 中注入"你已经尝试了 X 次相同的操作，请换一种方法"，给 Agent 一次自我纠正的机会
2. ==工具屏蔽==：临时禁用导致循环的工具，迫使 Agent 选择其他路径
3. ==上下文压缩==：循环往往是因为上下文太长导致 LLM "忘记"了之前的尝试，压缩历史后重试
4. ==优雅降级==：如果自我纠正失败，返回当前最优的中间结果 + 失败原因说明，而不是空结果

### 防无限循环的三道防线

| 防线 | 机制 | 触发条件 |
|------|------|---------|
| ==步骤计数器== | 全局最大步骤数（通常 20-50 步），超出强制终止 | 任何情况下都生效 |
| ==相似度检测== | 检测连续几步的 Action 是否高度相似，相似则判定为循环 | 连续 3 步 Action 相似度 > 0.9 |
| ==超时熔断== | 整个任务设置时间上限（如 5 分钟），超时强制终止 | 任何情况下都生效 |

==三道防线独立工作==，任意一道触发都会终止任务。终止时必须输出当前进度和失败原因，==不能静默退出==。

## 二、Fallback 策略：分层降级体系

==核心原则==：**宁可给一个不完美的结果，也不能给一个空结果或错误结果**。

Fallback 不是简单的 try-catch，而是一套==分层降级体系==。

### 四层 Fallback 架构

```
Layer 1: 工具级 Fallback（单个工具失败）
  → 重试（指数退避，最多 3 次）
  → 备用工具（如主搜索引擎挂了，切备用引擎）
  → 参数调整（缩小搜索范围、降低精度要求）

Layer 2: 步骤级 Fallback（某个推理步骤失败）
  → 跳过该步骤，用默认值或上一步结果继续
  → 换一种推理路径（如检索失败，改用 LLM 内部知识）

Layer 3: 任务级 Fallback（整个任务无法完成）
  → 返回部分结果 + 明确说明哪些部分未完成
  → 降级到更简单的任务版本（如无法生成完整报告，给出摘要）

Layer 4: 系统级 Fallback（Agent 本身不可用）
  → 直接调用 LLM 不走 Agent 流程
  → 返回预设的兜底回复 + 人工介入通知
```

### 关键设计原则

- 每层 Fallback 都要有**超时时间**，不能无限等待
- Fallback 触发时必须**记录日志**，包含失败原因、降级路径、最终结果质量评估
- 对用户要**透明**：告知"由于 XX 原因，本次结果可能不完整"，而不是假装一切正常

## 三、工具调用失败的四层决策策略

==上面 Fallback 策略讲架构层降级==，这里讲==LLM 决策层==——工具调用失败后 Agent 自己（在 ReAct 循环里）怎么决定下一步。两者互补：架构定义"系统能怎么降级"，决策定义"LLM 看到错误怎么选路"。

### 四层决策

| 层级 | 策略 | 触发条件 | 示例 |
|------|------|---------|------|
| ==L1 重试== | 相同工具、相同或修正参数再试 | 临时性错误（超时、网络抖动） | search 超时 → 等 2 秒重试 |
| ==L2 参数修正== | 分析错误信息，调整参数后重新调用 | 参数错误（格式不对、值越界） | 日期格式传错 → LLM 分析报错 → 修正格式重调 |
| ==L3 换工具== | 放弃当前工具，选替代工具完成同一目标 | 工具不可用或持续失败 | Google Search 挂了 → 换 Bing Search |
| ==L4 换策略== | 重新规划，绕过这步或改变整体方案 | 没有替代工具，或目标本身需调整 | 数据库查询权限不足 → 改从缓存文件读取 |

### 关键机制：错误信息回注上下文

工具调用失败后，==Harness 不是简单返回"失败"==，而是==把完整错误信息（错误码、错误描述、堆栈摘要）作为 `tool` 消息内容回注==。LLM 在下一轮分析这些信息，决定走哪一层策略。

```python
try:
    result = TOOLS[tool_name](**tool_args)
except Exception as e:
    result = f"ERROR: {type(e).__name__}: {str(e)}"
    # 把错误当成普通 observation 回注——LLM 自己决定怎么办
messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
```

==这就是 ReAct 模式天然支持错误处理的原因==——`tool` 消息不区分成功/失败，==都是 LLM 的决策输入==。详见 [[Function Calling#三、四种消息角色]]。

### 工程实现要点

- ==最大重试次数==：每个工具设上限（通常 2-3 次），防止陷入无限重试
- ==错误分类==：可重试错误（超时、限流）vs 不可重试错误（权限不足、资源不存在）—— 不可重试的直接跳到 L3/L4
- ==全局兜底==：整个任务的最大工具调用次数（比如 20 次），超过强制终止汇报进度
- ==优雅降级==：核心步骤失败无法绕过 → 输出"我完成了 X 和 Y，但 Z 因为某原因失败，需要你介入"，==别默默返回不完整结果==

## 四、错误累积防护

错误累积是 Agent 最致命的问题：第 1 步的小偏差，经过 5 步放大后可能完全偏离目标。这在长链推理中尤为严重。

### 错误累积的三种形态

1. ==事实错误传播==：第 2 步基于第 1 步的错误结论继续推理
2. ==上下文污染==：错误的中间结果留在上下文中，影响后续所有决策
3. ==目标漂移==：每一步都微小偏离原始目标，最终结果和需求完全不符

### 防护机制

**1. 中间结果验证（最有效）**

在关键步骤后插入验证环节：
- 用另一个 LLM 调用检查上一步结果是否合理（Generator-Evaluator 模式）
- 对可验证的结果做确定性检查（如代码能否编译、SQL 能否执行、数学计算是否正确）
- 设置置信度阈值，低于阈值的结果标记为"不确定"，后续步骤需要额外验证

==Coding Agent 的具体落地==：详见 [[LSP 与代码诊断#五、coding-agent-的诊断回注模式]]——`write_file` 后异步触发 LSP 诊断，回注下一轮 LLM。

**2. 计划锚定**

在执行前生成完整计划，每一步执行后回头检查是否偏离计划：
- 每 N 步做一次"计划对齐检查"：当前进度是否符合原始计划？
- 如果偏离，决定是修正回来还是更新计划（而不是继续在错误路径上走）

**3. 上下文隔离**

不把所有中间结果都堆在一个上下文里：
- 每一步只传入必要的上下文（上一步的最终结论），而不是完整的推理过程
- 用结构化的状态对象传递信息，而不是自然语言（自然语言容易引入歧义）

==Plan-and-Execute 的两层 context 分离==：详见 [[Plan-and-Execute 实现#4.4 Replan 上下文裁剪策略]]——Planner 看 metadata，Executor 看完整 artifact。

**4. 回滚机制（Agent 状态层）**

发现错误后能回到之前的正确状态：
- 每一步保存状态快照（LangGraph 的 checkpoint 机制）
- 检测到错误时回滚到最近的正确状态，从那里重新开始

**5. Side-History Git Snapshot（文件系统层，Coding Agent 特有）**

==LangGraph checkpoint 存的是 Agent 状态（变量 / 消息历史）==，但 Coding Agent 还需要回滚==文件系统状态==——LLM 改了 10 个文件，用户想撤回到改之前。==Side-History Git Snapshot== 是专门解决这个问题的模式。

==核心设计==：==独立 git 仓库，不写用户项目的 `.git`==

```
用户项目:  ~/code/myapp/.git          ← 用户自己的 git 历史,不动
Side-history: ~/.paicli/snapshots/<project_hash>/.git  ← Agent 专用快照仓库
```

==为什么不用用户的 `.git`==：
- 用户的 git 历史是干净的业务提交，Agent 的每轮快照会污染 `git log`
- 用户可能在 git 操作中途，Agent 快照会破坏 index 状态
- 独立仓库可以随时清理（`/snapshot clean`），不影响用户项目

==快照粒度==：

| 快照类型 | 时机 | 用途 |
|---------|------|------|
| ==pre-turn== | 每个 ReAct / Plan / Team turn 开始前 | ==用户可恢复到这里==（`/restore N`） |
| ==post-turn== | turn 结束后异步创建 | 记录 Agent 做了什么 |
| ==pre-restore== | 执行 `/restore` 前 | 防止恢复操作本身不可逆 |

==实现要点==（JGit / libgit2 / 纯 shell git）：

```python
def create_snapshot(project_root: str, label: str) -> str:
    snapshot_repo = get_or_init_snapshot_repo(project_root)

    # 1. 把项目文件 worktree 指向用户项目目录
    # 2. git add -A（含 .gitignore 排除的文件可选）
    # 3. git commit -m f"{label}:{timestamp}"
    # 4. 返回 commit hash

def restore_snapshot(project_root: str, n: int):
    # 1. 先创建 pre-restore 快照（安全网）
    create_snapshot(project_root, "pre-restore")
    # 2. 找到第 N 个 pre-turn 快照的 commit hash
    target = get_nth_pre_turn_commit(project_root, n)
    # 3. git checkout <hash> -- . （只恢复文件，不改 HEAD）
    restore_files_from_commit(project_root, target)
```

==与 LangGraph checkpoint 的对比==：

| 维度 | LangGraph checkpoint | Side-History Git Snapshot |
|------|---------------------|--------------------------|
| 存什么 | Agent 状态（消息 / 变量） | ==文件系统状态== |
| 恢复什么 | Agent 推理上下文 | ==磁盘上的文件== |
| 用户可见 | 不可见（内部状态） | ==可见（`/snapshot` 命令）== |
| 适用 | 断点续跑 / Replan | ==用户主动撤回 Agent 的修改== |

==两者互补==——生产 Coding Agent 通常同时用：LangGraph checkpoint 做 Agent 内部恢复，Side-History 做用户可见的文件回滚。

==`revert_turn` 工具纳入 HITL 危险工具链==——恢复文件是不可逆操作（会覆盖当前文件），必须走 HITL 单步审批，不能走"全部放行"。详见 [[Agent 安全模型]]。

## 五、LLM 幻觉与 Agent 可靠性

### 幻觉的三个根本原因

1. ==训练数据的统计偏差==：LLM 本质是在做"下一个 token 的概率预测"，它学到的是语言模式而非事实。当训练数据中某个错误说法出现频率高，模型就会"自信地"输出错误信息
2. ==知识截止==：模型只知道训练数据截止日期之前的信息，对之后的事实只能"编造"
3. ==长尾知识缺失==：对于训练数据中出现次数少的知识（冷门领域、最新研究），模型倾向于用相关但不准确的信息"填充"

==在 Agent 场景下幻觉更危险==：普通 LLM 幻觉只是输出错误文本，Agent 幻觉会导致**错误的行动**——比如 Agent "幻觉"出一个不存在的 API 端点并尝试调用，或者"编造"一个文件路径并尝试读取。

### Agent 场景的缓解方法

| 方法 | 原理 | 适用场景 |
|------|------|---------|
| ==RAG 增强== | 用检索到的真实文档作为 grounding，减少模型"编造"的空间 | 知识密集型任务 |
| ==工具验证== | Agent 的每个断言都通过工具调用验证（如"文件存在"→先 ls 确认） | 涉及外部状态的任务 |
| ==自我反思== | 生成后让 LLM 自己检查"这个回答有依据吗？"，标记不确定的部分 | 所有场景 |
| ==置信度标注== | 要求 Agent 对每个结论标注置信度，低置信度的结论不作为后续推理的前提 | 多步推理任务 |
| ==结构化输出== | 要求 Agent 输出 JSON 而非自然语言，减少"自由发挥"的空间 | 需要精确结果的任务 |
| ==人工兜底== | 关键决策点设置人工确认环节，Agent 不确定时主动请求人工介入 | 高风险任务 |

==实践中最有效的组合==：RAG（提供事实基础）+ 工具验证（确认外部状态）+ 置信度标注（识别不确定性）。三者配合可以将幻觉导致的错误行动降低 80% 以上。

## 六、Agent 评估方法

### 评估的三个维度

| 维度 | 衡量什么 | 指标 |
|------|---------|------|
| ==任务完成率== | 能不能完成任务 | 成功率（Success Rate）、部分完成率 |
| ==效率== | 用多少资源完成任务 | 平均步骤数、平均 token 消耗、平均耗时 |
| ==质量== | 完成得好不好 | 输出准确性、人工评分、与参考答案的相似度 |

### 主流 Benchmark

| Benchmark | 测什么 | 特点 |
|-----------|--------|------|
| ==SWE-bench== | 解决真实 GitHub Issue（代码修复） | 最贴近实际工程任务，难度高 |
| ==WebArena== | 在真实网站上完成任务（购物、搜索、表单填写） | 测试 Web 操作能力 |
| ==AgentBench== | 8 类任务的综合评测（代码、游戏、网页、数据库等） | 覆盖面广，适合横向对比 |
| ==ToolBench== | 工具调用能力评测，16000+ 真实 API | 专门测工具选择和参数生成准确率 |
| ==GAIA== | 通用 Agent 能力（多步推理 + 工具调用 + 网页浏览） | 综合任务 |
| ==τ-bench== | 工具使用可靠性 | 多轮对话中工具调用的准确性 |

> [!warning] Benchmark ≠ 生产能力
> SWE-bench Verified 上 Claude 4 解决率超过 72%，但这是在理想条件下（单 Issue、清晰描述、有测试用例）。生产环境中 Agent 面对的是模糊需求、多文件变更、缺少测试的场景，实际成功率会大幅下降。

### 生产环境评估方法

1. ==A/B 测试==：新版 Agent 和旧版并行运行，对比关键指标（完成率、成本、用户满意度）
2. ==人工抽检==：每天随机抽取 N 个任务，人工评审推理链是否合理、结果是否正确
3. ==自动化回归==：维护一套标准测试用例（输入+期望输出），每次发布前跑一遍，防止退化
4. ==用户反馈收集==：在 Agent 输出后加"这个结果有帮助吗？"的反馈按钮，收集真实用户评价
