# 代码分析工具

> 代码分析工具让 Agent 在大代码库里精准找到相关代码——是 Coding Agent 的核心能力。本文详解 grep / glob / LSP / codebase_search 等工具的底层实现、性能优化、产品差异。

---

## 一、工具清单与定位

| 工具 | 作用 | 底层 | 适用场景 |
|------|------|-----|---------|
| ==`grep`== / ==`code_search`== | 关键词搜索 | ripgrep | 找包含特定字符串的代码行 |
| ==`glob`== / ==`find_files`== | 按文件名模式找 | glob 库 / find 命令 | 找特定类型/名称的文件 |
| ==`codebase_search`== | 语义检索 | [[Code RAG]] 向量索引 | 自然语言描述找相关代码 |
| ==`get_definition`== | 跳转定义 | LSP / Tree-sitter | 找函数/类的定义位置 |
| ==`get_references`== | 找引用 | LSP / 静态分析 | 找某个函数/变量被哪里调用 |

==搜索策略分层==：

```
用户问题
   ↓
L1: grep 找确定关键词（如函数名、错误信息）   ← 最快、最准
   ↓ 没找到或结果太多
L2: glob 找相关文件（如 *Service.java）        ← 缩小范围
   ↓ 仍不确定
L3: codebase_search 语义检索                   ← 自然语言理解
   ↓ 需要跳转
L4: LSP get_definition / get_references        ← 精确跳转
```

## 二、grep：关键词搜索

### 2.1 为什么是 ripgrep 而不是 grep

==现代 Coding Agent 都用 [ripgrep](https://github.com/BurntSushi/ripgrep)==（命令是 `rg`）：

| 维度 | grep | ripgrep |
|------|------|---------|
| 速度 | 基准 | ==10x 更快== |
| 自动排除 | 需手动 `-exclude` | ==自动读 `.gitignore`== |
| 默认行为 | 递归需 `-r` | ==默认递归== |
| 输出格式 | 需 `-n` 显示行号 | ==默认带行号 + 彩色== |

==为什么快==：
- 用 Rust 写（零成本抽象）
- 并行搜索（多核）
- 跳过二进制文件和 `.git` 目录

### 2.2 实现

```python
import subprocess

def grep(pattern: str, path: str = ".") -> str:
    result = subprocess.run(
        ["rg", "--line-number", "--no-heading", pattern, path],
        capture_output=True,
        text=True,
        timeout=30
    )
    return result.stdout

# 调用示例
grep("def authenticate", "src/")
# 返回：
# src/auth.py:42:def authenticate(user, password):
# src/api/login.py:18:def authenticate_oauth(token):
```

### 2.3 高级参数

==生产环境常用参数==：

```python
def grep(
    pattern: str,
    path: str = ".",
    case_sensitive: bool = False,
    file_type: str = None,  # 如 "py" / "ts" / "java"
    max_results: int = 100
) -> str:
    cmd = ["rg", "--line-number", "--no-heading"]
    
    if not case_sensitive:
        cmd.append("--ignore-case")
    
    if file_type:
        cmd.extend(["--type", file_type])
    
    cmd.extend(["--max-count", str(max_results)])
    cmd.extend([pattern, path])
    
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=30)
    return result.stdout
```

### 2.4 返回格式设计

==必须带文件路径 + 行号==——方便 LLM 后续调用 `read_file` 或 `edit_file`：

```
src/auth.py:42:def authenticate(user, password):
src/auth.py:45:    if not authenticate_ldap(user):
src/api/login.py:18:def authenticate_oauth(token):
```

==不要只返回匹配行==——需要给 LLM 上下文：

```python
def grep_with_context(pattern: str, path: str = ".", context: int = 2) -> str:
    result = subprocess.run(
        ["rg", "--line-number", "--context", str(context), pattern, path],
        capture_output=True, text=True, timeout=30
    )
    return result.stdout

# 返回示例：
# src/auth.py:40:
# src/auth.py:41:def check_password(hash, input):
# src/auth.py:42:def authenticate(user, password):  ← 匹配行
# src/auth.py:43:    if not user:
# src/auth.py:44:        return False
```

## 三、glob：文件名模式匹配

### 3.1 什么是 glob

==glob== 是 Unix 老概念——用 `*` `?` `**` 匹配文件名：

- `*.py` — 当前目录所有 `.py` 文件
- `**/*.ts` — 递归找所有 `.ts` 文件
- `src/*/test_*.py` — `src` 下各子目录的测试文件

### 3.2 实现

==Python 内置 glob 库，零依赖==：

```python
import glob

def find_files(pattern: str, root: str = ".") -> list[str]:
    return glob.glob(f"{root}/{pattern}", recursive=True)

# 调用示例
find_files("**/*.ts", "src")
# 返回：
# ['src/app.ts', 'src/api/user.ts', 'src/utils/helper.ts']
```

### 3.3 常见场景

| 场景 | 模式 |
|------|------|
| 找所有测试文件 | `**/*test*.py` / `**/*.test.ts` |
| 找所有配置文件 | `**/*.json` / `**/*.yaml` |
| 找特定模块 | `src/auth/**/*.java` |
| 找所有 Service 类 | `**/*Service.java` |

### 3.4 与 grep 的配合

==先 glob 缩小范围，再 grep 精确搜索==：

```python
# LLM 思考：用户问"登录功能在哪"
   ↓
# Step 1: 找所有可能的文件
files = find_files("**/*auth*.py")
# → ['src/auth.py', 'src/api/auth_api.py', 'tests/test_auth.py']
   ↓
# Step 2: 在这些文件里搜关键词
grep("def login", " ".join(files))
# → src/auth.py:42:def login(user, password):
```

## 四、codebase_search：语义检索

### 4.1 为什么需要语义检索

==grep 只能找精确关键词==——用户问"怎么处理支付失败"，代码里写的是 `handle_transaction_error`，grep 找不到。

==语义检索==通过向量相似度找到语义相关的代码。

### 4.2 完整流水线

```
离线（建索引时）：
  ① 遍历代码库所有文件
  ② AST 切块（每个函数/类一个 chunk）—— 用 tree-sitter
  ③ 每个 chunk 调 embedding 模型（如 Qwen2.5-Coder-Embedding）
  ④ 存到向量库（Qdrant / Milvus / sqlite-vec）

在线（LLM 调 codebase_search 时）：
  ① 把 query 也做 embedding
  ② 向量库 ANN 检索 Top-50
  ③ Re-rank 模型重排，返回 Top-5
  ④ 把 5 个代码片段返回给 LLM
```

### 4.3 实现示例

```python
def codebase_search(query: str, top_k: int = 5) -> list[dict]:
    # ① query → embedding
    query_vec = embedding_model.encode(query)
    
    # ② 向量检索 Top-50
    results = vector_db.search(query_vec, top_k=50)
    
    # ③ Re-rank 重排
    reranked = rerank_model.rerank(
        query=query,
        docs=[r.text for r in results]
    )
    
    # ④ 返回 Top-5
    return [
        {
            "file": r.path,
            "line": r.line,
            "code": r.text,
            "score": r.score
        }
        for r in reranked[:top_k]
    ]
```

### 4.4 关键技术选择

| 组件 | 选择 | 原因 |
|------|------|------|
| ==切块== | tree-sitter | 按 AST 切（函数/类粒度），比行切更语义完整 |
| ==Embedding 模型== | Qwen2.5-Coder / StarCoder-Embedding | 代码专用，比通用模型好 |
| ==向量库== | Qdrant / Milvus / sqlite-vec | 百万级代码 chunk 够用 |
| ==Re-rank== | Cohere Rerank / bge-reranker | 向量召回是粗排，Re-rank 精排 |

### 4.5 Claude Code 为什么不用这套

==Claude Code 不预建代码向量索引==——靠 grep + read_file 现场检索。原因：

1. **索引建得慢、过期快**——代码变化频繁，索引要频繁重建
2. **Claude 的 1M context**——能直接装下中等代码库，检索精度比向量召回更高
3. **grep 是确定性的**——索引检索是概率性的，前者更可靠

==代价==：每次都要现搜，==大代码库（百万行级）会慢==。

详见 [[Code RAG#主流产品对比]]。

## 五、LSP：跳转定义与查找引用

### 5.1 什么是 LSP

==LSP（Language Server Protocol）== 是 Microsoft 2016 年推出的==语言服务协议==——用 JSON-RPC 提供：

- 跳转定义（go to definition）
- 查找引用（find references）
- 悬停信息（hover）
- 代码补全（completion）
- 诊断信息（diagnostics）

==Coding Agent 主要用前两个==。

### 5.2 架构

```
Coding Agent (LSP Client)
   ↓ JSON-RPC over stdio
Language Server (如 pyright / typescript-language-server)
   ↓
解析项目代码 → 返回定义位置/引用列表
```

==每种语言有自己的 Language Server==：

| 语言 | Language Server |
|------|----------------|
| Python | pyright / pylsp |
| TypeScript/JavaScript | typescript-language-server |
| Java | jdtls (Eclipse JDT) |
| Rust | rust-analyzer |
| Go | gopls |

### 5.3 实现：get_definition

```python
def get_definition(file_path: str, line: int, character: int) -> dict:
    # 调用 LSP Server 的 textDocument/definition
    result = lsp_client.send_request(
        method="textDocument/definition",
        params={
            "textDocument": {"uri": f"file://{file_path}"},
            "position": {"line": line - 1, "character": character}
        }
    )
    
    if result:
        return {
            "file": result["uri"].replace("file://", ""),
            "line": result["range"]["start"]["line"] + 1,
            "character": result["range"]["start"]["character"]
        }
    else:
        return {"error": "Definition not found"}
```

==LLM 调用示例==：

```
用户问：authenticate 函数定义在哪？
   ↓
LLM: grep("def authenticate")
   → 返回：src/api/login.py:18:    result = authenticate_oauth(token)
   ↓
LLM: get_definition("src/api/login.py", line=18, character=15)
   → 返回：{"file": "src/auth.py", "line": 42}
   ↓
LLM: read_file("src/auth.py", offset=40, limit=10)
   → 看到完整定义
```

### 5.4 实现：get_references

```python
def get_references(file_path: str, line: int, character: int) -> list[dict]:
    result = lsp_client.send_request(
        method="textDocument/references",
        params={
            "textDocument": {"uri": f"file://{file_path}"},
            "position": {"line": line - 1, "character": character},
            "context": {"includeDeclaration": False}
        }
    )
    
    return [
        {
            "file": ref["uri"].replace("file://", ""),
            "line": ref["range"]["start"]["line"] + 1
        }
        for ref in result
    ]
```

==LLM 调用示例==：

```
用户问：authenticate 函数被哪里调用了？
   ↓
LLM: get_references("src/auth.py", line=42, character=5)
   → 返回：[
       {"file": "src/api/login.py", "line": 18},
       {"file": "src/api/oauth.py", "line": 35},
       {"file": "tests/test_auth.py", "line": 12}
     ]
```

### 5.5 LSP 的另一个用途：编辑后诊断回注

==Coding Agent 还会用 LSP 做编辑后自动修复==：

```
LLM: write_file("src/app.py", "...")
   ↓
Harness: 调用 LSP textDocument/didChange
   ↓
LSP Server: 返回 diagnostics（语法错误 / 类型错误 / 未定义变量）
   ↓
Harness: 把 error/warning 注入下一轮 LLM
   ↓
LLM: 看到错误 → 自主修复
```

详见 [[LSP 与代码诊断]]。

## 六、产品对比：不同 Coding Agent 的搜索策略

| 工具 | grep | glob | codebase_search | LSP |
|------|------|------|----------------|-----|
| ==Claude Code== | ✓ ripgrep | ✓ | ✗ ==无索引== | ✓ |
| ==Cursor== | ✓ | ✓ | ✓ ==预建索引== | ✓ |
| ==Aider== | git grep | ✓ | ✗ | 部分 |
| ==Continue== | ✓ | ✓ | ✓ | ✓ |
| ==Windsurf== | ✓ | ✓ | ✓ | ✓ |

### 6.1 Claude Code："无索引模式"

==不预建向量索引，靠 grep + glob + read_file 现场检索==。

**优点**：
- 无需等待索引建立
- 代码变化立即生效（不需要重建索引）
- grep 是确定性的，更可靠

**缺点**：
- 大代码库（百万行级）搜索慢
- 自然语言查询效果不如语义检索

**适用**：中小代码库（<10 万行），或不在意搜索延迟。

### 6.2 Cursor："预建索引模式"

==启动时/代码变化后自动建索引==。

**优点**：
- 大代码库搜索快
- 支持自然语言查询

**缺点**：
- 首次索引需要等待（大项目可能几分钟）
- 索引过期时结果不准

**适用**：大代码库，或需要频繁自然语言查询。

### 6.3 Aider："极简模式"

==只用 git ls-files + cat==——没有 grep 工具、没有 codebase 检索。

**哲学**：让人决定看哪些文件，AI 只负责改。

**适用**：小项目，或希望严格控制 Agent 行为。

## 七、搜索工具的性能优化

### 7.1 ripgrep 性能调优

==大代码库搜索慢的常见原因==：

1. **没排除大文件夹**——手动排除 `vendor` / `third_party`
2. **没限制结果数**——用 `--max-count` 限制每文件匹配数
3. **正则太复杂**——简单字符串搜索比正则快

```python
# 优化后的 grep
def grep_optimized(pattern: str, path: str = ".") -> str:
    cmd = [
        "rg",
        "--line-number",
        "--max-count", "50",          # 每文件最多 50 个匹配
        "--max-filesize", "1M",       # 跳过大文件
        "--glob", "!vendor",          # 排除 vendor
        "--glob", "!node_modules",    # 排除 node_modules
        "--fixed-strings",            # 字面匹配（比正则快）
        pattern,
        path
    ]
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=10)
    return result.stdout
```

### 7.2 codebase_search 索引优化

==常见性能瓶颈==：

1. **切块粒度太细**——按行切会产生百万级 chunk，太多
2. **Embedding 模型太大**——用量化版本或更小模型
3. **向量库无索引**——确保用 HNSW / IVF 索引而非暴力搜索

==推荐配置==（10 万行代码库）：

| 维度 | 配置 |
|------|------|
| 切块粒度 | 函数/类级别（~5000 chunks） |
| Embedding 模型 | jina-embeddings-v2-small-zh（量化版）|
| 向量库 | sqlite-vec（轻量）或 Qdrant |
| 索引算法 | HNSW |
