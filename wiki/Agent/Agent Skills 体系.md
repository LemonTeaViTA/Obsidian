# Agent Skills 体系

> Skills 是把一类问题的解决方案固化下来，变成可反复调用的模块。文档给人看，Skills 给 AI 用——这是本质区别。

---

## 一、Skills 是什么？为什么会出现？

**Skills 是把一类问题的解决方案固化下来，变成可反复调用的模块。** 就像写代码时封装的工具类，一次开发、到处复用。

| 维度 | 文档 | Skills |
|------|------|--------|
| 使用对象 | 人 | AI |
| 执行确定性 | 理解偏差、记忆遗漏、应用变形，每个人执行效果都不一样 | 定义什么就做什么，不会走样 |
| 更新传播 | 需要重新学习、培训 | 更新 Skill 后所有人立刻生效 |

**为什么不用 Prompt？** 传统 Prompt 有三个硬伤：
- **散落难管理**：散落在各个同事的笔记、聊天记录里，版本不统一
- **静态不进化**：规范更新了，每个人都要手动同步自己的 Prompt
- **无法团队协作**：资深工程师的经验只能靠老带新口口相传，传播慢、容易失真

Skills 的核心价值三个词：**沉淀**（个人经验→团队资产）、**复用**（好实践快速传播）、**进化**（能力持续迭代）。

## 二、工作原理——渐进式披露

不是让 AI 知道更多，而是让 AI 在恰当的时间知道恰当的事：

1. **发现阶段**：AI 启动时只加载所有 Skills 的元数据（名称、描述、触发关键词），像图书馆目录
2. **语义匹配**：用户输入问题时，基于向量相似度匹配相关 Skills
3. **执行阶段**：匹配成功后才加载完整内容

**为什么不全量加载？** 上下文窗口是有限资源：
1. **空间挤占**：十几个 Skill 每个几千字，全塞进去占掉好几万 token
2. **注意力稀释**：信息越多，模型对每条指令的关注度越低
3. **性能和成本**：token 数量直接影响推理速度和 API 费用

核心逻辑和 RAG 一样——**在有限的上下文里放最有价值的信息，而不是最多的信息**。

### 索引段预算（生产实践细节）

==渐进式披露的"元数据常驻"==落地时有具体的工程边界:

| 维度 | 生产典型值 | 理由 |
|------|----------|------|
| ==启用 skill 上限== | 20 个左右 | 超过这个数,LLM 选择正确 skill 的准确率明显下降(选择困难) |
| ==索引段总大小== | ==≤ 4 KB==(约 1k tokens) | 索引吃掉 system prompt 太多 token,挤压主任务空间 |
| ==单个 skill 索引== | ==name + description ≤ 200 字符== | 长 description 直接截断,强制简洁 |

==索引段长这样==(注入 system prompt 的最末段):

```
## 可用 Skills(按需 load_skill 加载完整内容)

- web-access — 处理网页内容获取/SPA/防爬墙站点的决策手册...
- code-review — 代码审查的 checklist 和常见缺陷模式...
- ppt-generation — 把文档转成 PPT 的标准流程...
...(最多 ~20 条)
```

==关键认知==:==索引段是 prompt cache 友好的==——稳定不变,跟 system prompt + 工具描述一起进缓存(参考 [[长上下文工程#1-prompt-caching\:coding-agent-的成本第一杠杆]])。==Skill 的真实正文不进索引==,只有 LLM 调 `load_skill` 时才读入,这就是渐进式披露的"惰性"在缓存层的体现。

==上限超出怎么办==:==错误做法==是直接截断到前 20 个;==生产做法==是==按使用频率/最近调用时间==排序后取前 20——常用 skill 优先暴露给 LLM。

## 三、语义匹配机制

两步走：

**第一步，建立候选池**。系统启动时，所有 Skill 的名称和描述作为元信息常驻上下文。每个 Skill 的描述里写明适用场景和不适用情况。

**第二步，请求匹配**。用户发请求后，模型把请求语义和所有 Skill 描述做比对。不是简单的关键词匹配，而是基于模型自身的语义理解。比如用户说"帮我把报告整理成 PPT"，模型理解这是演示文稿生成任务，匹配到 pptx 这个 Skill。

关键设计细节——**排除条件**：好的 Skill 描述不光写能干什么，还写不干什么。通过排除提高匹配精度。一个请求可以同时匹配多个 Skill。

整个机制的核心：**让模型充当路由器**，基于语义理解做任务分发。Skill 描述写得越精准、边界越清晰，路由准确率越高。

## 四、SKILL.md 加载机制

模型通过语义匹配判断需要某个 Skill 后，会主动调用 view 工具读取对应路径下的 SKILL.md 文件，把内容拉进当前上下文窗口。

**关键点：读取动作是模型自己发起的，不是系统预先注入的。** 系统只提供 Skill 目录，真正触发读取是模型根据任务需求自主决定的。

时序：用户请求 → 模型扫 Skill 目录 → 判断该用哪个 → 调 view 工具读 SKILL.md → 根据指令执行任务。

本质是**工具调用驱动的动态上下文扩展**，和 RAG 思路一致：区别在于 RAG 检索外部知识，这里检索的是执行规范。

### 专用 `load_skill` 工具 vs 通用 `view` 工具

==早期实现==让 LLM 用通用 `view`/`read_file` 工具读 SKILL.md——==生产中很快暴露出问题==:

| 维度 | 通用 view | ==专用 `load_skill(name)`== |
|------|----------|---------------------------|
| LLM 输入 | 完整路径 `~/.paicli/skills/web-access/SKILL.md` | 仅 skill 名称 `web-access` |
| 路径错误率 | 高(LLM 容易拼错路径 / 不知道 user 还是 project 级) | ==零==(只传 name) |
| 三层位置查找 | LLM 要自己尝试 | ==Host 自动==(jar < user < project 顺序查找) |
| 内容截断 | 无(SKILL.md 太大会爆 prompt) | ==Host 统一截断 5KB==——超过部分提示"详见 references/" |
| 注入位置 | 当前 tool message | ==下一轮 user message 前置==(单独的 SkillContextBuffer) |
| 缓存友好 | 每次读全文 | ==Host 可缓存解析结果== |

==生产 Coding Agent 都用专用工具==——Claude Code / Cursor / PaiCli 都是。==专用工具的核心价值==:==把"如何加载"的复杂性从 LLM 转移到 Host==,LLM 只需要做"该不该用这个 skill"的语义判断,不操心实现细节。

==SkillContextBuffer 的设计==:Skill 正文不直接拼到 tool message(那样会和 tool result 混在一起)——而是==缓存到下一轮 user message 前置注入==:

```
轮次 N:
  assistant: tool_call(load_skill, {"name": "web-access"})
  tool: "已加载 skill: web-access(已注入下一轮上下文)"

轮次 N+1:
  user: <skill name="web-access">
        # 决策手册...
        # 工具选择表...
        </skill>

        [用户的实际消息]
```

==这样设计==:LLM 在轮次 N 看到 skill 已加载的确认,轮次 N+1 才真正用到 skill 内容——==既不污染 tool result,也保证 skill 内容在用户消息前的"指导位置"==。

## 五、创建流程（六步）

1. **Understanding**：搞清楚要解决什么问题、使用场景、触发条件
2. **Planning**：拆解成可复用的资源类型（scripts/、references/）
3. **Initializing**：调用脚本自动生成标准目录结构和模板
4. **Editing**：填充具体内容，description 写清触发条件，SKILL.md 不超过 500 行
5. **Packaging**：六重验证（文件存在性、Frontmatter 格式、YAML 解析、必填字段、命名规范 hyphen-case、长度限制 1024 字符）
6. **Iteration**：根据实际使用反馈持续优化

---

## 六、自由度分级

- **高自由度**：纯文本指令，适用于有多种有效方法的场景
- **中自由度**：伪代码或带参数的脚本，有推荐模式但允许变化
- **低自由度**：特定脚本，少量参数，适用于需要严格一致性的操作

---

## 七、如何写出高质量的 Skill？

1. **description 写清触发条件**：准确描述能解决什么问题、什么场景使用，同时写清不适用的场景。好的例子："Use when working with PDF documents for: (1) Extracting text and tables, (2) Merging or splitting PDFs, (3) Filling form fields"
2. **保持 SKILL.md 简洁**：不超过 500 行，详细内容放 references 目录，SKILL.md 只保留核心流程
3. **多写示例**：示例比说明更有用，越具体模型执行越准确
4. **控制粒度**：一个 Skill 专注一类问题，单一职责原则。"写文章"和"做 PPT"应拆成两个 Skill
5. **指令要具体到可执行**："输出要美观"模型无法执行，得具体到"标题用二号字体加粗，行间距 1.5 倍"
6. **善用 references**：复杂领域知识放 references 目录，按需加载
7. **持续迭代**：每发现一个边界情况就加一条约束，像写单元测试一样打磨

## 八、性能优化

**缓存机制**：所有 Skill 的名称和描述常驻上下文（描述索引级缓存）。SKILL.md 内容读取一次后就在上下文里，后续引用不需要再次读取（会话级缓存）。跨对话不缓存，每次新对话需要重新读取。

**增量加载**：大型 reference 文件不一次性全部加载，只加载需要的章节。先加载目录和概述，按需展开具体内容。

**脚本预编译**：Python 脚本优先使用预编译的 .pyc 字节码，省去编译时间。

## 九、安全机制

三层防护：

1. **沙箱执行**：脚本只在沙箱环境中运行，限制网络访问、文件访问和系统调用，防止恶意脚本破坏系统
2. **输入验证**：所有用户输入经过严格验证，检查危险字符（`;`、`&&`、`||`、反引号、`$(` 等），防止命令注入
3. **资源限制**：执行时间、内存、文件大小、网络请求次数都设上限，防止资源耗尽攻击。==具体数值取决于实现==——以某 Coding Agent 的沙箱默认配置为参考：执行时间 30 秒、内存 512MB、文件大小 100MB、网络请求 10 次（==均为该实现的默认值,非行业标准,生产按需调整==）

### 与 HITL 的协同：Skill 不引入新审批维度

==关键设计原则==:==Skill 是"使用说明书",不是"权限提升"==。Skill 内调用工具(如 `execute_command`、`fill_form`)==仍走该工具既有的 HITL 规则==,==不给 Skill 单独的审批维度==。

==典型场景==:

```
Skill: code-review (要求执行 `pytest` 验证测试通过)
   ↓ Skill 内容指示 LLM 调用 execute_command("pytest")
   ↓
Host 拦截:execute_command 是高危工具,走 HITL 规则
   ↓
用户已对 execute_command 选过"全部放行" → 直接执行
用户没放行 → 弹出单步审批
```

==设计原因==:

| 不引入 Skill 维度的理由 | 说明 |
|---------------------|------|
| ==避免授权矩阵爆炸== | Skill × Tool × HITL 三维授权 = 配置组合数爆炸,用户无法理解 |
| ==Skill 是行为模板,不是身份== | Skill 不应该让原本被拒绝的工具变成允许 |
| ==审批的本质是工具== | "能否执行 shell" 与"通过哪个 skill 触发"无关——风险来自工具本身 |
| ==简化心智模型== | 用户只需理解"工具的 HITL 状态",不用记每个 skill 各自的授权 |

==反面教材==:某些产品给 Skill 单独"信任"开关——结果用户==以为 trust 了 skill 就能畅通无阻==,实际上 skill 内的 `rm -rf` 工具调用还是触发拦截,造成==预期与实际不符==的体验灾难。

==与 [[Agent 工程实践#hitl-工具读写粒度细化]] 的关系==:HITL 是 N 维矩阵(粒度/读写/敏感度/可逆性)——==Skill 不在这些维度里==,它是==指令模板层==,与 HITL 决策层正交。

## 十、与 MCP 的关系：互补不是竞争

| | MCP | Skills |
|---|-----|--------|
| 解决什么 | AI 与外部系统的**连接**问题（调 GitHub API、查数据库、操作文件） | AI 的**能力封装**问题（代码审查能力、文档生成能力、技术方案评审能力） |
| 类比 | 给 AI 装上机械臂，让它能接触外部世界 | 给 AI 使用说明书，让它知道怎么做事 |

两者结合才能真正成为生产力工具：MCP 提供"手"，Skills 提供"怎么做"。

## 十二、目录位置

==生产 Coding Agent 普遍采用三层加载位置==——优先级递增,后者整体覆盖同名 skill:

| 层级 | 路径示例 | 来源 | 覆盖对象 |
|------|---------|------|---------|
| ==1. 内置层== | jar 内 `resources/skills/` 或安装目录 | ==产品出厂默认 skill==(如 web-access) | 无(最低优先级) |
| ==2. 用户级== | `~/.<product>/skills/<name>/` | 个人 skill,跨项目生效 | 覆盖内置层同名 |
| ==3. 项目级== | `<project>/.<product>/skills/<name>/` | 项目专属 skill,只对当前项目生效 | 覆盖用户级 + 内置层同名 |

==各产品的具体路径==:

| 工具 | 内置 | 个人级 | 项目级 |
|------|------|-------|-------|
| Claude Code | （无固定内置层，以用户/项目级为主） | `~/.claude/skills/` | `.claude/skills/` |
| Codex | codex 包内 | `~/.codex/skills/` | `.codex/skills/` |
| TRAE | TRAE 包内 | `~/.trae/skills/` | `.trae/skills/` |
| PaiCli | jar 内 | `~/.paicli/skills/` | `.paicli/skills/` |

> [!note] "内置层"不是每个产品都有
> 上表的内置层是==部分产品==的做法（PaiCli 把出厂 skill 打进 jar）。==Claude Code 官方文档并没有"包内内置 skill 层"==——它的 skill 主要在用户级（`~/.claude/skills/`）和项目级（`.claude/skills/`）。所谓"出厂示例"更多是 Anthropic 提供的样例供用户复制,而非运行时加载的内置层。

==关键原则:同名 skill 整体覆盖,不是字段级合并==

```
内置层:   web-access/SKILL.md (浏览哲学 v1)
                ↓ 用户在 ~/.paicli/skills/web-access/SKILL.md 覆盖
用户级:   web-access/SKILL.md (添加自己的微信公众号经验)
                ↓ 项目级再覆盖
项目级:   web-access/SKILL.md (本项目特定的内网站点经验)
```

==整体覆盖的理由==:==字段级合并语义模糊==——若用户级只覆盖 description 而 SKILL.md 内容继承,==容易出现"description 说能做 X 但内容里没讲怎么做 X"==的不一致。整体覆盖虽然牺牲灵活性,但==行为可预测==。

==派生模式==:用户想"基于内置 web-access 加点东西"——==推荐复制==整个目录到用户级再修改,而不是依赖部分覆盖。

==与 CLAUDE.md / 项目配置的覆盖思路一致==——参考 [[长上下文工程]] 中 CLAUDE.md 的层级覆盖。

`~` 是用户根目录（个人级，跨项目生效），不带 `~` 的是项目目录（项目级，只对当前项目生效），无路径的是产品内置层。

## 十三、企业级管理（SkillHub）

> [!info] ==以下是设想中的企业级方案,非某具体产品的既有实现==
> 截至 2026-06，主流 Coding Agent 的 Skills 仍以用户级 / 项目级文件为主，==没有官方的中心化 Skill 注册中心==。下面描述的是企业自托管场景下==合理的能力设想==,供架构参考。

企业级场景可以部署 SkillHub（自托管的 Skill 注册中心），典型能力：
- **全文搜索**：按命名空间、下载量、评分、更新时间过滤
- **权限控制**：组织/团队/全局作用域的命名空间，每个命名空间有所有者/管理员/成员角色
- **CLI 集成**：通过命令行进行 Skills 的发布、安装和管理

## 十四、典型案例：web-access skill

==举一个具体的内置 skill 例子==——前面章节的抽象设计落到一个真实案例上。`web-access` 是 ==Coding Agent 的高频内置 skill==(PaiCli / Claude Code / 类似产品都有),教 LLM 如何获取网页内容。

### 14.1 目录结构

```
web-access/
├── SKILL.md              # 决策手册(主入口,LLM 看到的第一份文件)
├── references/           # 按需读取的子文档
│   ├── mp.weixin.md     # 微信公众号经验(防爬墙 + 折叠屏样式)
│   ├── zhuanlan.zhihu.md# 知乎专栏经验(SPA + 登录墙)
│   ├── x.com.md         # 推特经验(SPA + cursor 分页)
│   ├── xiaohongshu.md   # 小红书经验
│   ├── github.md        # GitHub 经验(API + Web 双路径)
│   ├── juejin.md        # 掘金经验
│   └── cdp-cheatsheet.md# CDP 常用调用快速参考
└── (无 scripts/——纯决策手册型 skill)
```

==分层设计==:
- ==SKILL.md== = 浏览哲学 + 工具选择表 + 站点优先级(被 `load_skill` 截断到 5KB)
- ==references/*.md== = 单站点/单主题深度知识(LLM 按需 read_file 加载)

### 14.2 SKILL.md 内容结构(决策手册型)

```markdown
---
name: web-access
description: |
  获取网页内容时使用——决策"用什么工具读哪个站",处理 SPA / 防爬墙 / 登录墙。
  适用:网页摘要、知乎专栏、推特线程、微信公众号文章
  不适用:API 文档(用 read_documentation)、代码仓库(直接 git clone)
---

## 浏览哲学(四步法)

1. ==先 fetch_url 试一次==——80% 静态站点能搞定
2. 失败 → ==查 references/<domain>.md== 看有没有专门经验
3. 没经验或失败 → ==browser_connect 切 shared==——登录态 + Chromium 渲染
4. 浏览器仍失败 → ==Jina Reader 兜底==(`r.jina.ai/<url>`,代理转 Markdown)

## 工具选择表

| 站点类型 | 推荐工具 | 原因 |
|---------|---------|------|
| API 文档 / 静态博客 | fetch_url | 轻量、快 |
| SPA 公开页 | chrome-devtools (isolated) | 需 JS 渲染 |
| SPA 登录页 | chrome-devtools (shared) | 需登录态 |
| 防爬墙 | chrome-devtools (shared) + 经验文件 | 站点专属 hack |

## 浏览器优先级

shared 模式仅在==必要时切==——公开页面优先 isolated。详见 [[MCP 协议#isolated-vs-shared]]。

## Jina 兜底

- 适用:静态/低 JS 站点 + browser MCP 不可用时
- 限制:免费额度有限,生产推荐自托管 Reader
```

### 14.3 references 子文档:站点经验文件

==每个站点一个文件==,记录==踩过的坑和有效的方法==:

```markdown
# mp.weixin.qq.com 经验

## 防爬墙特征
- fetch_url 直接返回"环境异常"页
- 必须走 chrome-devtools

## 推荐流程
1. browser_connect 切 shared(用户的微信登录态)
2. navigate_page → 等待页面加载
3. take_snapshot 取主内容区(`#js_content` 选择器)
4. 文章末尾的"赞赏"按钮区域==排除==(易触发反爬)

## 已知坑
- 折叠章节需先 click 展开再 take_snapshot
- 视频文章只能截图,文本提取不可用
```

==这种 references 文件的价值==:==每次踩坑都沉淀进去==,下次遇到同站点 LLM 直接读经验,==不用重新探索==。

### 14.4 决策手册型 vs 流程型 skill

==web-access 是"决策手册型"==——给 LLM 一份判断框架,LLM 自己根据具体任务选路径。==对比"流程型 skill"==:

| 类型 | 例子 | 特点 |
|------|------|------|
| ==决策手册型== | web-access、code-review | LLM 看完手册自主决策每一步 |
| ==流程型== | ppt-generation、deploy | 写明严格步骤,LLM 按步执行 |

==选哪种==:
- ==执行路径多变 / 站点太多== → 决策手册型(给原则,不写死)
- ==流程固定 / 错一步全废== → 流程型(给步骤,严格遵守)

==web-access 适合决策手册型==——站点千变万化,流程型写不全。

### 14.5 这个 skill 体现的关键设计

| 设计点 | 体现 |
|--------|------|
| ==渐进式披露== | SKILL.md 5KB + references 按需加载,不一次性塞所有站点经验 |
| ==排除条件清晰== | description 明确"不适用"(API 文档 / 代码仓库) |
| ==复用宿主工具== | 调 `fetch_url` / `chrome-devtools.*` / `read_file`——不绑定特定实现 |
| ==沉淀机制== | references 是"团队共享的踩坑笔记",随项目增长 |

==面试时讲清==:"==Skill 不是工具==——它是==给 LLM 的工作经验文档==。`web-access` 这类决策手册型 skill 的价值是==把'团队对网页爬取的经验'结构化沉淀==,新人 Agent 启动就能用上老团队踩过的坑——这就是 [[#一、Skills 是什么？为什么会出现？]] 说的"个人经验→团队资产"的具体落地。"
