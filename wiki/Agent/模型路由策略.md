# 模型路由策略

> ==生产级 Agent 的隐藏标配==——根据任务类型动态选择模型（成本优化 / 质量优化 / 隐私优化）。
>
> 路由不是简单"if-else 选模型"，而是==Router LLM（轻量分类器）+ 主 LLM（执行）==的两层架构。
>
> 多厂商 LLM 适配协议见 [[Function Calling#七、厂商差异]]；多模型框架封装见 [[Agent 框架]]。

---

## 一、什么时候需要模型路由

==生产级 Agent 系统几乎都做模型路由==——五个驱动因素：

| 驱动 | 例子 |
|------|------|
| ==成本优化== | 简单问候用 Flash 模型（$0.001），复杂分析才用 Opus（$0.10），==省 70%+== |
| ==质量优化== | Worker 用 Sonnet，Reviewer 用 Opus（详见 [[Reflection 实现#3.2 Critic Model]]） |
| ==延迟优化== | 实时对话用流式 + 快模型，离线分析用慢但准的 |
| ==隐私优化== | 公司机密走本地 Qwen，通用问题走 API |
| ==多模态== | 文本走 Qwen-Plus，图像走 Qwen-VL，代码走 DeepSeek-Coder |

==没有路由的 Agent==：所有请求都用最贵的 Opus / GPT-4 → 成本爆炸 + 延迟劝退。

==有路由的 Agent==：80% 简单任务路到便宜模型，20% 复杂任务才用顶级模型 → ==成本省 70%+，延迟降 50%==。

## 二、三种路由模式

### 2.1 规则路由（最便宜，覆盖明显场景）

```python
def route_simple(task: str, context: dict) -> str:
    # 用户显式指定 → 直接路由
    if context.get("user_specified_model"):
        return context["user_specified_model"]

    # 强信号关键词命中
    if any(kw in task for kw in ["写代码", "代码生成", "debug"]):
        return "deepseek-coder"
    if "图" in task or context.get("has_image"):
        return "qwen-vl"

    # 长度阈值
    if len(task) < 100:
        return "glm-flash"   # 短任务走便宜的

    # 都没命中 → 走默认 / 走 LLM 路由
    return None
```

==✅ 优点==：零成本、零延迟、可解释。
==❌ 缺点==：覆盖不全——同一意图有无数表达方式（"帮我写个排序" / "实现快排" / "用 Python 排个序"），==关键词词典爆炸==。

==适用==：明显信号（用户显式指定 / 多模态判断 / 长度阈值）。

### 2.2 LLM 路由（==生产主流==，用便宜小模型做"分类器"）

```python
ROUTER_PROMPT = """分析任务类型，输出 JSON：
{"type": "code|qa|analysis|creative|simple", "complexity": "simple|medium|hard"}

任务：{task}
"""

ROUTING_TABLE = {
    ("simple", "simple"):    "glm-flash",          # 简单问候 → 最便宜
    ("qa", "simple"):        "deepseek-chat",      # 简单问答 → 便宜
    ("code", "medium"):      "deepseek-coder",     # 代码任务 → 专业
    ("code", "hard"):        "claude-opus",        # 复杂代码 → 顶级
    ("analysis", "hard"):    "claude-opus",        # 复杂分析 → 顶级
    ("creative", "*"):       "gpt-4o",             # 创意写作 → GPT
}

async def route_with_llm(task: str) -> str:
    # ★ 关键:用 Haiku 这种便宜小模型做路由(成本 ~$0.001 / 次)
    classification = await haiku.invoke(ROUTER_PROMPT.format(task=task))
    return ROUTING_TABLE[(classification.type, classification.complexity)]
```

==✅ 优点==：泛化最强，理解语义不依赖关键词。
==❌ 缺点==：每次都要调 Router LLM，有成本和延迟（虽然小）。

==适用==：复杂场景，规则覆盖不到的请求。

### 2.3 Embedding 路由（折中方案）

```python
# 离线:为每种任务类型预先建 prototype 向量
TASK_PROTOTYPES = {
    "code": embed("写代码 / 代码 debug / 代码生成 / 算法实现"),
    "qa": embed("回答问题 / 解释概念 / 是什么 / 为什么"),
    "creative": embed("写文章 / 创意写作 / 故事 / 文案"),
}

async def route_with_embedding(task: str) -> str:
    task_vec = await embed(task)
    # 跟 prototype 算相似度,选最近的
    best = max(TASK_PROTOTYPES, key=lambda k: cosine(task_vec, TASK_PROTOTYPES[k]))
    return ROUTING_TABLE[best]
```

==✅ 优点==：比规则灵活、比 LLM 便宜（embedding 比 LLM 调用便宜 100x）、延迟低（< 50ms）。
==❌ 缺点==：固定的 prototype 词集，新任务类型识别不出。

==适用==：任务类型相对稳定的产品。

## 三、★ 关键认知：Router LLM ≠ 主 LLM

==这是模型路由的核心认知==——很多人以为"模型路由让大 LLM 自己决定走哪个模型"，==错==。

==Router LLM 是专门做路由分类的便宜小模型==，不是干活的主 LLM：

| 维度 | Router LLM | 主 LLM |
|------|-----------|-------|
| 角色 | 决定"任务类型 + 复杂度" | 真正完成任务 |
| 模型 | ==Haiku / GLM-Flash / Gemini-Flash==（便宜） | ==GPT-4 / Claude Opus / Sonnet==（贵） |
| 调用 | 每个请求 1 次（==分类==） | 路由后才调（==执行==） |
| 单次成本 | ~$0.001 | $0.01 - $1 |
| 输出 | JSON（task type + complexity） | 实际答案/代码/文档 |

### 成本经济学（具体数字）

==假设==：1 万次请求

==无路由（全用 Opus）==：
```
10000 × $0.10 = $1000
```

==有路由（Router + 智能分流）==：
```
Router 成本：10000 × $0.001         = $10
80% 简单任务用 Flash: 8000 × $0.001  = $8
20% 复杂任务用 Opus: 2000 × $0.10    = $200
                                     ─────────
                              总计    = $218
```

==省 78% 成本==——==Router LLM 的成本被路由后省钱完全覆盖，且净省 70%+==。

==这就是为什么所有生产级 Agent 都做路由==——不做就是==字面意义上的烧钱==。

## 四、生产真实做法：三层级联

==没人用纯单一方案==——大厂都是==三层级联，按命中难度递进==：

```
请求进来
   ↓
==层 1：规则==（强信号关键词命中，占 50% 流量）
   命中 → 直接路由（零成本）
   不命中 ↓
==层 2：Embedding 路由==（覆盖 30% 流量，延迟低）
   高置信度命中 → 路由（< 50ms）
   低置信度 ↓
==层 3：Router LLM==（兜底 20% 复杂请求）
   Haiku/GLM-Flash 分类 → 路由到对应模型
```

```python
async def production_router(task: str, context: dict) -> str:
    # 层 1：规则
    if model := route_simple(task, context):
        return model

    # 层 2：Embedding
    embed_result = await route_with_embedding(task)
    if embed_result.confidence > 0.85:
        return embed_result.model

    # 层 3：Router LLM 兜底
    return await route_with_llm(task)
```

==每一层覆盖一部分流量==——总成本是==加权平均==，比纯 Router LLM 还便宜。

## 五、实际产品案例

| 产品 | 路由实现 |
|------|---------|
| ==OpenAI ChatGPT== | 内部路由 GPT-4 / GPT-4o / DALL-E / Code Interpreter——===Router LLM== 决定调哪个能力 |
| ==Anthropic Claude.ai== | sub-agent 路由（搜索/编程/分析等任务路由到不同 sub-agent） |
| ==Cursor== | Cmd+K（小修改）走便宜模型 vs Composer（大改）走 Sonnet——==规则路由== |
| ==Cline== | 用户配置不同任务用不同模型（Plan / Act / Reviewer 各自配） |
| ==RouterChat== | 开源 Router 项目，专门做模型路由 |
| ==LiteLLM== | 框架内置 fallback 路由（主模型挂了走备用） |

==Anthropic 的 Multi-Agent Research==：
> "==Lead Agent 用 Sonnet 做规划，Sub-Agent 用 Haiku 执行==——==成本降到原来的 1/5，质量基本无损==。"

## 六、路由的多个维度

==生产级路由不只看"任务复杂度"==——多维度同时考虑：

| 维度 | 路由策略示例 |
|------|------------|
| ==成本敏感== | 简单走 Flash，复杂走 Opus |
| ==延迟敏感== | 实时对话走流式快模型（GPT-4o-mini），离线分析走慢但准的（Opus） |
| ==质量敏感== | Worker 用 Sonnet，Reviewer 用 Opus（盲点互补） |
| ==隐私敏感== | 含 PII 走本地 Qwen，公开问题走 API |
| ==多模态== | 文本走 Qwen-Plus，图像走 Qwen-VL，代码走 DeepSeek-Coder |
| ==厂商容灾== | 主用 Anthropic，挂了 fallback 到 OpenAI（==单点风险== mitigation） |

==实际生产路由表==通常==是个 N 维矩阵==：

```
(task_type, complexity, has_image, contains_pii, latency_req, retry_count) → model
```

## 七、关键认知与面试要点

### 认知 1：Router LLM 是分类器，不是决策者

==很多人误解==："让大 LLM 决定走哪个模型"——错，==那等于双倍成本==。

==正确做法==：用 ==Haiku / GLM-Flash 这种便宜小模型做分类==，==1 次便宜调用决定路由==。

### 认知 2：路由的省钱来自"路由后选便宜模型"

==Router 调用本身有成本==（虽然小），==但路由后用便宜模型省下的钱完全覆盖路由成本==——==净省 70%+==。

### 认知 3：生产是三层级联，不是单一方案

==规则 + Embedding + LLM 三层级联==——按命中难度递进，==每层覆盖一部分流量==。

### 认知 4：路由是多维度而非单一维度

==成本 / 延迟 / 质量 / 隐私 / 多模态==同时考虑——生产路由表是 N 维矩阵。

### 面试时的高质量回答

==问==："你怎么管理多个 LLM？怎么决定用哪个？"

==❌ 平庸答案==："看任务类型选模型，简单的用便宜的。"

==✅ 高质量答案==:"==生产环境用三层级联路由==——规则覆盖明显信号(关键词/长度阈值/用户显式指定),Embedding 覆盖中等场景(prototype 相似度),最后用 Router LLM(==Haiku 或 GLM-Flash 这种便宜小模型==)兜底复杂请求。==关键认知是 Router LLM ≠ 主 LLM==——Router 用便宜模型做==1 次分类==,主 LLM 路由后才调用执行,==整体成本比全用 Opus 省 70%+==。Anthropic 的 Multi-Agent Research 用这个套路把成本降到 1/5。"
