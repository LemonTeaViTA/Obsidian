## 核心工作流

### 1. Raw 文档整理

从 `raws/staged/` 提取知识写入 wiki，完成后移到 `raws/processed/`。

**原则**：
- 一次只处理一个文件，人工审核
- 只提取核心知识（跳过教程、安装命令、个人经验）
- 保留数据来源和项目链接
- 不记录使用教程

### 2. 面试题目同步（三步规范）

每次在 wiki 文件中新增或修改面试题解，必须同步完成：

1. 在对应 wiki 文件中编写题解（`###` 主题目，`####` 子题目）
2. 在 `wiki/面试题目.md` 对应模块下追加 checklist 条目：
   `- [ ] 题目 ➡️ 👉 [[文件名#标题|查看核心解析]]`
3. 在 `优化记录.md` 当天日期下追加变更日志

**注意**：
- 锚点必须与 wiki 文件中的实际标题完全一致（包括空格和标点）
- 新增前先检查是否已有相同题目，避免重复
- 题目必须归入正确的模块分类

### 3. 优化记录写入规范

- **写入前先确认当前日期**（从系统上下文读取，不能凭记忆猜测）
- 新记录插入到文件顶部（`# 知识库优化记录` 标题后），保持最新日期在最前
- 格式：
  ```
  - `文件路径` — 一句话描述改动内容
    **原因**：一句话描述改动原因
  ```
- 同一天、同一文件的多项改动合并为一条

## 页面规范

### 知识颗粒度

采用"主题聚类"颗粒度：
- 不要拆分得过细（避免只有几行字的孤立碎片）
- 不要保留上万字的巨型文档
- 例：`List体系.md` 包含 ArrayList、LinkedList 及其对比，而不是拆成多个文件

### 标题层级

- `##` 用于主题分组（如"## ArrayList 与 LinkedList"）
- `###` 用于具体题目（如"### 🌟ArrayList 和 LinkedList 有什么区别？"）
- `####` 用于子问题
- 标题内不嵌入 `[[...]]` 链接（会导致锚点匹配失败）
- 标题内不使用加粗 `**...**`（同样导致锚点失败）

## 面试题解规范

- 题解结构：`###` 主题目 + `####` 子题目
- 每个题解有清晰的问题描述和答案
- 算法题解只放在 `wiki/算法/`，不加入 `面试题目.md`

---

## 算法笔记规范

严格采用以下标题层级：
- `### 题目名称`
- `#### 解题思路`（包含直观的物理隐喻和坑点说明）
- `#### 代码实现`（先基础解法，再进阶优化版）
- `#### 复杂度分析`（不能遗漏）

多面型题目（如既可用 DP 也可用双指针）必须在不同页面留双向链接。

## 防误操作规范

**批量编辑后必须验证**：
```bash
# 检查标题层级是否完整
python3 -c "
with open('wiki/面试题目.md') as f:
    for i, line in enumerate(f, 1):
        if line.startswith('## ') or line.startswith('### '):
            print(f'{i}: {line.rstrip()}')
"

# 检查题目总数
grep -c "^\- \[ \]" wiki/面试题目.md
```

**大段删除操作**：不要用 Edit 工具，改用 Python 脚本按行号操作，避免字符编码匹配失败。

**新增内容前**：先确认是否已有重复章节，避免重复插入。
