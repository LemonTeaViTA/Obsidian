## Boundaries

### Always do

- 修改 wiki 文件后更新 frontmatter 的 `last_reviewed` 为今天
- 新增题解后同步 `wiki/面试题目.md`（加 checklist 条目 + wikilink）
- 写入内容后检查编码：`grep -n $'\xef\xbf\xbd' 文件路径`
- 新增内容前确认无重复章节
- 变更后在 `优化记录.md` 当天日期下追加日志

### Ask first

- 删除超过 10 行的内容
- 合并或拆分已有 wiki 文件
- 修改面试题目.md 的模块分类结构
- 修改 OBSIDIAN_STYLE.md 或本文件的规则

### Never do

- 标题内嵌入 `[[...]]` 链接或 `**加粗**`（导致锚点匹配失败）
- 用 Edit 工具做大段删除（用 Python 脚本按行号操作）
- 凭记忆猜测日期（必须从系统上下文读取）
- 在算法题解以外的地方写算法题（算法题只放 `wiki/算法/`）
- 跳过编码检查直接提交

## 当你修改/新建 wiki 文件时

1. 确认 frontmatter 四字段齐全：`module`, `tags`, `difficulty`, `last_reviewed`
2. 遵循标题层级：`##` 主题分组 → `###` 具体题目 → `####` 子问题
3. 知识颗粒度：主题聚类（如 `List体系.md` 包含 ArrayList + LinkedList + 对比），不拆碎片也不堆万字
4. 对照 `OBSIDIAN_STYLE.md` 快速自检：高亮 ≤ 3 处/节，callout 不连续，wikilink 有效
5. 检查编码：`grep -n $'\xef\xbf\xbd' 文件路径`

**完成标准**：
```bash
# 验证 frontmatter 存在
head -5 wiki/模块/文件.md | grep -q "last_reviewed"
# 验证无损坏字符
grep -c $'\xef\xbf\xbd' wiki/模块/文件.md  # 应输出 0
```

## 当你新增面试题解时

1. 在对应 wiki 文件中写题解（`###` 主题目，`####` 子题目）
2. 在 `wiki/面试题目.md` 对应模块下追加：
   ```
   - [ ] 题目 ➡️ 👉 [[文件名#标题|查看核心解析]]
   ```
3. 锚点必须与 wiki 文件中的实际标题完全一致（含空格和标点）
4. 在 `优化记录.md` 当天日期下追加日志

**完成标准**：
```bash
# 验证锚点有效（标题存在于目标文件）
grep -q "### 你写的标题" wiki/模块/文件.md
# 验证面试题目.md 已更新
grep -q "题目关键词" wiki/面试题目.md
```

## 当你整理 raw 文档时

流程：`raws/staged/分类/文件` → 提取知识到 wiki → 移到 `raws/processed/`

1. 一次只处理一个文件
2. 只提取核心知识（跳过教程、安装命令、个人经验）
3. 保留数据来源和项目链接
4. 完成后按"新增面试题解"流程同步

详细操作见 `RAW_WORKFLOW_GUIDE.md`。

## 当你写优化记录时

- 先确认当前日期（从系统上下文读取）
- 新记录插入文件顶部（`# 知识库优化记录` 标题后）
- 格式：`- \`文件路径\` — 改动描述` + `**原因**：原因描述`
- 同一天同一文件的多项改动合并为一条

## 当你写算法笔记时

标题层级：
- `### 题目名称`
- `#### 解题思路`（含物理隐喻和坑点）
- `#### 代码实现`（基础解法 → 进阶优化）
- `#### 复杂度分析`（不能遗漏）

多解法题目必须在不同页面留双向链接。

## 防误操作

批量编辑后验证：
```bash
# 检查面试题目.md 标题层级完整
grep -n "^## \|^### " wiki/面试题目.md | tail -20
# 检查题目总数
grep -c "^\- \[ \]" wiki/面试题目.md
```

大段删除用 Python 脚本按行号操作，不用 Edit 工具。

## 知识库健康守护

> ==方法论文档==见 [[wiki-管理方法论]]——本节是==操作触发规则==。

### 主动检查时机

每次以下场景==必须主动==跑健康指标 + 给用户提醒（不等用户问）：

| 场景 | 检查什么 |
|------|--------|
| 新增 / 大改文档（> 50 行） | 该文档是否触发红灯 |
| 文档超过 ==600 行== | 立即提醒拆分 |
| 提到其他文档已讲的概念 ≥ 50 行 | SSoT 违反，提醒重写为引用 |
| 新增子目录 | 是否需要 WORKSPACE.md |
| 用户问"这个文档怎么样" | 跑全套健康体检 |

### 健康指标（红线）

| 指标 | 红线 |
|------|------|
| 单文档行数 | < 600 行 |
| 一级章节数 | ≤ 8 个 |
| Diátaxis 类型 | 1 类（最多 1.5） |
| SSoT 违反（同概念在 ≥ 2 文档重复 ≥ 50 行） | 0 |
| > 400 行文档顶部速览 | 必须有 |

### 主动告警的措辞模板

==发现红灯时主动说==：

```
⚠️ Wiki 健康提醒:

文档:[文档名]
触发:[超出哪个阈值]
红灯:[X 个]    黄灯:[Y 个]

主要问题:
- [具体问题 1]
- [具体问题 2]

建议:
- [具体整理动作]

是否要我做整理?(等待确认再动手)
```

### 整理优先级

| 优先级 | 问题 |
|--------|------|
| P0 | SSoT 违反 |
| P1 | 文档 > 800 行 |
| P2 | Diátaxis ≥ 2 类 |
| P3 | 缺顶部速览（> 400 行文档） |
| P4 | frontmatter 过期 |

==P0 / P1 必须立即提醒==，==P2-P4 可批量整理==。

==详细方法论 / 决策树 / 文档模板==见 [[wiki-管理方法论]]。

## 思考记录触发规则（==对话 → 知识==的转化）

> ==Personal Knowledge Management 核心实践==——把对话中的隐性知识==显式化==到 [[思考记录]]。
> 详细方法论见 [[wiki-管理方法论#八对话--知识的转化方法]]。

### 主动询问时机

==以下时机==必须==主动==问用户："==这值得记到思考记录吗==？"

| 时机 | 触发条件 |
|------|--------|
| ==误区被纠正后== | 用户或我意识到之前理解错了（如"Skill ≠ 命令脚本"） |
| ==设计决策做出后== | 尤其有反例 / 替代方案的（如"为什么不用 `00-` 前缀"） |
| ==学到新概念后== | 用户表达"我明白了" / "原来是这样"  |
| ==元洞察出现== | 用户问"==这个该怎么做==" 级别的元问题 |
| ==错误被承认== | 我之前说错了，需要修正记录 |
| ==对话告一段落== | 长对话结束前主动总结 |

==不主动问 = 知识流失==。

### 4 类知识的归属（决定放哪）

| 类型 | 例子 | 该去哪 |
|------|------|------|
| ==事实性知识== | "FTS5 是 SQLite 内置全文索引" | wiki/ 主文档 |
| ==设计决策== | "为什么不用 `00-` 前缀" | [[思考记录]] |
| ==误区纠正== | "Skill ≠ 命令脚本" | [[思考记录]] + 主文档 callout |
| ==元能力 / 方法论== | "对话→知识 4 类分流" | [[wiki-管理方法论]] |

### 思考记录条目格式（ADR）

```markdown
## YYYY-MM-DD — <标题>

> 一句话定位

### 背景
<触发这次思考的问题>

### 思考 / 误区纠正
<关键洞察 / 误区 / 真实情况>

### 决策（如果有）
<选了什么 / 为什么 / 替代方案为什么没选>

### 链接
- [[相关 wiki 文档]]
```

### 判断标准

==该记的==：
- ✅ 误区纠正、设计决策的"为什么"、反例 / 替代方案、错误修正、元洞察

==不该记的==：
- ❌ 一般性问答（wiki 已有）、流水账、临时讨论、调试记录

==判断==：==半年后回头看，还有价值吗==？
