# wiki 审计

本目录是对 `wiki/` 知识库做**方法论合规性 + 内容质量**审计的工具集与产物归档。**只读审计，只出报告，不修改任何 wiki 原文件。**

## 审计依据

规则来自 `wiki-管理方法论.md` §十「机械检查清单」与 `QUALITY_CHECKLIST.md`，分两类：

- **客观可校验**：UTF-8 乱码、frontmatter 四字段、行数阈值（>600🟡/>800🔴）、代码块语言标签、callout 类型/嵌套/连用、标题混 `[[]]`、wikilink 坏链、anchor 精确匹配、面试题双向同步、孤儿页、MOC 覆盖。
- **需专家判断**：Diátaxis 类型混杂、SSoT 重复、内容越界/求职方向对齐（Java + Agent）、标题层级逻辑、>400 行渐进式摘要、知识点准确性/时效性/面试深度。

## 工具集

### 快速检查脚本 `check.sh`

**用途**: 日常机械检查,无需启动完整审计工作流。可用于 git pre-commit hook 或 CI/CD。

```bash
# 全量检查
./wiki审计/check.sh --all

# 分项检查
./wiki审计/check.sh --encoding   # UTF-8 乱码
./wiki审计/check.sh --links      # wikilink 坏链(抽样)
./wiki审计/check.sh --structure  # 行数/H1/层级
./wiki审计/check.sh --callouts   # callout 规范
```

**输出**: 人类可读的彩色文本,列出发现的问题。

### 审计工作流 `_audit_workflow.js`

**用途**: 全量审计(3-6 个月执行一次),生成完整的领域报告。

**产出**: 21 个领域报告 + 2 个机械检查报告 + 1 个总报告,写入 `历史审计/YYYY-MM/`。

### 单元配置 `units.json`

**用途**: 定义 21 个审计单元(领域)的文件清单、报告路径、P0 提示。

**复用**: 下次审计前根据 wiki 结构变化(新增域、文件拆分)更新此 JSON,工作流自动读取。

**增量审计**: 临时注释掉不审的单元,只审部分域。

## 多 agent 分工

- **全局机械检查**（2 agent）：`机械检查报告.md`（格式类）、`链接同步检查报告.md`（链接/同步类）。
- **领域专家审计**（21 agent,配置来自 `units.json`）：每个戴「知识库管理专家 + 领域知识点专家」两顶帽子。
- **汇总**（1 agent）：读全部报告产出 `审计总报告.md`（红黄绿记分卡 + P0/P1/P2 清单 + 修复顺序）。

## ⚠️ 下次审计必读：5 个踩过的坑（防止重蹈覆辙）

> 这些坑都是 2026-06 审计中**实际花钱踩出来的**。详细案例见 [[工作流总结]] §三。

### 坑 1：机械检查脚本本身有 bug（最致命）

**症状**：check.sh 报告 41 个"多 H1"文件 → 实际只有 2 个真问题

**根因**：grep 没区分代码块内外，把 ```bash 里的 `# 注释` 也算 H1

**防坑动作**：
- 启动 agent 修复**之前**先抽样 1-2 个文件**人工核实**
- 如果误报率 >20%，先修脚本再重新扫描
- ✅ 遵循 [[wiki-管理方法论#十一、N-agent 验证法则]]

### 坑 2：Schema 强制返回会卡死 Agent

**症状**：Phase 1 的 21 个 agent 中 13 个"完成但未调用 StructuredOutput"

**根因**：编辑类任务的 agent 容易在最后步骤失败，丢失元数据

**防坑动作**：
- ❌ 不用 `agent(prompt, { schema })`
- ✅ 用 `agent(prompt + "请以纯文本格式返回: 字段1: X")`
- 重编辑任务尤其要避免 schema

### 坑 3：Agent 说做不一

**症状**：agent 日志说"修了 196 条链接"，git diff 实际只有 34 条

**根因**：agent 规划能力 > 执行能力，会**写应该做的步骤**但不实际执行

**防坑动作**：
- 修复后**用 git diff 对账**，看实际改动数 vs agent 声称数
- 大批量机械操作（如批量改链）**直接用 sed/awk 脚本**，不要交给 agent
- ✅ 规则：机械重复用脚本，语义判断用 agent

### 坑 4：Phase 间没 Barrier 会级联失败

**症状**：Phase 2 三个 agent 全撞 500 错（token 耗尽），但 Phase 3/4 照常跑，产出无效报告

**根因**：workflow 没检查 Phase 2 是否成功，盲目继续

**防坑动作**：
- workflow 每个 phase 后**加前置检查**：上一 phase 是否产出有效结果
- 使用 `resumeFromRunId` 续跑机制，避免重跑已完成的 agent
- ✅ 例：`if (phase2.status !== 'success') return`

### 坑 5：600 行不是红线，是健康信号

**症状**：本能反应想拆所有 >600 行文件 → 实际有些是单一主题深度文档，不该拆

**根因**：把工具阈值当成硬性规则，忽略**内容连贯性**

**防坑动作**：
- 16 个 >600 行文件经语义判断后：12 个保留 + 3 个拆 + 1 个边缘
- 判断依据：单一主题？Reference 性质？拆分后能独立查询？
- ✅ 遵循 [[wiki-管理方法论#十.4 机械检查的局限性]]

## 下次审计前的准备

> **⭐ 必读**：先看上面的「5 个踩过的坑」章节，避免重蹈覆辙。

1. **更新 `units.json`** — 如果 wiki 结构有变(新域、文件拆分、改名),同步到配置
2. **跑一次 `check.sh --all`** — 快速自查当前健康度
3. **抽样核实问题清单** — 启动 agent 前先看 1-2 个典型案例（避免坑 1）
4. **检查方法论** — `wiki-管理方法论.md` §十 机械检查清单是否有新规则
5. **启动审计工作流** — 新产物写入 `历史审计/YYYY-MM/`
6. **对比历史** — 用 `diff` 对比两次 `审计总报告.md`,追踪趋势
