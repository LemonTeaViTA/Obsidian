### P2 — PDF.md "扫描版每页就是一张 JPEG/PNG" 表述不严谨

- **文件**：`PDF.md` 第 86 行
- **问题**：扫描版 PDF 单页图像编码常用 **CCITT G4（B&W 传真编码）** / **JBIG2**（双层压缩）/ **JPEG2000**（彩色），而非简单 JPEG/PNG。这影响后续 OCR 工程实践——CCITT 黑白扫描件 OCR 难度低、JBIG2 字典压缩可能丢字、JPEG2000 解码兼容性差。
- **影响**：对初学者无害，但对 RAG/文档解析方向的求职者讲解 PDF 解析时这是必备知识。
- **修复建议**：把"每页就是一张 JPEG/PNG"改为"每页是一张栅格图像（黑白多为 CCITT G4 / JBIG2，彩色多为 JPEG / JPEG2000）"，或在后面加一个 `> [!info] 扫描版 PDF 的常见图像编码` callout。

### P2 — OOXML.md §四 4.6 表述"`r:id` 通过 `_rels/workbook.xml.rels` 映射到具体文件"可加严谨说明

- **文件**：`OOXML.md` 第 269–283 行
- **问题**：表述正确但 `r:id` 命名空间未明示（`r=` Office Relationships namespace `http://schemas.openxmlformats.org/officeDocument/2006/relationships`）。读者若读 OPC 规范会困惑为什么有时是 `r:id` 有时是 `Id`（rels 文件里）。
- **修复建议**：加 1 行括注 "（`r:` 是 Office Relationships 命名空间）"，并说明 `_rels/workbook.xml.rels` 里的 `<Relationship Id="rId1">` 与主文档里的 `r:id="rId1"` 通过同名匹配。

### P2 — XML.md §四 "JSON 约 150 字节、XML 约 300 字节" 数字未给出处

- **文件**：`XML.md` 第 141–142 行 callout
- **问题**：体积估算是估值但无明确条件（是否含缩进、字段数量是否一致）。对初学者无害，但作为面试素材需可复现。

## ③ 领域知识点准确性评价

### 准确性总评：**优秀**

四篇文章在数据格式领域的核心概念上准确且不过时，2026 年视角下没有重大技术错误。亮点：

- **XML.md**：把 SGML→HTML/XML 分化讲清；命名空间、良构规则、与 HTML/JSON 三方对比、命名空间 URI 仅作标识符不需要访问——这些都是面试常考且容易讲错的点，本文都把握住了。
- **OOXML.md**：是四篇里最有工程价值的一篇。
  - ECMA-376（2006）/ ISO/IEC 29500（2008）日期准确 ✅
  - OPC（Open Packaging Conventions）+ rId 间接寻址讲到位 ✅
  - xlsx `<c>` 元素的 `r` / `t` / `s` / `<v>` / `<f>` 属性全部覆盖，取值表（`n` / `s` / `b` / `d` / `str` / `e`）完整且准确 ✅
  - sharedStrings 共享字符串机制、styles.xml `cellXfs` 间接寻址、合并单元格 mergeCells 规则——都是 openpyxl 解析时绕不开的细节，讲得到位 ✅
  - pptx 三层继承（母版→布局→幻灯片）、备注页 notesSlides 分离——准确 ✅
- **DOM.md**：把"DOM 三层含义（数据结构 / API 标准 / 跨语言实现）"讲透，并强调 DOM 不是浏览器专属——这是 Java 后端面试容易踩的认知坑（很多候选人以为 DOM = JS 的东西），本文准确扳正。
- **PDF.md**：把"PDF 是页面描述语言，不是图片"这个核心认知讲透；BT/ET/Tf/Td/Tj 指令清单准确；数字版 vs 扫描版的区分对 RAG/文档解析方向非常关键。

### 可补但不算缺陷的进阶点（面试加分项）

按目标岗位 Java + Agent 方向，下列知识点目前未覆盖，未来若扩内容可考虑：

1. **OOXML.md** 未提 `Strict OOXML vs Transitional OOXML`（ISO 标准里实际有两个 conformance level，Office 默认 Transitional）。面试问"为什么开源库读 docx 偶尔报错"时是答案之一。
2. **XML.md** 未提 **XXE 注入攻击**（XML External Entity）。Java 安全面试高频考点（特别是 Spring / Mybatis 解析配置时禁用外部实体的 `setFeature` 设置）。建议补一个 §九 "安全：XXE 与禁用外部实体" 小节，约 30 行即可。**对求职方向高度相关，建议补。**
3. **DOM.md** 未提 **SAX / StAX 的对比**——Java 后端面试讲 XML 解析时 SAX（事件流）/ StAX（拉式）/ DOM（树形）三对比是经典题。可以在 §五 之后加一个表格对比，强调 DOM 内存占用大、SAX 适合大文件流式处理。**对求职方向高度相关，建议补。**
4. **PDF.md** 未提 **PDF/A**（长期归档子集）和 **PDF 数字签名**——电子合同、电子发票场景必备，弱相关，低优先。
5. **OOXML.md** 未提 `DrawingML` 在 docx/pptx 公式编辑（OMML）和图形（DrawingML）中的角色——技术深度补充，低优先。

### 求职方向对齐评价

数据格式单元定位准确：作为 RAG / 文档解析 / Spring 配置 / Mybatis 解析 的前置基础知识，服务于 **Java + Agent** 求职方向。没有塞入大模型研发岗、纯算法岗才用的内容（如 Tokenizer / SubwordEncoding 等不会出现这里）。XML 单元最末的"LLM Prompt 用 `<example>` 标签"算 Agent 方向的轻量呼应，恰当不喧宾。
