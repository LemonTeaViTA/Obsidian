## 二、逐条问题

### P0 红旗

#### P0-1 · 文件上传机制.md 文件结构严重错乱（7 个 H1）
- **文件**：`文件上传机制.md`
- **行号**：第 8/79/134/272/331/398/482 行均为 `# xxx` 一级标题
- **问题**：单个 .md 文件包含 **7 个 `#` 一级标题**，每个 H1 都是一个独立面试问答帖（文件大小限制 / 5MB 分片 / Bitmap 状态 / MySQL 存储 / MinIO 存储 / 断点续传 / 文档更新机制）。这违背 wiki "单文档单主题" 原则；同时 Obsidian outline 会渲染成 7 个并列根节点，破坏导航。顶层小节远超 "≤8" 限制（按 H1 算 7 个，按 H2 算 30+ 个）。
- **严重度**：P0（结构性错误，影响 anchor、导航、SSoT）
- **修复建议**：
  1. 将 7 个 H1 全部降为 `##`，统一在一个 `# 文件上传机制` 之下；或
  2. 拆成 3 个文件：`文件上传-分片与断点续传.md`、`文件上传-存储分层.md`、`文件上传-限制与版本.md`；
  3. 多个 `## 面试怎么说` 重复出现 4 次（266/325/392/476/548），按 Diátaxis 应集中到末尾的"面试要点"小节。

#### P0-2 · 文件上传机制.md Diátaxis 严重混类
- **文件**：`文件上传机制.md`
- **问题**：同文档内同时是 Reference（限制/字段表）+ How-to（断点续传流程）+ Explanation（为什么用 Bitmap）+ Tutorial 形式的"面试怎么说"逐题问答。≥3 类，红旗。

## 四、本单元小结

Baize 项目共 11 篇文档、3131 行，是当前 wiki 中最贴合用户求职目标（Java + Agent）的核心单元，**领域知识点准确性良好**，但**文档结构层面存在 1 个红旗 + 4 个黄旗**：

1. **最严重问题**：`文件上传机制.md` 是 7 个独立 FAQ 拼接成的"FAQ 大合集"，含 7 个 H1 段，Diátaxis 4 类混合 — 必须重构。
2. **第二严重**：`聊天助手模块.md` 中 LLM 选型（DeepSeek）与 `RAG管道设计.md` / `Qwen3接入排障.md` / `模型选型.md`（Qwen3.5-9B 本地）矛盾，是知识库可信度的硬伤。
3. **SSoT 问题**：BM25 / KNN 详解在 `对话上下文管理.md` 与 `RAG管道设计.md` 双份；`接口文档.md` 与 `用户管理与权限.md` / `聊天助手模块.md` 三方接口描述重复。
4. **结构问题**：`对话上下文管理.md` 文件名与 H1 完全不匹配，实质是"4 个 FAQ 合订本"。
5. **库表与正文不同步**：document_chunks vs document_vectors 命名、缺 file_upload/chunk_info 表。

**建议处理顺序**：① 立刻修 LLM 选型矛盾（一致性硬伤）→ ② 拆 / 重构 `文件上传机制.md` → ③ 拆 / 重命名 `对话上下文管理.md` → ④ 消除 BM25/KNN 双份讲解 → ⑤ 库表对齐 → ⑥ 杂项 P2。
