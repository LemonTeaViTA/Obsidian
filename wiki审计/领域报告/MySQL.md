## 二、逐条问题

### P1（计划整改）

**1. [准确性] B+树"空间复杂度是多少"答案概念错误**
- 文件：`索引机制详解.md` 第 166-168 行（### B+树的空间复杂度是多少？）
- 问题：答案写"O(logN)"，并用树高 h 来推导。B+ 树要存储 N 条数据，**空间复杂度是 O(N)**；O(logN) 是树高/单次查询的时间复杂度。此处把空间复杂度与树高/时间复杂度混为一谈，是明确的技术错误，面试官追问会翻车。
- 建议：改为"空间复杂度 O(N)（需存储���部 N 条记录），树高 O(log_m N) 决定的是查询的 I/O 次数"。

**2. [SSoT / 内容边界] Online DDL「大表加字段 / ALTER 锁表」在两篇重复讲解**
- 文件：`锁机制.md` 第 307-389 行（## DDL 与锁，含 Online DDL 三算法、pt-osc、gh-ost）与 `高可用与分库分表.md` 第 257-271 行（### 千万级大表如何添加字段，pt-osc + INSTANT）
- 问题：两处都在讲"大表加字段如何避免锁表 + pt-online-schema-change + INSTANT 算法"。`锁机制.md` 是全文（~80 行，权威单点），`高可用` 是 ~15 行精简版。虽未到 50 行重复的 P0 线，但属同一概念多点讲解，易产生维护漂移。
- 建议：`高可用与分库分表.md` 该节保留一句话结论 + callout 链接到 `[[锁机制#DDL 与锁]]`，删除重复的工具用法。

**3. [准确性] Buffer Pool 默认大小表述不准**
- 文件：`引擎与日志.md` 第 89 行
- 问题：写"1-4GB RAM 默认 25%，>4GB 默认 50% 但不超过 4GB"。这是**云厂商（如 AWS RDS）的自动调参规则**，并非 MySQL 原生默认值。原生 MySQL `innodb_buffer_pool_size` 默认就是 **128MB**。表述为"MySQL 默认"会误导。
- 建议：明确"原生默认 128MB；生产/云上常按物理内存 50%~75% 配置"。

**4. [准确性] DATETIME 占用字节数过时**
- 文件：`MySQL 基础与架构.md` 第 107 行
- 问题：写"DATETIME 占 8 个字节"。自 **MySQL 5.6.4 起 DATETIME 为 5 字节**（+小数秒 0~3 字节），8 字节是 5.6.4 之前的旧值。TIMESTAMP 4 字节正确。
- 建议：更正为 5 字节（不含小数秒），并可补一句版本变更。

**5. [准确性] VARCHAR"最多 65535 个字符"表述不精确**
- 文件：`MySQL 基础与架构.md` 第 87 行
- 问题：65535 是**单行的字节上限**（且为整行所有列共享），不是"字符数"。utf8mb4 下单个 varchar 实际最多约 16383 字符。原文虽提了字符集与长度字节，但"最多容纳 65535 个字符"开头表述仍属概念混淆。
- 建议：改为"最多 65535 个**字节**（整行共享），实际可存字符数取决于字符集"。

**6. [Diátaxis 混类] 优化与实战篇类型混杂**
- 文件：`优化与实战.md` 全篇
- 问题：同时包含 Tutorial 式 SQL 手写题（各班前十名、求名字）、Reference（EXPLAIN 字段大表、type/Extra 表）、How-to（慢查询三步、避免慢查询六条）。三类混在一篇，偏离"单篇 ≤1.5 类"。尚不到红旗（实战题量不大），记 P1 提示。
- 建议：可将 EXPLAIN 参考表抽到独立"EXPLAIN 速查"小节或并入索引篇，SQL 手写题与优化 how-to 分区更清晰。

## 三、领域知识点准确性评价

总体技术质量**较高**，B+ 树（矮胖/IO 次数/2000 万三层估算 1170 扇出）、MVCC（隐藏列 DB_TRX_ID/DB_ROLL_PTR、版本链、ReadView 四字段与三段可见性判断）、锁体系（记录锁/间隙锁/临键锁退化规律、意向锁兼容矩阵、等待图死锁检测）、redo/undo/binlog 与两阶段提交、Online DDL 三算法等核心考点讲解准确、面试深度足够，Spring `@Transactional`/`@Retryable`/MyBatis BigDecimal 等衔接对齐 Java + Agent 求职方向，体例良好。

需修正的硬错误/过时点集中在少数细节：
- **概念错误**：B+树空间复杂度 O(logN)（应 O(N)）—— 唯一明确技术错误（P1）。
- **过时/不准**：DATETIME 8 字节（应 5 字节，5.6.4+）；Buffer Pool 默认 25%/50%（原生为 128MB，属云厂商规则）；VARCHAR"65535 字符"（应字节）。
- **表述含糊**：RU 隔离级别加锁描述。
- redo log"默认 2 个文件、每个 48MB / ib_logfile0/1"在 MySQL 8.0.30+ 已被 `innodb_redo_log_capacity` 取代为多文件动态管理，可加版本提示（轻微）。

代码示例普遍正确可运行；雪花算法位分配（1+41+10+12）与移位正确；变量法分组排名注明仅适用 8.0 以下、`rank` 为保留字属已知边界。无伪造 API。
