**[P1-2] 并发/并行 定义不严谨（领域技术问题）**
- 文件：线程基础与ThreadLocal.md，L13-14
- 原文：「并行是多核 CPU 上的多任务处理…」「并发是单核 CPU 上的多任务处理，多个任务在同一时间段内交替执行」
- 问题：把「并发=单核、并行=多核」对立起来是常见误区。并发（concurrency）是**任务组织/处理多任务的能力**这一逻辑概念，多核上同样存在并发；并行（parallelism）是并发的子集，强调**物理上同时执行**。多核机器上既有并发也有并行。严谨面试官会追问，此处易被反驳。
- 修复建议：改为「并发指系统同时处理多个任务的能力（单核通过时间片轮转交替执行，多核也可并发）；并行指多个任务在多核上真正同时执行，是并发的一个子集」。

**[P2-2] ABA 示例未真正复现 ABA（领域代码问题）**
- 文件：锁.md，L65-84（ABAFix 示例）
- 问题：演���线程在 `Thread.sleep(100)` 之后才 `getStamp()`，此时 stamp 已被前一线程推进到 3，CAS("100"→"300", 3, 4) 会**成功**。该示例想说明 AtomicStampedReference「能识别 ABA」，但因为读 stamp 的时机在 A→B→A 之后，实际没有体现「我拿着旧 stamp，发现 stamp 变了所以失败」的关键点，演示效果与结论相反。
- 修复建议：让观察线程在另一线程修改**之前**就 `int stamp = ref.getStamp()`（拿到旧版本号 1），再 sleep 等对方完成 A→B→A，最后用旧 stamp 做 CAS，结果应为 false——这才证明 stamp 拦住了 ABA。

**[P2-3] ExchangerTest 代码 bug（领域代码问题）**
- 文件：并发工具类.md，L327
- 原文：`String A = exgr.exchange("B");`（交换的是字面量 "B"，而非变量 `B`="银行流水B"）
- 问题：导致 `A.equals(B)` 比较的是「银行流水A」与「银行流水B」恒为 false，且对照另一线程交换进来的其实是「银行流水A」。属从外部素材搬运时残留的 bug，示例输出会误导读者。
- 修复建议：改为 `String A = exgr.exchange(B);`。

**[P2-4] synchronized 锁标志位描述小瑕疵（领域技术问题）**
- 文件：锁.md，L381
- 原文：「锁状态为偏向锁，也就是锁类型为"01"，偏向锁标记为"0"的状态」
- 问题：Mark Word 中 `01` + 偏向标记位 `0` 表示的是**无锁/未偏向**，偏向锁应是 `01` + 偏向标记 `1`。此处把「未偏向」当成了偏向锁入口的描述，表述含混。
- 修复建议：明确「锁标志位 01、偏向标记 1 = 偏向锁；偏向标记 0 = 无锁」。锁标志位整体：01(无锁/偏向)、00(轻量级)、10(重量级)、11(GC 标记)。

## ③ 领域知识点准确性评价（Java 并发专家视角）

总体**准确度较高**，覆盖面贴合 Java 后端面试主线（JMM/锁升级/AQS/线程池七参数/ThreadLocal 内存泄漏/ConcurrentHashMap 7→8 演进），与求职方向（Java + Agent）契合，无纯算法/纯大模型偏题内容。

- 线程池 5 种状态及状态码（RUNNING=111、SHUTDOWN=000、STOP=001、TIDYING=010、TERMINATED=011）准确。
- ThreadLocalMap：HASH_INCREMENT=0x61c88647 黄金分割、线性探测、弱引用 Key/强引用 Value 泄漏机理、remove() 源码、rehash「先清理再扩容 / size≥3/4 阈值」均正确。

- synchronized 锁升级不可逆、偏向锁撤销需 Safepoint/STW、JDK15 默认关闭偏向锁——callout 内容准确且有面试增量。

- volatile：可见性 + 禁止重排 + DCL 单例必须加 volatile + 不保证原子性，讲解完整、是本单元 SSoT 处理得当的范例（锁.md 与 Java 内存模型.md 均以链接回指，未重复全文）。
