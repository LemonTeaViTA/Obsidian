## 三、技术难点与解决方案

### 3.1 Bug #1: StructuredOutput 协议卡死

**现象**: Phase 1 的 21 个 agent 中 13 个"完成但未调用 StructuredOutput"

**根因**: schema 强制结构化返回，重编辑任务容易在最后步骤失败

**修复**:
```javascript
// 错误方式（schema 强制）
const result = await agent(prompt, {
  schema: {
    type: 'object',
    properties: {
      fixed_count: {type: 'number'},
      files_modified: {type: 'array'}
    }
  }
})

// 正确方式（纯文本约定）
const result = await agent(prompt + `

请在最后以这个格式返回:
---RESULT---
修复数量: X
修改文件: file1.md, file2.md
`)
```

**教训**: 重编辑任务用纯文本返回 + prompt 尾部约定格式，避免 schema 卡死

---

### 3.2 Bug #2: check.sh 多 H1 误报（39/41）

**现象**: grep '^# ' 检测到 41 个"多 H1"文件，实际只有 2 个真问题

**根因**: 未区分代码块内外，把代码注释的 `#` 也算作 markdown H1

**修复**:
```bash
# 错误写法（误报）
local h1_count=$(grep -c '^# ' "$file")

# 正确写法（区分代码块）
local h1_count=$(awk '/^```/{in_code=!in_code; next} !in_code && /^# /{c++} END{print c+0}' "$file")
```

**教训**: 启动 41 个 agent 前先抽样核实，发现误报率 >95%，修正脚本后重新扫描

**衍生**: 这个案例直接催生了 [[wiki-管理方法论]] §十一 "N-agent 验证法则"

---

### 3.3 Bug #3: check.sh 链接提取漏尾巴

**现象**: `[[file]]` 提取后变成 `wiki/file]].md`（末尾多 `]]`）

**根因**: sed 's/\[\[\(.*\).*/\1/' 只去除了开头 `[[`，未去除末尾和锚点/别名

**修复**:
```bash
# 错误写法（不完整）
local link=$(echo "$full_link" | sed 's/\[\[\([^#|]*\).*/\1/')

# 正确写法（完整清理）
local link=$(echo "$full_link" | \
  sed -e 's/^\[\[//' \        # 去开头 [[
      -e 's/\]\]$//' \        # 去末尾 ]]
      -e 's/[#|].*//')        # 去锚点和别名
```

**教训**: 链接提取需处理 4 个符号：`[[` / `]]` / `#anchor` / `|alias`

---

### 3.4 Bug #4: Agent "说做不一"现象

**现象**: 残留修复 agent 日志声称"修复 196 条链接"，实际只修了 34 条

**根因**: agent 规划能力 > 执行能力，会写"应该做"但实际不彻底

**缓解**: 用 sed 脚本绕过 agent，直接批量替换

**未来方向**: 建立"说做校验"机制 — agent 输出 vs git diff 对账

---

### 3.5 Bug #5: Phase 2 token 耗尽 500 错

**现象**: 整合/MOC/改链三个 agent 全部撞 500 Internal Server Error

**根因**: 服务端 token 池临时不足

**缓解**: resumeFromRunId 续跑，已完成 agent 返回缓存结果

**教训**: workflow 加前置检查，phase 间有 barrier，避免级联失败

### 4.2 配置外部化：units.json（21 单元）

**设计理念**: 审计单元（域列表 + 规则阈值）外部化，workflow 只负责编排

```json
{
  "审计单元": [
    {"域": "Agent", "重点": "SSoT违反 + MCP/LSP重复"},
    {"域": "RAG", "重点": "超长文件拆分"},
    ...
  ],
  "阈值": {
    "单文档行数": 600,
    "一级章节数": 8,
    "...": "..."
  }
}
```

**优势**:
- 下次审计改 JSON 不改 workflow 代码
- 新增域只需在 units.json 追加一行
- 阈值调整无需重写脚本逻辑

## 八、关键认知总结

### 8.1 架构设计

1. **Barrier 隔离依赖** — Phase 间严格串行，Phase 内完全并发
2. **配置外部化** — 审计单元/阈值分离到 units.json，workflow 只负责编排
3. **工具独立运行** — check.sh 脱离 workflow，支持日常自查

### 8.2 质量管理

1. **N-agent 验证法则** — 启动批量修复前先核实真实 N 值
2. **机械检查有局限** — 600 行阈值是工具建议，需语义判断
3. **Agent 说做校验** — agent 输出 vs git diff 对账（未来方向）

### 8.3 Token 经济

1. **机械重复用脚本** — sed/awk 替代 agent 批量修改
2. **语义判断用 agent** — SSoT 违反 / 内容拆分需要领域专家
3. **抽样核实省 95%** — 误报率高的清单需先修脚本再分发

### 8.4 可持续性

1. **归档历史快照** — 支持健康度趋势追踪
2. **方法论自演进** — 新规则直接加入 wiki-管理方法论.md
3. **工具版本管理** — check.sh 保留 changelog，记录 bug 修复
