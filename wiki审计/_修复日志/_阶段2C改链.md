## 七、修复方法论总结

### 7.1 工具链

1. **发现阶段**：
   - `grep -rohE '\[\[[^\]]+\]\]' wiki/ | sort -u` — 提取所有 wikilink
   - `find wiki/ -name "*.md"` — 列出所有合法目标文件
   - 比对差异 → 生成死链清单

2. **修复阶段**：
   - **sed** — 简单路径前缀批量替换（如 `LLM/LLM基础与训练` → `LLM/LLM 基础与训练`）
   - **Python re** — 复杂锚点重指（需语义判断的场景）
   - **Read 工具** — 人工核��目标文件实际标题

3. **验证阶段**：
   - `grep -r` 反向验证：搜索已知死链模式，确保清零
   - 抽检关键文件（面试题目.md）的 10% 样本

### 7.2 决策原则

1. **内容主题优先**：锚点失效时，根据语义判断内容实际迁移到哪个新文件（如 `Agent 核心概念#Memory` → `Agent Memory 系统`）
2. **文件存在性**：目标文件不存在且无合理替代 → 改为纯文本或删除
3. **磁盘为准**：Phase 1 部分 agent 未写日志，修复时以 `find wiki/` 实际文件名为真相源
4. **最小修改**：只改链接（`[[...]]` 内容），不动正文

### 7.3 Phase 2C 特有挑战

- **锚点语义判断**：`Agent 核心概念` 拆分后，12 条锚点需逐条分析内容归属（ReAct/Memory/MCP/Planning 等）
