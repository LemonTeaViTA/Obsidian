# Wiki 管理方法论

> ==这是 wiki 的"宪法"==——给 Agent 看的==知识库管理规范==。==所有文档==的写作 / 整理 / 重构==都对照本文==。
>
> ==读者==：Agent（Claude Code / Codex / Cursor 等）+ 维护者本人。==位置==：仓库根（与 OBSIDIAN_STYLE.md / QUALITY_CHECKLIST.md 并列）。

> [!tip] ==Wiki 管理方法论速览（一分钟读完）==
>
> ==1. Diátaxis 4 类==——文档==只能定位 1 类==
>
> | 类型 | 给谁用 | 关键问题 |
> |------|------|--------|
> | ==Tutorial== | 学习者 | 怎么开始 |
> | ==How-to== | 有目标的工程师 | 怎么完成 X |
> | ==Reference== | 查询者 | X 是什么 |
> | ==Explanation== | 想理解原理的人 | 为什么 |
>
> ==2. 4 条核心原则==
>
> - ==SSoT==：同一事实只在一个地方写，其他地方用 wikilink 引用
> - ==Progressive Disclosure==：> 400 行文档==必有顶部速览==
> - ==Bounded Context==：每个文档==有清晰边界==——超出 scope 链接出去
> - ==Atomic Notes==：每个文档==聚焦一个核心概念==
>
> ==3. 健康指标（红线）==
>
> | 指标 | 阈值 |
> |------|------|
> | 单文档行数 | ==< 600 行==（流水账除外） |
> | 一级章节数 | ≤ 8 个 |
> | Diátaxis 类型 | 1 类（最多 1.5） |
> | SSoT 违反 | 0 |
> | > 400 行文档顶部速览 | 必须有 |
>
> ==4. 整理决策树==
>
> ```
> > 600 行?
> ├── 是 → 找最大子主题(>20%)
> │   ├── 独立成文合理? → ★ 拆成新文档,主文档加 callout
> │   └── 否? → 看是否冗余 → 链接到现有文档
> └── 否 → 检查 SSoT 违反 → 修
> ```
>
> ==5. 主动监控==——==Agent 必须主动==:
>
> - 大改文档（>50 行）后跑健康指标
> - 发现红灯==立即告警==（不等用户问）
> - 文档 > 600 行立即提醒拆分
>
> ==6. 对话→知识 4 类分流==
>
> | 类型 | 该去哪 |
> |------|------|
> | 事实性知识 | wiki/ 主文档 |
> | 设计决策 | [[思考记录]] |
> | 误区纠正 | 思考记录 + 主文档 callout |
> | 元能力 / 方法论 | 本文档 |
>
> ==7. 反模式（避免）==
>
> - ❌ 复制粘贴（同段话多处出现）
> - ❌ 超大杂烩文档（1000+ 行混 4 类）
> - ❌ 无顶部摘要（大文档读者翻几百行才知全貌）
> - ❌ 文件夹只放 1 个文件
> - ❌ 排序前缀（`00-` / `01-`）

> [!note] ==自指声明：本方法论自身遵守自己的规则==
>
> ==避免无限套娃陷阱==——本文档==自身==也是 wiki 文档，==按本方法论的规则写==：
>
> - ==Diátaxis 类型==：Reference + 少量 Explanation（==1.5 类==）
> - ==SSoT==：不与其他文档重复（[[OBSIDIAN_STYLE]] 讲格式 / [[QUALITY_CHECKLIST]] 讲单文档质量 / 本文讲==组织结构与方法==）
> - ==Bounded Context==：只讲"==怎么管理 wiki==" — ==不讲具体内容怎么写==
> - ==自指演进==：发现规则不合适，==直接改本文==，==不创建"方法论的方法论"==
>
> ==这就是"==自洽=="——==方法论自己==遵守==自己定义的规则==。
>
> ==类比==：宪法不需要"宪法的宪法"——宪法的修订程序写在宪法自己里。==发现失效就改本文==，==不堆叠新文档==。详见 [[思考记录]] "==知识管理套娃陷阱的解=="。

## 一、Diátaxis 文档分类法（最权威）

==`Diátaxis`==（==希腊语"==分类法=="）是 ==technical 文档==领域的==权威方法论==——==Django / Cloud Native / Stripe / GitLab== 等都用。

==核心思想==：==文档分 4 类==，==绝不能混在一个文档里==。

| 类型 | 给谁用 | 关键问题 | 例子 |
|------|------|--------|------|
| ==Tutorial==（教程） | ==学习者==（不懂） | "==怎么开始==" | "Agent 入门：从零写一个" |
| ==How-to Guide==（操作指南） | ==有目标的工程师== | "==怎么完成 X==" | "怎么部署 Agent 到生产" |
| ==Reference==（参考手册） | ==查询者== | "==X 是什么 / 长什么样==" | "所有内置工具列表 / 协议规范" |
| ==Explanation==（理论解释） | ==想理解原理的人== | "==为什么 / 原理是什么==" | "为什么 Coding Agent 不做沙箱" |

==混在一起的后果==：

```
学习者打开 → 看到 Reference → 不知从何下手
工程师打开 → 看到 Explanation → 找不到操作步骤
查询者打开 → 看到 Tutorial → 想要的字段在哪
```

==解决方法==：==每个文档定位 1 类==（最多 1.5 类——主类型 + 少量补充）。==其他类型链接出去==。

### 1.1 判断文档属于哪一类

==看主体内容是回答哪个问题==：

| 主要内容 | 类型 |
|---------|------|
| ==列举字段 / API 签名 / 配置项== | Reference |
| ==分步骤做某事== | How-to |
| ==从零教学 + 概念引入== | Tutorial |
| ==解释为什么 / 设计理念== | Explanation |

### 1.2 当前 wiki 各文档的类型

| 文档 | 类型 | 健康 |
|------|------|------|
| [[Agent 核心概念]] | Explanation | ✅ |
| [[Agent Memory 系统]] | Explanation + 顶部 速览 callout | ✅ |
| [[Memory 实现对比]] | Reference + Explanation | ✅ 主题统一 |
| [[Function Calling]] | Reference | ✅ |
| [[MCP 协议]] | Reference + 少量 Explanation | ✅ |
| [[LSP 与代码诊断]] | Explanation + 顶部 速览 | ✅ |
| [[Coding Agent 工具集]] | ==4 类混合== | ⚠️ 待整理 |

==混合 ≥ 2 类的文档需要重构==。

## 二、4 条核心原则

### 2.1 Single Source of Truth（SSoT，唯一真相来源）

==同一事实==只在==一个地方==写，==其他地方用 wikilink 引用==。

==正面例子==：

- ✅ Memory 实现细节 → ==只在== [[Memory 实现对比]]，[[Agent Memory 系统]] 用 callout 链接
- ✅ Stdio / WebSocket / Streamable HTTP → ==只在== [[计算机网络]]，MCP / LSP 链接过去
- ✅ raws 整理规则 → ==只在== `raws/WORKSPACE.md`，其他文档不重复

==反面例子==：

- ❌ Skill 详情同时在 [[Agent Skills 体系]] 和 [[Coding Agent 工具集]] §5.3 → ==重复==
- ❌ HITL 三档授权同时在 [[Agent 安全模型]] §3 和 [[Coding Agent 工具集]] §5 → 重复

==发现重复时==：选==一个 canonical 位置==保留全文，==其他位置改成 callout==。

### 2.2 Progressive Disclosure（渐进披露）

==入门读者看摘要==，==深度读者看详情==——同一文档内==分层==。

==实现方式==：

1. ==文档顶部加"速览 callout"==（一分钟读完核心）
2. ==复杂章节顶部再加小 callout==
3. ==代码示例先放简化版，再放完整版==

==正面例子==：

- ✅ [[Agent Memory 系统]] 顶部 速览 callout（7 个 section 一分钟读完）
- ✅ [[LSP 与代码诊断]] 顶部 三层关系速览
- ✅ [[Memory 实现对比]] 文档说明 + 总览表

==所有 > 400 行的文档都应该有顶部速览==。

### 2.3 Bounded Context（界限清晰）

==每个文档==有==清晰的边界==——==文档名就是 scope==，==超出 scope 的内容链接出去==。

==判断方法==：==每个章节问"这章和文档主题真的强相关吗"==。==弱相关 == 链接出去==。

==反面例子==：

- ❌ [[Coding Agent 工具集]] §4.8 SSRF 防御（80 行）——==SSRF 是安全话题==，应在 [[Agent 安全模型]]
- ❌ [[Coding Agent 工具集]] §五 Commands/Skills 详细实现（190 行）——==Skills 是上层封装==，应在 [[Agent Skills 体系]]
- ❌ [[Coding Agent 工具集]] §八 可靠性（15 行）——==可靠性是独立话题==，应在 [[Agent 可靠性设计]]

==核心判断==：==如果章节内容"==独立成文也合理=="==，==那就独立成文==。

### 2.4 Atomic Notes（原子笔记 / Zettelkasten）

==每个文档==聚焦==一个核心概念==——==Zettelkasten 方法的核心==。

==判断方法==：==能用一句话说清这个文档的 scope 吗==？

| 文档 | 一句话 scope | 健康 |
|------|------------|------|
| [[Agent Memory 系统]] | "Agent 怎么记住事" | ✅ |
| [[MCP 协议]] | "Agent 与工具的标准协议" | ✅ |
| [[Coding Agent 工具集]] | "Coding Agent 实际有哪些工具 + 工具怎么实现"（==但 §五 偏题==） | ⚠️ |

==说不清的就是混了==——==拆==。

## 三、健康指标（量化标准）

==每个文档==都要==量化体检==。==指标 + 阈值==：

| 指标 | 阈值 | 超标处理 |
|------|------|--------|
| ==单文档行数== | ==< 600 行== | 超 → 拆分独立子主题 |
| ==一级章节数== | ==≤ 8 个== | 超 → 合并相关章节 |
| ==Diátaxis 类型== | ==1 类==（最多 1.5 类） | 超 → 按类型拆 |
| ==SSoT 违反== | ==0 处== | 有 → 选 canonical 位置，其他链接 |
| ==顶部速览 callout==（>400 行文档） | ==必须有== | 缺 → 补 |
| ==独立子主题占比== | ==< 30%== | 超 → 拆出去 |
| ==frontmatter `last_reviewed`== | ==6 个月内== | 过期 → 更新 |
| ==断链 wikilink== | ==0== | 有 → 修 |

### 3.1 红黄绿等级

| 等级 | 触发条件 | 处理 |
|------|--------|------|
| 🟢 **绿** | 0 红灯 + ≤ 1 黄灯 | 健康 |
| 🟡 **黄** | 1 红灯 或 2-3 黄灯 | 计划整理 |
| 🔴 **红** | ≥ 2 红灯 或 Diátaxis ≥ 2 类 | ==立即整理== |

### 3.2 流水账文档的特殊规则

==流水账文档==（==优化记录、思考记录==等==按时间累积==的文档）==不受 600 行限制==——它们==天然按时间增长==，==不是知识内容==。

==特殊规则==：

| 流水账类型 | 健康阈值 | 处理 |
|----------|--------|------|
| ==优化记录==（变更日志） | ==当前活跃 < 1500 行== | 超过 → ==按月归档==到 `优化记录_归档/{YYYY-MM}.md` |
| ==思考记录==（ADR） | ==无硬限制== | 按主题分组即可，不归档 |
| ==session 日志==（如果有） | ==当前活跃 < 1000 行== | 按周归档 |

==归档策略==：

```
优化记录.md             ← 当前月(活跃)
优化记录_归档/
   ├── 2026-04.md       ← 4 月归档
   ├── 2026-05.md       ← 6 月初归档
   └── ...
```

==归档时机==：

- ==每月初==（1 号 / 月初第一次操作时）归档上月
- ==文件超 1500 行==立即归档老内容
- ==归档时==顶部加==归档目录链接 callout==，==不让老记录消失==

==与知识内容文档的区别==：

| 维度 | 知识内容（wiki/） | 流水账（优化记录） |
|------|----------------|----------------|
| ==内容性质== | 概念 / 规范 / 决策 | 时间事件 |
| ==读者目的== | 学习 / 查询 | 追溯历史 |
| ==600 行限制== | ✅ 严格 | ❌ 不适用 |
| ==整理方式== | 拆分 / 链接 | ==归档== |
| ==阅读模式== | 通读 / 跳读 | 倒序看最新 |

## 四、整理决策树（What to do）

==发现红灯文档时==，==按这个决策树走==：

```
单文档 > 600 行?
├── 是 → 找最大的子主题(占比 > 20%)
│   ├── 这个子主题"独立成文也合理"吗?
│   │   ├── 是 → ★ 拆出去成独立文档,主文档加 callout 链接
│   │   └── 否 → 看是否有其他文档已经讲了 → 删本地内容,链接过去
│   └── 没有大子主题 → 看是否冗余(同义重复) → 合并去重
│
├── 否(<600 行)
└── 但 SSoT 违反? → 选 canonical 位置 + 其他改 callout
```

==关键认知==：==删 / 链接 比 重写 重要==——==不要复制内容==，==要建立引用==。

### 4.1 拆分独立文档的判断标准

==该拆==:
- 子主题 > ==50 行==
- 独立成文==有读者价值==（不只是"工具集的一部分"）
- 与其他文档==有交叉引用==的可能

==不拆==:
- 子主题 < ==30 行==
- 完全依附主文档（脱离上下文不可读）
- 只是==简短列表==

### 4.2 链接而不是复制的写法

==错误写法==（在文档 A 里复制文档 B 的内容）：

```markdown
## 5.3 Skill 怎么实现

Skill 是给 LLM 看的专家手册...(repeat 100 行)
```

==正确写法==（链接 + 一句话定位）：

```markdown
## 5.3 Skill 是 Tool 的上层封装

==Skill== = ==给 LLM 看的专家手册==（不是命令脚本）。==与 Command 的区别==：
- Command：用户主动触发的快捷指令
- Skill：LLM 自动激活的决策框架

==完整机制==（渐进式披露 / SKILL.md / 三层加载）见 [[Agent Skills 体系]]。
```

==简短 + 关键差别 + 链接==——==读者要深入==点链接，==不要深入==也能==立刻明白本文档要表达什么==。

## 五、主动监控机制（给 Agent 的指令）

==Agent 在以下时机==必须==主动检查==并==告警==：

### 5.1 触发主动检查的时机

| 场景 | 检查什么 |
|------|--------|
| ==每次新增 / 大改文档（> 50 行）== | 该文档==健康指标==是否触发红灯 |
| ==每次新增子目录== | 是否需要 ==WORKSPACE.md== |
| ==文档里出现 ≥ 50 行讲其他文档已有的概念== | ==SSoT 违反==，提醒重写为引用 |
| ==文档超过 600 行== | ==立即==提醒拆分 |
| ==用户问"这个文档怎么样"== | 跑健康体检 |
| ==每月 / 每 N 期 PaiCli==（重大批改后） | 全 wiki 体检 |

### 5.2 主动告警的措辞模板

==Agent 必须主动说==（不等用户发现）：

```
⚠️ Wiki 健康提醒:

文档:[文档名]
触发条件:[超出哪个阈值]
红灯:[X 个]    黄灯:[Y 个]

主要问题:
- [具体问题 1]
- [具体问题 2]

建议:
- [具体整理动作]

是否要我做整理?(等待确认再动手)
```

### 5.3 整理优先级

==发现红灯多个文档时==，按==优先级==排：

| 优先级 | 问题类型 | 理由 |
|--------|--------|------|
| ==P0== | SSoT 违反（同概念在多文档重复） | 修改一处其他全错，==最危险== |
| ==P1== | 文档体量爆炸（> 800 行）| 读不下去，==立即降低门槛== |
| ==P2== | Diátaxis 类型 ≥ 2 | 读者迷惑 |
| ==P3== | 缺顶部速览（> 400 行文档）| 入门门槛高 |
| ==P4== | frontmatter 过期 | 不影响阅读 |

## 七、违反方法论的常见模式（==避免==）

| 反模式 | 表现 | 解决 |
|--------|------|------|
| ==复制粘贴== | 同一段话在多文档出现 | ==选 canonical + 其他改 callout== |
| ==超大杂烩文档== | 1000+ 行混 4 类内容 | ==按 Diátaxis 拆== |
| ==无顶部摘要== | 大文档读者要翻几百行才知道全貌 | ==加速览 callout== |
| ==文件夹只放 1 个文件== | 形式主义 | ==文件直接放上层目录== |
| ==排序前缀==（`00-` / `01-`） | 风格不一致 + 视觉杂乱 | ==自然命名 + 用索引文件导航== |
| ==索引和内容混在一起== | 索引文件含大量内容 | ==索引专做导航 + 内容拆出独立文档== |

## 八、对话 → 知识的转化方法

> ==对话和思考是 fleeting==（短暂的，会忘）——==必须显式提炼==才能成为==durable== 的知识资产。
>
> ==Personal Knowledge Management（PKM）的核心实践==：把对话中的隐性知识==显式化==。

### 8.1 对话能产生 4 类知识

==不是所有对话内容都该塞进 wiki==——==分类==才能==决定放哪==：

| 类型 | 内容 | 例子 | 该去哪 |
|------|------|------|------|
| ==1. 事实性知识== | 客观事实 / 概念 / 规范 | "FTS5 是 SQLite 内置全文索引" | ==wiki/ 主文档== |
| ==2. 设计决策== | 为什么这么选 + 替代方案 | "为什么不用 `00-` 前缀" | ==思考记录.md== |
| ==3. 误区纠正== | 常见误解 + 真实情况 | "Skill ≠ 命令脚本" | ==思考记录== + 主文档 callout |
| ==4. 元能力 / 方法论== | 怎么做事的方法 / 流程 | "对话→知识 4 类分流" | ==方法论文档==（本文） |

==关键认知==：==只有 1 类（事实性）==该进 wiki 主文档。==其他 3 类==是==隐性知识==——==必须显式记录==。

### 8.2 文档分工

| 文件 | 内容 | 例子 |
|------|------|------|
| ==优化记录.md== | ==流水账==——改了什么 | "新增 wiki-管理方法论.md 329 行" |
| ==思考记录.md== | ==决策记录==——==为什么==这么改 + 误区 | "为什么不用 `00-` 前缀" |
| ==wiki/ 主文档== | ==事实性知识== | "FTS5 是什么 + 怎么工作" |
| ==wiki-管理方法论==（本文） | ==怎么管理 wiki 的方法== | "Diátaxis 分类法" |

### 8.3 判断标准：半年后还有价值吗

==该记的==：

- ✅ ==误区纠正==（==读 wiki 学不到==的）
- ✅ ==设计决策的"==为什么=="==
- ✅ ==反例 / 替代方案==（==`00-` 前缀==的反例）
- ✅ ==错误修正==（==我之前说 Codex 闭源是错的==）
- ✅ ==关键概念的精确定义==（==chunk 不是按段落切==）
- ✅ ==元能力 / 元洞察==（==对话本身是知识资产==）

==不该记的==：

- ❌ 一般性问答（==MCP 是什么==这种 wiki 已有的）
- ❌ 流水账（==我改了 X 文件==——这是优化记录的事）
- ❌ 临时讨论（==今天该写哪个话题==）
- ❌ 调试记录（==encoding 检查==这种）

==判断标准==：==这条记下来==，==半年后回头看==，==还有价值吗==？

### 8.4 ADR 格式（决策记录的行业标准）

==每个思考记录条目==遵循==ADR（Architecture Decision Record）格式==：

```markdown
## YYYY-MM-DD — <标题>

> 一句话定位

### 背景
<触发这次思考的问题>

### 思考 / 误区纠正
<关键洞察 / 误区是什么 / 真实情况是什么>

### 决策（如果有）
<选了什么 / 为什么这样选 / 替代方案为什么没选>

### 链接
- [[相关 wiki 文档]]
```

==这是 Web 框架（Spring / React / Kubernetes）== 的==标准做法==——==记录"==为什么 X 这么设计=="==这种思考过程。

### 8.5 提炼时机（==Agent 必须主动询问==）

==以下时机==必须==主动==问"==这值得记到思考记录吗=="：

| 时机 | 触发条件 |
|------|--------|
| ==对话告一段落== | 长对话结束前 |
| ==决策做出后== | 尤其有反例 / 替代方案的 |
| ==误区被纠正后== | 用户或我意识到之前理解错了 |
| ==学到新概念后== | 用户表达"我明白了" |
| ==元洞察出现== | 用户问"==这个该怎么做=="==级别的元问题 |
| ==错误被承认== | 我之前说错了，需要修正记录 |

==不主动问 = 知识流失==。

### 8.6 提取流程

```
对话过程中或结束时:
  1. 识别有"==半年后还有价值=="的内容
  2. 分类(4 类知识)
  3. 决定放哪(主文档 / 思考记录 / 方法论 / 优化记录)
  4. 按格式记录(ADR for 思考记录)
  5. 双向链接到相关 wiki 文档
```

==关键==：==不是流水账==——==是提炼==。==记录的是结论 + 推理过程==，==不是对话过程==。

## 九、AI 辅助知识管理的底层架构

> [!tip] 来源启发
> 以下原则提炼自 Karpathy 的 AI 知识库实践与 Obsidian 社区的深度思考——补充本文 §八「对话→知识的转化方法」的上游边界（什么内容由 AI 维护、什么内容必须靠人内化）。

### 9.1 三层架构（raw / wiki / schema）

==AI 辅助的知识库应分三层==——==每一层对 AI 的开放程度不同==：

| 层 | 角色 | AI 权限 | 说明 |
|---|------|------|------|
| ==raw== | 原料层 | ==只读== | 原始文章 / 笔记 / 剪藏，AI 不修改 |
| ==wiki== | 知识层 | ==可维护== | 结构化的知识库，AI 负责摄入、组织、更新 |
| ==schema== | 规则层 | ==只读 + 受控修订== | 定义 AI 的行为边界（本文档 + AGENTS.md + OBSIDIAN_STYLE 等） |

==AI 在此架构中执行三种操作==：

- ==digest==——摄入新原料到 wiki（raw → wiki 的提炼）
- ==query==——基于 wiki 回答问题（不读 raw）
- ==lint==——健康检查（发现矛盾、孤立节点、过时内容）

==与本文 §三「健康指标」配套==：lint 操作的具体红黄绿阈值就是 §三的指标。

### 9.2 "两个大脑"分离原则

==知识库应分为两个独立空间==：

| 空间 | 别名 | 内容性质 | 追求 |
|------|------|--------|------|
| ==个人思考库== | ==内脑== | 深度内化的知识，经过自己思考与验证 | ==深度== |
| ==AI 研究库== | ==外脑== | AI 辅助收集整理的广度知识 | ==覆盖面 + 速度== |

==反模式==：==混合两者==——==个人库会被未消化的机器输出膨胀==，==知识密度下降==。

==本仓库的对应==：

- `wiki/` ≈ 外脑（事实性知识 / 综述 / 对比 / 工程参数）
- [[思考记录]] ≈ 内脑（决策 ADR / 误区纠正 / 元洞察）
- ==判断标准==：==如果一条内容"半年后我自己还会用到"==才进思考记录（详见 §8.3）。

### 9.3 知识代谢循环

==知识管理是持续的代谢过程，不是一次性产出==：

```
采矿（收集原料 → raws/）
   ↓
破碎筛选（过滤噪音 → inbox/Clippings）
   ↓
冶炼提纯（提取核心知识点 → staged/）
   ↓
锻造成型（结构化输出 → wiki/）
```

==每个阶段都需要人类主动注入"能量"==——==判断 / 筛选 / 追问==。==纯自动化流水线==产出的是==信息堆积==，==不是知识==。

==对应到本仓库流程==：详见 [[RAW_WORKFLOW_GUIDE]] 的 raws → wiki 全链路。

## 十、机械检查清单（Automated Checklist）

> ==本清单与 §三「健康指标」对齐==——每项都可通过脚本验证，是审计工作流的输入规则。

### 10.1 客观规则（grep/wc/python 可自动验证）

#### 文档结构类
- [ ] **单文档行数** < 600 行（>600 🟡 黄旗 / >800 🔴 红旗）
  - 豁免：流水账文档（优化记录 / 思考记录）
- [ ] **顶层小节数** ≤ 8 个（H2 标题数量）
- [ ] **单文件单 H1**（多个 `#` 标题视为结构混乱）
- [ ] **标题层级不跳级**（不允许 `##` 直接跳到 `####`）
- [ ] **>400 行文档有顶部速览 callout**（`> [!tip]` 或 `> [!info]`，含章节导航）

#### Frontmatter 类
- [ ] **frontmatter 四字段齐全**：`module` / `tags` / `difficulty` / `last_reviewed`
- [ ] **last_reviewed ≤ 6 个月**（过期需更新）
- [ ] **module 与所在域一致**（如 `wiki/Agent/` 下文件 `module: Agent`，不是 `LLM`）

#### 编码与格式类
- [ ] **零 UTF-8 乱码**（`U+FFFD` 替换字符数 = 0）
- [ ] **代码块有语言标签**（\`\`\` 后必须跟 java/python/bash/sql 等）
- [ ] **代码块前后留空行**（提升可读性）

#### Callout 规范类
- [ ] **callout 类型仅限 5 种**：`tip` / `info` / `note` / `warning` / `danger`
- [ ] **每 `###` 小节下 callout ≤ 3 个**（过多降低信息密度）
- [ ] **禁止连用 callout**（两个 callout 间无正文，视觉疲劳）
- [ ] **禁止嵌套 callout**（`> > [!...]` 格式破坏）

#### 链接与引用类
- [ ] **标题不内嵌 `[[wikilink]]` 或 `**bold**`**（破坏锚点匹配）
- [ ] **所有 wikilink 指向存在的文件**（零死链）
- [ ] **锚点文本与目标标题精确一致**（`[[file#anchor]]` 的 `anchor` 需逐字符匹配）

### 10.2 半自动规则（需 agent 判断）

以下规则需领域专家 + 知识库管理专家联合判定：

- [ ] **Diátaxis 类型 ≤ 1.5 种**（混合 ≥2 类需拆分或重定位）
- [ ] **域内 SSoT 无重复**（同一概念在本域 ≥2 篇文档各 ≥50 行重复讲解）
- [ ] **内容边界清晰**（无跨域内容、无安装教程、无营销文案）
- [ ] **求职方向对齐**（本库目标 Java + Agent 岗，非纯算法/纯大模型研发）

### 10.3 检查脚本使用

机械检查可通过 `wiki审计/check.sh` 独立执行（不启动完整审计工作流）：

```bash
# 全量检查
./wiki审计/check.sh --all

# 分项检查
./wiki审计/check.sh --encoding   # UTF-8 乱码
./wiki审计/check.sh --links      # wikilink 坏链
./wiki审计/check.sh --structure  # 行数/H1/层级
./wiki审计/check.sh --callouts   # callout 规范
```

输出格式为 JSON，可被 CI/CD 消费（见 [[QUALITY_CHECKLIST]]）。

### 10.4 机械检查的局限性

> ==机械检查产生的是"待判断清单"，不是"必修问题清单"==。

==600 行阈值是工具建议，不是内容规则==——判断标准应该是==内容关联性==而非==行数==。

#### 正确判断逻辑

| 场景 | 行数 | 判断 | 原因 |
|------|------|------|------|
| ==高度关联的概念体系== | 800 | ✅ 保持完整 | 拆分会破坏理解连贯性 |
| ==单一技术栈深度解析== | 700 | ✅ 保持完整 | 脱离上下文后部分内容不可读 |
| ==多个独立主题硬拼== | 400 | ⚠️ 需拆分 | 违反 Atomic Notes 原则 |
| ==Diátaxis 混类== | 500 | ⚠️ 需拆分 | Tutorial + Reference 混在一起 |

#### 实际判断流程

```
机械检查报告 "X.md 超过 600 行"
  ↓
人类/agent 判断：
  1. 这些内容属于同一个核心概念吗？
     └── 是 ��� 检查是否有独立子主题（占比 >30%）
     └── 否 → 拆分（违反 Atomic Notes）
  
  2. 内容类型是否统一（Diátaxis）？
     └── 否 → 拆分（违反 1 类原则）
  
  3. 是否有 SSoT 违反（域内重复）？
     └── 是 → 合并或链接，消除重复
```

==关键认知==：==行数只是健康信号==，==不是修复指令==。机械检查的作用是==发现需要人类判断的对象==，而非==给出确定性结论==。

#### 真实案例（本次审计）

- ❌ **多 H1 误报**：grep 检测到 41 个"多 H1"文件 → 实际核实后只有 2 个真问题（39 个是代码块内 `#` 注释）
- ✅ **600 ���判断**：线程基础 842 行 → ThreadLocal 是独立概念且占比 >40% → 值得拆分
- ✅ **600 行保留**：某技术栈深度解析 750 行 → 内容高度关联 + 无独立子主题 → 保持完整

==结论==：==启动批量修复前，先抽样核实 1-2 个典型案例==。

## 十一、N-agent 验证法则（Meta-level Practice）

> ==工具产生的"问题清单"自带偏差——检测脚本可能误报，审计 agent 可能漏检。==
>
> ==在按清单分发 N 个 agent 之前，抽样核实 1-2 个典型案例，经常发现 N 的真实值是声称值的 5% 或 500%。==

### 11.1 法则描述

==启动 N 个 agent 之前，先验证 N 的真实大小。==

| 阶段 | 操作 | 目标 |
|------|------|------|
| **Phase 0** | ==抽样核实 1-2 个==典型案例 | 验证问题清单的准确性 |
| **Phase 1** | 修正检测逻辑（如果误报率 >20%） | 更新脚本/prompt，重新扫描 |
| **Phase 2** | 按==真实 N==分发 agent | 避免浪费 token + 避免遗漏真问题 |

### 11.2 为什么需要这个法则

==工具输出≠真实问题==：

| 误差来源 | 表现 | 后果 |
|--------|------|------|
| ==脚本逻辑缺陷== | grep 误算代码块内注释 | 误报率 >90% |
| ==规则过于机械== | 所有 >600 行都标红 | 淹没真正需要拆的文档 |
| ==上下文缺失== | agent 只看单文档，不知域内重复 | 漏检 SSoT 违反 |
| ==阈值设定不当== | callout ≤3 个/节过于严格 | 大量假阳性 |

==不核实的代价==：

- **Token 浪费**：启动 41 个 agent 处理误报 → 只需 2 个 agent 处理真问题
- **真问题被淹没**：大量假阳性让维护者失去信任 → 真问题被忽略
- **过度修正**：机械执行错误规则 → 破坏合理的文档结构

### 11.3 真实案例（2026-06 审计）

==问题==：check.sh 报告 41 个"多 H1"文件

==Phase 0（核实）==：
1. 抽样读取 `Git.md` 和 `Baize/文件上传机制.md`
2. 发现 Git.md 确实有多 H1（真问题），但大部分文件是代码块内 `#` 注释被误报

==修正检测逻辑==：
```bash
# 错误写法（误报）
grep -c '^# ' "$file"

# 正确写法（区分代码块）
awk '/^```/{in_code=!in_code; next} !in_code && /^# /{c++} END{print c+0}' "$file"
```

==Phase 1（重新扫描）==：
- 真实问题：2 个（Git.md / Baize/文件上传机制.md）
- Token 节省：~95%（从 41 → 2）

==Phase 2（精准修复）==：
- 启动 2 个 agent 并行修复 + 1 个验收 agent
- 多 H1 清零 ✅

### 11.4 应用场景

==什么时候必须执行 Phase 0 核实==：

| 场景 | N 的来源 | 是否核实 |
|------|--------|--------|
| ==机械检查脚本输出== | grep / awk / python 脚本 | ==✅ 必须== |
| ==Agent 审计报告== | 领域 agent 扫描后的问题清单 | ==✅ 必须== |
| ==用户手工标记== | 用户明确说"这 5 个文件有问题" | ❌ 直接信任 |
| ==Git diff 变更== | 最近提交修改的文件列表 | ❌ 直接信任 |

==核实方法==：

1. **抽样 1-2 个典型案例** — 一个"明显有问题"，一个"边缘情况"
2. **人类/expert-agent 判断** — 不依赖原脚本，用新视角重新检查
3. **计算误报率** — 如果 ≥20%，修正检测逻辑并重新扫描

### 11.5 与其他方法论的关系

- ==与 §十.4 配套==：机械检查有局限 → 需要人类语义判断 → N-agent 验证是执行层实践
- ==与 §五 主动监控配套==：Agent 不盲目执行清单 → 先核实 → 再告警 → 最后执行
- ==与 §三 健康指标配套==：红黄绿等级是工具输出 → N-agent 验证决定真实优先级

==关键认知==：==这是 meta-level 的方法论==——不只是"怎么管理 wiki"，而是=="怎么管理 Agent 的工作流"==。
