# Obsidian 风格指南

本文档定义 wiki 内容的 Obsidian 格式规范，确保不同时期、不同作者（包括 AI 协作）产出的内容风格一致。

> [!tip] 使用方式
> 新增或修改 wiki 内容时，对照本文档检查；AI 协作时，将本文档作为 context 提供。

---

## 1. Frontmatter（元数据）

每个 wiki 文件**必须**以 frontmatter 开头，格式固定：

```yaml
---
module: <所属模块>
tags: [<主分类>, <子主题1>, <子主题2>, ...]
difficulty: easy | medium | hard
last_reviewed: YYYY-MM-DD
---
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `module` | ✅ | 所属模块名，如 `JVM`、`集合框架`、`并发编程`、`Java基础`、`LLM` |
| `tags` | ✅ | 第一个 tag 是主分类，后续是子主题；每个文件 3-8 个 tag 为宜 |
| `difficulty` | ✅ | **知识点理解难度**（读这篇文件需要多少前置知识）：`easy` 无需特殊前置 / `medium` 需要数据结构或算法基础 / `hard` 需要理解底层原理（JVM、编译器、OS 等） |
| `last_reviewed` | ✅ | 最近一次审核日期。修改内容时同步更新 |

**示例：**

```yaml
---
module: JVM
tags: [JVM, JIT, 字节码, 分层编译, 方法内联, invokedynamic]
difficulty: hard
last_reviewed: 2026-05-07
---
```

> [!warning] 不要写的字段
> 不要加 `author`、`created`、`aliases` 等字段，保持 frontmatter 精简统一。版本信息走 Git。

## 2. 高亮 `==关键词==`

> [!info] 本 vault 自定义规则
> `==高亮==` 是 Obsidian 独有语法，Google/Microsoft/GitLab 等业界 style guide 均不涉及。以下规则为本 vault 自行约定，无外部标准依据。

### 使用场景（什么时候该高亮）

| 情境 | 是否高亮 | 例子 |
|------|---------|------|
| **核心定义**（一句话概括概念本质） | ✅ | `==写屏障是 JVM 在对象引用赋值前后插入的一段代码==` |
| **面试高频考点术语** | ✅ | `==漏标的充要条件==`、`==分层编译==` |
| **反直觉结论** | ✅ | `==HotSpot 的栈上分配主要通过标量替换实现==` |
| 一般代码注释、普通解释 | ❌ | — |
| 已经用 **加粗** 的词 | ❌ | 不要重复强调 |
| 长整句（超过 20 字） | ❌ | 高亮要精准，长句子高亮等于没高亮 |

### 密度控制

- ==单个三级标题（`###`）下最多 2-3 处高亮==
- 一整段文字里最多 1 处高亮
- 表格单元格内尽量不用高亮（表格本身已有视觉区分）

### 与加粗（`**`）的区别

| 语法 | 语义 | 用途 |
|------|------|------|
| `==高亮==` | 视觉强调，"这一句是考点" | 面试考点、核心定义、反直觉结论 |
| `**加粗**` | 结构强调，"这是关键词" | 术语、参数名、分点小标题 |

**同一段里两者都可以出现，但不要给同一个词同时加粗+高亮。**

## 3. Callout（提示块）

### 使用类型 `[业界共识]`

使用以下 5 种类型，与 GitHub/Microsoft/GitLab 标准对齐：

| 类型 | 语义 | 使用场景 |
|------|------|---------|
| `> [!tip]` | 最佳实践/反直觉洞察 | 帮助读者做得更好、更快；反直觉结论；深度解释"为什么" |
| `> [!info]` | 背景信息/扩展知识 | 补充上下文，不影响主线理解 |
| `> [!note]` | 一般补充 | 读者跳过也不影响理解，但知道了更好 |
| `> [!warning]` | 潜在问题/易踩的坑 | 可能导致意外行为、线上事故、版本兼容性问题 |
| `> [!danger]` | 不可逆风险 | 数据丢失、安全漏洞、严重后果 |

### 格式规范

```markdown
> [!tip] 标题（可选，但推荐）
> 正文内容，可以多行
> 也可以有第二段

> [!warning] 直接内存 OOM 是排查难点
> 直接内存不走正常的堆 GC，泄漏时：
> - `jmap -heap` 看不到异常
> - 堆 dump 里找不到大对象
```

### 密度控制 `[业界共识]`

Microsoft Learn 明确建议"每篇文章 1-2 个，不要连续出现"。我们的文件内容更长，放宽到 **3-8 个**，但保留"不连续"原则：

- 单个文件 callout 总数 3-8 个
- 不要连续两个 callout
- 不要在 callout 内部嵌套 callout

> [!warning] 不要滥用 callout
> callout 用多了失去强调效果，变成装饰。每个 callout 都应该承担"视觉前景"的角色——读者扫到它时会专门停下看。

## 4. Wikilink（内部链接）

### 格式规范

| 场景 | 写法 | 示例 |
|------|------|------|
| 同模块 | `[[文件名]]` | `[[HashMap核心原理]]` |
| 跨模块 | `[[模块/文件名]]` | `[[JVM/JVM 垃圾收集]]` |
| 锚点链接 | `[[文件名#标题]]` | `[[JVM 调优#GC 日志逐行解读]]` |
| 带显示文本 | `[[文件名#标题\|展示文字]]` | `[[JVM JIT与字节码#为什么需要 JIT？\|查看核心解析]]` |

### 禁忌

- ==标题里不要放 wikilink==：会导致锚点失效
  - ❌ `### [[HashMap核心原理\|哈希表]] 的扩容机制`
  - ✅ `### 哈希表的扩容机制`（在正文里写 `详见 [[HashMap核心原理]]`）
- ==不要嵌套 wikilink==：`[[xxx|[[yyy]]]]` 会渲染错乱
- ==跨模块链接必须用相对路径==：不要用绝对路径如 `wiki/JVM/xxx`

### 每个文件末尾的"相关链接"区块

文件末尾**必须**有一个"## 相关链接"区块，列出跨模块双链：

```markdown
## 相关链接

- [[JVM 内存管理]] — 堆内存分代结构是 GC 的基础
- [[线程基础与ThreadLocal]] — GC 的 STW 会暂停所有用户线程
- [[锁]] — Mark Word 与锁升级的关系
```

格式：`- [[目标文件]] — 一句话说明关联关系`

## 5. 标题层级

### 使用规则

```
# 一级标题  ← 整个文件的大标题，可选（frontmatter 后的第一行）
## 二级标题  ← 主章节（"HashMap 基础"、"HashMap 进阶"）
### 三级标题 ← 具体问题（"HashMap 的 put 流程知道吗？"）
#### 四级标题 ← 问题内的子问题（"只重写 equals 会发生什么？"）
##### 五级标题 ← 极少用（超过四级说明章节切分不合理）
```

### 三级标题的命名规范（面试题文件）

==三级标题就是面试题的问题本身==，要能当问题直接问出来：

- ✅ `### HashMap 的 put 流程知道吗？`
- ✅ `### 为什么 HashMap 的容量是 2 的幂次方？`
- ❌ `### put 流程`（太简略，不是完整的面试问题）
- ❌ `### HashMap 相关`（太泛）

### 🌟 emoji 标记 `[本 vault 自定义]`

高频重点题在三级标题前加 🌟：

```markdown
### 🌟HashMap 的 put 流程知道吗？
```

按重要性标记，无数量上限。但同一文件内 🌟 题目应**明显少于**非 🌟 题目——如果大多数题都加了星，强调效果就消失了。

## 6. 表格

### 使用场景

表格用于**横向对比**，不要用表格组织线性内容。

| 适合用表格 | 不适合用表格 |
|-----------|-------------|
| 对比（ArrayList vs LinkedList） | 流程步骤（用编号列表） |
| 分类（5 种 invoke 指令） | 层次结构（用缩进列表） |
| 参数字典（GC 参数） | 代码示例（用代码块） |

### 格式规范

- 表头用 `| 列名 |`，表头下必须有 `| --- |` 或 `|---|` 分隔行
- 列数较多时，在 Markdown 源码里手动对齐，可读性更好
- 单元格内可以有 `==高亮==` 和 **加粗**，但==不要用 callout==（会渲染错）
- 单元格内的 wikilink 用 `\|` 转义分隔符：`[[锁\|CAS]]`

## 7. 代码块

### 语言标签

==代码块必须带语言标签==，不要裸用 ``` ```：

````markdown
```java
public int sum() { return a + b; }
```

```bash
-XX:MaxInlineSize=35
```

```yaml
---
module: JVM
---
```
````

### 代码注释风格

- 中文注释更适合面试场景的知识讲解
- 注释写"为什么"，不写"什么"（代码已经说明了什么）

```java
// ❌ 不好：注释写"什么"
int n = 16;  // 设置 n 为 16

// ✅ 好：注释写"为什么"
int n = 16;  // HashMap 默认初始容量，必须是 2 的幂次方
```

## 8. 列表

### 有序 vs 无序

| 用什么 | 什么时候 |
|--------|---------|
| 有序列表 `1. 2. 3.` | 步骤、流程、优先级排序 |
| 无序列表 `- ` | 枚举、并列关系 |

### 缩进

- 统一使用 **4 个空格** 或 **Tab** 缩进子列表
- 不要混用 2 空格和 4 空格
- 嵌套不要超过 3 层

## 9. 文件命名

### 命名规范

| 文件类型 | 命名风格 | 示例 |
|---------|---------|------|
| 普通知识文件 | 中文或中英混合 | `HashMap核心原理.md`、`JVM 调优.md` |
| MOC（Map of Content） | `XX索引_MOC.md` | `JVM索引_MOC.md`、`算法_MOC.md` |
| 规范/元信息 | 全大写英文 | `QUALITY_CHECKLIST.md`、`AGENTS.md` |

### 空格约定

- 中文和英文之间==可以有空格==，但同一个模块要统一
  - JVM 文件夹：`JVM 内存管理.md`（带空格）
  - Java基础文件夹：`包装类与 Object.md`（带空格）
- 文件名内不要有特殊字符（`<>:"/\|?*`）

## 10. 内容长度与结构

### 拆分原则 `[本 vault 自定义]`

不设硬性行数上限。按内容是否可以**自然拆分**判断：

- ==子主题独立性强==（如 HashMap 和 TreeMap 可以分开理解）→ 拆文件
- ==内容有明确的进阶层次==（基础 vs 进阶）→ 一个文件内用二级标题分章
- ==只是题目多==（10 个相关问题）→ 不用拆，知识点本身就是一个整体

MOC 索引文件（`XX索引_MOC.md`）只放导航，不放内容，保持 30-80 行。

## 11. 内容组织 `[业界共识]`

### 先概念后应用（Introduce Before Use）

一个概念必须在它被引用或依赖之前先完整介绍。如果读者到达某一段时需要"跳回去看"才能理解，说明组织顺序有问题。

- ✅ 先讲三色标记原理 → 再讲 CMS/G1 如何使用三色标记
- ❌ 先讲 CMS 的"并发标记阶段" → 中间隔了 ZGC → 再回头讲三色标记是什么

**自检方法**：从头到尾读一遍文件，如果某一段需要"跳回前面"才能理解，说明顺序需要调整。

### 单一信息源（DRY）

同一个知识点只在一个文件的一个位置详细讲解：

- **详解位置**：选最自然的归属文件（如逃逸分析的详解放在 JIT 文件，因为它是 JIT 优化）
- **其他引用处**：一句话概括 + `[[文件名#标题]]` 链接到详解位置
- ❌ 两个文件都贴完整的代码示例和解释

**自检方法**：`grep -rn "关键词" wiki/模块/` 看同一概念是否出现在多处，如果两处都有详细解释就需要去重。

### 就近原则（Proximity Principle）

密切相关的知识紧挨放置，不要被不相关的内容隔开：

- ✅ CMS 收集器介绍 → 紧接着讲 CMS 的 Concurrent Mode Failure
- ❌ CMS 收集器介绍 → G1 → ZGC → 然后回头讲 CMS 的问题

**自检方法**：同一个父概念下的子知识点之间，不应该隔超过 2 个无关章节。

## 12. AI 协作的额外要求

如果使用 Claude/GPT 等 AI 协作生成内容，需要：

1. ==把本文档作为 context 提供给 AI==
2. 审查 AI 产出的内容是否：
   - frontmatter 完整
   - 高亮和 callout 符合密度规范
   - wikilink 指向真实存在的文件/标题
   - 没有无意义的套话（"综上所述"、"总而言之"）
3. 大批量修改后，运行 `.claude/scripts/validate-md.sh`（如果存在）

## 检查清单（快速自检）

新写或改完一个文件后，对照以下清单：

- [ ] frontmatter 四个字段齐全，`last_reviewed` 是今天
- [ ] `difficulty` 填的是**知识点理解难度**，不是题目难度
- [ ] 每个三级标题能当面试问题直接问出来
- [ ] 🌟 标记的题目明显少于未标记的题目
- [ ] 高亮密度：单个 `###` 下 ≤ 3 处，且只用于核心定义/考点/反直觉结论
- [ ] callout 总数 3-8 个，无连续两个 callout，类型只用 tip/info/note/warning/danger
- [ ] 所有 wikilink 都指向存在的文件/标题
- [ ] 文件末尾有"## 相关链接"区块
- [ ] 代码块都有语言标签
- [ ] 表格只用于横向对比，不用作线性内容组织
- [ ] 内容组织：无需"跳回前面"才能理解（先概念后应用）
- [ ] 同一知识点只在一处详解，其他处用链接引用（DRY）
