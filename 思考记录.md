## 2026-05-31 — 元洞察：对话本身是新型知识资产

> ==对话不是流水账==——==是 fleeting thought==，==必须显式提炼==才能成为==durable== 的知识。

### 背景

==传统观念==：wiki = 学到的事实。对话 = 学习过程，==记录"过程"似乎是流水账==。==但实际==上，对话中有大量==读 wiki 学不到==的隐性知识。

### 思考

==对话能产生 4 类知识==：

| 类型 | 例子 | 该去哪 |
|------|------|------|
| ==事实性知识== | "FTS5 是 SQLite 内置全文索引" | wiki 主文档 |
| ==设计决策== | "为什么不用 `00-` 前缀" | 思考记录 |
| ==误区纠正== | "Skill ≠ 命令脚本" | 思考记录 + 主文档 callout |
| ==元能力 / 方法论== | "对话→知识 4 类分流" | 方法论文档 |

==只有 1 类（事实性）==该进 wiki 主文档。==其他 3 类==是==隐性知识==——==必须显式记录==才不会流失。

### 决策

- 建立 ==思考记录.md==（本文档）专门记录==决策 / 误区 / 元洞察==
- 在 [[wiki-管理方法论]] 加 §九 "==对话→知识的转化方法=="
- 在 [[AGENTS.md]] 加触发规则——==关键时机主动提议记录==

## 2026-05-31 — 知识管理"套娃陷阱"的解：方法论自洽

> ==方法论自身==遵守==自己定义的规则==——不需要再来一层来管它。

### 背景

担心 ==wiki-管理方法论.md== 不完美 → 是否需要"==管方法论的方法论==" → 是否需要"==元元方法论==" → ...==无限套娃==？

### 思考

==Infinite regress==（无限回溯）是==哲学难题==，==但工程上有止步条件==：

1. ==方法论自洽==——方法论自身遵守自己的规则。
   - ==类比==：宪法不需要"宪法的宪法"——宪法的修订程序写在宪法自己里
   - ==类比==：编程语言用语言自己描述（self-hosted compiler）
2. ==实践驱动 > 理论完美==——发现失效时==直接改方法论==，==不创新文档==。
3. ==投入产出==——元元层级没有读者，==自然止步==。

### 决策

- ==wiki-管理方法论 加自指声明==："本文档自身也按本规则写"
- ==不创建=="方法论的方法论"
- ==演进 = 直接改本文==，==不堆叠新文档==

- [[wiki-管理方法论]]

## 2026-05-31 — 文件夹 vs 单文件：何时该建文件夹

> ==单文件不需要文件夹==——文件夹是==预期 ≥ 3 个相关文件==时才建。

### 背景

最初提议建 `wiki/00-知识库管理/wiki-管理方法论.md`——用 `00-` 前缀强制排序。

### 误区 / 反例

==错误的==：

1. ==`00-` 前缀==破坏了 wiki 已有的==自然命名风格==（Java基础 / Agent / RAG ...）
2. ==单文件==建文件夹==是形式主义==——只为"==排序前缀=="服务

==`00-` / `01-`== 这种==排序前缀==只在==排序敏感==的工作流系统（如 ThirdSpace）有意义，==知识 wiki 不需要==。

### 决策

- ==方法论文件==放仓库根，==与 OBSIDIAN_STYLE.md / QUALITY_CHECKLIST.md / 优化记录.md 并列==
- ==自然命名==（`wiki-管理方法论.md`）==而非==编号
- ==只有 ≥ 3 个相关文件==才建文件夹

### 何时建文件夹的判断

| 场景 | 该建文件夹 |
|------|---------|
| ==预期 ≥ 3 个相关文件== | ✅ 例如 `主流 Coding Agent 实现对比/` |
| ==独立子主题需要 WORKSPACE== | ✅ 例如 `raws/` |
| ==只有 1 个文件== | ❌ |
| ==文件夹只为"排序前缀"== | ❌ |

## 2026-05-29 — Skill ≠ 命令脚本

> ==Skill 是给 LLM 看的"专家手册"==，==不是已确定好的命令序列==。

### 背景

==容易误以为==：Skill = 一个流程包装起来，里面有==一系列已确定好的命令==。

### 误区纠正

==Skill 实际==：

| 类比 | 例子 |
|------|------|
| ==Command== | ==按钮==——用户按了执行固定流程（"==给我一杯咖啡=="） |
| ==Skill== | ==咖啡师工作手册==——告诉 LLM"==遇到这种情况，考虑这些原则==" 但==怎么做 LLM 自己决定== |

==SKILL.md 文件==里==没有=="先调 A 再调 B"——==只有原则和约束==。==LLM 看完==自己决定怎么用 Tool 完成。

### Command vs Skill 真实区别

| 维度 | Command | Skill |
|------|---------|-------|
| 触发方式 | ==用户主动==输入 `/xxx` | ==LLM 自动==激活（看上下文判断） |
| 内容形态 | ==prompt 模板 + 参数== | ==专家手册 / 经验文档== |
| 执行决策权 | 大部分==写死的流程== | ==LLM 自己决定==怎么做 |
| 谁是消费者 | ==用户==（点按钮） | ==LLM==（看手册） |

## 2026-05-29 — Memory L2→L3 不一定靠 LLM

> ==评分驱动（OpenClaw）==也是 L2→L3 的有效方式——==纯数据投票，不靠 LLM 自动 extract==。

### 背景

==默认认为==：L2→L3（短期 → 长期记忆）==都是 LLM 读取关键内容保存==。

### 思考 / 真实情况

==四个项目实际有 5 种 L2→L3 方式==：

| 方式 | 项目 | 靠 LLM? |
|------|------|------|
| ==后台 forked agent extract== | Claude Code | ✅ |
| ==Agent 主动调 memory tool== | Hermes | ✅ |
| ==memory nudge==（每 N 轮提醒） | Hermes | ✅ |
| ==评分提升==（基于召回次数 + unique queries） | OpenClaw | ❌ ==纯数据== |
| ==只做日志==（用户手写） | Codex | ❌ ==用户手写== |

### 关键认知

==OpenClaw 的 short-term-promotion==：

```typescript
DEFAULT_PROMOTION_MIN_SCORE = 0.75
DEFAULT_PROMOTION_MIN_RECALL_COUNT = 3       // 至少召回 3 次
DEFAULT_PROMOTION_MIN_UNIQUE_QUERIES = 2     // 至少 2 个不同 query
DEFAULT_RECENCY_HALF_LIFE_DAYS = 14          // 14 天指数衰减
```

==这种方式==更准（==被多次使用 + 跨场景==的才被巩固），==但需要长期数据==才生效。

## 2026-05-29 — chunk 不是按段落切，是按字符累积

> ==OpenClaw 的 chunkMarkdown 不识别段落==——==只看字符上限累积==。

### 背景

==容易认为==：分块是==按段落 / heading 切==——==这是 RAG 进阶做法==，==不是 OpenClaw 默认==。

### 误区纠正

==实际代码==：

```typescript
const lines = content.split("\n");
for (const line of lines) {
  if (currentChars + line.length > maxChars) {
    flush();  // ★ 累积到字符上限就切
  }
  current.push(line);
}
```

==没有==：
- ❌ 看是否遇到段落分隔符（空行）
- ❌ 看是否遇到 markdown heading
- ❌ 看是否句号

==只有==：✅ 字符够了就切。

==所以==：==chunk = N 行打包成的一段==（按字符上限），==不是每行一个 chunk==，==也不是按段落切==。

### 关键认知

| 分块方式 | 怎么切 |
|---------|------|
| ==字符上限==（OpenClaw） | 按字符累积 |
| 段落感知（按 `\n\n`） | 遇空行分段 |
| Heading 感知（按 `#`） | 按 markdown heading 切 |
| 语义分块（embedding） | 模型判断语义边界 |

==OpenClaw 选最朴素==——==memory 文件小==，==字符上限够用==。

## 2026-05-29 — FTS5 倒排索引：整段文本 tokenize，没有"关键字字段"

> ==chunk.text 整段被 tokenize==，每个 token 都是潜在的搜索关键字。

### 背景

==容易认为==：要检索就需要"==关键字字段=="——==输入关键字 → 匹配该字段==。

### 误区纠正

==FTS5 实际==：

1. ==chunk.text== 整段被自动 tokenize（trigram 三字符分词）
2. ==每个 token== 都建==倒排索引==
3. ==用户搜任何词==——FTS5 用 BM25 排序找最相关 chunk

==所以==：==没有"关键字字段"==——==整段文本就是被索引的内容==。

### 倒排索引的本质

| 方向 | 例子 |
|------|------|
| ==正向==（普通字典） | chunk → 它包含哪些词 |
| ==倒排==（FTS5 / Google 搜索） | "整理" → 哪些 chunk 包含它 |

==为什么叫"倒排"==：==查询时是从词反查文档==，==不是从文档查词==。

## 2026-05-29 — stdio 不是网络协议，是 OS 进程间通信

> ==Stdio==（Standard I/O）是==操作系统层面的 IPC==，==不是网络协议==——==只能本机==。

### 背景

==Stdio / Streamable HTTP / WebSocket== 经常一起出现在 MCP / LSP 文档里，==容易认为它们都是网络协议==。

### 误区纠正

| 协议 | 层级 | 通信范围 |
|------|------|------|
| ==stdio== | ==OS 进程间通信==（IPC） | ==仅本机== |
| ==Streamable HTTP== | 应用层（HTTP） | 跨网络 |
| ==WebSocket== | 应用层（从 HTTP 升级） | 跨网络 |

==Stdio 的本质==：父进程启动子进程 → ==通过 stdin/stdout/stderr 三根管道通信==。==像 shell 里==的 `ls | grep`。

==关键认知==：

- ==stdio 是字节流==——OS 不知道消息边界
- ==MCP / LSP 都用 `Content-Length` 头==解决粘包问题
- ==生产中 MCP 80% 用 stdio==（本地工具）

## 2026-05-28 — Hermes Frozen Snapshot：解决"写 memory 破坏 prompt cache"矛盾

> ==Hermes 的工程亮点==——session 启动读 MEMORY.md → 进 system prompt → ==整个 session 不变==。

### 背景

==问题==：写 memory 后==下一轮 system prompt 会变==——==prompt cache 全部 miss==。

==但==：memory 又必须及时持久化==——==不能不写==。

### Hermes 的解

==Frozen Snapshot Pattern==：

```
session 启动:
  读 MEMORY.md / USER.md → 快照进 system prompt → 整个 session 不变

session 中 Agent 调 memory tool:
  立即写入磁盘(durable) → 但 system prompt 不变 → ==prompt cache 100% 命中==

下次 session 启动:
  读最新的 MEMORY.md → 新快照
```

### 关键认知

==Claude Code 没这么做==——所以 Claude Code 写 memory 后==下一轮 system prompt 会变==（cache miss）。

==Hermes 的取舍==：==session 中写的 memory 下次会话才生效==——==换来 prompt cache 100% 命中==。
